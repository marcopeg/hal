# 060 — Stale session empty-response recovery

**Status**: completed

## Description

Bring the Claude adapter in line with the stale-session recovery behavior already
implemented for Codex and Copilot. When Claude attempts to send a message over an
existing session and that continuation attempt is no longer usable, HAL should
discard the stale session, retry the same request immediately in a fresh session,
and include a warning in the Telegram response explaining that session continuity
was reset.

This scenario occurs reliably when the Claude CLI is configured against a non-native
backend (e.g. OpenRouter via `ANTHROPIC_BASE_URL`) and the stored session ID is from
a previous native Claude API session. In that case, the CLI may exit with code 0
but produce no assistant content, or otherwise fail the continuation attempt even
though a fresh run would succeed. Today the user either gets a blank reply or an
opaque engine failure instead of a seamless recovery.

## Goals

- Detect when Claude is trying to continue an existing session and that continuation
  attempt fails because the session is stale or otherwise no longer resumable.
- Clear the stale continuation state immediately so future messages do not keep
  retrying the broken session.
- Retry the same prompt once in a fresh session during the same Telegram request.
- Return the fresh-session response to the user together with a warning that
  continuity was reset.
- Keep the mechanism uniform with Copilot and Codex rather than introducing a
  Claude-only reset-button flow.

## Detection Criteria

This task should apply only when Claude was attempting to continue an existing
session. That includes both continuation modes HAL can use:

1. Per-user continuation via `--resume <sessionId>`.
2. Shared continuation via `--continue`.

Treat the continuation attempt as stale-session recovery territory when HAL tries to
continue an existing Claude session and receives a result that indicates the prior
session is unusable. At minimum, the refined implementation must cover:

1. `result.success === true` but assistant output is empty or blank after trimming.
2. An explicit engine/CLI failure returned from a continued session attempt, when the
   failure is attributable to resuming/continuing a stale or invalid prior session.

Normal fresh-session failures must not be reclassified as stale-session recovery.
The retry path should be limited to one automatic fresh retry per user message.

## Constraints and Expectations

- Detection is **Claude adapter-specific**; other adapters have different empty-output
  semantics and must not be affected.
- The Claude recovery UX should match the established Copilot/Codex pattern:
  automatic fresh retry plus warning, not an inline Telegram reset button.
- Stale continuation state must be cleared using the same user/project scoping as
  normal session persistence so one user's stale Claude session cannot affect other
  users.
- The existing `/reset` and `/clean` commands must continue to work as before.
- The retry must re-run the same prompt with continuation disabled so the user does
  not need to send the message again manually.
- If the detection fires, the stale-session event should be logged at `warn` level
  with the project slug, user ID, continuation mode, and the failure signal
  (empty output and/or relevant error details).
- If the fresh retry also fails, HAL should return the fresh-run failure normally;
  it must not loop or keep masking the underlying error.

## Acceptance Criteria

- When Claude attempts to continue an existing session and returns success with empty
  output, HAL clears the stale continuation state, retries once in a fresh session,
  and returns the fresh response to Telegram with a warning that the previous Claude
  session could not be resumed and continuity was reset.
- When Claude attempts to continue an existing session and returns a stale-session
  continuation failure, HAL performs the same one-time fresh retry and warning flow.
- The warning is surfaced through the existing `EngineResult.warning` / Telegram
  warning path rather than via a new callback-button interaction.
- The condition is not triggered for fresh-session Claude runs, for unrelated Claude
  failures that are not continuation-specific, or for other engine adapters.
- Shared Claude continuation (`--continue`) and per-user Claude continuation
  (`--resume <sessionId>`) both use the same stale-session recovery behavior.
- The stored stale session marker/ID is cleared before future messages, so the next
  user message does not keep retrying the broken session.
- No regression in normal Claude fresh-session behavior, existing `/reset` and
  `/clean` commands, or other adapters' continuation semantics.
