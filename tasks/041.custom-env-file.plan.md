# Plan: 041 — Custom env file

## Goal

Implement a single-source env loading model: either config-dir `.env`/`.env.local` only, or a single explicit `env` path (plus its `.local` sibling). Remove per-project env loading. Add conflict detection, strict boot errors for missing/unreadable custom file, and watcher support for the custom path (including when `.local` appears after startup).

---

## Phase 1 — Schema and path resolution

### 1.1 Add `env` to config schema

- In `src/config.ts`:
  - Add optional `env: z.string().optional()` to `MultiConfigFileSchema` (top-level).
  - Add optional `env: z.string().optional()` to the local config schema used by `LocalConfigFileSchema` (the object that has `globals`, `providers`, `context`, `projects`), so local config can override the base `env` path.
- Ensure merged config type includes `env?: string` (it will follow from the schema).

### 1.2 Merge `env` in local config

- In `mergeLocalIntoBase`: when building the return value, set `env: local.env ?? base.env` (local overrides base when present). Include `env` in both return branches (with and without merged projects).

### 1.3 Path resolution helper

- Add a small helper (e.g. in `config.ts` or next to env loading) that:
  - Takes `configDir` (config file directory) and the raw `env` string from config.
  - Resolves the path: if absolute, use as-is (after tilde expand); if relative, resolve against `configDir`. Apply tilde expansion (`~` → `os.homedir()`) when the path starts with `~`.
  - Returns the normalized absolute path for the main env file.
- Derive the `.local` sibling path as: same directory as the resolved path, filename equal to the base filename plus `.local` (e.g. `path/to/secrets.env` → `path/to/secrets.env.local`). Do not add `.local` in the middle of the extension (e.g. not `secrets.local.env`).

---

## Phase 2 — Env loading rewrite

### 2.1 Replace `loadEnvFiles` contract and implementation

- Change `loadEnvFiles` signature from `(configDir: string, projectCwds: string[])` to something that accepts the chosen mode and optional custom path, e.g. `loadEnvFiles(configDir: string, options: { envPath?: string })`. Caller will pass `envPath` when `merged.env` is set (after resolving the path).
- **When `envPath` is not set (default mode):**
  - Candidates: `[join(configDir, ".env"), join(configDir, ".env.local")]` (only these two; no project cwds).
  - Load in that order; later wins when merging into `vars`. Skip missing files; optional read errors can remain non-fatal for these two files if that matches current behavior, unless the task requires otherwise (task says strict boot fail only for "custom file" when `env` is set).
- **When `envPath` is set (explicit mode):**
  - Resolve the path (Phase 1.3). Compute `.local` sibling path.
  - **Conflict check:** If `configDir` has `.env` or `.env.local` and the resolved main path (after normalization) is not equal to `join(configDir, ".env")`, throw `ConfigLoadError` with a clear message: e.g. config specifies env file X but config directory also has `.env` / `.env.local`; use only one source; link to configuration documentation (e.g. `docs/config/README.md#environment-variable-substitution` or the project's public doc URL).
  - **Missing/unreadable main file:** If the main env file (the one specified) is missing or unreadable, throw `ConfigLoadError` with a strong message and link to configuration documentation. The `.local` file remains optional.
  - Candidates: `[resolvedMainPath, resolvedLocalPath]`. Load in that order; later wins. Only load these two; do not load config-dir `.env` in this mode.
- Return type remains `EnvSources` (`{ vars, loadedFiles }`). Include all loaded file paths in `loadedFiles` for logging.

### 2.2 Integrate into `loadMultiConfigInternal`

- After merging base + local (so `merged` has `env`), **before** calling `loadEnvFiles`:
  - If `merged.env` is set, resolve the custom path using the config file directory (use `configDir`; base config path is under `configDir` so dirname is `configDir`). Pass the resolved path into `loadEnvFiles` as `envPath`.
  - If `merged.env` is not set, call `loadEnvFiles(configDir, {})` (no `envPath`).
- Remove use of `projectCwds` for env loading: stop computing `rawCwds` for env purposes (or keep it only if still needed elsewhere in this function). Call `loadEnvFiles(configDir, { envPath: resolvedCustomPath })` when `env` is set, otherwise `loadEnvFiles(configDir, {})`.
- Ensure `substituteEnvVars` still receives the combined `envSources.vars` and that `loadedFiles` from env loading are still appended to the result's `loadedFiles`.

## Phase 3 — Config watcher

### 3.1 Extend watcher to accept extra paths

- In `src/config-watcher.ts`:
  - Change `startConfigWatcher(configDir: string, onConfigChange: ...)` to accept an optional third parameter, e.g. `extraPaths?: string[]`. Alternatively use an options object: `startConfigWatcher(configDir, onConfigChange, { extraPaths?: string[] })`.
  - Build the list of watched paths as: `CONFIG_FILES` mapped to `join(configDir, f)` **plus** `extraPaths` (when provided). Pass the combined array to `chokidar.watch` (chokidar accepts multiple paths). This ensures the custom env file and its `.local` sibling are watched even when the `.local` file does not exist yet (chokidar will emit `add` when it is created).

### 3.2 CLI: pass custom env paths when starting the watcher

- In `src/cli.ts`, after a successful config load, when starting the config watcher:
  - If `loaded.config.env` is set, resolve the custom env path (same resolution as in config load: relative to configDir, tilde expand). Compute the two paths: main file and `.local` sibling.
  - Call `startConfigWatcher(configDir, onConfigChange, { extraPaths: [resolvedMain, resolvedLocal] })` with those two paths when `env` is set; otherwise call without `extraPaths`.
  - Use a small shared helper for "resolve custom env path + .local sibling" so both config load and CLI use the same logic (either export from `config.ts` or a tiny util used by both).

## Phase 4 — Documentation and examples

### 4.1 Config reference and schema docs

- In `docs/config/reference.yaml`: add a top-level commented key `env` with a short description and link to the Environment variable substitution section. Example: `# env: "~/.hal/secrets.env"  # optional; see README Environment variable substitution`.
- In `docs/config/README.md`, replace or heavily update the **Environment variable substitution** section so that it:
  - Describes the **single-source** model: either config-dir only (no `env` key) or explicit path only (`env` set). No per-project `.env` loading.
  - Lists resolution order for default mode: `{config-dir}/.env`, then `{config-dir}/.env.local` (`.local` overrides). For explicit mode: configured file then its `.local` sibling (same directory, base name + `.local`).
  - Documents the **conflict** case: when `env` is set and points to a different file than `{config-dir}/.env`, and `{config-dir}/.env` or `.env.local` exists, boot fails with instructions to use only one source. Link to this section from the error message (anchor or path).
  - Documents **missing/unreadable** custom file: when `env` is set and the main file is missing or unreadable, boot fails with a clear error and link to this doc.
  - Documents **watcher** behavior: when `env` is set, the config watcher also watches the custom env path and its `.local` sibling; changes or creation of the `.local` file trigger a config reload.
  - Removes or rewrites any mention of per-project `.env` / `.env.local` (no migration section required per task).

### 4.2 Example config

- In `examples/hal.config.yaml`: add an optional commented example for `env`, e.g. `# env: ~/.hal/secrets.env` with a one-line comment that it's optional and that when set, only that file (and its `.local` sibling) is used for substitution.

### 4.3 Error message doc link

- Ensure boot error messages for conflict and for missing/unreadable custom file include a stable link to the configuration documentation (e.g. the repo's `docs/config/README.md#environment-variable-substitution` or whatever URL the project uses for deployed docs). If the codebase uses a constant like `HAL_DOCS_URL`, extend it or add a fragment/path for the env section so the printed link is correct.

## Phase 5 — Tests and sanity checks

### 5.1 Unit or integration tests (if applicable)

- Add or extend tests that cover: (a) default mode: only config-dir `.env` and `.env.local` are loaded; (b) explicit mode: only the configured file and its `.local` sibling are loaded; (c) conflict: when `env` points elsewhere and config-dir has `.env`, load fails with a clear error; (d) missing custom file: when `env` is set and the main file is missing, load fails; (e) optional `.local` for custom path: main file required, `.local` optional; (f) same path as config-dir `.env`: when `env` is e.g. `.env`, no conflict and only that path (and `.env.local`) are used.
- If the project uses snapshot or error-message tests, ensure the new error texts and doc link are asserted.

### 5.2 Manual checks

- Run the bot with no `env` and only `configDir/.env` and `configDir/.env.local`; confirm substitution and log line show only those two.
- Run with `env: "~/.hal/secrets.env"` (or a path that exists), confirm only that file (and optional `.local`) are used and no config-dir `.env` is loaded when path differs.
- Trigger conflict (e.g. `env: "~/.hal/other.env"` and existing `configDir/.env`); confirm boot error and doc link.
- Trigger missing file (`env` set to a non-existent path); confirm boot error and doc link.
- With `env` set, create the `.local` file after startup; confirm reload is triggered (watcher).
