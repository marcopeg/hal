# 057 — Status/info command with feature flags

**Status**: completed

## Description
Add a new built-in command `/info` (not `/status`) that is enabled by default and shows runtime project information.

The command output must include:
- project name
- CWD
- current engine and model
- session setting value
- current resolved context given to the project

## Context and constraints
- Command name is `/info` only.
- Config key must be `commands.info`.
- For session details, show only the configured session mode/value (`true`, `false`, `shared`, or `user`).
- If `commands.info.enabled` is `false`, `/info` must reply: `This command is disabled.`
- Env/source-aware redaction is out of scope for this task iteration; `/info` may show resolved context values as-is.

## Output formatting (Telegram readability)
- `/info` should send output in two messages when `commands.info.context` is enabled:
	- Message 1: compact human-readable summary (project/cwd/engine+model/session)
	- Message 2: context dump only
- Context dump must be rendered as a fenced code block for clean monospaced display in Telegram.
- Context entries in the code block should be sorted by key for stable, scannable output.
- If context output is too large for one Telegram message, it should be split into multiple fenced code-block messages.
- If `commands.info.context` is disabled, only Message 1 is sent.
- Keep labels concise and consistent (for example: `Project`, `CWD`, `Engine`, `Model`, `Session`).
- Use Telegram Markdown-compatible formatting only.

## Decisions captured during refinement/planning
- Implement command as `/info` only (no `/status` alias).
- Use `commands.info` as the canonical config node.
- Ship without env-derived redaction for now and document this limitation.
- Use a two-message format when context is enabled: summary first, context code block second.
- Split oversized context output into multiple fenced code-block messages instead of truncating.

## Expected config shape
`commands.info` should support:
- `enabled: boolean` (default `true`)
- `cwd: boolean` (default `true`)
- `engineModel: boolean` (default `true`)
- `session: boolean` (default `true`)
- `context: boolean` (default `true`)

These flags control visibility of their respective sections in `/info` output.

## Acceptance criteria
- `/info` appears as a built-in HAL command and is registered by default.
- `/info` output always includes project name.
- `/info` includes/excludes `cwd`, `engineModel`, `session`, and `context` sections based on `commands.info.*` flags.
- `commands.info.enabled: false` disables command wiring and returns `This command is disabled.` when `/info` is invoked.
- Session section shows only the configured session mode/value (`true|false|shared|user`), not extra session internals.
- When context is enabled, context is sent as a separate second message in a fenced code block.
- Context code block contains key-value pairs for resolved context, sorted by key.
- Large context output is split into multiple fenced code-block messages without breaking Markdown rendering.
- Defaults for all `commands.info` section flags are `true`, so a default config shows all sections.
- Documentation/config reference is updated to include `commands.info` and its fields.
- Documentation explicitly notes that env/source-aware context redaction is not implemented in this task.
