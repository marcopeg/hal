# 077 — Architectural Refactoring for Maintainability

## Status

ready

## Summary

A surgical, zero-feature refactoring of `/src` to eliminate code duplication, reduce coupling, and establish clean extension points. Every change preserves existing behaviour. The refactoring was previously planned, executed, and reviewed in detail — then rolled back when the `/btw` work that ran alongside it was abandoned. All the thinking and decisions are preserved here. The code just needs to be applied again from the current clean baseline.

---

## Problem

After 70+ tasks of incremental development, several structural patterns have accumulated:

1. **Handler duplication** — `photo.ts`, `document.ts`, `voice.ts`, and `text.ts` each copy-paste ~80% of the same LLM dispatch pipeline (session loading, progress tracking, engine execution, session saving, chunked response, file sending). Any change to dispatch requires editing 4 files in lockstep.

2. **Engine adapter boilerplate** — All 6 adapters repeat identical patterns for `check()`, process spawning, stdout/stderr collection, error handling, and process lifecycle. Only arg-building and output-parsing differ per engine.

3. **Monolithic files** — `cli.ts` (902 lines) mixes CLI parsing, init, config orchestration, banner, wizard, and hot-reload. `config.ts` (600+ lines) mixes Zod schemas, file I/O, env resolution, and validation.

4. **Scattered engine identity** — Adding a new engine requires touching 5+ files: `engine/types.ts`, `engine/registry.ts`, `config.ts`, `cli.ts`, `default-models.ts`. No single source of truth.

5. **Progress/stop pattern duplication** — The `onProgress` + throttled Telegram message editing pattern is copy-pasted verbatim in 5+ locations.

6. **Context resolver duplication** — `resolveContext()` and `buildCronContextVars()` share 90% of the same pipeline but are separate functions with duplicated logic.

7. **Tight command registration in bot.ts** — Every built-in command is wired procedurally. Adding a new command requires editing `bot.ts`.

---

## Goals

- Eliminate structural duplication without changing any external behaviour
- Establish clean seams for future plugin/provider registration
- Keep every phase independently testable
- Maintain 100% backward compatibility — no config, CLI, or user-facing changes

## Constraints

- Zero feature additions — pure structural improvement
- No config schema changes
- No new external dependencies
- All existing tests must pass after each phase

## Phase 1 — Extract Shared LLM Dispatch Pipeline

**Problem**: `text.ts`, `voice.ts`, `photo.ts`, `document.ts` each contain a near-identical LLM dispatch pipeline (~80 lines duplicated 4×).

### Step 1.1 — Create `src/bot/handlers/progress.ts`

Extract the throttled progress/stop pattern that every handler copy-pastes. Interface:

```ts
interface ProgressTracker {
  onProgress: (message: string) => Promise<void>;
  onStop: () => void;
}

function createProgressTracker(
  gramCtx: Context,
  statusMsgId: number,
  throttleMs?: number, // default 2000
): ProgressTracker
```

Internal state: `lastProgressUpdate`, `lastProgressText`, `stoppedByBtw` flag. `onStop` sets the flag and deletes the status message. Do NOT expose `isStopped()` — it was identified as dead code in review.

### Step 1.2 — Create `src/bot/handlers/dispatch.ts`

Move `dispatchMessage()` (currently the `async function dispatchMessage(...)` inside `createTextHandler` in `text.ts`) into a standalone `dispatch.ts`. Refactor it to use `createProgressTracker`. Signature stays the same:

```ts
export function createDispatch(ctx: ProjectContext) {
  return async function dispatchMessage(
    gramCtx: Context,
    text: string,
    existingStatusMsgId?: number,
  ): Promise<void>
}
```

This function already handles the full pipeline: `ensureUserSetup` → session load → status message → progress/stop tracking → `engine.execute` → status delete → session save → `sendChunkedResponse` → `sendDownloadFiles`.

Export `createDispatch` from `src/bot/handlers/index.ts`.

### Step 1.3 — Refactor `photo.ts`

Currently 157 lines. After: ~50 lines. Keep only: download image, save to uploads, build prompt. Replace the LLM pipeline with `await dispatch(gramCtx, prompt)`.

### Step 1.4 — Refactor `document.ts`

Currently 206 lines. After: ~70 lines. Keep only: validate MIME type, download file, save to uploads, build prompt. Replace the LLM pipeline with `await dispatch(gramCtx, prompt)`.

### Step 1.5 — Refactor `voice.ts`

Currently 244 lines. After: ~90 lines. Keep only: download voice, ffmpeg → WAV, Whisper transcription, show transcription. Replace the LLM pipeline with `await dispatch(gramCtx, transcription)`. Note: `voice.ts` previously used `result.sessionId` from the engine result for session saving — `dispatch` handles this internally.

### Step 1.6 — Refactor `text.ts`

Already has `dispatchMessage` inline — just import it from `dispatch.ts` after the move. No other changes.

**Verify**: `npm run typecheck && npm run lint && npm run test`. Manual: send text, photo, document, voice — all produce correct responses. Session continuity across messages. File downloads sent correctly.

## Phase 2 — Extract Engine Adapter Base

**Problem**: All 6 adapters repeat the same ~60-line process lifecycle block.

### Step 2.1 — Create `src/engine/adapters/base.ts`

Three shared helpers:

**a) `checkEngineAvailable(cmd: string, engineLabel: string): void`**

```ts
// Replaces the identical check() in all 6 adapters:
try { execSync(`${cmd} --version`, { stdio: "pipe" }); }
catch { throw new Error(`${engineLabel} command "${cmd}" not found...`); }
```

**b) `defaultParse(result: EngineResult): ParsedResponse`**

```ts
// Replaces the identical parse() in Codex, OpenCode, Cursor (3 adapters):
if (!result.success) return { text: result.error || "An unknown error occurred" };
return { text: result.output || "No response received" };
```

Copilot, Claude, and Antigravity keep their custom parsers.

**c) `spawnAndCollect(options: SpawnAndCollectOptions): Promise<SpawnResult>`**

```ts
interface SpawnAndCollectOptions {
  cmd: string;
  args: string[];
  cwd: string;
  envFile?: string;
  env?: NodeJS.ProcessEnv;
  onStdoutChunk?: (chunk: string) => void;
  onProcessStart?: (proc: ChildProcess) => void;
  onProcessExit?: (proc: ChildProcess) => void;
  logger: pino.Logger;
  engineName: string;
}

interface SpawnResult {
  stdout: string;
  stderr: string;
  exitCode: number | null;
}
```

Handles: `spawnEngineProcess()`, stdout/stderr accumulation, `onProcessStart`/`onProcessExit` callbacks, `proc.on("close")`, `proc.on("error")`. Adapters pass an `onStdoutChunk` callback for streaming progress; post-process the returned `SpawnResult` for their own output parsing.

### Step 2.2 — Refactor adapters

Order: Copilot and OpenCode (simplest) → Cursor → Codex (JSONL progress) → Claude → Antigravity.

All adapters use `checkEngineAvailable` for `check()`. Codex, OpenCode, Cursor use `defaultParse`. Claude and Antigravity keep custom `parse()`.

**Important note from review**: Pass `onProcessStart` and `onProcessExit` directly (e.g. `onProcessStart: options.onProcessStart`) — do NOT wrap in arrow functions (`(proc) => options.onProcessStart?.(proc)`). The direct pass-through pattern is what all other adapters use.

**Important note from review**: In Antigravity, check `spawnError` FIRST, then fall through to `lastResult` — matching the Claude adapter's order. Checking `lastResult` first (the original mistake) causes wrong behaviour on spawn failure.

**Verify**: `npm run typecheck && npm run lint && npm run test`. Manual: run at least one adapter (Claude), verify progress messages appear, response comes back, `/btw` still kills the process.

## Phase 3 — Unify Engine Identity (Engine Manifest)

**Problem**: Adding a new engine requires touching 5+ files. No single source of truth.

### Step 3.1 — Create `src/engine/manifest.ts`

```ts
export interface EngineManifest {
  name: EngineName;
  displayName: string;
  defaultCommand: string;
  halDefaultModel?: string;      // HAL-injected default (undefined = let CLI decide)
  providerDefaultModel: string;  // shown in /model and init template
  factory: (command?: string, model?: string) => EngineAdapter;
}

export const ENGINE_MANIFESTS: Record<EngineName, EngineManifest> = { ... };
export const ENGINE_NAMES = Object.keys(ENGINE_MANIFESTS) as [EngineName, ...EngineName[]];
```

Values per engine:

| Engine | displayName | defaultCommand | halDefaultModel | providerDefaultModel |
|--------|-------------|----------------|-----------------|---------------------|
| claude | Claude Code | claude | "default" | "sonnet" |
| copilot | GitHub Copilot | copilot | undefined | "gpt-5-mini" |
| codex | Codex | codex | undefined | "gpt-5.2-codex" |
| opencode | OpenCode | opencode | undefined | "opencode/gpt-5-nano" |
| cursor | Cursor | agent | undefined | "auto" |
| antigravity | Antigravity | gemini | undefined | "gemini-2.0-flash" |

### Step 3.2 — Derive `EngineNameSchema` from manifest

In `src/config.ts` (or `src/config/schemas.ts` if Phase 7 has been done):

```ts
import { ENGINE_NAMES } from "./engine/manifest.js";
const EngineNameSchema = z.enum(ENGINE_NAMES as [string, ...string[]]);
```

This closes the loop: adding a new engine only needs a manifest entry + adapter file.

### Step 3.3 — Replace `DEFAULT_PROVIDER_MODEL` in `cli.ts`

```ts
import { ENGINE_MANIFESTS } from "./engine/manifest.js";
function getDefaultProviderModel(engine: EngineName): string {
  return ENGINE_MANIFESTS[engine].providerDefaultModel;
}
```

### Step 3.4 — Replace `default-models.ts`

```ts
import { ENGINE_MANIFESTS } from "./engine/manifest.js";
export function getDefaultEngineModel(engine: EngineName): string | undefined {
  return ENGINE_MANIFESTS[engine].halDefaultModel;
}
```

Or inline the one-liner at the 3 call sites and delete `default-models.ts`. The file is only 17 lines.

### Step 3.5 — Update `engine/registry.ts`

```ts
import { ENGINE_MANIFESTS } from "./manifest.js";

export function getEngine(name: EngineName, command?: string, model?: string): EngineAdapter {
  const manifest = ENGINE_MANIFESTS[name];
  if (!manifest) throw new Error(`Unknown engine "${name}"`);
  return manifest.factory(command ?? manifest.defaultCommand, model);
}
```

**Verify**: `npm run typecheck && npm run lint && npm run test`. Manual: `/engine` shows all 6 with correct display names. `npx hal init --engine claude` uses correct default model.

## Phase 6 — Declarative Command Registration in `bot.ts`

**Problem**: `bot.ts` has 170+ lines of procedural command wiring that must be manually extended for every new command.

### Step 6.1 — Create `src/bot/commands/registry.ts`

```ts
interface BuiltinCommandDef {
  name: string;
  enabledKey: keyof CommandEnabledFlags;
  placement: "before-seq" | "after-seq";
  createHandler: (ctx: ProjectContext, deps: CommandDeps) => Handler;
  callbackHandler?: (ctx: ProjectContext, deps: CommandDeps) => Handler;
  guardWhenDisabled?: boolean; // generate bot.hears() guard for disabled state
}
```

Populate the table from current registrations:
- `btw`: before-seq, guardWhenDisabled=true
- `start`, `help`, `clean`, `info`: before-seq
- `reset`: before-seq (command) + after-seq (callback)
- `model`, `engine`: before-seq, guardWhenDisabled=true, has callback
- `git_*`: before-seq (commands + callback)
- `npm`: before-seq (command + callback)

Export `registerCommands(bot, projectCtx, deps)` which iterates the table.

### Step 6.2 — Replace procedural registration in `bot.ts`

Replace the ~100-line procedural block with `registerCommands(bot, projectCtx, { bot, debounceActiveUsers })`. The middleware stack ordering is preserved:

```
WAL middleware
→ auth middleware
→ rate-limit middleware
→ registerCommands() [before-seq handlers]
→ sequentialize()
→ reset callback [after-seq]
→ LLM handlers (text, photo, document, voice)
→ error handler
```

**Verify**: All commands work when enabled. Disabled commands reply "This command is disabled." `/btw` runs concurrently (before sequentialize). `/reset` confirmation flow works.
