# 069 — Add `/btw` system command that interrupts the ongoing prompt by killing the child process — Plan

## Phase 1: Config schema + resolved command shape

### 1.1 — Extend commands schema in `src/config.ts`
- Add a new `btw` command config entry under `CommandsConfigSchema`.
- Shape:
  - `enabled?: boolean`
  - `message?: { stopped?: string; nothingToDo?: string }`
- Keep compatibility with existing pattern: globals + per-project override.

### 1.2 — Extend `ResolvedProjectConfig.commands`
- Add:
  - `btw: { enabled: boolean; message: { stopped?: string; nothingToDo?: string } }`

### 1.3 — Resolve defaults + inheritance in `resolveProjectConfig`
- Resolve project > globals > defaults:
  - `commands.btw.enabled` default `true`
  - `commands.btw.message.stopped` default at handler layer (`The process stopped.`)
  - `commands.btw.message.nothingToDo` default at handler layer (`No active process to stop, nothing to do.`)

## Phase 2: Active LLM run registry (per-project, per-user)

### 2.1 — Add a shared runtime registry module
- Create a small module (e.g. `src/bot/llm-run-registry.ts`) that tracks the currently running LLM child process by user id within one bot/project.
- API sketch:
  - `register(userId, proc)`
  - `clear(userId, proc)` (idempotent; only clears if same proc)
  - `stop(userId): boolean` (returns whether anything was running)
  - `has(userId): boolean`
- Stop strategy in `stop(userId)`:
  - send `SIGINT`, then escalate to `SIGTERM`, then `SIGKILL` with short timers (same safety pattern used in npm exec helper).

### 2.2 — Attach registry to project runtime context
- Extend `ProjectContext` in `src/types.ts` with a runtime object for LLM run tracking.
- Initialize this runtime state when starting each bot (inside `startBot`), so runs are isolated per bot/project.

### 3.1 — Extend engine execute options for process hooks
- Extend `EngineExecuteOptions` with optional lifecycle hooks (e.g. `onProcessStart`, `onProcessExit`) or equivalent callback contract.
- Goal: handlers can register the spawned child process without engine-specific coupling.

### 3.2 — Implement hook calls in all adapters
- Update all CLI adapters that use `spawnEngineProcess`:

- Ensure `finally` cleanup so stale references are not left behind.

## Phase 4: Implement `/btw` command and Telegram wiring

### 4.1 — Add command handler
- Create `src/bot/commands/btw.ts`.
- Behavior:
  - If command disabled, reply `This command is disabled.` (same built-in pattern).
  - If caller has active run: stop it and reply `commands.btw.message.stopped` (default `The process stopped.`).
  - If no active run: reply `commands.btw.message.nothingToDo` (default `No active process to stop, nothing to do.`).

### 4.2 — Register command in bot bootstrap
- In `src/bot.ts`:
  - register `bot.command("btw", createBtwHandler(projectCtx))` when enabled.
  - add disabled catch behavior similar to `/info`/`/model`/`/engine` for `/btw`.
- Keep `/stop` unregistered and unexposed.

### 4.3 — Include `/btw` in command discovery lists
- Update `src/bot/commands/loader.ts`:
  - `CommandEnabledFlags` add `btw`
  - builtin commands list add `btw` with description
  - enabled map includes `btw`
- Update all `enabled` object constructors to include `btw` (e.g. `src/bot.ts`, `src/bot/commands/message.ts`).
