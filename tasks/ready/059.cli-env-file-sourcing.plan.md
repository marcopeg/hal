# Plan: 059 — CLI env file sourcing

## Goal

Add per-project engine runtime env-file sourcing via `projects.<key>.engine.envFile`, resolved from project `cwd` when relative, used only for the spawned engine CLI child process, with boot failure limited to active projects when the file is invalid.

## Scope Decisions (locked)

- Config key: `projects.<key>.engine.envFile`.
- Runtime target: engine child process only (no HAL process env mutation).
- Loading mode: shell sourcing semantics before running engine CLI.
- Validation behavior: fail boot for active projects only; inactive invalid projects do not block startup.
- Existing top-level `env` (config substitution) remains unchanged.

## Phase 1 — Config Surface and Resolved Model

1. Extend config schemas in `src/config.ts`:
- Add `envFile: z.string().optional()` under `EngineConfigSchema`.
- Ensure local overrides accept it via existing `.partial()` flow.

2. Extend resolved project config type:
- Add `engineEnvFile?: string` to `ResolvedProjectConfig`.
- Populate it in `resolveProjectConfig` using project-over-globals precedence.

3. Keep compatibility:
- Do not change behavior when `engineEnvFile` is undefined.

## Phase 2 — Env File Resolution and Active-Project Validation

1. Add helper for runtime env-file resolution (in `src/config.ts` or a small shared util):
- Input: `project.cwd` (already resolved absolute) and raw `engineEnvFile`.
- Behavior: absolute path stays absolute; relative resolves from `project.cwd`.
- Output: normalized absolute file path.

2. Validate only active projects during boot/reload path in `src/cli.ts` (`runBotsForConfig`):
- After filtering to active projects, validate each configured `engineEnvFile` path exists and is readable.
- Throw `ConfigLoadError` with project slug/name and resolved path on failure.

3. Logging/source visibility:
- Include resolved `engine.envFile` (for projects where configured) in startup source logs to aid troubleshooting.

## Phase 3 — Child Process Shell Sourcing Execution Path

1. Introduce a shared spawn utility for adapters (new file, e.g. `src/engine/spawn.ts`):
- Provide a function that executes engine commands with optional shell-sourced env file.
- No env file: use current direct spawn behavior.
- With env file: run through shell so env is sourced before command execution.

2. Shell behavior details:
- Use explicit shell execution on Unix-like systems (HAL current target).
- Ensure variable export semantics so values from sourced file are available to engine process.
- Preserve adapter-provided env overrides (for example OpenCode's `OPENCODE_DISABLE_CLAUDE_CODE_PROMPT`).

3. Safety and correctness:
- Avoid command injection by robust command/arg quoting strategy in the shell path.
- Keep stdout/stderr streaming behavior unchanged for adapters.

## Phase 4 — Adapter Integration

1. Update adapters to use shared spawn path with `config.engineEnvFile`:
- `src/engine/adapters/claude.ts`
- `src/engine/adapters/copilot.ts`
- `src/engine/adapters/codex.ts`
- `src/engine/adapters/opencode.ts`
- `src/engine/adapters/cursor.ts`
- `src/engine/adapters/antigravity.ts`

2. Preserve current behavior:
- Existing args/session semantics remain unchanged.
- Existing env maps remain effective and are merged into child process env.

## Phase 5 — Docs, Example, and Validation Notes

1. Docs updates:
- `docs/config/reference.yaml`: document `projects.<key>.engine.envFile`.
- `docs/config/README.md`: add `engine.envFile` to project keys table and explain runtime semantics vs top-level `env` substitution.

2. Example update:
- `examples/hal.config.yaml`: add a commented project-level `engine.envFile` example.

3. Manual verification checklist (add/update under `manual-tests/`):
- Relative path resolves from project cwd.
- Absolute path is used as-is.
- Missing/unreadable env file fails boot for active project.
- Inactive project with invalid env file does not block startup.
- Behavior unchanged when `engine.envFile` is omitted.

## Implementation Order

1. Phase 1 (schema + resolved config field).
2. Phase 2 (active-only validation + path helper).
3. Phase 3 (shared shell-sourcing spawn utility).
4. Phase 4 (adapter migration to shared utility).
5. Phase 5 (docs/example/manual-tests + lint/build verification).

## Risks and Mitigations

1. Shell quoting mistakes could break commands or create injection risk.
- Mitigation: centralize quoting in one utility and reuse across all adapters.

2. `source` semantics differ across shells.
- Mitigation: use a single explicit shell strategy and document expected platform behavior.

3. Boot validation location could accidentally include inactive projects.
- Mitigation: perform validation only after active filtering in `runBotsForConfig`.

## Exit Criteria

- Config accepts `projects.<key>.engine.envFile`.
- Active projects fail at boot/reload when env file is invalid.
- Engine CLI commands run with sourced env file when configured.
- Existing behavior remains unchanged when not configured.
- Docs and examples clearly distinguish top-level `env` substitution from project engine runtime `envFile`.
