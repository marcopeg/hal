# 030 — Provider default model propagation

## Status

draft

## Summary

Support a default model **per provider** by allowing one model in each `providers.<engine>` list to be marked with `default: true`. When `engine.model` is not set, pass that provider’s default model name to the active engine adapter (e.g. Codex CLI `--model`, Claude, etc.). Validate at boot that at most one model per provider list has `default: true`.

## Context

- **Current behavior**: `engine.model` is resolved from project or globals and exposed as `config.engineModel`. Engine adapters accept an optional `model`; when `engine.model` is unset, some engines use HAL defaults from `default-models.ts`, others get no model and use the engine’s built-in default.
- **Project-level model**: Resolved as `project.engine?.model ?? globals.engine?.model`; this value takes precedence over any provider default.
- **Existing `providers.<engine>`**: Array of `{ name, description? }` used for the `/model` command picker. Same list is used to define the default: one entry may have `default: true`, so the default is always one of the listed models (minimal schema change, maximum backward compatibility).

## Requirements

- Extend provider model schema: each entry is `{ name, description?, default?: boolean }`. Optional `default` marks that model as the provider default for that engine.
- **Boot-time validation**: For each provider list (per engine, in globals and in each project), at most one model may have `default: true`. If two or more models in the same list have `default: true`, fail at boot with a clear `ConfigLoadError` (hard crash).
- **Provider default for the active engine**: When `engine.model` is not set, resolve the provider list for the active engine (project over globals); if exactly one model in that list has `default: true`, use that model’s `name` and pass it to the engine adapter. Otherwise (no list, empty list, or no model marked default), do not pass a model and rely on the engine’s built-in or HAL default.
- If `engine.model` is explicitly set (project or globals), it always takes precedence; provider default is only used when `engine.model` is absent.
- **Documentation:** User-facing docs (e.g. `docs/engines`, `docs/config`) must describe the `default` field on provider model entries, boot validation (at most one default per list), and precedence (`engine.model` overrides provider default).
- All engines that accept a model parameter can use a provider default; behavior is uniform (no engine-specific branching beyond “does this engine get a model?”).

## Acceptance Criteria

- **Documentation (requirement):** User-facing docs (e.g. `docs/engines`, `docs/config`) must describe the `default` field on provider model entries, boot validation (at most one default per list), and precedence (`engine.model` overrides provider default).
- Config parsing accepts optional `default: true` on provider model entries (`ProviderModelSchema` or equivalent).
- Boot validation: for every `providers.<engine>` array (globals and each project), at most one entry has `default: true`; otherwise throw `ConfigLoadError` with a clear message (e.g. which engine and which list).
- Resolved project config exposes the provider default model for the active engine when applicable (the `name` of the model with `default: true` in the resolved provider list), so that at runtime the adapter receives that model when `engine.model` is unset and the resolved provider list has exactly one default.
- When no default is available (no model with `default: true` in the resolved list), the engine is invoked without a model override; engine uses its own or HAL default.
- Project-level `engine.model` continues to override provider default.
- Same behavior for every engine: effective model = `engine.model` ?? `providerDefaultModel` ?? HAL default (if any); no special-casing by engine name beyond existing adapter behavior.
- User-facing documentation (e.g. `docs/engines/README.md` model list and model defaults, `docs/config/README.md` if it references providers) describes: optional `default: true` on provider entries, at-most-one-default-per-list validation and `ConfigLoadError`, and that explicit `engine.model` overrides the provider default.
