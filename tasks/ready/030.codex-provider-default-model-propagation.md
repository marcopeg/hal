# 030 — Codex provider default model propagation

## Status

draft

## Summary

Support a default model for the Codex engine by marking one of the models in `providers.codex` with `default: true`. When `engine.model` is not set, pass that model's `name` to the Codex CLI via `--model`. Validate at boot that at most one model per provider list has `default: true`.

## Context

- **Current behavior**: `engine.model` is resolved from project or globals and exposed as `config.engineModel`. Codex adapter already supports an optional `model` and passes it as `-m <model>` to the CLI. When `engine.model` is unset, Codex gets no `--model` (no entry in `default-models.ts`).
- **Project-level model**: Resolved as `project.engine?.model ?? globals.engine?.model`; this value takes precedence over any provider default.
- **Existing `providers.<engine>`**: Array of `{ name, description? }` used for the `/model` command picker. Same list is used to define the default: one entry may have `default: true`, so the default is always one of the listed models (minimal schema change, maximum backward compatibility).

## Requirements

- Extend provider model schema: each entry is `{ name, description?, default?: boolean }`. Optional `default` marks that model as the provider default for that engine.
- **Boot-time validation**: For each provider list (per engine, in globals and in each project), at most one model may have `default: true`. If two or more models in the same list have `default: true`, fail at boot with a clear `ConfigLoadError` (hard crash).
- When the engine is Codex and `engine.model` is not set: resolve the provider list for Codex (project over globals); if exactly one model in that list has `default: true`, use that model's `name` and pass it to the Codex CLI as `--model <name>`. Otherwise (no list, empty list, or no model marked default), do not pass `--model` and rely on the engine's built-in default.
- If `engine.model` is explicitly set (project or globals), it always takes precedence; provider default is only used when `engine.model` is absent.
- No behavior change for non-Codex engines.

## Acceptance Criteria

- Config parsing accepts optional `default: true` on provider model entries (`ProviderModelSchema` or equivalent).
- Boot validation: for every `providers.<engine>` array (globals and each project), at most one entry has `default: true`; otherwise throw `ConfigLoadError` with a clear message (e.g. which engine and which list).
- Resolved project config exposes the provider default model for the active engine when applicable (e.g. the `name` of the model with `default: true` in the resolved provider list), so that at runtime the Codex adapter receives `--model <name>` when `engine.model` is unset and the Codex provider list has exactly one default.
- When no default is available (no model with `default: true` in the resolved list), Codex is invoked without `--model`; engine uses its own default.
- Project-level `engine.model` continues to override provider default.
- No behavior change for non-Codex engines.
