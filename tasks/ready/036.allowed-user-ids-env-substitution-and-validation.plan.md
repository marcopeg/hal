# Plan: 036 — Allowed user IDs: env substitution and numeric validation

## Phase 1: Schema — accept string or number for `allowedUserIds`

**Goal:** Config parsing accepts both number and string entries so that env-substituted values (strings) pass the schema.

1. In `src/config.ts`, change **GlobalsFileSchema** `access.allowedUserIds` from `z.array(z.number())` to `z.array(z.union([z.number(), z.string()]))`.
2. Change **ProjectFileSchema** `access.allowedUserIds` the same way.
3. **LocalProjectSchema** inherits from `ProjectFileSchema.partial()`, so it will automatically allow `(number | string)[]` for `allowedUserIds`; no change needed there.
4. Ensure the inferred file types still allow the merged/substituted config to be parsed; the parsed config will have `allowedUserIds?: (string | number)[]` until normalization.

**Outcome:** Base and final Zod parse succeed when `allowedUserIds` contains numbers and/or strings (e.g. `["${MY_ID}", 123456789]`). No behavior change yet for invalid values (normalization in Phase 2).

---

## Phase 2: Normalize and validate after substitution

**Goal:** After the final parse, convert every `allowedUserIds` entry to a number using strict rules; throw `ConfigLoadError` with path and value on invalid data; return config with `number[]` only.

1. **Add a single-value validator/normalizer** (e.g. `parseTelegramUserId(value: string | number, path: string): number`):
   - Coerce to string if number (for consistent exact-match check).
   - Parse to number (e.g. `Number(resolved)`); reject `NaN`, `Infinity`, non-finite.
   - Require integer (e.g. `Number.isInteger(num)`).
   - **Exact string match:** `String(num) === String(resolved).trim()` is too loose — task says no spaces. So: for string input, after trim (or no trim per task "formatting drift" = reject), require `String(num) === resolved` (no trim so spaces reject). For number input, no string to match; just require integer and optional range.
   - Optional: require `1 <= num <= 0xFFFFFFFFF` (Telegram user ID range).
   - On failure: throw `ConfigLoadError` with message including `path` and the resolved value (e.g. `invalid allowedUserIds entry at globals.access.allowedUserIds[1]: "12.34"`).
   - Return the number.

2. **Add config-wide normalizer** (e.g. `normalizeAllowedUserIdsInConfig(config)`):
   - Accept the parsed multi-config (with `allowedUserIds?: (string | number)[]` in globals and each project).
   - For `config.globals?.access?.allowedUserIds`: iterate with index, call the single-value normalizer with path `globals.access.allowedUserIds[i]`, replace array in place with `number[]`.
   - For each `config.projects[j].access?.allowedUserIds`: same with path `projects.<j>.access.allowedUserIds[i]` (use project name or index for clarity).
   - Skip missing/undefined `access` or `allowedUserIds` (no change).
   - Ensure the object is mutated so returned references have `number[]` (or build a new config and return it). Downstream and `resolveProjectConfig` expect `number[]`.

3. **Integrate into load flow** in `loadMultiConfigInternal`:
   - After step 6 (final `MultiConfigFileSchema.safeParse(substituted)`), call `normalizeAllowedUserIdsInConfig(finalResult.data)`.
   - Normalizer throws `ConfigLoadError` on any invalid entry; that propagates and is reported by `loadMultiConfig` / `tryLoadMultiConfig`.
   - TypeScript: after normalization, config has `number[]` at runtime. Use a type assertion or a small type (e.g. `NormalizedMultiConfigFile`) when returning so `LoadedConfigResult.config` is typed with `allowedUserIds: number[]` for consumers.

**Outcome:** Env-substituted strings and numeric entries become validated numbers; invalid entries fail at load with a clear path and value; existing numeric-only configs keep working.

## Phase 3: Hot reload — exit process on config error

**Goal:** When hot reload encounters a config load/validation error, exit the process instead of only logging.

1. In `src/cli.ts`, in the config watcher callback's `catch` block (around the "Reload failed" log):
   - After logging the error, call `process.exit(1)` so that any failed reload (including invalid `allowedUserIds` after substitution) stops the process.
   - Optionally: only exit when the error is `ConfigLoadError` (or has a recognizable config-error property), so other reload failures don't exit. Task says "if hot reload gets into trouble, it should exit the system"; treating any reload failure as fatal is acceptable unless we want to distinguish config vs other errors. Prefer exiting on any reload failure for simplicity, or explicitly on `ConfigLoadError` if we want to avoid exiting on unrelated errors.

**Outcome:** Invalid config on reload stops the process; behavior matches boot (fail-fast).

## Phase 4: Verification and docs

**Goal:** Confirm behavior and document the new option.

1. **Manual verification:** Run with `allowedUserIds` as `[123456789]`, `["123456789"]`, `["${TELEGRAM_USER_ID}"]` (env set), and mixed; confirm valid cases work. Try invalid: `["12.34"]`, `[" 123 "]`, `[""]`, `["${EMPTY}"]`, leading zeros; confirm process exits with a clear message including path and value.
2. **Docs:** In `docs/config/README.md` (and any config table that mentions `allowedUserIds`), state that entries can be numbers or strings (strings allow env substitution); after substitution they are validated and normalized to numeric Telegram user IDs, and invalid values cause config load to fail with an error that includes the path and value.
