# 027 — YAML config support — Development Plan

## Phase 1: Format detection infrastructure

Extract the hardcoded filename logic from `loadMultiConfigInternal` and `loadLocalConfig` into a reusable detection layer.

### Step 1.1: Add `yaml` dependency

- `pnpm add yaml`

### Step 1.2: Create `resolveConfigFile` helper

Add a function (in `src/config.ts` or a new `src/config-format.ts` if preferred) that, given a config dir and a slot basename (e.g. `"hal.config"` or `"hal.config.local"`), does the following:

1. Scan for all supported extensions: `.json`, `.yaml`, `.yml` (leave a slot for `.jsonc` that task 028 will fill).
2. Collect all files that exist on disk.
3. If zero files found → return `null` (not found).
4. If exactly one file found → return `{ path, format }` where format is `"json" | "yaml"`.
5. If more than one found → throw `ConfigLoadError` naming all conflicting files.

### Step 1.3: Create `parseConfigContent` helper

A function that takes `(content: string, format: "json" | "yaml", filePath: string)` and returns `unknown`:

- `"json"` → `JSON.parse(content)`
- `"yaml"` → `yaml.parse(content)` (from the `yaml` npm package)
- Wraps parse errors in `ConfigLoadError` referencing the actual filename.

### Step 1.4: Refactor `loadMultiConfigInternal`

Replace the hardcoded `hal.config.json` lookup (lines ~759–793) with:

1. Call `resolveConfigFile(configDir, "hal.config")`.
2. If `null` → throw existing "not found" error (updated to list all supported extensions).
3. Read file, call `parseConfigContent` with detected format.
4. Rest of the pipeline (Zod validation, env substitution, etc.) stays unchanged — it operates on the parsed `unknown` object regardless of source format.

### Step 1.5: Refactor `loadLocalConfig`

Replace the hardcoded `hal.config.local.json` lookup (lines ~677–701) with:

1. Call `resolveConfigFile(configDir, "hal.config.local")`.
2. If `null` → return `null` (no local config).
3. Read file, call `parseConfigContent` with detected format.
4. Zod validation and merge logic unchanged.

### Step 1.6: Update error messages

All error messages currently hardcoding `hal.config.json` or `hal.config.local.json` should reference the actual detected filename. The `resolveConfigFile` return value provides this.

## Phase 2: Config watcher update

### Step 2.1: Update `CONFIG_FILES` in `config-watcher.ts`

The watcher currently watches a static list: `["hal.config.json", "hal.config.local.json", ".env", ".env.local"]`.

Replace with a list that includes all supported config extensions:
```
hal.config.json
hal.config.yaml
hal.config.yml
hal.config.local.json
hal.config.local.yaml
hal.config.local.yml
.env
.env.local
```

Chokidar handles non-existent files gracefully (they just don't trigger events until created), so watching all candidates is safe.

---

## Phase 3: Config writer update

### Step 3.1: Update `config-writer.ts`

`updateProjectModel` currently hardcodes `hal.config.local.json` and `hal.config.json`. It reads JSON and writes JSON.

Update to:

1. Use `resolveConfigFile` to detect which format exists for each slot.
2. Read and parse using `parseConfigContent`.
3. Write back in the same format: JSON via `JSON.stringify`, YAML via `yaml.stringify`.

If no local config exists and the base is YAML, write a new `hal.config.local.yaml` (matching the base format) — or fall back to JSON. Document this choice.

## Phase 4: CLI update

### Step 4.1: Update help text in `cli.ts`

The help text references `hal.config.json` in several places. Update to say `hal.config.{json,yaml}` or similar, so users know YAML is supported.

### Step 4.2: Update `init` command

The `runInit` function creates `hal.config.json`. Consider:

- Keep `init` generating JSON by default (it's the simpler format for first-time users).
- No changes needed unless we want a `--format yaml` flag (out of scope for this task).

### Step 4.3: Update startup log line

The loaded-files log (line ~244) that annotates `[local override]` based on filename — update the suffix detection to handle `.yaml`/`.yml` endings too.
