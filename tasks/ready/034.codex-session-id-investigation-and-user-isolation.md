# 034 — Validate Codex session ID isolation

## Status

ready

## Summary

Validate and harden HAL's existing experimental Codex per-user session isolation so Codex sessions are safely segregated per Telegram user, documented clearly, and verified with a real two-client manual test.

## Problem

Codex no longer needs greenfield investigation: HAL already contains an experimental per-user implementation that scans `~/.codex/sessions/` after successful runs, extracts a session UUID for the project `cwd`, stores it per Telegram user, and resumes via `codex exec resume <UUID>` on later messages.

The remaining problem is confidence, not feasibility. The current implementation relies on Codex internal filesystem artifacts and has not yet been fully validated as a solid, production-safe isolation mechanism. Without that validation, HAL risks either:

- leaking context between Telegram users if the wrong Codex session is resumed,
- silently degrading to unsafe behavior when UUID discovery fails,
- or documenting behavior more strongly than the implementation actually guarantees.

This task is Codex-specific. It is not about designing a generic non-Claude isolation layer.

## Current Context

- Existing implementation lives in `src/engine/adapters/codex.ts` and `src/engine/codex-sessions.ts`.
- Existing docs already describe Codex `session: "user"` as experimental and based on scanning `~/.codex/sessions/`.
- The implementation appears to have been introduced during the session-configuration work around task 040 rather than through this draft task.
- Historical notes in `vibed/2603031018.open-session.md` show the earlier Codex work first focused on shared continuation (`resume --last`); later task 040 added the experimental per-user UUID discovery design/implementation.
- Codex CLI stores local session data under `~/.codex/`, with session transcripts under `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl`.
- Codex session transcript files are JSONL and may contain prompts, responses, tool calls, timestamps, token usage, and a `cwd` field that HAL can use to associate a session transcript with a project working directory.
- HAL runs Codex non-interactively via `codex exec ...`; for resumed non-interactive sessions the relevant CLI surface is `codex exec resume [SESSION_ID|--last]`, not the top-level interactive `codex resume` command.

## Product Decisions

- Codex should default to per-user session mode. Omitting `engine.session` for Codex should resolve to the equivalent of `"user"`, and this behavior change must be documented.
- Shared Codex sessions remain supported as an explicit opt-in via `engine.session: "shared"` and should continue to behave as they do today using Codex's last-session continuation semantics.
- In Codex `"user"` mode, `/clear` should only delete the user's local `session.json` and should **not** perform an active reset call to Codex.
- In Codex `"user"` mode, if HAL cannot discover a real Codex session UUID after a fresh run, it must not silently fall back to shared continuation.
- Instead, HAL should warn the Telegram user that a user-specific session ID could not be recovered, so continuity is unavailable and subsequent messages will run as new anonymous sessions until session discovery works again.
- The same failure should be logged normally in HAL logs/terminal output for operator visibility.

## Goals

- Verify whether the current Codex UUID discovery logic works reliably on a real Codex installation.
- Confirm that `engine.session: "user"` provides actual per-user session segregation for Codex in HAL.
- Identify failure modes and edge cases in the current implementation, especially around filesystem discovery and resume behavior.
- Tighten fallback behavior so isolation guarantees are explicit and safe when UUID extraction fails.
- Ensure docs match the real implementation, risks, and operational expectations.

## Scope

- Validate the existing implementation in `src/engine/adapters/codex.ts` and `src/engine/codex-sessions.ts`.
- Review Codex CLI behavior and local session artifacts only as needed to confirm or reject the current approach.
- Change Codex session semantics so the default mode is per-user and shared mode is explicit.
- Define and implement safe failure behavior for per-user mode when session UUID discovery fails.
- Update documentation if behavior, caveats, or setup expectations are inaccurate or incomplete.
- Run a manual verification scenario with 2 Telegram clients/users against Codex.

Out of scope:

- generic session-isolation design for other engines,
- introducing per-user isolation for Copilot/OpenCode,
- broad refactors unrelated to Codex session handling.

## Deliverables

- Verified assessment of whether the current Codex isolation implementation works as intended.
- Clear description of the exact mechanism HAL relies on for Codex session segregation.
- List of discovered risks, edge cases, and any required code hardening.
- Documentation updates covering setup, caveats, guarantees, and fallback behavior for Codex `session: "user"`.
- Updated behavior definition for Codex default session mode, explicit shared mode, and `/clear` semantics in per-user mode.
- Manual test evidence/checklist for a 2-client Telegram scenario demonstrating either successful isolation or a reproducible failure.

## Expected Behavior

- In Codex `"user"` mode, HAL should read `currentSessionId` from the user's `session.json` before each message.
- If `currentSessionId` exists and is a real Codex session identifier, HAL should resume with `codex exec resume <SESSION_ID>`.
- If no stored session identifier exists, HAL should run a fresh non-interactive Codex session for that message.
- After a successful fresh run, HAL should inspect the latest Codex session transcript files under `~/.codex/sessions/`, identify the newly created session for the configured `cwd`, extract the session identifier, and store it in that user's `session.json`.
- In Codex `"user"` mode, `session.json` should store a real Codex session identifier, not the placeholder `"active"` marker used by shared/session-marker flows.
- In Codex `"shared"` mode, HAL may continue using the existing shared continuation behavior.
- In Codex `"user"` mode, `/clear` should only remove `session.json`; the next message should start a fresh Codex session without any active reset call.
- In Codex `"shared"` mode, `/clear` may continue to actively start a fresh shared Codex session.
- If HAL cannot recover a session ID in Codex `"user"` mode, it should warn the user in Telegram and continue without session continuity rather than attaching that user to a shared/project-level session.

## Acceptance Criteria

- The current implementation path is exercised on a real Codex setup, not only reviewed statically.
- A manual test with 2 Telegram clients/users is executed and documented.
- The test shows whether separate users resume separate Codex sessions when `engine.session: "user"` is enabled.
- Codex default session behavior is changed to per-user and this breaking/default change is documented in engine and session docs.
- Shared Codex behavior remains available via explicit `engine.session: "shared"` configuration and is documented.
- In Codex `"user"` mode, `session.json` stores a real Codex session ID when discovery succeeds.
- In Codex `"user"` mode, `/clear` only deletes `session.json` and does not trigger an active Codex reset message.
- If session ID discovery fails in Codex `"user"` mode, HAL does not silently fall back to shared `--last` continuation for that user.
- If session ID discovery fails in Codex `"user"` mode, HAL sends a Telegram warning explaining that user-specific continuity is unavailable and messages will run in fresh anonymous sessions until recovery succeeds.
- The same failure is visible in normal HAL logs.
- Documentation clearly labels the feature as experimental and explains the dependency on Codex internal session files under `~/.codex/sessions/`.
- Risks of relying on unofficial/internal Codex artifacts are explicitly documented.
