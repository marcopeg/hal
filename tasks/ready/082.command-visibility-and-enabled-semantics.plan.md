# Plan for 082 — Built-in command visibility and enabled semantics

## Goal

Separate command routing from command visibility for built-in commands, add independent `showInMenu` and `showInHelp` controls with config inheritance, preserve existing runtime auto-availability rules for `/model` and `/engine`, and make disabled built-ins fall through to lower-precedence custom commands before the agent. Also align the npm-derived command surface, Telegram menu publication, `${HAL_COMMANDS}`, tests, examples, and docs with the new semantics.

## Phase 1 — Lock the target command model and affected surfaces

### Step 1.1 — Trace the current command lifecycle end to end

- Confirm how command config schemas and resolved defaults are defined in `src/config.ts`, including which command entries currently support only `enabled` and which ones carry extra fields.
- Confirm how built-ins and custom commands are merged in `src/bot/commands/loader.ts`, especially the current behavior that deletes disabled built-ins from the merged command map.
- Confirm where Telegram slash commands are published in `src/bot.ts` and `src/bot/commands/watcher.ts`, and where `${HAL_COMMANDS}` is built in `src/bot/commands/message.ts`.
- Confirm where slash text falls through to project/global custom commands and skills in `src/bot/handlers/text.ts`, because disabled built-ins must now rely on that resolver path instead of hard-coded “disabled” replies.

### Step 1.2 — Translate the task into implementation rules

- Treat `enabled` strictly as a routing flag for built-ins and npm-derived commands, not as an implicit visibility switch.
- Treat `showInMenu` and `showInHelp` as independent visibility controls that default to `true` unless a command-specific default overrides them.
- Keep `/start` interceptable by default while hidden from both menu/help by default.
- Preserve existing runtime availability rules for `/model` and `/engine` so their effective command presence still depends on available engines/models, even when config visibility defaults are true.
- Treat disabled built-ins as non-routable built-ins only; allow project custom commands first, then system/global custom commands, then the agent.
- Treat `commands.npm` as one config entry controlling a derived command surface from whitelist/blacklist resolution, not as a guaranteed single `/npm` launcher shape.

## Phase 2 — Extend config schema and resolved command metadata

### Step 2.1 — Add visibility fields to command schemas

- Extend the command config schemas in `src/config.ts` so built-in command entries support `showInMenu` and `showInHelp` wherever this task requires them.
- Preserve global-to-project inheritance for the new fields using the same override pattern already used for `enabled` and existing per-command options.
- Decide whether to model the visibility fields through a shared helper schema for built-ins or explicit per-command schema updates; keep the change minimal and consistent with current file style.

### Step 2.2 — Expand `ResolvedProjectConfig` for routing and visibility

- Add resolved visibility metadata for built-ins and npm-derived commands to the `ResolvedProjectConfig["commands"]` shape in `src/config.ts`.
- Encode command-specific defaults there: `/start` hidden in menu/help, `/reset` disabled by default, `/help` `/clear` `/info` visible by default, `/model` and `/engine` visible by default subject to runtime availability, `commands.npm.enabled` defaulting to `false`.
- Preserve the existing runtime-derived `model.enabled` and `engine.enabled` computation while attaching resolved menu/help visibility fields to the effective entries.
- Re-check any exported types that consume `ResolvedProjectConfig`, since command shape changes will cascade through `src/types.ts` and tests.

## Phase 3 — Refactor command loading into routing and visibility views

### Step 3.1 — Separate “routable commands” from “visible commands” in loader utilities

- Refactor `src/bot/commands/loader.ts` so built-in entries are not filtered solely by `enabled` when the caller actually needs the broader command universe for visibility or precedence logic.
- Introduce explicit filtering helpers instead of one shared `commandsForTelegramMenu` path for everything. The implementation likely needs distinct selectors for Telegram menu visibility and HAL help visibility.
- Ensure custom `.mjs` commands and Telegram-enabled skills continue to participate in precedence and menu/help output independently of built-in routing flags.

### Step 3.2 — Model npm-derived command entries explicitly

- Replace or extend the current single built-in `/npm` entry so loader output can represent the effective npm-derived command surface produced from `package.json` scripts plus whitelist/blacklist filtering.
- Decide where script names are discovered and converted into `CommandEntry` records so the same derived surface can feed Telegram menu publication and `${HAL_COMMANDS}`.
- Preserve the current `package.json` error handling path in `src/bot/commands/npm/scripts.ts` and avoid making menu/help generation crash when scripts are missing or unreadable.
- Keep Telegram naming constraints in mind if npm script names need sanitization, aliasing, or explicit filtering before becoming slash commands.

### Step 3.3 — Update menu/help builders to use the new visibility helpers

- Update `src/bot.ts` startup publication to use the menu-specific visibility filter rather than the current shared command list.
- Update `src/bot/commands/watcher.ts` to republish from the same menu-specific selector so hot reload matches startup behavior.
- Update `src/bot/commands/message.ts` to build `${HAL_COMMANDS}` from the help-specific visibility selector rather than mirroring Telegram menu logic.
- Preserve existing grouping and sort behavior where possible so this task changes semantics, not unrelated presentation.

## Phase 4 — Rework routing semantics for disabled built-ins

### Step 4.1 — Remove hard-coded disabled replies that block fallback

- Update `src/bot.ts` so disabled `/model`, `/engine`, and `/info` no longer respond with `This command is disabled.` when the task now requires fallback to custom commands or the agent.
- Re-check other built-ins registered only when enabled, since their disabled behavior should align with the new routing rule rather than implicit interception gaps.
- Keep callback-query handling sensible for stale built-in inline buttons; only preserve “disabled” callback responses where there is no meaningful fallback path for a callback event.

### Step 4.2 — Make slash interception honor precedence

- Update `src/bot/handlers/text.ts` so slash-command resolution continues to prefer project `.mjs` commands over global `.mjs` commands, then Telegram-enabled skills, before falling through to the agent.
- Ensure this precedence now works even when the slash name matches a disabled built-in such as `/help` or `/reset`.
- Verify that enabled built-ins still intercept first through Grammy command handlers, so custom-command shadowing remains blocked only while the built-in is effectively enabled.

## Phase 5 — Adapt npm runtime handling to the chosen surface

### Step 5.1 — Align command handlers with the derived npm command surface

- Update `src/bot/commands/npm/index.ts` and related registration in `src/bot.ts` to match whatever routable npm command shape the task requires after script derivation.
- Preserve concurrency locking, whitelist/blacklist enforcement, timeout handling, and large-output file sending.
- If the implementation still keeps `/npm` as an execution entry point for some flows, document exactly how that coexists with the exposed derived commands so behavior remains unambiguous.

### Step 5.2 — Keep command publication resilient around package state

- Decide how missing `package.json`, invalid scripts, or filtered-empty npm surfaces should affect Telegram menu and HAL help output.
- Prefer graceful omission plus runtime error messaging over startup failures unless the existing code already requires a hard error for the same condition.

## Phase 6 — Add focused test coverage

### Step 6.1 — Extend config-resolution tests

- Update `src/config.test.ts` to cover inheritance and defaults for `showInMenu` and `showInHelp` at both global and project scope.
- Add coverage for `/start` default hidden visibility, `/reset` default disabled routing, and `commands.npm.enabled` default false.
- Keep existing `/model` and `/engine` runtime availability assertions intact while validating their visibility defaults independently.

### Step 6.2 — Add command-surface and routing tests

- Add or extend tests around loader/menu/help filtering so Telegram menu and `${HAL_COMMANDS}` can diverge correctly.
- Add targeted coverage for the disabled-built-in fallback rule in `src/bot/handlers/text.test.ts`, especially same-name project command, same-name global command, and final agent fallback.
- Add coverage for npm-derived command exposure and any sanitization/filtering logic introduced for menu/help publication.

### Step 6.3 — Run project validation

- Run the repo’s relevant validation commands after implementation: build/typecheck, tests, and lint as appropriate for touched files.
- Fix only regressions introduced by this task.

## Phase 7 — Update maintained docs and examples

### Step 7.1 — Update command configuration docs

- Update `docs/config/commands/README.md` to explain the split between routing (`enabled`) and visibility (`showInMenu`, `showInHelp`).
- Update `/info`, `/model`, `/engine`, `/start`, `/reset`, and npm sections so the user-facing behavior matches the new routing and visibility semantics.
- Update the `${HAL_COMMANDS}` section to state that help output now uses its own visibility rules, not the Telegram menu filter.

### Step 7.2 — Update custom-command docs and examples

- Update `docs/custom-commands/README.md` so command precedence reflects: enabled built-ins first, then project custom commands, then global custom commands, then Telegram-enabled skills, then agent; disabled built-ins should no longer be described as always blocking same-name custom commands.
- Update `examples/hal.config.yaml` and any maintained config reference/examples to show the new visibility keys where they clarify defaults or supported shape.
- Review other maintained docs for `/npm` wording if the visible command surface changes from the current single `/npm` description.

### Step 7.3 — Record the change in release notes

- Update `CHANGELOG.md` with the built-in visibility split, routing fallback behavior, and npm command-surface change.

## Risks to watch

- Accidentally tying menu and help visibility back together through one shared helper.
- Regressing `/model` and `/engine` auto-availability by conflating runtime availability with visibility defaults.
- Making disabled built-ins swallow slash commands through leftover `bot.hears(...)` handlers or callback interceptors.
- Publishing invalid Telegram command names from npm script names without sanitization or filtering.
- Making Telegram menu publication depend on `package.json` in a way that breaks bot startup or watcher republish unnecessarily.
- Updating docs/examples to the new shape while leaving contradictory older wording in command precedence or `/npm` behavior sections.
