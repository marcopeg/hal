# Development plan — 055 NPM commands wrapper

## Phase 1 — Config schema + resolution

### Step 1.1 — Zod schema for `commands.npm`

In `src/config.ts`:

- Create `NpmConfigSchema` (mirrors `GitConfigSchema` pattern but with extra keys):
  - `enabled: z.boolean().optional()` (default `false`)
  - `whitelist: z.array(z.string()).optional()`
  - `blacklist: z.array(z.string()).optional()`
  - `timeoutMs: z.number().positive().optional()` (default `60000`)
  - `maxOutputChars: z.number().positive().optional()` (default TBD, safe Telegram limit)
  - `sendAsFileWhenLarge: z.boolean().optional()` (default `true`)
- Add `npm: NpmConfigSchema` to `CommandsConfigSchema`.

### Step 1.2 — Extend `ResolvedProjectConfig.commands`

- Add `npm` to the `commands` type in `ResolvedProjectConfig`:
  ```
  npm: {
    enabled: boolean;
    whitelist: string[] | undefined;
    blacklist: string[] | undefined;
    timeoutMs: number;
    maxOutputChars: number;
    sendAsFileWhenLarge: boolean;
  }
  ```
- In `resolveProjectConfig()`, resolve `npm` from project > globals > defaults (follow `git` pattern; default `enabled: false`).

### Step 1.3 — Extend `CommandEnabledFlags`

- Add `npm: boolean` to `CommandEnabledFlags` in `src/bot/commands/loader.ts`.
- Add `"npm"` to `BUILTIN_ENABLED_MAP` so the loader respects the flag.

### Step 1.4 — Register built-in command entry

- Add an `npm` entry to `BUILTIN_COMMANDS` (or a new dedicated array like `GIT_COMMANDS`) with:
  - `command: "npm"`, `description: "Run an npm script"`, `source: "builtin"`.
- Ensure the loader includes/excludes it based on `enabled.npm`.

## Phase 3 — Script discovery + allowlist resolution

### Step 3.1 — `package.json` reader

Create a helper (e.g. `src/bot/commands/npm/scripts.ts`):

- `readPackageScripts(cwd: string): Record<string, string>` — reads `<cwd>/package.json`, parses it, returns `scripts` map.
- Throws a typed error if:
  - `package.json` does not exist or is unreadable.
  - `scripts` key is missing or empty.

### Step 3.2 — Allowlist resolver

- `resolveAllowedScripts(available: string[], whitelist?: string[], blacklist?: string[]): string[]`
  - If `whitelist` present: `allowed = available ∩ whitelist`.
  - Else: `allowed = available`.
  - If `blacklist` present: `allowed = allowed \ blacklist`.
  - Return `allowed` (may be empty; caller decides error handling).

### Step 3.3 — Validation in handler

Inside the handler, before execution:

1. Call `readPackageScripts(config.cwd)` — on failure, reply with error + return.
2. Call `resolveAllowedScripts(...)` with config whitelist/blacklist.
3. If `allowed` is empty, reply with an error listing discovered scripts and active filters.
4. Parse `<script>` from message text (everything after `/npm `).
5. **If `<script>` is missing** (bare `/npm`): reply with an inline keyboard where each button is one allowed script. Use callback data prefixed `npm:<script>` (e.g. `npm:build`, `npm:test`). Return early — no execution yet.
6. If `<script>` not in `allowed`, reply with "script not allowed" + allowed list.

### Step 3.4 — Callback handler for script buttons

- Create `createNpmCallbackHandler(ctx: ProjectContext)` (e.g. in `src/bot/commands/npm/callback.ts`).
- Listen for `callback_query:data` matching the `npm:` prefix.
- Extract the script name from callback data, then run the same execution flow as a typed `/npm <script>`.
- Answer the callback query and remove/update the keyboard after selection.
- Wire the callback handler in `src/bot.ts` inside the `cmd.npm.enabled` block:
  ```ts
  bot.on("callback_query:data", createNpmCallbackHandler(projectCtx));
  ```

## Phase 4 — Subprocess execution + timeout

### Step 4.1 — npm runner

Create `src/bot/commands/npm/exec.ts`:

- `npmExec(cwd: string, script: string, timeoutMs: number): Promise<NpmResult>`
- `NpmResult = { stdout: string; stderr: string; exitCode: number | null; timedOut: boolean; durationMs: number }`
- Use `child_process.spawn("npm", ["run", script], { cwd, stdio: "pipe" })`.
- Collect stdout + stderr into buffers.
- On timeout:
  - Send `SIGINT`, wait 3s.
  - If still alive, `SIGTERM`, wait 2s.
  - If still alive, `SIGKILL`.
  - Set `timedOut = true`.
- Record `durationMs` from spawn to exit/kill.

### Step 4.2 — Concurrency lock

- Maintain a `Map<string, boolean>` keyed by project slug (`activeNpmRuns`).
- Before spawning, check if the project already has an active run. If yes, reply "A command is already running for this project" and return.
- Set the lock before spawn, clear it in a `finally` block.
- The map can live as module-level state in the handler file (same as `resetPrompt.ts` pattern).

## Phase 5 — Output formatting + delivery

### Step 5.1 — Format the result message

Build a summary block:

```
📦 npm run <script>
Status: ✅ success (exit 0) | ❌ failed (exit 1) | ⏱ timed out
Duration: 12.3s

<logs — last N chars>
```

- Wrap logs in a code block (triple backticks, no language tag for plain terminal output).
- Truncate logs to `maxOutputChars` from the **tail** (most recent output is most useful).

### Step 5.2 — Large output as file

- If total output exceeds `maxOutputChars` and `sendAsFileWhenLarge` is true:
  - Send the full summary message (with truncated tail inline).
  - Attach the full log as a `.txt` document via `gramCtx.replyWithDocument()`.
- If `sendAsFileWhenLarge` is false, just truncate inline.

### Step 5.3 — Immediate feedback

- Before spawning, send a short status message: `⏳ Running npm run <script>...`
- After completion, edit or reply with the result (editing the "running" message is cleaner if output fits).
