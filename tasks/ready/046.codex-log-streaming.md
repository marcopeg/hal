# 046 — Codex log streaming

## Status

draft

## Summary

Investigate and implement streaming log updates for the Codex engine, sending incremental progress messages to Telegram as the CLI runs rather than waiting for full completion.

## Problem

The Codex adapter currently buffers all stdout until the process exits before returning a result. Users see no feedback during long-running sessions, making the experience feel unresponsive. Claude Code already streams progress via `onProgress` callbacks using its `stream-json` output format — Codex needs an equivalent approach.

## Context (reference implementation)

**Codex adapter today** (`src/engine/adapters/codex.ts`):

- Spawns `codex exec` (or `codex exec resume --last` / `resume <UUID>`) with permission flags, `-C <cwd>`, and prompt. Destructures only `continueSession` and `sessionId` from `EngineExecuteOptions`; **does not use `onProgress`**. Listens to `proc.stdout` and `proc.stderr`, appends to string buffers; resolves only on `proc.on("close")` with the buffered stdout as output. No line-by-line or streaming handling.

**Claude adapter** (`src/engine/adapters/claude.ts`) — pattern to follow:

- Passes `--output-format stream-json` (and `--verbose`) to the CLI. Reads stdout in `proc.stdout.on("data")`, splits on newlines, parses each line as JSON. Handles event types: `system` (init/session_id), `assistant` (message.content blocks), `user` (tool_result logging), `result` (final result). For `assistant` + `block.type === "tool_use"`, builds a short human-readable `progressMsg` (e.g. "Reading: path", "Running: cmd...") and calls `if (onProgress) onProgress(progressMsg)`. Final result is taken from a `result` event or from accumulated `lastAssistantText` on process close. Same `EngineExecuteOptions` contract: `onProgress?: (message: string) => void`.

**Telegram side**: Handlers (text, voice, photo, document) pass `onProgress` into the engine; they send a status message and then call `editMessageText` from `onProgress` (throttled, e.g. every 2s) so each progress string **replaces** the previous status message.

## Progress UX (explicit expectation)

Progress in Telegram should appear as **reasoning in steps**: each update is a single status message that is **replaced** by the next step forward. The user should see a sequence of messages (e.g. groups of lines or logical steps) where each new step overwrites the previous one, not a single static "Codex is working..." line. Prefer deriving these steps from structured events or meaningful line groups (e.g. reasoning chunks, tool use, phase changes); if only raw lines are available, aggregate or summarize them into step-wise messages so the status still advances clearly. Progress is primarily for **perceived performance** — we don't need to show everything, just keep the user informed; progress messages should be **short**.

## Constraints and behaviour

- **Preferred mode**: Add `--json` to the `codex exec` arguments. In `--json` mode, Codex emits JSONL events to stdout; the adapter parses these for progress. Also add `--color never` to suppress ANSI codes on stderr.
- **Throttling**: Throttle inside the Codex adapter: call `onProgress` at most once every **3 seconds**. Handlers may throttle further; the adapter ensures we don't update the user more often than every 3s.
- **Progress message length**: Keep progress strings short (cap/truncate or summarize) so they fit a single Telegram status message and stay readable.
- **Streams to monitor**: In `--json` mode, parse stdout JSONL for progress events. Also watch stderr for semantic label prefixes as a supplementary or fallback signal.
- **Step granularity**: Use a **minimum interval** of 3s between progress steps to avoid flicker; prefer time-based throttling over line-count. Skip duplicate consecutive messages.
- **Semantic marker detection (JSONL mode)**: Map JSONL `item.started` / `item.completed` events by item type to short human-readable strings:
  - `command_execution` → `"Running: <command (truncated)>"`
  - `file_change` → `"Updating: <path>"`
  - `reasoning` → `"Thinking..."`
  - `agent_message` → `"Responding..."`
  - `mcp_tool_call` → `"Tool: <tool name>"`
  - `web_search` → `"Searching: <query>"`
- **Semantic marker detection (stderr / plain-text fallback)**: Watch for known label prefixes in stderr lines: `exec`, `file update`, `thinking`, `tool`, `hook`, `mcp`, `→`, `✓`, `🌐`. Emit the matched line (stripped of ANSI, truncated).
- **Elapsed-time fallback**: When no streaming signals or semantic markers are detected, emit a progress message of the form **`"Codex is working on it... {N}s"`** where N is elapsed seconds since execution started. Update this message every 3 seconds (replacing the previous one). The initial message fires immediately at t=0 as **`"Codex is working..."`** (no elapsed yet), then subsequent ticks add the elapsed time.
- **Final result**: Unchanged from today — the adapter still resolves with the **full buffered stdout** as `output` on success. Note: in `--json` mode, the full buffered stdout will contain JSONL events, not plain text; the final answer must be extracted from the `agent_message` item in the last `item.completed` event (or equivalent). Ensure the `parse()` / resolve logic handles this correctly.
- **On error**: When the run fails, clear the progress state and show the final error message as the handlers do today (no special handling for "last progress" — error replaces it).

## Findings (pre-research summary — confirm with actual runs)

Research conducted prior to implementation. Implementer should verify these findings with real `codex exec` runs and update accordingly.

### `--json` flag (primary mode)

- `codex exec --json` switches stdout to a **JSONL stream** (one JSON object per line). This is the machine-readable mode; without it, stdout only contains the final plain-text answer.
- Each line has a `"type"` field. Key event types:
  - `thread.started` — `{ thread_id }`
  - `turn.started` / `turn.completed` — `{ usage: { input_tokens, output_tokens } }`
  - `item.started` / `item.updated` / `item.completed` — `{ item: { id, type, ...details } }`
  - `turn.failed` — `{ error: { message } }`
- Item types within events: `agent_message`, `reasoning`, `command_execution`, `file_change`, `mcp_tool_call`, `web_search`, `todo_list`, `error`.
- `command_execution` items include: `command`, `aggregated_output`, `exit_code`, `status`. Note: real-time streaming of subprocess stdout within a running command is **not yet implemented** (issue #4751); output appears only on `item.completed`.
- Final answer is the `text` field of the last `agent_message` item.

### stderr (plain-text mode / supplementary)

- Without `--json`, Codex emits all progress to stderr in ANSI-colored plain text; stdout contains only the final answer.
- With `--json`, stderr is silent. Add `--color never` to suppress ANSI codes.
- Known stderr label prefixes: `codex`, `thinking`, `exec`, `tool`, `collab`, `hook`, `file update`, `mcp`, `mcp startup:`.
- Progress step symbols: `→` (in-progress), `✓` (completed), `•` (pending).
- Other markers: `🌐 Searching the web...`, `tokens used`, `context compacted`, `task interrupted`.

### Streaming limitation

- No real-time streaming of long-running command stdout/stderr within a `command_execution` item (GitHub issue #4751, open). A Unix socket workaround exists but is not standard. Implementer should verify current state.

### Flags confirmed

| Flag | Purpose |
|------|---------|
| `--json` | JSONL event stream on stdout |
| `--color never` | Suppress ANSI in stderr |
| `--output-last-message <path>` | Write final answer to file |
| `--ephemeral` | Skip session file persistence |
