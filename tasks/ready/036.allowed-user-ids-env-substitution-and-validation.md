# 036 — Allowed user IDs: env substitution and numeric validation

## Status

ready

## Summary

Expand `allowedUserIds` config handling to support environment-variable substitution by allowing string input, while enforcing that effective values are valid Telegram numeric user IDs.

## Context

- **Telegram user ID format (official):** [Bot API / core.telegram.org](https://core.telegram.org/api/bots/ids) defines user IDs as MTProto/Bot API integers in range **1 to 0xFFFFFFFFF** (1 to 1,099,511,627,775) inclusive. No leading zeros in standard representation; at most 52 significant bits, so JavaScript `Number` is safe.
- **Leading zeros:** Telegram does not assign leading-zero forms. Validation can treat IDs as numbers; the "exact string match" rule (parsed number’s string representation must equal the resolved string) naturally rejects leading zeros (e.g. `"0123456789"` → parse gives `123456789` → `"123456789"` ≠ original). No need to store or compare IDs as strings.
- **Config load flow:** Invalid `allowedUserIds` must stop the process: at boot, config load already exits on error; on hot reload, validation failure should exit the process (same fail-fast behavior).

## Requirements

- Allow `allowedUserIds` entries to be provided as strings (not only numbers), so env-var substitution can be used.
- After substitution/resolution, parse each value to a number.
- Enforce strict numeric integrity:
  - If parsed numeric content does not exactly match the original resolved string representation, treat it as invalid (rejects spaces, signs, decimals, scientific notation, leading zeros, mixed text).
  - Invalid entries must produce a clear configuration error and **stop the process** (boot or hot reload).
- Optionally validate that the integer lies in Telegram’s user ID range (1 to 0xFFFFFFFFF); if not feasible or ambiguous, requiring a positive integer that passes exact-match is sufficient.
- Preserve compatibility for existing numeric `allowedUserIds` values.
- Keep runtime access checks numeric after parsing/validation.

## Validation Expectations

- Accept values like `"123456789"` and `123456789`.
- Reject values with non-numeric content or formatting drift (e.g. spaces, signs, decimals, scientific notation, mixed text, leading zeros).
- Error message must identify the failing entry (path and value), e.g. `globals.access.allowedUserIds[1]` and the resolved value, so the user can fix config.
- On validation failure: process exits (boot or hot reload).

## Acceptance Criteria

- Config schema accepts string-or-number forms for `allowedUserIds`.
- Env-substituted values are validated and normalized to numeric IDs.
- Invalid values fail fast at config load with actionable errors (path + value); process exits on error.
- Existing setups with numeric IDs continue to work without changes.
