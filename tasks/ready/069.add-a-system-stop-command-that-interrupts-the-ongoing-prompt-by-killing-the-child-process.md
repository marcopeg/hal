# 069 — Add `/btw` system command that interrupts the ongoing prompt by killing the child process

## Status

draft

## Summary

Add a system command named `/btw` that lets users interrupt an active prompt by terminating the underlying child process.

## Problem

There is no reliable way to stop a long-running prompt once it has started. Users need a clear, system-level command that can immediately interrupt the active child process and return control without waiting for completion.

The command name must follow the Claude Code pattern and be exposed as `/btw` (not `/stop`).

## Goals

- Provide a system command `/btw` that is discoverable alongside other system commands.
- Terminate the active child process cleanly when a prompt is running.
- Return a clear confirmation message to the user when `/btw` succeeds.
- Support command enable/disable via built-in command config, enabled by default.

## Scope Clarifications

- Slash command name: only `/btw` should be exposed. `/stop` is not part of this task.
- Config key: `commands.btw.enabled`.
- Message keys:
	- `commands.btw.message.stopped` for the success case (active process was stopped)
	- `commands.btw.message.nothingToDo` for the no-op case (no active process)
- Default: `true`.
- Config inheritance: `globals.commands.btw` is inherited by all projects unless overridden by `projects.<key>.commands.btw`.
- Stop scope: only the caller's currently active LLM child process in the current bot/project context.
- User-facing outcomes:
	- If a process is active and is stopped: `commands.btw.message.stopped` (default: `The process stopped.`)
	- If no process is active: `commands.btw.message.nothingToDo` (default: `No active process to stop, nothing to do.`)

## Constraints

- Avoid interrupting idle sessions or unrelated background work.
- Keep existing prompt execution behavior unchanged unless the stop command is invoked.
- Ensure the stop command is safe to call when no prompt is active.
- Keep behavior per-user/per-project scoped; do not stop another user's run.

## Acceptance Criteria

- `/btw` is available as a built-in command.
- `commands.btw.enabled` is supported in both `globals.commands` and `projects.<key>.commands`, with project override over globals.
- `commands.btw.enabled` defaults to `true` when unset.
- `commands.btw.message.stopped` and `commands.btw.message.nothingToDo` are supported in both `globals.commands` and `projects.<key>.commands`, with project override over globals.
- When a prompt is active for the calling user in the current project, invoking `/btw` kills that child process and replies `commands.btw.message.stopped` (default `The process stopped.`).
- When no prompt is active for the calling user in the current project, invoking `/btw` replies `commands.btw.message.nothingToDo` (default `No active process to stop, nothing to do.`) and does not crash.
- `/stop` is not exposed by this task.
