## Phase 2: Pending confirmation state module

Create `src/bot/handlers/voice-pending.ts`.

This module holds a **module-level `Map<number, PendingTranscript>`** (keyed by `userId`) so that the voice handler (fire-and-forget) and the callback handler share state without coupling.

```ts
interface PendingTranscript {
  transcript: string;
  chatId: number;
  msgId: number;          // id of the permanent confirmation message
  timer: NodeJS.Timeout;  // 60 s expiry timer
  execute: () => Promise<void>; // closure that runs the engine call
}
```

Exports:
- `setPending(userId, data)` — store; cancels previous timer if one existed
- `getPending(userId)` — retrieve
- `clearPending(userId)` — cancel timer and delete from map
- `expirePending(userId, api)` — cancel timer, edit message to strip buttons + append `\n\n_[expired]_`, then clear

## Phase 3: Voice handler fork

In `src/bot/handlers/voice.ts`, after successful transcription, fork on `config.transcript.sticky`:

**Sticky = false (default — existing flow, unchanged):**
- Edit status msg to show transcript + processing text
- Call engine, show progress via edits, delete status msg, send response

**Sticky = true (new flow):**
1. Delete the volatile status message (`deleteMessage`).
2. Build inline keyboard:
   ```ts
   new InlineKeyboard()
     .text("✅ Use it", `tc:use:${userId}`)
     .text("❌ Cancel", `tc:cancel:${userId}`)
   ```
3. Send a **new permanent message** with the transcript text (plain, no parse_mode trickery needed) + inline keyboard. Save `confirmMsgId`.
4. Build an `execute` closure that captures `(gramCtx, transcription, session, ...)` and runs the engine call (same code path as the non-sticky flow, minus the status message lifecycle — a new status message is sent for engine progress).
5. Register a 60 s timer: on expiry, call `expirePending(userId, gramCtx.api)`.
6. Call `setPending(userId, { transcript, chatId, confirmMsgId, timer, execute })`.
7. Return — handler is done. Engine will be invoked by the callback or never.

## Phase 4: Callback handler

Create `src/bot/handlers/voice-callback.ts` — a Grammy `CallbackQueryMiddleware` that handles `tc:` prefixed data.

Data format:
- `tc:use:{userId}` — user confirmed
- `tc:cancel:{userId}` — user cancelled

**Use it flow:**
1. Parse `userId` from data; verify it matches `ctx.from.id` (security: only the owner can confirm).
2. `answerCallbackQuery()` immediately (Telegram requires this).
3. Get pending state; if none → answer with "Confirmation expired" and return.
4. `clearPending(userId)` (cancels timer).
5. Edit the confirmation message to remove the inline keyboard (keep transcript text visible).
6. Call `pending.execute()` — this triggers the engine.

**Cancel flow:**
1. Parse `userId`; verify ownership.
2. `answerCallbackQuery()`.
3. Get pending state; if none → answer with "Already handled" and return.
4. `clearPending(userId)`.
5. Edit confirmation message to remove inline keyboard (keep transcript text).

Both flows wrap `editMessageReplyMarkup` with `try/catch` (silent fail; safe to ignore if message was already edited by the expiry timer racing in).

## Phase 5: Expiry middleware

In `src/bot.ts`, register a **middleware before the message handlers** (after auth/rate-limit):

```ts
bot.on("message", async (ctx, next) => {
  const userId = ctx.from?.id;
  if (userId && getPending(userId)) {
    await expirePending(userId, ctx.api);
  }
  return next();
});
```

This ensures that when a user sends any new message while a confirmation is pending, the old confirmation is expired before the new message is processed.
