# Plan: 046 — Codex log streaming

## Phase 1: Investigate Codex CLI output

1. **List CLI flags**  
   Run `codex --help` and `codex exec --help` (or equivalent) and capture output. Note any flags related to output format, streaming, verbose, or JSON (e.g. `--output-format`, `--stream`, `--json`).

2. **Capture live stdout/stderr**  
   Run a short non-interactive `codex exec` in a test project (e.g. simple prompt that does one tool call or times out quickly). Log or tee stdout and stderr separately (e.g. in adapter temporarily or via a small script) to see:
   - Which stream(s) receive data during execution (not only at the end).
   - Line format: JSON-one-line-per-event vs plain text vs mixed.
   - Whether output is incremental (line-by-line) or only at the end.

3. **Document findings in task**  
   Update the "Findings" section in the task file (e.g. `tasks/ready/046.codex-log-streaming.md`) with:
   - Which stream(s) carry progress (stdout, stderr, or both).
   - Line/event structure (example shapes if JSON).
   - Any streaming/output-format flags to use.
   - Conclusion: can we derive step-wise progress messages, or fallback to buffered-only?

---

## Phase 2: Design progress extraction and throttling

4. **Define progress message rules**  
   From Phase 1 findings, decide:
   - If JSON events: map event types (e.g. tool_use, phase, step) to short human-readable strings (e.g. "Reading: path", "Running: cmd...").
   - If plain text: define how to group or summarize (e.g. last meaningful line, strip prefixes, take first N chars).
   - Set a max length for progress strings (e.g. 200–400 chars) and truncate/summarize when needed.

5. **Define throttling in adapter**  
   - Keep `lastProgressTime` (timestamp of last `onProgress` call).
   - Call `onProgress(msg)` only when: (1) `onProgress` is provided, (2) we have a new progress message, and (3) at least 5 seconds have passed since `lastProgressTime` (then update `lastProgressTime`).
   - Optionally skip duplicate consecutive messages to avoid redundant UI updates.

6. **Handle both streams if needed**  
   If findings show progress on stderr, ensure stderr is parsed/processed the same way (or with stream-specific rules) and merged into one progress stream (throttling applies to combined updates).

---

## Phase 3: Implement in Codex adapter

7. **Wire `onProgress` and throttling**  
   In `src/engine/adapters/codex.ts` `execute()`:
   - Destructure `onProgress` from `options`.
   - Add variables: `lastProgressTime = 0` (or similar), optionally `lastProgressMsg`.
   - Implement a helper e.g. `maybeReportProgress(msg: string)` that checks 5s throttle and optional dedup, then calls `onProgress(msg)`.

8. **Process stdout (and stderr) incrementally**  
   - Keep existing behaviour: accumulate full stdout (and stderr for error path) for final result and error message.
   - Add line-by-line (or chunk) processing: for each line or event that indicates progress, build a short message and call `maybeReportProgress(shortMessage)`.
   - If no streaming format is available, do not call `onProgress` (fallback to current behaviour).

9. **Preserve result contract**  
   - Do not change how `resolve()` is called: same `EngineResult` shape, full buffered stdout as `output` on success, stderr/exit code as `error` on failure. Session ID logic unchanged.
   - On process error or non-zero exit, resolve as today (no extra progress cleanup in adapter; handlers already show final error).

10. **Tests**  
    - Prefer a quick manual test script or existing dev flow: run bot with Codex, send a message that triggers a longer run, confirm status updates (at most every 5s, short messages) and final response/error unchanged.
    - Optionally add a unit test that mocks spawn and feeds sample stdout/stderr lines, asserting `onProgress` is called with expected messages and throttle is respected.
