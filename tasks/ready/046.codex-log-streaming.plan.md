## Phase 2: Implement `--json` mode and JSONL progress parsing

4. **Add `--json` and `--color never` flags to adapter args**
   In `src/engine/adapters/codex.ts`, add `--json` (and `--color never`) to the `args` array before the prompt. These flags apply to all runs (fresh and resume).

5. **Destructure `onProgress` from options**
   Update `const { continueSession, sessionId } = options` to also include `onProgress`.

6. **Add throttle state and helper**
   Add `let lastProgressTime = 0` and a helper `maybeProgress(msg: string)` that calls `onProgress(msg)` only if `onProgress` is defined and at least 3000ms have elapsed since `lastProgressTime`, then updates `lastProgressTime`. Also track `let lastProgressMsg = ""` to skip duplicate consecutive messages.

7. **Parse JSONL stdout line-by-line for progress**
   Replace the current simple `stdout += data.toString()` handler with a line-accumulator that splits on `\n` and calls `JSON.parse` on each line (catching errors). For each parsed event, call `maybeProgress` with a short message based on event type and item type:

   | Event `type` | Item `type` (if `item.*`) | Progress message |
   |---|---|---|
   | `item.started` | `command_execution` | `"Running: <command truncated to 60 chars>"` |
   | `item.started` | `file_change` | `"Updating files..."` |
   | `item.started` | `reasoning` | `"Thinking..."` |
   | `item.started` | `mcp_tool_call` | `"Tool: <tool name>"` |
   | `item.started` | `web_search` | `"Searching: <query truncated to 50 chars>"` |
   | `item.completed` | `agent_message` | `"Responding..."` |
   | `turn.started` | — | `"Codex is reasoning..."` |

   Continue accumulating the raw JSONL into the `stdout` buffer for final result extraction.

8. **Extract final answer from JSONL output**
   On process close (code 0), find the last `item.completed` event with `item.type === "agent_message"` in the accumulated JSONL lines and use its `item.text` as the `output` in `EngineResult`. Fall back to full raw stdout if no such event is found. This replaces the current `stdout.trim()` as output.

## Phase 3: Implement elapsed-time fallback

9. **Fire initial progress message at t=0**
   Immediately after spawning the process (before any data arrives), if `onProgress` is provided, call `onProgress("Codex is working...")`. Record `startTime = Date.now()`.

10. **Start elapsed-time interval**
    Start a `setInterval` (3000ms) that, while the process is running and no JSONL progress events have fired recently (i.e. `lastProgressTime === 0` or no event has updated it in the last tick), calls `onProgress(\`Codex is working on it... ${Math.round((Date.now() - startTime) / 1000)}s\`)`.

    Logic: if `lastProgressTime > 0` (JSONL progress is actively firing), skip the elapsed-time tick — JSONL updates are preferred. If `lastProgressTime === 0` (no events seen yet) or the last event was more than 6s ago, emit the elapsed-time message.

11. **Clear interval on process close or error**
    In both `proc.on("close")` and `proc.on("error")`, call `clearInterval` on the elapsed-time timer before resolving.

## Phase 4: Stderr fallback (supplementary)

12. **Parse stderr for semantic label prefixes**
    In the `proc.stderr.on("data")` handler, after appending to `stderrOutput`, also scan lines for known prefixes: `exec`, `file update`, `thinking`, `tool`, `hook`, `mcp`, and symbols `→`, `✓`, `🌐`. Strip ANSI escape codes (simple regex: `/\x1B\[[0-9;]*m/g`), truncate to 80 chars, and call `maybeProgress(line)`. This fires only if JSONL mode is not yielding progress (since both throttle through `maybeProgress`).
