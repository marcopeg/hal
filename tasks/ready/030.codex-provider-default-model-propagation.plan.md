# Plan: 030 — Provider default model propagation

## Phase 1: Config schema and types

**Goal:** Add optional `default` to provider model entries and expose it through types.

1. **Extend `ProviderModelSchema`** in `src/config.ts`:
   - Add `default: z.boolean().optional()` so each provider model entry may have `default: true`.
2. **Update `ProviderModel` interface** in `src/config.ts`:
   - Add optional `default?: boolean` so resolved config and consumers can rely on it.
3. **Preserve backward compatibility:** Existing configs without `default` on any model remain valid; no model is treated as default unless explicitly marked.

---

## Phase 2: Boot-time validation (at most one default per list)

**Goal:** Fail at boot with a clear error if any provider list has more than one model with `default: true`.

4. **Add `validateProviderDefaultUniqueness`** in `src/config.ts`:
   - Accept the merged multi-project config (e.g. `MultiConfigFile` or the parsed structure with `globals` and `projects`).
   - For `globals.providers`: for each engine key (codex, claude, copilot, opencode, cursor, antigravity), if the value is an array, count entries with `default === true`; if count > 1, throw `ConfigLoadError` with a message that includes the engine name and that it's the globals provider list (e.g. `globals.providers.<engine>`).
   - For each `project` in `config.projects`: same check on `project.providers`; on error include project identifier (name or cwd) and engine (e.g. `projects[<name>].providers.<engine>`).
   - If an array has 0 or 1 default, do nothing.
5. **Call the new validation from startup** in `src/cli.ts`:
   - After loading config (`loaded.config`) and before building `resolvedProjects`, call `validateProviderDefaultUniqueness(loaded.config)` (or the appropriate export from config). Ensure it runs after merge/substitution so it sees the same structure that resolution uses.
   - Export `validateProviderDefaultUniqueness` from `src/config.ts` if it is not already part of the public API used by `cli.ts`.

---

## Phase 3: Resolve and expose provider default model

**Goal:** Resolved project config exposes the provider default model for the active engine (the `name` of the model with `default: true` in the resolved list, if exactly one).

6. **In `resolveProjectConfig`** in `src/config.ts`:
   - When building `providerModels` from `rawProviderModels`, map each entry to include `default: m.default` so the resolved list retains the flag.
   - After resolving `providerModels`, compute `providerDefaultModel`: from the resolved list for the active engine, find the single entry with `default === true`; if there is exactly one, set `providerDefaultModel` to that entry's `name`, otherwise `undefined`.
   - Add `providerDefaultModel: string | undefined` to the returned `ResolvedProjectConfig` object.
7. **Update `ResolvedProjectConfig` interface** in `src/config.ts`:
   - Add `providerDefaultModel?: string` (or `string | undefined`) to the type definition.

---

## Phase 4: Use provider default for the active engine at runtime

**Goal:** When `engine.model` is not set, pass the provider default model (if any) to the active engine adapter; otherwise use HAL default or omit model. Same logic for every engine.

8. **Compute `effectiveModel` for the active engine** in `src/cli.ts`:
   - Where `effectiveModel` is currently `config.engineModel ?? getDefaultEngineModel(config.engine)`, change to:
     - `config.engineModel ?? config.providerDefaultModel ?? getDefaultEngineModel(config.engine)`.
   - So: explicit `engine.model` wins; then provider default (for the active engine’s resolved list); then HAL default from `getDefaultEngineModel`. No engine-specific branching.
9. **Verify no other call sites need changes:** Ensure every engine adapter is only ever invoked with the model from this `effectiveModel` (already done via `getEngine(..., effectiveModel)`). All engines that accept a model benefit from provider default when applicable.

---

## Phase 5: Tests and examples (optional but recommended)

10. **Config examples:** Add or update an example config (e.g. `examples/hal.config.jsonc` or `examples/_hal.config.yaml`) to show one provider with a model marked `default: true` (e.g. `providers.codex` or `providers.claude`).
11. **Manual or automated checks:** Optionally add a test or script that (a) loads a config with two models marked `default: true` in the same provider list and asserts a `ConfigLoadError`; (b) loads a config with one default for a given engine and asserts resolved `providerDefaultModel` and that the adapter receives the model when `engine.model` is unset.

## Phase 6: Documentation

**Goal:** Ensure user-facing docs describe the new `default` option, validation, and precedence so users can configure provider defaults correctly.

12. **Analyze and update documentation:**
   - In `docs/engines/README.md`: In the **Model list (`providers` key)** section, add the optional `default` field to the provider entry table and the example JSON (one entry with `default: true`). Explain that at most one model per `providers.<engine>` list may have `default: true`; otherwise HAL fails at boot with a clear `ConfigLoadError`. In **Model defaults**, add provider default to the precedence: when `engine.model` is omitted, the model used is the provider default (if exactly one entry in the resolved list has `default: true`), then HAL default or engine default as today.
   - In `docs/config/README.md`: If the globals or project tables mention `providers`, add a brief note that provider entries may include `default: true` (at most one per list) and that `engine.model` overrides it. If there is no providers detail there, a short pointer to the engines doc is enough.
   - Use the [docs skill](.agents/skills/docs/SKILL.md) for structure and cross-linking (relative links, one topic per folder, etc.).
