# Plan: 030 — Codex provider default model propagation

## Phase 1: Config schema and types

**Goal:** Add optional `default` to provider model entries and expose it through types.

1. **Extend `ProviderModelSchema`** in `src/config.ts`:
   - Add `default: z.boolean().optional()` so each provider model entry may have `default: true`.
2. **Update `ProviderModel` interface** in `src/config.ts`:
   - Add optional `default?: boolean` so resolved config and consumers can rely on it.
3. **Preserve backward compatibility:** Existing configs without `default` on any model remain valid; no model is treated as default unless explicitly marked.

---

## Phase 2: Boot-time validation (at most one default per list)

**Goal:** Fail at boot with a clear error if any provider list has more than one model with `default: true`.

4. **Add `validateProviderDefaultUniqueness`** in `src/config.ts`:
   - Accept the merged multi-project config (e.g. `MultiConfigFile` or the parsed structure with `globals` and `projects`).
   - For `globals.providers`: for each engine key (codex, claude, etc.), if the value is an array, count entries with `default === true`; if count > 1, throw `ConfigLoadError` with a message that includes the engine name and that it's the globals provider list (e.g. `globals.providers.<engine>`).
   - For each `project` in `config.projects`: same check on `project.providers`; on error include project identifier (name or cwd) and engine (e.g. `projects[<name>].providers.<engine>`).
   - If an array has 0 or 1 default, do nothing.
5. **Call the new validation from startup** in `src/cli.ts`:
   - After loading config (`loaded.config`) and before building `resolvedProjects`, call `validateProviderDefaultUniqueness(loaded.config)` (or the appropriate export from config). Ensure it runs after merge/substitution so it sees the same structure that resolution uses.
   - Export `validateProviderDefaultUniqueness` from `src/config.ts` if it is not already part of the public API used by `cli.ts`.

---

## Phase 3: Resolve and expose provider default model

**Goal:** Resolved project config exposes the provider default model for the active engine (the `name` of the model with `default: true` in the resolved list, if exactly one).

6. **In `resolveProjectConfig`** in `src/config.ts`:
   - When building `providerModels` from `rawProviderModels`, map each entry to include `default: m.default` so the resolved list retains the flag.
   - After resolving `providerModels`, compute `providerDefaultModel`: from the resolved list for the active engine, find the single entry with `default === true`; if there is exactly one, set `providerDefaultModel` to that entry's `name`, otherwise `undefined`.
   - Add `providerDefaultModel: string | undefined` to the returned `ResolvedProjectConfig` object.
7. **Update `ResolvedProjectConfig` interface** in `src/config.ts`:
   - Add `providerDefaultModel?: string` (or `string | undefined`) to the type definition.

---

## Phase 4: Use provider default for Codex at runtime

**Goal:** When the engine is Codex and `engine.model` is not set, pass the provider default model (if any) to the Codex CLI via `--model`; otherwise omit `--model`.

8. **Compute `effectiveModel` for Codex** in `src/cli.ts`:
   - Where `effectiveModel` is currently `config.engineModel ?? getDefaultEngineModel(config.engine)`, change to:
     - `config.engineModel ?? (config.engine === 'codex' ? config.providerDefaultModel : undefined) ?? getDefaultEngineModel(config.engine)`.
   - So: explicit `engine.model` wins; for Codex only, provider default is used next; then engine default from `getDefaultEngineModel` (Codex has none, so no `--model` when no default).
9. **Verify no other call sites need changes:** Ensure the Codex adapter is only ever invoked with the model from this `effectiveModel` (already done via `getEngine(..., effectiveModel)`). No behavior change for non-Codex engines.

---

## Phase 5: Tests and examples (optional but recommended)

10. **Config examples:** Add or update an example config (e.g. `examples/hal.config.jsonc` or `examples/_hal.config.yaml`) to show one Codex model with `default: true` in `providers.codex`.
11. **Manual or automated checks:** Optionally add a test or script that (a) loads a config with two models marked `default: true` in the same provider list and asserts a `ConfigLoadError`; (b) loads a config with one default for Codex and asserts resolved `providerDefaultModel` and that the adapter receives `--model` when `engine.model` is unset.
