# 084 — Rename /clean command to /clear — Plan

## Phase 1: Rename command surface in runtime code

### 1.1 — Rename command handler API to `clear`
- In `src/bot/commands/session.ts`, rename `DEFAULT_CLEAN_TEMPLATE` and `createCleanHandler(...)` to `DEFAULT_CLEAR_TEMPLATE` and `createClearHandler(...)`.
- Update comments/docstrings that still describe the command as `/clean`.
- Keep session reset behavior identical (no logic changes beyond naming).

### 1.2 — Register `/clear` and remove `/clean` routing
- In `src/bot.ts`, switch command registration from `bot.command("clean", ...)` to `bot.command("clear", ...)`.
- Update imports/usages to the renamed clear handler.
- Ensure no fallback route keeps `/clean` active.

### 1.3 — Keep command-list internals in sync
- In `src/bot/commands/loader.ts`, rename built-in command metadata from `clean` to `clear`.
- Rename `CommandEnabledFlags.clean` to `CommandEnabledFlags.clear` and update all usages.
- Verify Telegram menu generation only includes `/clear` when enabled.

---

## Phase 2: Migrate config key from `commands.clean` to `commands.clear`

### 2.1 — Add canonical `clear` key in config schema/types
- In `src/config.ts`, rename command schema/type fields from `clean` to `clear` in:
  - `CommandsConfigSchema`
  - `ResolvedProjectConfig["commands"]`
  - resolved command object construction.
- Update message-template labels from `commands.clean` to `commands.clear`.

### 2.2 — Support deprecated legacy key for one release cycle
- During config resolution, accept both `commands.clear` (canonical) and legacy `commands.clean`.
- Apply deterministic precedence:
  - project `commands.clear`
  - project `commands.clean` (legacy)
  - globals `commands.clear`
  - globals `commands.clean` (legacy)
- Ensure resolved runtime config only exposes `commands.clear`.

### 2.3 — Emit deprecation warning when legacy key is used
- Add logger warning(s) when `commands.clean` is detected in globals or project config.
- Warning should explicitly instruct migration to `commands.clear` and mention one-release compatibility window.
