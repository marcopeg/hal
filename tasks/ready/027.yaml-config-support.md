# 027 — YAML config support

## Status

draft

## Summary

Add support for YAML configuration files (`hal.config.yaml` / `hal.config.local.yaml`) alongside existing JSON, and introduce multi-format detection infrastructure in the config loader.

## Context

- Config is currently loaded via `JSON.parse()` in `src/config.ts`: `loadMultiConfigInternal` (base config) and `loadLocalConfig` (local overrides).
- The loader hardcodes filenames `hal.config.json` and `hal.config.local.json`.
- Task 028 adds JSONC as another format. This task should introduce the format-detection pattern so both 027 and 028 plug in cleanly.

## Decisions

- **Per-slot exclusivity**: each config slot (`hal.config.*` and `hal.config.local.*`) may exist in exactly one format. If multiple formats are found for the same slot (e.g., both `hal.config.json` and `hal.config.yaml`), the loader throws a boot error.
- **Independent slots**: base and local configs can use different formats (e.g., `hal.config.yaml` base + `hal.config.local.json` local overrides).
- **Full feature parity**: YAML configs go through the same pipeline as JSON — env var substitution (`${VAR}`), local override deep merging, and Zod schema validation all work identically.
- **YAML parser**: a `yaml` npm package (e.g., `yaml` by eemeli — standard, well-maintained).

## Scope

- Introduce a format-detection layer in `src/config.ts` that scans for supported extensions (`.json`, `.yaml`, `.yml`) per config slot and errors if more than one exists.
- Add a YAML parser path: when the detected format is `.yaml` or `.yml`, parse with the YAML library instead of `JSON.parse()`.
- The detection layer should be extensible so task 028 can add `.jsonc` to the supported extensions without restructuring.
- Update error messages to reference the actual filename/format detected (not hardcoded `hal.config.json`).
- Add `examples/hal.config.yaml` as a YAML equivalent of the existing JSON example.

## Execution Order

This task should be implemented **before task 028** (JSONC config support). It introduces the multi-format detection layer that 028 plugs into. Parallel implementation on separate branches is discouraged — both tasks modify the same loader functions in `src/config.ts`.

## Acceptance Criteria

- `hal.config.yaml` loads and produces the same resolved config as the equivalent `hal.config.json`.
- `hal.config.local.yaml` works for local overrides, including deep merge behavior.
- Env var substitution (`${VAR}`) works in YAML configs.
- Plain `hal.config.json` continues to work unchanged (no regressions).
- Per-slot format exclusivity covers all supported extensions (`.json`, `.jsonc`, `.yaml`, `.yml`). If more than one format exists for the same slot, the loader throws a clear boot error naming the conflicting files.
- Base and local slots can independently use different formats without error.
- `examples/hal.config.yaml` exists and parses without errors.
- No new linter or type errors introduced.
