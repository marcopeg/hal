## Summary

Add JSONC support to config loading so `hal.config.jsonc` and `hal.config.local.jsonc` can contain inline comments (`//`, `/* */`) and trailing commas. The primary motivation is documentation â€” enabling a richly commented example config that serves as inline reference.

## Context

- Config is currently loaded via `JSON.parse()` in two places in `src/config.ts`: `loadMultiConfigInternal` (base config) and `loadLocalConfig` (local overrides).
- Task 027 introduces multi-format config support (`json`, `jsonc`, `yaml`). Only one format exists at a time per config file â€” the loader detects which one is present.
- This task adds `.jsonc` as a supported format in that detection chain.

## Decisions

- **Parser**: `strip-json-comments` (sindresorhus, 69M+ weekly downloads). Preprocessor approach: strips comments and trailing commas, then feeds clean JSON to `JSON.parse()`. No AST, no extra features â€” just what we need.
- **Options**: `{ trailingCommas: true }` â€” accept both `//`/`/* */` comments and trailing commas. Nothing else (no unquoted keys, no single quotes, etc.).
- **File extension**: `.jsonc` â€” standard extension recognized by VS Code and most editors for JSON-with-comments. One format per config file; the loader picks whichever exists.
- **Example config**: Convert `examples/hal.config.json` to `examples/hal.config.jsonc`. A fully-documented example config is out of scope (separate task).

## Scope

- Add `strip-json-comments` as a dependency.
- Create a shared parse helper (e.g. `parseJsonc(content: string): unknown`) that uses `strip-json-comments` with `{ trailingCommas: true }` followed by `JSON.parse()`.
- Integrate with the config format detection from task 027: when the detected format is `jsonc`, use the JSONC parser instead of plain `JSON.parse()`.
- If task 027 is not yet implemented, add `.jsonc` detection directly in `loadMultiConfigInternal` and `loadLocalConfig` alongside the existing `.json` lookup.
- Convert `examples/hal.config.json` to `examples/hal.config.jsonc` (rename + add a few representative comments, no content changes).
- Preserve existing error reporting: `strip-json-comments` replaces comments with whitespace by default, so `JSON.parse` error positions remain accurate.

## Execution Order

This task should be implemented **after task 027** (YAML config support), which introduces the multi-format detection layer. If implemented before 027, the fallback approach in scope applies, but rework will be needed when 027 lands. Parallel implementation on separate branches is discouraged â€” both tasks modify the same loader functions in `src/config.ts`.

## Acceptance Criteria

- `hal.config.jsonc` with `//` line comments, `/* */` block comments, and trailing commas loads correctly.
- `hal.config.local.jsonc` works the same way.
- Plain `hal.config.json` (no comments) continues to work unchanged.
- Per-slot format exclusivity covers all supported extensions (`.json`, `.jsonc`, `.yaml`, `.yml`). If more than one format exists for the same slot, the loader throws a clear boot error naming the conflicting files.
- `examples/hal.config.jsonc` exists and parses without errors.
- No new linter or type errors introduced.
