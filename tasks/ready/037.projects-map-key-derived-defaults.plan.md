# Plan — 037 Projects map with key-derived defaults

## Overview

Refactor `projects` from array to map keyed by project key; validate keys as slug-like; derive default `name` and `cwd` from key; use key as `slug` in resolved config; apply same map shape and key-based merge for local config.

## Phase 1: Schema and key validation

1. **Define slug-key validation**
   - Add a Zod schema or refine for project map keys: only `[a-zA-Z0-9_-]+` (e.g. regex `^[a-zA-Z0-9_-]+$`). Reuse for both base and local `projects` keys.
   - Document in code that this keeps default `cwd` safe as a path segment.

2. **Change base config schema**
   - Replace `projects: z.array(ProjectFileSchema).min(1)` with `projects: z.record(ProjectKeySchema, ProjectFileSchema)` and ensure at least one key (refine or min on keys length).
   - In `ProjectFileSchema`: make `cwd` optional (when omitted, default from key); keep `name` optional (default from key). Ensure `cwd` still has `.min(1)` when present.

3. **Change local config schema**
   - Replace `projects: z.array(LocalProjectSchema).optional()` with `projects: z.record(ProjectKeySchema, LocalProjectSchema).optional()`.

4. **Update types**
   - `MultiConfigFile.projects` becomes `Record<string, ProjectFileEntry>` (or equivalent inferred from Zod). Ensure `ProjectFileEntry` and `LocalProjectSchema` types reflect optional `name`/`cwd` where appropriate.

5. **Error messages**
   - Ensure schema/refine errors for invalid keys mention slug-like requirement and point to the key (e.g. `projects.<key>`).

---

## Phase 2: Normalization and merge

1. **Normalize base config after parse**
   - After successfully parsing base config, normalize each project entry: for each `[key, entry]`, set `entry.name = entry.name ?? key`, `entry.cwd = entry.cwd ?? key`. So internally every entry has `name` and `cwd` set (for simpler downstream code). Decide whether to mutate in place or build a new object; keep a single place where this normalization runs (e.g. right after base parse, before merge).

2. **Update `normalizeAllowedUserIdsInConfig`**
   - Iterate over `Object.entries(config.projects)` instead of array indices. Use paths like `projects.<key>.access.allowedUserIds[<i>]` in error messages.

3. **Update `mergeLocalIntoBase`**
   - Base and local both have `projects` as records. Build merged map: start from `{ ...base.projects }`. For each key in `local.projects`, if key is not in base, throw `ConfigLoadError` (local key must exist in base). Otherwise set `merged[key] = deepMerge(base.projects[key], local.projects[key])`. After merge, normalize again so each merged entry has `name = entry.name ?? key`, `cwd = entry.cwd ?? key`.

4. **Update `loadMultiConfigInternal`**
   - After merge, `merged.projects` is a record. When building `rawCwds` for env loading, use `Object.values(merged.projects)` (or equivalent) to get project list; ensure order is stable (e.g. use same key order when later resolving projects).
   - Ensure the substituted config is still validated with the same record schema (projects as record). No second parse that expects an array.

5. **Ordering**
   - Preserve deterministic order when iterating projects (e.g. `Object.keys(merged.projects).sort()` or rely on insertion order) so startup and logs are consistent. Document choice.

## Phase 3: Resolution, iteration, and validation

1. **Change `resolveProjectConfig` signature and body**
   - New signature: accept project key as first argument, e.g. `resolveProjectConfig(key: string, project: ProjectFileEntry, globals, configDir, rootContext)`.
   - Inside: set `slug = key`. Use `project.name ?? key` and `project.cwd ?? key` for name/cwd (if not already normalized before this call, do it here). Compute `resolvedCwd`, `logDir`, `dataDir` using `slug` and `cwd` as today. Return `ResolvedProjectConfig` with `slug` always equal to `key`.

2. **Remove or restrict `deriveSlug`**
   - No longer use `deriveSlug(project.name, project.cwd)` for project identity. Either remove the function if unused elsewhere, or keep it for backward compatibility but do not use it when resolving projects from the map. Check for other call sites (e.g. context resolver) and update or leave as-is.

3. **Update CLI / startup**
   - Replace `multiConfig.projects.map((project) => resolveProjectConfig(project, ...))` with iteration over the map: e.g. `Object.entries(multiConfig.projects).map(([key, project]) => resolveProjectConfig(key, project, ...))`.
   - When filtering by `active`, use the same iteration order and check `project.active === false` to skip inactive projects. Adjust log message to use `project.name ?? key` or `key` as needed.

4. **Validation**
   - `validateProjects(resolvedProjects)` and `validateAccessPolicies(resolvedProjects)` already work on `ResolvedProjectConfig[]`; no change to their logic. Optionally relax or update duplicate-name check now that `slug` is always unique (duplicate name might still be worth flagging if we allow explicit names that differ from key).

## Phase 4: Config writer

1. **Update `updateProjectModel`**
   - Prefer identifying project by slug (key): e.g. `updateProjectModel(configDir: string, projectSlug: string, engine: EngineName, model: string)`. Callers that have `ResolvedProjectConfig` can pass `project.slug`.

2. **Handle map shape in config-writer**
   - After reading and parsing the target config, if `data.projects` is an object and not an array, treat it as a map: get `entry = data.projects[projectSlug]`. If entry exists, update `entry.engine` (name/model). If not, either throw (e.g. "project not found") or do not add new keys (per task: local keys must exist in base). Do not push to an array.

3. **Call sites of `updateProjectModel`**
   - Find all callers and change them to pass `projectSlug` (e.g. `project.slug`) instead of `{ name, cwd }`. Update `ProjectIdentifier` usage if it is no longer needed for this function.

## Phase 5: Examples, init template, and help text

1. **Example config**
   - Update `examples/_hal.config.yaml`: convert `projects` from array to map (e.g. `obsidian:`, `timetracker:` with same content as current array items, minus redundant `name`/`cwd` if they equal the key).

2. **Init template**
   - In `src/cli.ts`, update the generated `hal.config.json` template so `projects` is a map, e.g. `"projects": { "my-project": { "telegram": { ... }, "access": { ... } } }`. Omit `name`/`cwd` so they default to the key.

3. **Help and error messages**
   - Update any CLI help or init instructions that show `projects: [ ... ]` to show the map format. Update the init success message if it references `projects[0]` to reference the project by key (e.g. first key or "my-project").

## Phase 7: Testing and edge cases

1. **Config tests**
   - Update existing config tests to use map format. Add or extend tests for: invalid key (non–slug-like) rejected with clear error; key-derived defaults (omit `name`/`cwd`, assert resolved config has `slug`/name/cwd equal to key); explicit `name`/`cwd` overrides preserved; local config merge by key (local key must exist in base, else error).

2. **Edge cases**
   - Empty map: schema or refine should require at least one project. Local key not in base: clear `ConfigLoadError`. Duplicate bot token / duplicate cwd: existing `validateProjects` still applies.

3. **Regression**
   - Run full test suite and manual smoke test: start with example config (map format), ensure all projects resolve and bots start; run config writer (e.g. model switch) and verify file stays map-shaped.
