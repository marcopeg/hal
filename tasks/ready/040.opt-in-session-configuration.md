# 040 — Opt-in session configuration

## Status

ready

## Summary

Introduce explicit session configuration object for engines, replacing the
current boolean `engine.session` field. Sessions are enabled by default;
engine-specific sub-options control shared vs. per-user isolation and
experimental per-user session discovery.

## Engine session capabilities (research summary)

| Engine | Session ID in output | Resume by ID flag | Per-user isolation | Notes |
|---|---|---|---|---|
| **Claude** | Yes — `session_id` on `result` event | `--resume <id>` or `--continue` (most recent in cwd) | Via `--resume <id>` | `--continue` gives shared-for-all-users behavior |
| **Codex** | Indirect — UUID in `~/.codex/sessions/YYYY/MM/DD/rollout-<ts>-<UUID>.jsonl` filename, JSONL body contains `cwd` | `codex exec resume <UUID>` | Scan file system post-run to extract UUID | Default `--last` is already cwd-scoped (Jan 2026+) |
| **Cursor** | Yes — `session_id` UUID on every `stream-json` event incl. terminal `result` | `--resume <session_id>` | Parse from output | Currently ignored; `"active"` placeholder used |
| **OpenCode** | No clean CLI resume-by-ID for `opencode run` | Only `-c` (continue last, cwd-scoped) | Not feasible via CLI | Sessions in SQLite + `~/.local/share/opencode/storage/session/<project-id>/`; no public `run --session <id>` flag |
| **Antigravity (Gemini)** | Yes — `sessionId`/`session_id`/`id` on `init` event | `--resume <UUID>` (or `gemini --resume` for latest) | Already implemented in adapter | Adapter already captures and returns real session IDs; sessions in `~/.gemini/tmp/<project_id>/chats/` |
| **Copilot** | No real session ID; uses `--continue` flag | `--continue` for last session only | Not feasible | Adapter uses `"active"` placeholder |

## Context & Constraints

- This is a **breaking change**: no backward compatibility for the old
  `engine.session: boolean` form. All existing configs, examples, and docs
  must be updated to use the new nested shape.
- The `engineSession: boolean` field on `ResolvedProjectConfig` must be
  replaced by a richer resolved object covering all engine-specific options.
- Engine-specific session sub-options live inside `engine.session.*`.
- The `shared` flag is only meaningful for Claude; `experimentallyEnableUserSession`
  is only meaningful for Codex and Cursor. Other engines either already handle
  sessions correctly (Antigravity) or lack CLI support for per-user resume
  (OpenCode, Copilot). The schema accepts all flags on any engine config but
  resolvers apply them only to the relevant engines.

## Requirements

### Core schema change

- Replace `engine.session: z.boolean()` in `EngineConfigSchema` with a session
  object:
  ```
  engine.session:
    enabled: boolean                         # default true when omitted
    shared: boolean                          # Claude only; default false (per-user)
    experimentallyEnableUserSession: boolean # Codex & Cursor only; default false
  ```
- `engine.session` itself remains optional; omitting it entirely means
  `enabled: true` with all sub-flags at their defaults.
- `enabled: false` → stateless mode: no session IDs or continue flags are
  passed to engines, no `session.json` is written to user directories.
- `enabled: true` → session-aware mode (current default); engine-specific
  flags further configure behavior.

### Claude — `session.shared`

**`shared: true` (default)**

- Use `--continue` (`-c`) instead of `--resume <id>` when invoking Claude in
  print mode: `claude -c -p "query"`.
- Claude's `--continue` flag automatically resumes the most recent session in
  the project's `cwd`, making all users in that project share the same
  conversation history. No per-user `session.json` is read or written.
- `engineSessionMsg` (the "hi!" warm-up ping on `/clean`) is still sent
  without `--continue` to start a fresh shared session.

**`shared: false`**

- Per-user session isolation: read/write `session.json` in each user's
  `userDir`, pass `--resume <id>` on subsequent calls.
- Session IDs are sourced from Claude's `stream-json` output (`session_id`
  field on the `result` event).

### Codex — `session.experimentallyEnableUserSession`

**`false` (default)**

- Existing behavior: use `codex exec resume --last` for continued sessions.
  As of Jan 2026, `--last` is scoped to the current `cwd` automatically.

**`true`**

- After each successful `codex exec` run, scan `~/.codex/sessions/` recursively
  for JSONL files whose filename matches `rollout-<timestamp>-<UUID>.jsonl`, read
  each file to find the one whose embedded `cwd` field matches `config.cwd`,
  extract the UUID from the filename, persist it in the user's `session.json`,
  and use `codex exec resume <UUID>` on the next call.
- Scan is sorted newest-first; stop at first match to avoid reading many files.
- This is "experimental" because it relies on Codex's internal file-system
  session storage layout (`~/.codex/sessions/YYYY/MM/DD/*.jsonl`), which is
  undocumented and may change.
- If no matching session file is found after a run, fall back to `--last`.

### Cursor — `session.experimentallyEnableUserSession`

**`false` (default)**

- Existing behavior: no real session ID stored; `"active"` placeholder
  effectively disables resume (current adapter behaviour).

**`true`**

- The Cursor Agent CLI already emits `session_id` (UUID) on every `stream-json`
  event, including the terminal `result` event. Parse the `session_id` from the
  `system/init` or `result` event, persist it in the user's `session.json`, and
  pass `--resume <session_id>` on subsequent calls.
- This is "experimental" because the `--resume` flag's stability is not
  guaranteed by Cursor docs.

### OpenCode — no `experimentallyEnableUserSession`

- OpenCode's `run` command exposes only `-c` (continue last session), with no
  `--session <id>` equivalent for non-interactive execution.
- Session data lives in a SQLite database + JSON files under
  `~/.local/share/opencode/storage/session/<project-id>/`, but there is no
  stable public API to resume by ID via CLI.
- The existing `-c` flag (continue last in cwd) remains the only continuation
  mechanism. `experimentallyEnableUserSession` is **not** applicable to OpenCode.

### Antigravity (Gemini CLI) — already per-user, no new flag needed

- The Gemini CLI adapter already captures the real session UUID from the `init`
  event (`event.sessionId ?? event.session_id ?? event.id`) and returns it in
  `EngineResult.sessionId`.
- The existing `--resume <sessionId>` path in the adapter already provides
  full per-user session isolation — no new flag is required.
- The only change needed is replacing the `config.engineSession` boolean guard
  with the new `config.engineSession.enabled` flag.
- Sessions are stored by Gemini CLI at `~/.gemini/tmp/<project_id>/chats/`.

### Copilot — no `experimentallyEnableUserSession`

- Copilot uses `--continue` for session continuation with an `"active"`
  placeholder. There is no per-session-ID resume mechanism available.
- `experimentallyEnableUserSession` is **not** applicable to Copilot.

### Stateless mode (`enabled: false`) — all engines

- Pass no session-related flags to any engine.
- Do not call `getSessionId` or `saveSessionId`.
- Do not write `session.json` to the user's directory.
- All handlers (`text.ts`, `photo.ts`, `document.ts`, `voice.ts`) guard on
  `config.engineSession.enabled`.

## Resolved config shape

`ResolvedProjectConfig` must expose the new resolved flags. Replace the flat
`engineSession: boolean` with:

```ts
engineSession: {
  enabled: boolean;
  shared: boolean;                          // resolved default: false (Claude — preserves per-user behaviour)
  experimentallyEnableUserSession: boolean; // resolved default: false (Codex, Cursor)
};
```

## Acceptance Criteria

- `engine.session: boolean` (old form) is **rejected** by Zod schema validation
  with a clear error message.
- Omitting `engine.session` entirely resolves to
  `{ enabled: true, shared: false, experimentallyEnableUserSession: false }`.
- `enabled: false`: no session flags passed to any engine; no `session.json`
  written; all handlers skip session read/write paths.
- **Claude, `shared: true`**: `--continue` is passed; no per-user session file
  is used.
- **Claude, `shared: false`**: `--resume <id>` used with per-user `session.json`;
  session ID sourced from `stream-json` `result` event.
- **Codex, `experimentallyEnableUserSession: false`**: `codex exec resume --last`
  behaviour unchanged.
- **Codex, `experimentallyEnableUserSession: true`**: scans
  `~/.codex/sessions/` JSONL files, matches on `cwd`, extracts and persists
  UUID per user, uses `codex exec resume <UUID>`.
- **Cursor, `experimentallyEnableUserSession: false`**: `"active"` placeholder
  behaviour unchanged.
- **Cursor, `experimentallyEnableUserSession: true`**: parses `session_id` from
  `stream-json` output and persists per user; uses `--resume <id>` next call.
- **Antigravity**: existing per-user session ID capture/resume unaffected;
  only `config.engineSession` → `config.engineSession.enabled` access updated.
- **OpenCode / Copilot**: continue-flag behaviour unchanged; no per-user
  session ID mechanism added.
- All example configs (`examples/`) updated to new session object shape.
- Documentation updated to describe new session configuration options per engine.
- TypeScript compilation succeeds; no linter errors.
