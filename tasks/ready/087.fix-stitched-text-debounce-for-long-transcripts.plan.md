# Plan - 087 Fix stitched text debounce for long transcripts

## Objective

Fix the text debounce flow so rapid transcript-like multi-message input is stitched into one logical prompt and executed exactly once downstream, using a `600ms` baseline quiet window and a documented distinction between fragment-style concatenation and burst-style joining.

## Phase 1 - Confirm implementation constraints

1. Re-read `src/bot/handlers/text.ts`, `src/bot/middleware/rateLimit.ts`, and `src/bot.ts` together so the handler, debounce state, and middleware counting rules are treated as one system.
2. Preserve the already-refined task decisions as hard constraints:
	- incoming text may be buffered behind a fixed quiet window,
	- the baseline window to validate first is `600ms`,
	- uncertain classification falls back to `join("")`,
	- the preferred config surface is `telegram.message.debounceMs`,
	- slash commands must still bypass debounce completely.
3. Record the implementation-level rule that will distinguish:
	- true fragments of one logical message, and
	- burst-style separate text messages that should still be collapsed into one prompt.
4. Keep the Telegram research requirement scoped to evidence that affects the implementation, not generic long-message background.

## Phase 2 - Reproduce and pin down the bug

1. Add targeted automated coverage around `createTextHandler()` using fake timers and mocked engine execution.
2. Reproduce the current failing scenario where the first chunk is below `4096` and rapid follow-up messages still cause multiple downstream executions.
3. Assert at the engine boundary how many times execution occurs and exactly what prompt text is forwarded.
4. Add one control scenario proving that spaced-apart messages outside the quiet window still execute independently.

## Phase 3 - Redesign the trigger and stitch model

1. Replace the current exact-threshold activation rule with an arrival-pattern model that can capture both:
	- fragmented long-message inputs, and
	- rapid burst-style user text sequences.
2. Implement the new debounce flow around a fixed quiet window with `600ms` as the initial default under `config.telegram.message.debounceMs`.
3. Keep buffering scoped per user and preserve ordering by `message_id` before reconstruction.
4. Implement and document the stitch classifier that decides between:
	- fragment concatenation via `join("")`, and
	- newline joining for burst-style separate messages.
5. If the classifier is uncertain, explicitly fall back to `join("")`.
6. Make the engine boundary explicit: only the final reconstructed prompt may cross into `ctx.engine.execute(...)`.

## Phase 4 - Protect adjacent behavior

1. Re-check the interaction with `debounceActiveUsers` and `createRateLimitMiddleware()`.
2. Decide whether the middleware contract still counts exactly one logical message when every candidate text can enter the debounce window.
3. Confirm that the fixed-delay UX is intentional and documented, rather than an accidental regression.
4. Keep the existing processing-status UX coherent (`Buffering...` / `Processing...`) or intentionally revise it with clear rationale.
5. Re-check slash-command bypass and mixed sequences so commands are never pulled into the debounce path.

## Phase 5 - Validate and document

1. Cover at least these scenarios in tests:
	- failing-before-fix rapid multi-message input with first chunk below `4096`,
	- fragment-style stitching that should use `join("")`,
	- burst-style stitching that should use newlines,
	- uncertain classification falling back to `join("")`,
	- separate messages outside the quiet window,
	- slash-command bypass,
	- rate-limit interaction for buffered inputs.
2. Run the relevant validation commands for touched code.
4. Document the final trigger model, the classifier rule, the chosen debounce window, and the config-key migration path so the task, code, and tests all describe the same behavior.
4. Include a concise Telegram research note in the implementation handoff covering what is known, what remains ambiguous, and why HAL handles reconstruction at the application layer.
