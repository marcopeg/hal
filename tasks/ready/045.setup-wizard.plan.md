# Plan: 045 — Interactive Setup Wizard

## Execution Order

> **Requires task 040 (Opt-in Session Configuration) to be completed first.**
> Phase 5 (session step) cannot be implemented until the structured `engine.session` object exists.
> Phases 1–4 and 6 can be built independently of 040; wire in the session step once 040 lands.

## Overview

Add a `wiz` CLI command that interactively guides users through HAL configuration. The wizard auto-triggers when `start` is invoked with no config or an incomplete config. Built with `@clack/prompts` for a polished CLI UX, using a modular step-pipeline architecture.

---

## Phase 1: Dependency + module skeleton

**Goal:** Install `@clack/prompts`, create the `src/wizard/` directory structure with the step interface and type definitions.

### Steps

1. **Install `@clack/prompts`**
   - `pnpm add @clack/prompts`
   - Verify it's added to `dependencies` in `package.json` (it's a runtime dependency, not dev-only — the wizard runs from the published CLI).

2. **Create wizard types and step interface**
   - Create `src/wizard/types.ts` with:
     ```ts
     export interface WizardContext {
       cwd: string;                    // working directory
       existingConfig: PartialConfig | null; // parsed existing config (null = fresh)
       prefill: PrefillFlags;          // CLI pre-fill flags
       reset: boolean;                 // --reset flag
       results: Record<string, unknown>; // accumulates step outputs
     }

     export interface PrefillFlags {
       engine?: string;
       model?: string;
       botKey?: string;
       userId?: string;
       session?: string;
     }

     export interface WizardStep {
       id: string;
       label: string;
       isConfigured(ctx: WizardContext): boolean;
       shouldSkip?(ctx: WizardContext): boolean;
       run(ctx: WizardContext): Promise<void>;
     }
     ```
   - `PartialConfig` is the loosely-typed raw config object (pre-Zod), read directly from the YAML file. Define it as a minimal type covering the fields the wizard cares about (engine name, bot token, user IDs, project cwd, session).

3. **Create wizard runner skeleton**
   - Create `src/wizard/runner.ts` — exports `runWizard(ctx: WizardContext, steps: WizardStep[]): Promise<void>`.
   - Stub implementation: iterates steps, calls `isConfigured()` to skip, shows progress ("Step N of M"), calls `run()`.
   - Handles `@clack/prompts` `isCancel()` for graceful Ctrl+C exit at any point.
   - When `ctx.reset` is true, skip the `isConfigured` check (always prompt).

4. **Create step registry**
   - Create `src/wizard/steps/index.ts` — exports an ordered array of all wizard steps. Initially empty; steps are added in subsequent phases.

5. **Create wizard entry point**
   - Create `src/wizard/index.ts` — exports `startWizard(cwd: string, prefill: PrefillFlags, reset: boolean): Promise<void>`.
   - Reads existing config (if any) into `PartialConfig`, builds `WizardContext`, imports step registry, calls `runWizard()`.

## Phase 2: Config analyzer (completeness check)

**Goal:** Detect whether an existing config is "complete" from the wizard's perspective, and determine which fields are missing.

### Steps

1. **Create config analyzer**
   - Create `src/wizard/analyzer.ts`.
   - Export `analyzeConfig(cwd: string): { configExists: boolean; missingFields: string[] }`.
   - Logic:
     1. Look for `hal.config.{json,jsonc,yaml,yml}` in `cwd` (reuse `resolveConfigFile` from `src/config.ts` — export it if not already public).
     2. If no file found → `configExists: false`, `missingFields: ["all"]`.
     3. If file found → parse it (raw, no env substitution — just structural check), inspect wizard-coverable fields:
        - `projects.*[0].cwd` present and not placeholder
        - `projects.*[0].telegram.botToken` present and not a literal `${...}` placeholder without a resolving env var
        - `globals.access.allowedUserIds` or any project's `access.allowedUserIds` has at least one real ID (not the template placeholder)
        - `globals.engine.name` or project's `engine.name` is set
        - Session config is present (once 040 lands)
     4. Return list of missing field IDs (matching step IDs: `"cwd"`, `"bot-token"`, `"user-id"`, `"engine"`, `"session"`).

2. **Export `needsWizard` helper**
   - `needsWizard(cwd: string): boolean` — returns `true` when `missingFields.length > 0` or `configExists === false`.
   - This is the function `runStart()` in `cli.ts` will call to decide whether to auto-trigger the wizard.

## Phase 3: Wizard steps (cwd, bot token, user ID, engine/model)

**Goal:** Implement the core wizard steps (1–5 from task requirements), each as a self-contained module.

### Steps

1. **Step: Project `cwd`** (`src/wizard/steps/cwd.ts`)
   - `isConfigured`: check `ctx.existingConfig?.projects` has at least one entry with a non-placeholder `cwd`.
   - `shouldSkip`: skip if prefill flag not applicable (cwd uses `--cwd` from the global CLI args, not a wizard-specific flag).
   - `run`: use `@clack/prompts` `select` to list subdirectories of `ctx.cwd` (via `readdirSync` + `statSync` to filter dirs only), plus a "Use current directory (.)" option. Store chosen path in `ctx.results.cwd`.

2. **Step: Telegram bot token** (`src/wizard/steps/bot-token.ts`)
   - `isConfigured`: check existing config has a bot token that is not a `${...}` placeholder, OR the `TELEGRAM_BOT_TOKEN` env var is set.
   - `shouldSkip`: skip if `ctx.prefill.botKey` is set and valid.
   - `run`: use `@clack/prompts` `text` with validation (non-empty, matches Telegram bot token format `digits:alphanumeric`). Offer a `confirm` follow-up: "Need help creating a bot?" → if yes, open `HAL_DOCS_TELEGRAM_CREATE_BOT` in the default browser (reuse `open` / `xdg-open` logic from existing `openFileInEditor`). Store in `ctx.results.botToken`.
   - Suggest storing the token in `.env` rather than inline in the config file, with a note explaining why (secrets shouldn't be committed).

3. **Step: Telegram user ID** (`src/wizard/steps/user-id.ts`)
   - `isConfigured`: existing config has at least one numeric entry in `access.allowedUserIds`.
   - `shouldSkip`: skip if `ctx.prefill.userId` is set and valid.
   - `run`: use `text` prompt with numeric validation (1–0xFFFFFFFFF range, reuse `parseTelegramUserId` logic). Offer help link similar to bot token step. Store in `ctx.results.userId`.

4. **Step: Additional allowed users** (`src/wizard/steps/additional-users.ts`)
   - `isConfigured`: always returns `false` (this step is additive — you can always add more users). But skip if not in `--reset` mode and existing config already has users configured.
   - `run`: loop — `confirm` "Add another user?", if yes → `text` prompt for ID, validate, accumulate. When user says no, store array in `ctx.results.additionalUserIds`.

5. **Step: Engine / model** (`src/wizard/steps/engine.ts`)
   - `isConfigured`: existing config has `engine.name` set.
   - `shouldSkip`: skip if `ctx.prefill.engine` is set and valid.
   - `run`:
     1. `select` prompt with all valid engines (`claude`, `copilot`, `codex`, `opencode`, `cursor`, `antigravity`).
     2. If `ctx.prefill.model` is set, apply it silently. Otherwise, show a `text` prompt for model name with the engine's default pre-filled (from `DEFAULT_PROVIDER_MODEL`). User can accept the default or type a custom model.
     3. Store `ctx.results.engine` and `ctx.results.model`.

6. **Register steps in the step registry**
   - Update `src/wizard/steps/index.ts` to import and export all five steps in order.

## Phase 4: Config generation, confirmation, and file writing

**Goal:** After all steps run, build the YAML config, show a summary, ask for confirmation, write to disk, and offer to start the bot.

### Steps

1. **Config builder** (`src/wizard/config-builder.ts`)
   - Export `buildConfigFromResults(ctx: WizardContext): string`.
   - Construct a config object from `ctx.results` + any existing config values (for idempotent runs where some fields were preserved).
   - Use the `yaml` package (already a dependency) to serialize to YAML with comments (doc-link header, section comments similar to `init-template.yaml`).
   - For bot token: if the user provided it inline, write `botToken: "${TELEGRAM_BOT_TOKEN}"` in the YAML and create/append to `.env` with the actual value. If user explicitly wants inline, allow that too.

2. **Confirmation + write step** (`src/wizard/confirm-and-write.ts`)
   - Show the generated YAML using `@clack/prompts` `note` (or plain `console.log` with syntax-highlighted box).
   - Show a summary of all values (including pre-filled ones) before the YAML.
   - `confirm` prompt: "Write this configuration?"
   - If yes: write `hal.config.yaml` to `ctx.cwd`. If `.env` entries were added, write those too.
   - If no: abort with a friendly message.
   - Create engine instructions file if missing (reuse logic from `runInit`).

3. **Post-write action**
   - `select` prompt: "Start the bot now?" / "Exit"
   - If start: return a signal to the caller so `cli.ts` can proceed to `runStart()`.
   - If exit: `process.exit(0)`.

## Phase 5: Session configuration step (depends on task 040)

**Goal:** Add the session configuration wizard step, using the structured `engine.session` object from task 040.

> **Blocked until task 040 is complete.** This phase can be deferred and added after the initial wizard ships with steps 1–5.

### Steps

1. **Step: Session configuration** (`src/wizard/steps/session.ts`)
   - `isConfigured`: existing config has `engine.session` object set (not boolean — that's the old form rejected by 040).
   - `shouldSkip`: skip if `ctx.prefill.session` is set and valid, or if `ctx.reset` is false and session is already configured.
   - `run`:
     1. Read `ctx.results.engine` to determine which sub-options to show.
     2. `confirm`: "Enable session persistence?" (default: yes).
     3. If engine is `claude`: `select` for `shared` (true/false) with descriptions explaining the difference.
     4. If engine is `codex` or `cursor`: `confirm` for `experimentallyEnableUserSession` with a warning about its experimental nature.
     5. Other engines: no sub-options.
     6. Store `ctx.results.session` as `{ enabled, shared, experimentallyEnableUserSession }`.

2. **Register in step registry**
   - Add session step after the engine step in `src/wizard/steps/index.ts`.

3. **Update config builder**
   - `buildConfigFromResults` must emit the session object in the YAML when session results are present.

## Phase 6: CLI integration (wiz command + auto-trigger)

**Goal:** Wire the wizard into `src/cli.ts` as both an explicit `wiz` command and an auto-trigger on incomplete config.

### Steps

1. **Extend `ParsedArgs` and arg parser**
   - Add `"wiz"` to the command union: `command: "start" | "init" | "wiz"`.
   - Parse `wiz` as a command in the args loop.
   - Parse wizard-specific flags: `--bot-key`, `--user-id`, `--session`, `--reset`. (Existing `--engine`, `--model`, `--cwd` are already parsed.)
   - Extend `ParsedArgs` with `botKey?: string`, `userId?: string`, `session?: string`, `reset?: boolean`.

2. **Add `wiz` command handler in `main()`**
   - After `parseArgs()`, if `command === "wiz"`, import and call `startWizard()` from `src/wizard/index.ts` with the parsed prefill flags and reset.
   - If the wizard signals "start the bot", call `runStart(cwd)`.

3. **Add auto-trigger in `runStart()`**
   - Before `tryLoadMultiConfig()`, call `needsWizard(configDir)` from the analyzer.
   - If it returns `true` and `process.stdin.isTTY` (interactive terminal), import and run the wizard in gap-filling mode (only missing fields).
   - If not a TTY (e.g., running in CI/Docker), skip the wizard and let the existing `ConfigLoadError` flow handle it (so non-interactive environments don't hang).
   - After the wizard completes, retry `tryLoadMultiConfig()`.

4. **Update help text**
   - Add `wiz` to the commands list in `showHelp()`.
   - Add wizard-specific flags to the options list.
   - Update the "no config found" error message in `loadMultiConfigInternal` to suggest `npx @marcopeg/hal wiz` instead of (or in addition to) `init`.

5. **Deprecation notice for `init`**
   - When `init` is invoked, print a deprecation notice: "Note: `init` is deprecated. Use `wiz` for an interactive setup experience."
   - Keep `init` functional — no behavioral changes.

6. **Build script update**
   - Verify `pnpm run build` succeeds with the new `src/wizard/` directory.
   - No special handling needed — TypeScript will compile the new files. The `init-template.yaml` copy in the build script is unaffected.

## Phase 7: Validation and polish

**Goal:** Ensure everything compiles, lints, and the UX is polished.

### Steps

1. **TypeScript and lint**
   - Run `pnpm run build` and `pnpm run lint`.
   - Fix any issues.

2. **UX polish**
   - Verify progress display shows correct step counts (dynamic based on which steps are active — e.g., session sub-options only count when the engine supports them).
   - Verify cancellation (Ctrl+C) at any prompt exits gracefully with a friendly message.
   - Verify pre-fill flags skip correctly and show in the summary.
   - Verify `--reset` re-asks all questions even when values exist.

3. **Edge cases**
   - Existing config in JSON/JSONC format: the wizard reads it but always writes YAML. If the original was JSON/JSONC, the wizard should note this and ask whether to keep the old file or replace it.
   - Empty directory (no subdirectories for cwd picker): offer only "Use current directory".
   - Invalid pre-fill values: fall back to prompting.
