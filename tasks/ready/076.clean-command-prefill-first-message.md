# 076 — Allow /clear to pass first message for new session

**Status**: draft

## Problem

`/clear` resets the current session, but if the user wants to immediately start a new flow they must send a second message. This adds friction and makes "clear + start new prompt" a two-step process.

## Goal

Allow `/clear` to accept optional text and automatically send it as the **first message** of the new session after cleanup.

## Context & Constraints

- `/clear` should still work with no extra text (current behavior).
- The provided text must be treated exactly like a normal user message for a new session (same pipeline, logging, and engine routing).
- The message must **not** be attached to the previous session transcript.
- Scope for this task is **Telegram only**.
- When `/clear` includes a payload, that payload overrides the configured `commands.clear.message` reply for that invocation.
- Session reset semantics must remain engine-correct:
  - engines with active reset behavior must still perform the real reset step first;
  - payload dispatch must start a new session (no continuation).

## What to do

### Command parsing

- Support `/clear <message>` where everything after `/clear ` is the new first message.
- Treat everything after `/clear ` as a single message and send it to the LLM as normal user input.
- If the message is empty or only whitespace, behave like `/clear` with no payload.

### Post-clear dispatch

- After session cleanup completes, enqueue the message for the engine as if the user had just sent it.
- Ensure the UI/telegram response shows the exact normal message processing/status flow.
- For `/clear <message>`, do not send an extra configured clear confirmation; dispatch the payload immediately after successful reset.

### Safety & UX

- If cleanup fails, do not send the message; surface the error.
- Reuse existing message-handling validation/limits from the normal text pipeline.

## Acceptance criteria

- [ ] `/clear` without a message behaves exactly as today.
- [ ] `/clear <message>` resets the session and sends `<message>` as the first message in the new session.
- [ ] Everything after `/clear ` is treated as one message and passed to the LLM as normal user input.
- [ ] The dispatched message appears in the new session history only.
- [ ] For `/clear <message>`, the configured `commands.clear.message` reply is not sent; the payload is processed immediately.
- [ ] Task scope is Telegram command handling only; no CLI command-surface changes are required.
- [ ] Errors during cleanup prevent dispatch and return a clear error to the user.
- [ ] Engine session behavior remains correct: reset first, then payload runs as a fresh turn without continuation.
