- Run `pnpm add strip-json-comments`.

- No `@types` package needed — ships its own TypeScript types.

## Phase 2: JSONC parser integration

### Step 2.1 — Create a `parseJsonc` helper in `src/config.ts`

Add a function that wraps the preprocessing + parse steps:

```typescript
import stripJsonComments from "strip-json-comments";

function parseJsonc(content: string): unknown {
  return JSON.parse(stripJsonComments(content, { trailingCommas: true }));
}
```

The `whitespace: true` default is preserved (keeps character positions for accurate `JSON.parse` error messages).

### Step 2.2 — Register `.jsonc` in the format detection layer

Add `".jsonc"` to the list of supported extensions in the format detection introduced by task 027. When the detected format is `.jsonc`, use `parseJsonc` instead of `JSON.parse`.

This applies to both config slots:
- `hal.config.jsonc` (base config, loaded in `loadMultiConfigInternal`)
- `hal.config.local.jsonc` (local overrides, loaded in `loadLocalConfig`)

The per-slot exclusivity check already covers all registered extensions — adding `.jsonc` to the list automatically makes `hal.config.json` + `hal.config.jsonc` a boot error.

## Phase 4: Config writer

### Step 4.1 — Make `src/config-writer.ts` format-aware for reading

`updateProjectModel` currently reads the config with `JSON.parse`. If the file is `.jsonc`, this would fail on comments. Update the read path to use `parseJsonc` when the detected file has a `.jsonc` extension.

### Step 4.2 — Clarify write behavior: comments are not preserved

`JSON.stringify` cannot preserve comments. This is acceptable because:
- The config-writer is only used by the `/model` command (saves engine/model changes).
- It already prefers `hal.config.local.*` over the base config when the local file exists.
- Users who care about preserving comments in their base `.jsonc` should ensure a `hal.config.local.*` file exists — machine writes go there.

No additional code needed for this — just document the behavior.

### Step 4.3 — Detect the correct file path per format

Currently the writer hardcodes `.json` paths:

```typescript
const localPath = join(configDir, "hal.config.local.json");
const basePath = join(configDir, "hal.config.json");
```

Update to scan for whichever extension actually exists (reuse the detection helper from task 027 if available), falling back to `.json` when creating a new local file.
