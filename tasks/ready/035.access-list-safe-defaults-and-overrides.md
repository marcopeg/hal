# 035 — Access list safe defaults and overrides

## Status

ready

## Summary

Harden access control defaults so the bot is not open to everyone unless explicitly and dangerously configured.

## Ordering

This task must be executed **before** task 036 (env substitution for allowedUserIds). Task 035 locks down the security model with numeric-only IDs; task 036 then layers string/env-var support on top.

## Requirements

- Default behavior must be restrictive, not open.
- Access is allowed only when at least one of these is true:
  - `access.allowedUserIds` contains at least one user ID, or
  - `projects.<name>.access.allowedUserIds` contains at least one user ID, or
  - `access.dangerouslyAllowUnrestrictedAccess` is explicitly set to `true`, or
  - `projects.<name>.access.dangerouslyAllowUnrestrictedAccess` is explicitly set to `true`.
- Introduce/configure the explicit dangerous flag name: `dangerouslyAllowUnrestrictedAccess`.
- For the `access` key specifically, project-level config must be a full replacement of global `access` (not a merge/extension).
- Precedence rule:
  - if project defines `access`, use only project `access`;
  - otherwise use global `access`.
- Within an effective access object, `allowedUserIds` takes precedence over `dangerouslyAllowUnrestrictedAccess`.
  - If `allowedUserIds` is present/non-empty, only listed users are allowed.
- `allowedUserIds` remains `z.array(z.number())` in this task (string support comes in task 036).

## Boot-time validation (fail-fast)

- Every resolved project config must have a valid access policy. If any project's effective access has no `allowedUserIds` and `dangerouslyAllowUnrestrictedAccess` is not `true`, the bot **refuses to start** with a clear error identifying the misconfigured project(s).
- This validation runs both at initial boot and after config hot-reload. A hot-reload that introduces an invalid access config is rejected.

## Project-level `access` replacement semantics

- If a project omits the `access` key entirely → falls through to global `access`.
- If a project defines `access` (even as `access: {}`), that fully replaces global `access` for that project.
  - Consequence: `access: {}` with no `allowedUserIds` and no `dangerouslyAllowUnrestrictedAccess` is a boot-time validation error for that project, even if global config is valid.

## Acceptance Criteria

- If no effective `access` permits users and no dangerous flag is true, bot **fails to start** with a clear error.
- Boot-time validation also applies after hot-reload (invalid reload is rejected).
- Global `dangerouslyAllowUnrestrictedAccess: true` no longer applies when a project defines its own `access`.
- Project `access` fully replaces global `access` (not merged).
- `access: {}` at project level triggers full replacement and results in a validation error (no IDs, no dangerous flag).
- Non-empty `allowedUserIds` is enforced even when dangerous flag is true in the same effective scope.
- Error messages clearly identify which project(s) have misconfigured access.
