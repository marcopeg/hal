# 062 — Wizard env file precedence when both .env and .env.local exist

**Status:** draft

## Goal

Define and implement a safe, explicit env-file selection strategy for the setup wizard, and centralize all env-file documentation so it is accurate and consistent across the project.

## Context

The docs currently say the wizard writes to `.env` (or `.env.local` if that already exists). The implementation in `src/wizard/confirm-and-write.ts` (`pickEnvPath()`) prefers `.env` when it exists and only falls back to `.env.local` when `.env` is missing.

The core concern is **accidental overwrites of existing secrets**. If `.env` already contains user secrets, silently editing it is unsafe. The wizard should make the user's choice explicit whenever there is ambiguity.

## Decision: env file selection rules

| State on disk | Wizard behavior |
|---|---|
| Neither `.env` nor `.env.local` exists | Write to `.env` (no prompt) |
| Only `.env.local` exists | **Ask user**: write to `.env.local`, or stop |
| Only `.env` exists | **Ask user**: write inside `.env`, or create `.env.local`, or stop |
| Both `.env` and `.env.local` exist | **Ask user**: write to `.env`, write to `.env.local`, or stop |

"Stop" means abort without writing any env file (the wizard exits cleanly with a message).

## Scope

### Implementation

- Update `pickEnvPath()` in `src/wizard/confirm-and-write.ts` to implement the four-case logic above
- When a prompt is needed, use the existing `select` primitive (clack) with three options: the two file paths + "Stop / don't write"
- If the user selects "Stop", skip env file writing and exit cleanly (no config write either — or at minimum log clearly that env was skipped)
- Add a small note in the wizard output about how this interacts with the `env` config key (custom env path), which bypasses the selection logic entirely

### Documentation

- Perform an extensive search across `README.md` and all files under `docs/` to locate every mention of `.env`, `.env.local`, env file loading, wizard env behavior, and secrets storage
- Centralize the canonical description of env file behavior (loading precedence + wizard selection rules) in a **single dedicated section or page** (e.g. `docs/env-files.md` or a dedicated section in the config reference)
- All other pages that currently describe env behavior should be simplified to a one-liner that references the canonical location — no duplication
- The canonical section must cover:
  - Runtime loading precedence (which file wins at startup)
  - Wizard selection logic (the four cases above)
  - The `env` config key (custom path override)
  - `.gitignore` guidance for secrets

## Out of scope

- Changing the runtime env loading precedence
- Modifying other parts of the wizard flow unrelated to env file choice

## Acceptance criteria

- [ ] `pickEnvPath()` implements the four-case matrix exactly as specified
- [ ] When a user prompt is needed, three options are offered: file A, file B, stop
- [ ] "Stop" exits cleanly with a clear message; no partial writes occur
- [ ] All env-related docs are consolidated: one canonical source, all others reference it
- [ ] No doc page contradicts another regarding env file behavior
- [ ] Manual smoke-test covers all four disk states (neither / only .env / only .env.local / both)
