# 067 — Rename config-dir flag to config

## Status

draft

## Summary

Replace the CLI flag `--config-dir` with `--config`, with no backward compatibility. `--config` accepts either a directory path or a direct path to a specific config file, and infers `configDir` and `configFile` from the value. Update all help text, docs, examples, and `package.json` scripts accordingly.

## Problem

`--config-dir` is verbose. A shorter `--config` flag is clearer and opens the door to pointing directly at a config file rather than only a directory.

## Goals

- Replace `--config-dir` with `--config` everywhere (source, docs, scripts).
- Remove `--config-dir` entirely — no alias, no deprecation shim.
- Make `--config` smart about its value:
  - If the value has **no file extension** → treat as a directory; use it as `configDir` and keep default filename resolution (`hal.config.*`).
  - If the value has a **recognised config extension** (`.json`, `.yaml`, `.yml`, etc. — whatever formats the app already supports) → treat as an explicit file path; derive `configDir` as the parent directory and use the filename directly.
  - If the value has an **unrecognised extension** → crash with a clear error indicating the extension is not supported.
- Update CLI help, inline comments, error messages, README, docs, and `package.json` dev scripts to use `--config`.

## Constraints

- Remove `--config-dir` completely — do not keep it as an alias.
- Path resolution and validation logic must remain equivalent for the directory case.
- The file-path case must correctly derive both `configDir` (parent) and the config filename so downstream code that reads/writes the config continues to work.
- When `--config` points to a file, crash if the file does not exist — do not attempt to create it.
- When `--config` points to a directory, existing directory-creation behaviour is preserved.
- File extension must be one of the app's supported config formats; any other extension is a hard error.

## Context

Current implementation in `src/cli.ts` parses `--config-dir` as a simple directory override (lines ~216–220). The resolved path flows into the rest of the boot sequence as `configDir`.

`package.json` likely references `--config-dir` in dev/test scripts — these must be updated too.

## Acceptance Criteria

- `--config <dir>` behaves identically to the old `--config-dir <dir>`.
- `--config <path/to/hal.config.json>` sets `configDir` to `path/to/` and uses `hal.config.json` as the config filename.
- `--config <path/to/hal.config.json>` crashes with a clear error if the file does not exist.
- `--config <path/to/hal.config.xyz>` crashes with a clear error if the extension is not a supported config format.
- `--config-dir` is gone — passing it produces an "unknown option" error (or is simply ignored per normal arg parsing).
- CLI help text uses `--config <path>` with a description covering both directory and file forms.
- All docs and `package.json` scripts reference `--config`, not `--config-dir`.
- Existing tests pass; new test cases cover the file-path form of `--config`.
