# 037 — Projects map with key-derived defaults

## Status

in progress

## Summary

Refactor project configuration to use a `projects` map keyed by project key instead of an array, and derive default `name` and `cwd` from that key. Resolved project identity uses the map key as `slug`.

## Context

- **Base config** (`hal.config.{json,yaml}`) and **local config** (`hal.config.local.{json,yaml}`) both use the same map shape: `projects: { <key>: { ... } }`.
- Local config entries merge into the base project with the same key (no array matching by name/cwd).
- Map keys are the single source of project identity; resolved config exposes that as `slug` everywhere (paths, errors, routing).

## Constraints

- **Key format**: Map keys must be slug-like: only letters, numbers, dashes, and underscores. This keeps default `cwd` safe as a path segment and avoids ambiguous keys. Reject or clearly error on invalid keys.

## Requirements

- Replace array-style project configuration with object/map style:
  - from: `projects: [{ ... }]`
  - to: `projects: { <projectKey>: { ... } }`
- Use the map key as the canonical project identifier (exposed as `slug` in resolved config).
- If `name` is not explicitly set, default `name` to the map key.
- If `cwd` is not explicitly set, default `cwd` to the map key.
- Preserve support for explicitly provided `name` and `cwd` overrides.
- Apply the same map shape and key-based merging for local config.
- Ensure project resolution, validation, and command routing continue to work with the new structure (using `slug` = map key).
- **Extensively update documentation** so that all user- and developer-facing docs describe the new `projects` map format, key-derived defaults, key format rules, and local config merge by key (no array or name/cwd matching).

## Acceptance Criteria

- Config schema accepts `projects` as a map keyed by project key; keys are validated as slug-like (alphanumeric, `-`, `_`).
- Loaded project objects include key-derived defaults for `name` and `cwd` when omitted.
- Resolved project config: `slug` is always the map key (not derived from name/cwd).
- Existing behavior remains unchanged when `name`/`cwd` are explicitly defined.
- Local config uses the same map shape; merging is by key (local key must exist in base).
- Errors are clear when invalid keys, duplicate/invalid map entries, or local keys not in base are provided.
- Documentation (config guide, README, Telegram docs, and any other references to `projects` or project matching) is updated to the map format, key rules, and key-based local merge.
