# 089 — Cron scheduler local timezone and context

## Status

draft

## Summary

Define and document cron scheduling so HAL evaluates cron schedules in the host computer's local timezone by default. Expose timezone information in HAL context as both `local timezone` and `cron scheduler timezone` to remove ambiguity for agents, and update the cron authoring skill so future cron work uses the correct timezone assumptions consistently.

---

## Goals

- Explicitly define that the cron scheduler uses the computer's local timezone.
- Document that behavior in the relevant user-facing and contributor-facing cron docs.
- Add timezone context fields so HAL can surface both:
  - local timezone
  - cron scheduler timezone
- Update the cron skill instructions so cron authoring and cron-related task execution use the same timezone rule.

## Problem

The current cron behavior and documentation do not clearly state which timezone a cron schedule is evaluated against. That creates avoidable ambiguity when a user asks HAL to schedule something "at 9am" or when an agent authors a cron file and assumes UTC or some other implicit timezone.

This ambiguity is especially risky in two places:

- runtime behavior, where the scheduler might execute at a different wall-clock time than the user expects;
- agent context and skills, where HAL or a cron-authoring skill might describe or generate cron schedules using the wrong timezone assumption.

## Desired Behavior

- Cron schedules are interpreted using the host computer's local timezone unless a future task explicitly introduces an override mechanism.
- HAL context includes the machine's local timezone in a clearly labeled field.
- HAL context also includes a separate `cron scheduler timezone` field, even if it currently resolves to the same value as local timezone.
- Cron documentation explains the timezone behavior clearly, including that "local timezone" means the computer running HAL.
- The cron skill is updated so it explicitly reads and preserves this rule when generating, editing, or explaining cron jobs.

## Scope

- Audit the cron scheduler implementation and make the timezone behavior explicit in code where needed.
- Update the relevant cron docs and examples to state the local-timezone rule.
- Update HAL context-building so timezone data is available to agents in a stable, documented way.
- Update the cron skill instructions to reference the local timezone and cron scheduler timezone context fields.

## Out of Scope

- Introducing per-cron timezone overrides.
- Adding a user-configurable timezone setting unless required to support the local-timezone rule cleanly.
- Reworking existing cron syntax or scheduler architecture beyond what is needed to make timezone behavior explicit and documented.

## Requirements

- The effective timezone for cron scheduling must be the host computer's timezone.
- The implementation must make that behavior explicit rather than relying on undocumented library defaults.
- HAL context must include both `local timezone` and `cron scheduler timezone`.
- If both context values are currently identical, that should be intentional and documented.
- The cron skill must instruct future agents to use the scheduler timezone context when reasoning about schedules.
- Relevant documentation for cron behavior must explain the timezone rule in plain language.

## Acceptance Criteria

- A reader can determine from the docs that cron schedules run in the computer's local timezone.
- The runtime scheduler behavior matches that documented rule.
- HAL context exposes both timezone fields for agent consumption.
- Cron-related prompts and skills can reference the scheduler timezone without ambiguity.
- The cron skill is updated so it does not assume UTC or invent another timezone when authoring cron files.

## Notes

- Adding both `local timezone` and `cron scheduler timezone` is intentionally redundant for now so future changes can distinguish them without breaking prompt contracts.
- This task should verify the exact timezone identifier format exposed in context, for example IANA zone names when available.
