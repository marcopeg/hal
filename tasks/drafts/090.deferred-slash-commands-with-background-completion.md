# 090 — Deferred slash commands with background completion

## Status

draft

## Summary

Allow a slash command to defer its heavy work so HAL can acknowledge immediately with a short assistant response such as "working on it in the background", release the chat flow back to the user, and deliver the final result later when the engine task finishes.

The task should turn this into a reusable capability rather than a one-off command hack. The likely direction is to extract the deferred execution mechanics into a shared library or runtime primitive that can be reused by slash commands and by cron-related flows that already need similar background execution semantics.

---

## Problem

Current command execution is primarily request-response oriented. A slash command either handles the message immediately or yields to the normal engine flow for a synchronous answer.

That is not a good fit for commands that need to:

- kick off a long-running engine task;
- tell the user immediately that work has started;
- let the user continue chatting and using HAL while that work runs;
- send the final result only after the background task completes.

Without an explicit deferred mode, command authors have to choose between blocking the current interaction until the engine finishes or building ad hoc background logic in each command. That increases complexity and makes it harder to converge on one coherent execution model.

The same underlying need already exists around cron-triggered work and other asynchronous flows. If deferred command execution is added only at the slash-command layer, HAL risks creating duplicate background-job mechanisms instead of one reusable library.

## Goal

Introduce a first-class deferred execution path for slash commands where a command can start an engine-backed task asynchronously, return an immediate assistant acknowledgement, and later deliver the final engine result to the user when the background work completes.

The implementation should be designed as shared infrastructure that can also support cron-related deferred work, so the project can begin consolidating asynchronous task execution into one reusable library/API instead of embedding special cases in `hal_cron` or individual commands.

## Desired Behavior

- A slash command can explicitly choose a deferred execution mode.
- When deferred mode is used, HAL sends an immediate assistant reply confirming that work continues in the background.
- The deferred path starts the engine task without blocking the user from sending more messages or using other commands.
- When the background task finishes, HAL sends the final result back to the same user synchronously as a normal assistant message.
- Deferred execution preserves enough context to correlate the completion message with the original command invocation.
- The deferred execution primitives are reusable by command handlers and by cron-related/background scheduling flows.

## Scope

- Define the command/runtime contract needed for deferred execution from `.mjs` slash commands.
- Implement the immediate acknowledgement plus background engine kickoff flow.
- Implement completion delivery back to the originating user/session when the engine run finishes.
- Define the lifecycle, persistence, and correlation data needed to track deferred tasks safely.
- Extract the reusable deferred/background execution logic into a shared module or library layer rather than leaving it embedded in one command handler.
- Review `hal_cron`-related behavior and align it with the same reusable background execution approach where appropriate.
- Document the command-authoring contract for deferred commands.

## Out of Scope

- Building a full generic job queue UI or dashboard unless required for minimal operability.
- Introducing distributed workers or multi-process orchestration beyond what HAL currently needs.
- Solving every background-task use case in one pass if a narrower reusable abstraction is sufficient.
- Redesigning unrelated slash-command semantics beyond what is required to support deferral cleanly.

## Requirements

- A slash command must be able to start an engine task asynchronously and return control to HAL immediately.
- The user must receive an immediate acknowledgement message indicating that the task is running in the background.
- The user must be able to continue interacting with HAL while the deferred task is running.
- When the deferred engine task completes, HAL must send the resulting assistant output back to the correct user/session.
- The implementation must define how deferred tasks are identified, tracked, and cleaned up.
- The implementation must define how failures are surfaced to the user when a deferred task errors or is cancelled.
- Deferred execution must reuse shared infrastructure rather than duplicating bespoke logic in each command.
- The design must explicitly consider `hal_cron` as a consumer of the same background execution primitives, even if cron migration is only partial in this task.
- Command-authoring docs/examples must explain when to use deferred execution and what response contract to return.

## Acceptance Criteria

- A `.mjs` command can trigger a deferred engine-backed task and HAL immediately replies with a background-work acknowledgement.
- The command invocation does not block the user from continuing to chat or issuing other commands.
- The engine work runs in the background and completes independently of the immediate acknowledgement.
- On completion, HAL sends the final result to the same user without requiring them to poll manually.
- Errors in deferred execution produce a clear user-visible failure message or status outcome.
- The implementation introduces a reusable deferred/background execution layer instead of a command-specific one-off.
- The task documents how the new mechanism relates to existing cron/background behavior and identifies any follow-up needed for fuller `hal_cron` consolidation.
- At least one example command and automated tests cover the deferred acknowledgement and later completion flow.

## Notes

- The immediate acknowledgement text does not have to be exactly "working on it in the background", but HAL should standardize the behavior and make it easy for command authors to opt into a consistent UX.
- The key architectural constraint is reuse: this should begin converging command deferral and cron-style asynchronous execution toward one shared model.
