# 071 — Enforce project cwd in agent prompts

## Status

draft

## Summary

Inject a universal system-level instruction into every HAL prompt that anchors all agent file operations to the project `cwd`. Add an opt-out config flag (`context.enforceCwd: false`) for monorepo setups where the agent legitimately needs to roam.

## Problem

When a user sends a natural-language prompt like "add a file somewhere" or "create the config for this project", the agent may resolve "somewhere" or "the project" to the git repository root rather than the HAL-configured `cwd`. This is not a bug in the engine — it is expected behaviour. All supported engines are git-aware: they run `git rev-parse --show-toplevel` internally and treat the git root as the authoritative "project root". When the HAL `cwd` is a subdirectory of a larger repository (e.g. `examples/my-project/` inside a monorepo), the agent silently writes files at the repo root.

This was first observed with Copilot (session log `vibed/2603140703.open-session.md`): the agent placed `AGENTS.md` at `~/Projects/ccpa-telegram/` instead of `examples/copilot/` when asked to "write the AGENTS.md for the project". The agent itself confirmed: *"the git repository root was provided in the environment context"* — it discovered the git root through its own git introspection and acted on it.

## Root cause analysis

HAL already injects `project.cwd` into every prompt via `formatContextPrompt` (`src/context/resolver.ts:384`). The rendered block looks like:

```
# Context
- project.cwd: /path/to/examples/my-project
- project.name: my-project
- engine.name: copilot
...

# User Message
add a file somewhere
```

`project.cwd` is a **data point** in a flat key-value list — it carries no instruction weight. Nothing in the prompt tells the agent that this path is the exclusive root for file operations. The agent reads it as ambient metadata and continues to use its own git-root discovery as the authoritative project boundary.

## Why instruction files don't solve it uniformly

The natural alternative is to write a cwd rule in the project's instruction file. But there is no single instruction file that all engines read:

| Engine | Instruction file |
|--------|-----------------|
| Claude Code | `CLAUDE.md` only |
| Copilot, Codex, Cursor, OpenCode | `AGENTS.md` |
| Antigravity | `GEMINI.md` only |

A rule in `AGENTS.md` is invisible to Claude and Antigravity. A rule in `CLAUDE.md` is invisible to every other engine. Per-engine instruction files require the user to maintain multiple files with the same content, and still do not cover engine switches at runtime.

## What HAL already does (reference point)

`buildContextualPrompt` (`src/engine/prompt.ts:41`) already injects one imperative system instruction:

```ts
if (downloadsPath) {
  return `${contextualPrompt}\n\n[System: To send files to the user, write them to: ${downloadsPath}]`;
}
```

This pattern — a `[System: ...]` line appended to the prompt — is the existing mechanism for cross-engine directives. It is engine-agnostic: it works for Claude, Copilot, Codex, Cursor, OpenCode, and Antigravity equally because it is injected into the prompt text that every engine receives.

## Proposed solution

Inject a cwd anchor instruction at the **start** of the context block (before the key-value list) in `buildContextualPrompt`. The instruction should be:

```
[System: Your working directory is {project.cwd}. All file read and write operations
must be relative to this path. Do not create, edit, or delete files outside this
directory unless the user explicitly provides an absolute path outside it.]
```

Placement matters: the beginning of the prompt is weighted more heavily by models than the end. The existing downloads path instruction appends at the end, which is acceptable for a secondary directive. The cwd anchor is a primary constraint and should lead.

**On/off behaviour:** The instruction should be on by default, with an explicit opt-out:

```yaml
globals:
  context:
    enforceCwd: false   # disable for monorepo setups
```

Or per-project:

```yaml
projects:
  monorepo-root:
    context:
      enforceCwd: false
```

This aligns with the `copilot.allowAllPaths` pattern: safe by default, explicit opt-out for power users.

## Design options considered

**Option A — Always-on instruction (no config flag).** Simplest. No user configuration needed. Risk: may be surprising for monorepo setups where the agent intentionally needs to work at the git root.

**Option B — On by default with opt-out (recommended).** Balances safety and flexibility. Consistent with how `copilot.allowAllPaths` works.

**Option C — Free-text context instructions field.** Expose a `context.instructions: [...]` array in config for arbitrary system-level instructions. More flexible but a broader feature. Solves a different problem space and can be a follow-up.

## Relationship to engine-level path enforcement

This feature is **complementary** to the engine-level structural fixes done in the same session:

- Copilot: `--allow-all-paths` removed (hard OS boundary, default on)
- Codex: `--full-auto` sandbox (hard OS boundary, default on)
- Antigravity: `--sandbox` flag available as opt-in
- Claude Code: `allowedPaths` in `.claude/settings.json` (user-managed)
- Cursor, OpenCode: no CLI mechanism available

For engines with hard boundaries, the injected instruction is redundant but harmless — a belt-and-suspenders guard. For engines without hard boundaries (Cursor, OpenCode, Claude without `allowedPaths`), the injected instruction is the **only available HAL-level control**.

## Scope

- Add `enforceCwd` boolean to config schema (default `true`), resolved in `ResolvedProjectConfig`.
- In `buildContextualPrompt`, prepend the `[System: ...]` cwd instruction when `config.context.enforceCwd !== false`.
- Update `docs/config/reference.yaml` and `docs/config/README.md` with the new field.
- Update engine docs to note that the cwd instruction is injected by HAL and can be disabled.

## Acceptance criteria

- Every prompt sent to any engine includes a `[System: Your working directory is {cwd}...]` instruction at the top of the context block, unless `enforceCwd: false` is set.
- Setting `enforceCwd: false` at globals or project level suppresses the instruction for that scope.
- The instruction does not appear for cron executions that do not include a gram context (or is handled gracefully if it does).
- Existing prompt structure (context block → user message → downloads path) is preserved; the cwd instruction is the new first line.
- Docs updated.
