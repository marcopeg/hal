# 041 ‚Äî Custom env file

## Status

draft

## Summary

Add a new top-level config property `env` that specifies the path to a custom
env file used for variable substitution. Env loading is simplified to a single
source per run: either the config directory‚Äôs `.env` (CLI cwd) or the
explicitly configured file (plus its `.local` sibling). Conflicts between both
being present are rejected at boot with a clear error and documentation link.

## Context

- Config loading already supports env-var substitution (see task 002).
- **Current behavior (to change):** Env is loaded from per-project
  `<cwd>/.env`, `<cwd>/.env.local` and config-dir `<configDir>/.env`,
  `<configDir>/.env.local` (later wins). This task removes per-project env
  loading and introduces a single-source model with optional explicit path.
- **Target behavior:**
  - Env is loaded from **one** of two modes:
    1. **Default (no `env` in config):** Only the CLI‚Äôs cwd (config directory):
       `<configDir>/.env`, `<configDir>/.env.local` (`.local` overrides `.env`).
    2. **Explicit (`env` set):** Only the configured file and its sibling
       `.local` variant (e.g. `secrets.env` ‚Üí `secrets.env` and
       `secrets.env.local` in the same directory). Config-dir `.env` is **not**
       loaded in this mode.
  - If `env` is set and the resolved path is **different** from the config
    directory‚Äôs `.env` **and** a `.env` (or `.env.local`) exists in the config
    directory, boot must fail with a conflict error telling the user to choose
    one approach and linking to the configuration docs.
- Tilde expansion for `~` in the `env` path is desirable if straightforward.
- Path is resolved relative to the config file‚Äôs directory, or as absolute if
  given as absolute.

## Requirements

- Top-level config may include an optional `env` property: path (string) to a
  single env file.
- **Path resolution:** Relative to the config file‚Äôs directory; absolute paths
  allowed; tilde expansion for `~` if straightforward.
- **Loading rules:**
  - **No per-project env:** Do not load `.env` / `.env.local` from any
    project `cwd`. Only the config directory (CLI cwd) or the explicit `env`
    path is used.
  - **When `env` is unset:** Load only `<configDir>/.env` and
    `<configDir>/.env.local` (`.local` overrides `.env`). Same merge order as
    today for these two files.
  - **When `env` is set:** Load only the resolved path and its sibling `.local`
    file (e.g. `path/to/file.env` and `path/to/file.env.local`). `.local`
    overrides the base file. Do not load config-dir `.env` in this mode.
- **Conflict (boot error):** If `env` is set and the resolved path is
  **different** from `<configDir>/.env` (after normalizing paths), and
  `<configDir>/.env` or `<configDir>/.env.local` exists, fail at config load
  with a clear error message that explains the conflict and instructs the user
  to use only one source (either remove `env` and use config-dir `.env`, or
  remove/rename config-dir `.env` and use `env`). Include a link to the
  configuration documentation.
- **Missing or unreadable custom file (boot error):** If `env` is set but the
  main file (the one specified, not the optional `.local`) is missing or
  unreadable, fail at config load with a strong, clear error message and a
  link to the configuration documentation. Do not treat as non-fatal.
- **Watcher:** When `env` is set, the config watcher must watch the resolved
  custom env path and its `.local` sibling (if it exists). Changes to either
  file trigger a config reload (same debounce as other config files).
- Schema, examples, and **thorough** documentation updated to describe:
  - The single-source model (config-dir only vs explicit path only).
  - Resolution order (base file then `.local` overrides).
  - Conflict case and how to resolve it.
  - Missing-file error and doc link.
  - Watcher behavior for the custom path.

## Out of scope

- Multiple custom env files or per-project custom env paths (single top-level
  path only for this task).
- Changing how engines receive `process.env` at run time; this is only for
  config substitution.
