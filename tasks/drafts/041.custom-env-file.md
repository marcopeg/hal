# 041 — Custom env file

## Status

draft

## Summary

Add a new top-level config property `env` that specifies the path to a custom
env file used for variable substitution. This allows using secrets (or any
variables) from a file outside the config directory and project `cwd`, e.g.
`~/.hal/secrets.env` or a path outside the repo.

## Context

- Config loading already supports env-var substitution (see task 002). Env
  values are currently loaded only from:
  - Per-project: `<cwd>/.env`, `<cwd>/.env.local`
  - Config directory: `<configDir>/.env`, `<configDir>/.env.local`
- There is no way to point at an env file outside these locations. Users who
  keep secrets in a single file outside the project (e.g. under home or a
  shared vault path) cannot use them for substitution without copying or
  symlinking.

## Requirements

- Top-level config may include an optional `env` property: path (string) to a
  single env file.
- Path is resolved relative to the config file’s directory (or as absolute if
  given as absolute). Tilde expansion for `~` is desirable if straightforward.
- That file is loaded and merged into the env used for substitution. Priority
  vs. existing sources (project .env, config-dir .env) must be defined and
  documented (e.g. custom file loaded first so project/local .env can override,
  or last so custom overrides).
- If `env` is set but the file is missing or unreadable, behavior should be
  explicit: either error at load time or treat as non-fatal and document it.
- Schema, examples, and docs updated to describe `env` and resolution order.
- Config watcher: consider whether the custom env path should trigger reload
  when the file changes.

## Out of scope

- Multiple custom env files or per-project custom env paths (single top-level
  path only for this task).
- Changing how engines receive `process.env` at run time; this is only for
  config substitution.
