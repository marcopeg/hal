# 032 — Cron jobs and scheduled actions

## Status

draft (parent task — broken into sub-tasks)

## Summary

Add cron-job support to schedule tasks in time, including one-off timers and recurring schedules. Jobs survive hot reload and process restarts via file-based definitions. Jobs can be defined at three levels (system, project, user), execute prompts or programmatic handlers, and log every execution to disk.

## Shared Specifications

The sections below are the single source of truth for cross-cutting concerns. Sub-tasks reference this file rather than duplicating these definitions.

### Cron Source Format

There is no YAML-based cron format and no inline config key. All cron jobs are **file-based**, living in a `.hal/crons/` directory at the appropriate tier. Two file types are supported:

#### `.md` — Prompt-based cron

YAML frontmatter defines scheduling and targets; the Markdown body is the prompt submitted to the project engine.

```markdown
---
name: daily-summary             # optional; defaults to filename without extension
enabled: true
schedule: "0 9 * * *"           # cron expression (recurring) — OR —
# runAt: "2026-03-10T10:00:00Z" # ISO 8601 absolute datetime (one-off)

targets:
  - projectId: my-project       # which project engine to call (implicit for project/user tiers)
    userId: 123456              # optional; inject this user's context into the prompt
    flowResult: true            # optional; if true, send agent response to userId via Telegram DM
---

The prompt body goes here.
```

**`flowResult` rule:** `flowResult: true` requires `userId`. If `userId` is absent, this is a hard error at boot. During hot reload, log the error and skip the job — do not crash.

#### `.mjs` — Programmatic cron

ES module exporting a schedule declaration and a handler function.

```js
export const schedule = "0 9 * * *"; // recurring — OR —
// export const runAt = "2026-03-10T10:00:00Z"; // one-off

export const name = "daily-maintenance"; // optional; defaults to filename

export async function handler(ctx) {
  // ctx.config                    — full computed HAL configuration
  // ctx.projects                  — map keyed by project name/slug
  // ctx.projects[key].config      — project config slice
  // ctx.projects[key].bot         — Grammy Bot instance (full Telegram API access)
}
```

### Scheduling

- `schedule` — cron expression, recurring.
- `runAt` — ISO 8601 absolute datetime, one-off.
- A job with `runAt` in the past is **silently skipped** at load time (debug log written). To re-run, manually update `runAt` to a future datetime; the file watcher re-arms it automatically.
- Source files are never mutated by the scheduler after execution.

### Session Behaviour

| Cron tier    | `.md` execution          | `.mjs` execution                          |
|--------------|--------------------------|-------------------------------------------|
| System       | Anonymous `agent.call()` | Handler called with `CronContext`         |
| Project      | Anonymous `agent.call()` | Handler called with `CronContext`         |
| User         | User session (owning)    | N/A — `.mjs` at user tier is manual-only |

`userId` in `.md` targets injects user-scoped context data into the anonymous call; it does not change the session owner.

`runAs: <userId>` support for project/system crons (run with a specific user's context) is deferred to 032b.

### Output and Notification

- `flowResult: true` (`.md`) → agent response sent as Telegram DM to `userId`.
- `flowResult` absent or `false` → silent (log-only).
- User crons (032c) default to sending output back to the owning user.
- `.mjs` handlers manage their own output via `ctx.projects[key].bot`.

### Execution Logging

```
{tierRootDir}/.hal/logs/crons/{job-name}/{timestamp}.{job-name}.txt
```

Where `tierRootDir` is:
- System: `{configDir}`
- Project: `{projectCwd}`
- User: `{dataDir}/{userId}`

- One folder per job name. Timestamp-first filename for sorting; job name repeated for portability.
- Plain text. Respects `logging.persist` setting.
- Contents: job name, source file path, start/end timestamps, full output, exit status.

### Cron Source Tiers

| Tier    | File-based location                   | Mutable from chat |
|---------|---------------------------------------|-------------------|
| System  | `{configDir}/.hal/crons/*.{md,mjs}`   | No                |
| Project | `{projectCwd}/.hal/crons/*.{md,mjs}`  | No                |
| User    | `{dataDir}/{userId}/crons/*.md`       | Yes (owner only)  |

All tiers use the same `.md`/`.mjs` format. All sources are hot-reloaded via file watchers.

User-tier crons are `.md` only (`.mjs` can be placed manually but slash commands only generate `.md`).

### Lifecycle and Hot Reload

- **Startup:** load all tiers, validate, schedule all jobs with future triggers.
- **File watchers:** detect file add/edit/delete across all tiers and reschedule.
  - Add → schedule new job.
  - Edit → replace existing scheduled job.
  - Delete → remove scheduled job.
  - Validation error on hot reload → log error, skip, do not crash.
- **One-off jobs:** not rescheduled after firing. Re-arm by editing `runAt` to a future date.
- **Process restart:** all definitions re-read from disk.
