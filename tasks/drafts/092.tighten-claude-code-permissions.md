# 092 — Tighten Claude Code permissions

## Status

draft

## Summary

Harden HAL's Claude adapter so projects can enforce stricter Claude Code runtime
boundaries from `hal.config.yaml`, especially when a project `cwd` is a
subdirectory inside a larger git repository.

The current behavior sets process `cwd` and injects a prompt-level cwd
instruction, but this is not a hard boundary for Claude Code. Claude still
loads user/global settings and slash-command context unless explicitly disabled,
and Bash/tooling can escape the project directory.

This task introduces explicit Claude security controls at HAL config level and
documents the exact guarantees and limitations.

## Problem

Using Claude in `examples/claude-code-or` exposed parent-repo context and
commands/skills that are not part of the example project. This creates
confusion and weakens isolation expectations:

- users expect `project.cwd` to mean "stay inside this folder"
- Claude can still inherit global/user context
- prompt-level `engine.enforceCwd` is soft guidance, not enforcement
- strict isolation currently requires manual per-project Claude settings and CLI
  flag knowledge

## Goal

Provide a first-class, project-scoped way to tighten Claude behavior from HAL
config with safe defaults and explicit tradeoffs.

## Desired behavior

### New Claude-specific config surface

Add a `engine.claude` config group (global + project scopes, with normal HAL
inheritance rules) to control Claude launch hardening:

- `settingSources`: controls Claude settings source loading
  - value maps to `claude --setting-sources ...`
  - default keeps current behavior
- `disableSlashCommands`: disables Claude slash commands
  - maps to `claude --disable-slash-commands`
  - default `false`
- `allowedTools`: optional allow-list for Claude tools
  - maps to `--allowed-tools ...`
- `disallowedTools`: optional deny-list for Claude tools
  - maps to `--disallowed-tools ...`

### Boundary profile for strict mode

HAL docs should define a recommended strict profile for Claude subdirectory
projects:

- `settingSources: project,local`
- `disableSlashCommands: true`
- `disallowedTools: ["Bash"]`
- project `.claude/settings.json` with `allowedPaths: ["."]` (or absolute cwd)

### Clarify guarantees

Documentation must clearly distinguish:

- `cwd` process working directory (always applied)
- prompt-level instruction (`engine.enforceCwd`) as soft guard
- Claude-native settings/permissions as hard guard for file tools
- residual risk when Bash is enabled

## Scope

### In scope

- extend config schema/types for Claude-specific runtime controls
- support globals-to-project inheritance for all new options
- plumb new options into Claude adapter CLI args
- update docs/config and docs/engines/claude with precise behavior and examples
- add/update tests covering:
  - inheritance and overrides
  - arg building for Claude adapter
  - backward compatibility when options are unset

### Out of scope

- implementing OS-level sandboxing for Claude from HAL
- changing non-Claude engine behavior
- auto-writing `.claude/settings.json` files for projects

## Acceptance criteria

- HAL config supports `engine.claude.settingSources`,
  `engine.claude.disableSlashCommands`, `engine.claude.allowedTools`,
  `engine.claude.disallowedTools` at global and project scope.
- Project-level values override global values; missing project values inherit.
- Claude adapter passes corresponding CLI flags when configured.
- Existing Claude configs without `engine.claude.*` remain unchanged.
- Docs explain that `cwd` + `engine.enforceCwd` alone is not strict isolation.
- Docs include a step-by-step "tight permissions" recipe for subdirectory
  projects.
- Tests cover config resolution and adapter argument generation.

## Notes

- User report context: Claude running in `examples/claude-code-or` inherited
  parent-repo context/skills and appeared to operate beyond project boundary.
- This task is about reducing surprise and making secure behavior explicit and
  easy to configure.
