# 091 — Project command and skill visibility signature

## Status

draft

## Summary

Extend project-level custom command and skill authoring so they can expose the same visibility controls now available for built-in HAL commands.

Project `.mjs` commands should be able to optionally export:

- `enabled`
- `showInMenu`
- `showInHelp`

with all three defaulting to `true` when omitted.

Skills should continue supporting the existing boolean `telegram: true` shorthand, but should also support an object form:

- `telegram.showInMenu`
- `telegram.showInHelp`

The object form should allow a skill to remain enabled while being hidden from one or both Telegram-facing surfaces. Basically if either of the object based flags is true, the skill is enabled but only for that specific integration.

## Problem

After task 082, built-in HAL commands and npm-derived commands have explicit routing and visibility controls, but project `.mjs` commands and skills still use older, less granular exposure rules:

- project `.mjs` commands are auto-discovered and shown in menu/help by default with no file-level visibility signature
- skills still rely on `telegram: true` as a single combined exposure flag

This creates an inconsistent authoring model:

- built-ins can be enabled but hidden
- project commands cannot be temporarily disabled without deleting or renaming the file
- skills cannot independently control menu vs help visibility

## Goal

Unify the visibility model for all Telegram-exposed command surfaces while preserving backward compatibility for existing command and skill files.

## Desired behavior

### Project `.mjs` commands

Allow `.mjs` command modules to optionally export:

- `enabled`
- `showInMenu`
- `showInHelp`

Defaults when omitted:

- `enabled: true`
- `showInMenu: true`
- `showInHelp: true`

Examples:

```js
export const description = "Show project status";
export const enabled = false;
```

```js
export const description = "Show project status";
export const showInMenu = false;
export const showInHelp = true;
```

Semantics:

- `enabled: false` means HAL should not route that custom command at slash-command interception time
- `showInMenu: false` hides the command from Telegram `setMyCommands`
- `showInHelp: false` hides the command from `${HAL_COMMANDS}`

### Skills

Keep supporting the existing shorthand:

```yaml
telegram: true
```

This should continue to mean:

- `enabled: true`
- `showInMenu: true`
- `showInHelp: true`

Also support an object form:

```yaml
telegram:
  enabled: true
  showInMenu: false
  showInHelp: true
```

Object-form defaults when the object exists but a field is omitted:

- `enabled: true`
- `showInMenu: true`
- `showInHelp: true`

If `telegram` is omitted entirely, the skill should remain engine-only and not be exposed in Telegram surfaces, matching current behavior.

## Backward compatibility

### Project commands

Existing `.mjs` commands that only export `description` plus a handler must continue working unchanged.

Missing optional exports should behave as:

- `enabled: true`
- `showInMenu: true`
- `showInHelp: true`

### Skills

Existing skills with:

```yaml
telegram: true
```

must keep working exactly as before.

Existing skills with no `telegram` field must remain hidden from Telegram menu/help and direct skill invocation.

## Routing semantics

### Project commands

When a project or global `.mjs` command exports `enabled: false`:

- HAL must not treat that file as routable for slash-command interception
- resolution should continue to lower-precedence handlers:
  - project command
  - global command
  - skill
  - engine

This should mirror the built-in command semantics introduced by 082.

### Skills

For skills, `telegram.enabled: false` should mean:

- the skill is not exposed as a Telegram slash command
- it is not listed in menu/help
- it is not directly invokable through the Telegram slash-command shortcut path

The skill still exists for engine-side use in its normal engine skill directory.

## Scope

### In scope

- add optional metadata exports for `.mjs` project/global custom commands:
  - `enabled`
  - `showInMenu`
  - `showInHelp`
- add structured `telegram` object support to skills:
  - `telegram: true`
  - `telegram.enabled`
  - `telegram.showInMenu`
  - `telegram.showInHelp`
- preserve current defaults and compatibility behavior
- update command loading, menu publication, help rendering, and slash-command routing
- document the new authoring signatures for project commands and skills
- update examples to show the new syntax where useful

### Out of scope

- changing the typed return contract for `.mjs` commands
- changing built-in HAL command config again
- changing engine-side skill semantics outside Telegram exposure

## Acceptance criteria

- A project `.mjs` command may export `enabled`, `showInMenu`, and `showInHelp`.
- If those exports are omitted, the command behaves exactly as it does today.
- A project `.mjs` command with `enabled: false` is not routed by HAL and falls through to lower-precedence handlers.
- A project `.mjs` command with `showInMenu: false` is hidden from the Telegram menu.
- A project `.mjs` command with `showInHelp: false` is hidden from `${HAL_COMMANDS}`.
- Skills still support `telegram: true` with current behavior unchanged.
- Skills also support:
  - `telegram.enabled`
  - `telegram.showInMenu`
  - `telegram.showInHelp`
- `telegram: true` is treated as shorthand for all three values being `true`.
- A skill can be shown in help but hidden from the Telegram menu, or vice versa.
- A skill with `telegram.enabled: false` is not directly exposed through Telegram command routing.
- Existing project commands and skills continue to work without modification.
- Docs for project commands and skills explain the new signatures and defaults.
