# 053 — Resilient config hot-reload

## Status

draft

## Summary

Keep the process alive when configuration loading fails at startup or during a hot-reload cycle, and continue watching config files so the system can recover automatically when the configuration is fixed — without operator intervention.

## Problem

Currently a bad configuration is a hard failure: the process exits immediately. This means a single typo in `hal.config.yaml` (or `.env`) takes the entire service down and requires a manual restart after fixing the file. Task 015 added hot-reload for valid config changes, but that loop still exits the process on parse or validation errors.

The desired behaviour is: crash the bots (they won't work with a bad config), but keep the process alive, log the error clearly, and keep watching the config files. When the operator saves a corrected config, the watcher fires, the new config loads cleanly, and bots restart — no manual process restart needed.

## Goals

- Prevent `process.exit` on config load/validation failure (both at startup and during reload cycles).
- Keep the config file watcher running after a failed reload attempt.
- Restart bots automatically once a subsequent config change produces a valid configuration.
- Distinguish between a "degraded" state (process alive, no bots running) and a "healthy" state (bots running).
- Surface clear log messages when entering/leaving the degraded state.

## Scope

- Wrap config loading in the hot-reload loop so errors are caught and logged rather than propagated to an unhandled rejection or explicit `process.exit`.
- Introduce a "degraded" state: all bots stopped, watcher still active, waiting for next file change.
- On each config file change: attempt reload; on success → start bots and transition to healthy; on failure → stay degraded, log the error.
- Handle the initial startup case: if config is invalid on first load, enter degraded state immediately (do not exit) and wait for a corrected file.
- SIGINT/SIGTERM still stop everything and exit cleanly regardless of state.

## Deliverables

- Updated hot-reload loop in `src/cli.ts` (or equivalent orchestration layer) with error-resilient config loading.
- Clear log output distinguishing degraded vs healthy state transitions.
- Notes confirming correct behaviour for: bad config at startup, bad config during reload, fix applied while degraded.

## Acceptance Criteria

- Process does not exit when `hal.config.yaml` contains invalid YAML, unknown fields, or fails Zod validation.
- Log clearly reports the error and indicates the system is waiting for a corrected config.
- Saving a valid config while the process is in degraded state causes bots to start automatically.
- SIGINT/SIGTERM exit cleanly in both healthy and degraded states.
- No regression in the normal hot-reload flow (valid → valid config change restarts bots correctly).
