# 053 — Resilient config hot-reload

## Status

draft

## Summary

Keep the process alive when a **hot-reload** cycle fails (config load/validation or bot startup), and keep the config watcher running so the next config file change can trigger a fresh reload and recover without operator restart. First load from CLI remains a hard failure on any error.

## Problem

Task 015 added hot-reload: on config file change, bots are stopped, config is reloaded, and bots are started again. If reload fails (invalid YAML, Zod validation, or bot startup failure), the process currently exits. A single typo or misconfiguration after the process is already running takes the whole service down and requires a manual restart.

The desired behaviour for **reload only**: on failed reload, do not exit — stop the bots, log the error clearly, stay in a "degraded" state with the config watcher still active. When the operator saves a corrected config, the watcher fires, the next reload succeeds, and bots start again. First load from CLI continues to exit on error (no resilient behaviour at startup).

## Context and constraints

- **Startup (first load from CLI):** Unchanged. Invalid config or bot startup failure at initial run → log error and `process.exit(1)`. No degraded state at startup.
- **Hot-reload path only:** When a config file change is detected, the process stops current bots, attempts to reload config and start bots. If **either** config load/validation **or** bot startup fails → do not exit; enter/remain in degraded state (no bots), keep the config file watcher running, log the error. A subsequent config file change will trigger another reload attempt.
- **Bot startup failure during reload:** Treated the same as config failure — stay alive, degraded, watcher keeps running, retry on next file change.

## Goals

- Prevent `process.exit` on reload failure (config load/validation or bot startup).
- Keep the config file watcher running after a failed reload attempt.
- Restart bots automatically when a subsequent config change leads to a successful reload.
- Distinguish between "degraded" (process alive, no bots, watcher active) and "healthy" (bots running).
- Surface clear log messages when entering/leaving the degraded state.
- Leave startup behaviour unchanged: first load errors still exit the process.

## Scope

- In the hot-reload callback only: wrap the reload cycle (load config + run bots) in error handling so failures are caught and logged, not propagated to unhandled rejection or `process.exit`.
- Introduce a degraded state for the reload path: after a failed reload, no bots, watcher still active, waiting for the next file change.
- On each config file change: attempt reload; on success → start bots and transition to healthy; on failure → stay degraded, log the error.
- **Out of scope:** Changing startup behaviour. Invalid config or bot startup failure on first load continues to exit the process.
- SIGINT/SIGTERM stop all current bots (if any), close the watcher, and exit cleanly in both healthy and degraded states.

## Deliverables

- Updated hot-reload loop in `src/cli.ts` (or equivalent) with error-resilient reload (catch config and bot-startup failures, no exit).
- Clear log output distinguishing degraded vs healthy state transitions on reload.
- Notes confirming: startup still exits on error; failed reload leaves process degraded with watcher; fix applied while degraded causes bots to start on next change.

## Acceptance Criteria

- **Startup:** Invalid config (e.g. invalid YAML, unknown fields, Zod validation) or bot startup failure at first load still causes the process to exit (no change).
- **Reload:** If config is invalid or bot startup fails during a hot-reload cycle, the process does not exit; it logs the error, enters/remains in degraded state (no bots), and keeps the config watcher running.
- Log clearly reports the error and indicates the system is waiting for a corrected config when in degraded state.
- Saving a valid config (and fixing any bot-startup issues) while degraded causes the next reload attempt to start bots automatically.
- SIGINT/SIGTERM exit cleanly in both healthy and degraded states (watcher closed, no bots).
- No regression in the normal hot-reload flow (valid → valid config change restarts bots correctly).
