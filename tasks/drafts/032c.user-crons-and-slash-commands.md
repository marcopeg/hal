## Summary

Add user-level cron support and the full set of `/cron_*` slash commands. Users can create, list, edit, pause, resume, and remove their own crons via AI-assisted chat commands. User crons are stored as `.md` files in the user's data directory and hot-reloaded.

## Dependencies

- [032a — Core scheduler + system-level crons](./032a.core-scheduler-and-system-crons.md) must be completed first.
- [032b — Project-level crons + logging refinement](./032b.project-crons-and-logging.md) should be completed first (provides `runAs` session and multi-project scheduler patterns).

## Scope

### In scope

- User-level cron source:
  - **File-based:** `{dataDir}/{userId}/crons/*.md`, loaded at boot for known users.
  - Hot-reloaded via file watcher on `{dataDir}/*/crons/` directories.
  - Mutable only by the owning user through slash commands.
  - `.mjs` files may exist at the user tier (manually placed) but are not creatable via slash commands.
- `user` session mode for user-defined `.md` crons: execute with the owning user's session context.
- User crons default to sending output back to the owning user (`flowResult: true` with the owning `userId` injected automatically).
- Slash commands (built-in, registered like other HAL commands):
  - `/cron_add` — AI-assisted creation.
  - `/cron_list` — programmatic listing of all visible crons (user + project + system).
  - `/cron_edit` — AI-assisted editing.
  - `/cron_remove` — delete a user-owned cron.
  - `/cron_pause` — set `enabled: false` in the cron's frontmatter.
  - `/cron_resume` — set `enabled: true` in the cron's frontmatter.
- AI-assistance skill/context prompt: a dedicated skill file describing the `.md` frontmatter format and guiding the engine to produce valid cron definitions.
- Add a `commands.cron` config option (with `enabled` flag) to control whether cron commands are registered.

### Out of scope

- Modifying project or system crons from chat (these remain immutable via slash commands).
- Generating `.mjs` cron files via slash commands.

## Slash Command Details

### `/cron_add <natural-language description>`

1. User sends something like: `/cron_add every morning at 9am check git status and tell me`.
2. Bot constructs a prompt combining the user's input + the AI-assistance skill describing the `.md` frontmatter format and examples.
3. Engine generates a valid `.md` cron definition (frontmatter + prompt body).
4. Bot validates the output against the user-tier `.md` frontmatter Zod schema.
5. Bot writes the file to `{dataDir}/{userId}/crons/{name}.md`.
6. File watcher picks up the new file and schedules it.
7. Bot confirms with a summary of the created cron.

**Note:** consider adding a confirmation step where the bot shows the generated definition before writing.

### `/cron_list`

Programmatic (no AI). Displays a formatted list of all crons visible to the user:
- **User crons** — owned by the requesting user.
- **Project crons** — defined for the current project.
- **System crons** — global definitions.

Each entry shows: name, tier (user/project/system), schedule/timing, prompt summary or handler filename, enabled status, next scheduled run.

### `/cron_edit <cron-name> <change description>`

1. Bot loads the existing `.md` cron file by name (must be user-owned).
2. Constructs a prompt with the current definition + user's change request + AI-assistance skill context.
3. Engine produces the updated `.md` content (frontmatter + body).
4. Bot validates and overwrites the file.
5. File watcher detects the change and reschedules.
6. Bot confirms with a summary of what changed.

### `/cron_remove <cron-name>`

Deletes the `.md` file from `{dataDir}/{userId}/crons/`. Rejects if the cron is project-level or system-level.

### `/cron_pause <cron-name>` / `/cron_resume <cron-name>`

Reads the user's `.md` cron file, sets `enabled: false` or `true` in the YAML frontmatter, writes back. File watcher handles reschedule.

## Requirements

- Implement user cron loading: on boot, scan `{dataDir}/*/crons/` for existing user `.md` (and optionally `.mjs`) files.
- Add file watcher on `{dataDir}/*/crons/` to detect user cron changes.
- Implement `user` session mode: execute `.md` prompt with the owning user's session context.
- Implement auto-injection of `flowResult: true` + owning `userId` for user crons that don't explicitly set these.
- Register all six `/cron_*` commands as built-in commands (enabled via `commands.cron.enabled` config flag).
- Create the AI-assistance skill/context prompt file describing the `.md` cron format with examples and edge cases.
- `/cron_add` and `/cron_edit` must validate AI-generated output against the user-tier `.md` frontmatter Zod schema before writing.
- `/cron_list` must aggregate crons from all three tiers with clear tier labeling and next-run info.
- `/cron_remove`, `/cron_pause`, `/cron_resume` must enforce user-ownership (reject non-user crons with a clear message).

## Acceptance Criteria

- `/cron_add` creates a valid `.md` file in the user's crons directory via AI assistance.
- The created cron is automatically picked up by the file watcher and scheduled.
- `/cron_list` shows user, project, and system crons with status and next-run info.
- `/cron_edit` modifies an existing user cron via AI and the scheduler updates.
- `/cron_remove` deletes a user cron file and the scheduler removes it.
- `/cron_pause` and `/cron_resume` toggle `enabled` in the frontmatter and the scheduler reacts.
- Attempting to remove/edit/pause a project or system cron via slash command is rejected with a clear message.
- User crons execute with the user's session by default and output is sent back to the owning user.
- User crons survive process restart (reloaded from files on boot).
- Invalid AI-generated cron content is caught by schema validation and the user is informed.
- The cron commands appear in `/help` and the Telegram command menu when enabled.

## Notes

- The AI-assistance skill prompt quality is critical for `/cron_add` and `/cron_edit` — it should include full frontmatter field descriptions, valid examples, and common edge cases (e.g. `flowResult` without `userId`).
- User-tier `.md` frontmatter omits `projectId` in targets (implicit — the cron belongs to the user's current project context). Confirm during planning whether user crons are project-scoped or run across all user-accessible projects.
- One-off user crons (`runAt`) that have passed are skipped on reload. The user must `/cron_edit` them to a future date to re-arm.
