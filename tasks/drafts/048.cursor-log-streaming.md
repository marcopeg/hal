# 048 — Cursor log streaming

## Status

draft

## Summary

Implement streaming log updates for the Cursor engine, sending incremental progress messages to Telegram as the CLI runs rather than waiting for full completion.

## Problem

The Cursor adapter currently buffers all output until the process exits. Users see no feedback during long-running sessions. Claude Code already streams progress via `onProgress` callbacks — Cursor needs an equivalent approach.

## Output format support (JSON / stream-json)

- **CLI support:** Cursor Agent CLI supports `--output-format json` (single JSON object at completion with `result`, `session_id`, timing) and `--output-format stream-json` (NDJSON with real-time events). Both require `--print`. See [Cursor CLI output format](https://docs.cursor.com/cli/reference/output-format).
- **Current use in HAL:** We do **not** use these flags. The Cursor adapter passes `--print` but no `--output-format`; stdout is consumed as plain text (and we optionally parse lines for `session_id` when `engineSession === "user"`).
- **Recommendation:** Using `--output-format stream-json` would provide real-time events suitable for progress updates and a clear final result. Using at least `--output-format json` would give a single final object with a `result` field, improving reliability of the final response. For log streaming, adopting `stream-json` is recommended so we can map event types to incremental Telegram messages.

## Goals

- Investigate what output Cursor emits to stdout/stderr while running (structured JSON, plain text, progress events, etc.).
- Determine whether Cursor provides a streaming or event-driven output mode (prefer adopting `stream-json` — see above).
- Identify meaningful log lines or events that can be surfaced as Telegram progress update messages.
- Implement `onProgress` callback integration in the Cursor adapter.

## Scope

- Research Cursor CLI output format and any available streaming flags; prefer adopting `--output-format stream-json` (see Output format support above).
- Prototype event (or log-line) parsing to extract actionable progress signals.
- Implement incremental Telegram message updates via the existing `onProgress` mechanism.
- Preserve existing buffered behavior as a fallback.

## Deliverables

- Findings on Cursor CLI output structure during execution.
- Updated `src/engine/adapters/cursor.ts` with streaming/progress support.
- Test notes covering typical Cursor runs with progress updates visible in Telegram.

## Acceptance Criteria

- Telegram users see incremental status messages while a Cursor session is running.
- The final response is still delivered correctly after streaming completes.
- Fallback to buffered mode works when no streaming signals are detected.
