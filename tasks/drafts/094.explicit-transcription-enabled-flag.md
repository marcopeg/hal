# 094 — Explicit `transcription.enabled` flag

## Status

draft

## Summary

Add an explicit `transcription.enabled` boolean so voice transcription is opt-in instead of being implicitly active whenever the voice handler runs.

---

## Goals

- Introduce a clear on/off switch for transcription.
- Make the default behavior disabled unless the user enables it.
- Support the flag at both globals and project scope with normal inheritance.
- Keep the change small and low-risk.

## Problem

HAL currently has transcription mode/model settings, but no explicit enablement flag. In practice that makes transcription feel implicitly on, which is a poor default for a feature that can require local dependencies, CPU time, and potentially future remote-service credentials.

The config should make the intent explicit: transcription is either enabled for a project or it is not.

## Desired Behavior

- `transcription.enabled` exists at globals and project scope.
- Default is `false`.
- Project-level value overrides globals when set; otherwise it inherits globals.
- When disabled, voice messages should not enter the transcription pipeline.
- Existing `transcription.mode` and `transcription.model` remain meaningful only when transcription is enabled.

## Scope

- Extend config schema, types, and resolution for `transcription.enabled`.
- Update the voice handler to gate transcription on the resolved flag.
- Update docs and config examples to show the explicit enablement.

## Out of Scope

- Changing transcript UX modes.
- Adding remote transcription providers.
- Reworking the rest of the voice pipeline beyond the enable/disable gate.

## Requirements

- The flag must be supported at both globals and project scope.
- The default must be `false`.
- The change must preserve the existing inheritance model used by other config options.
- Disabled transcription must fail cleanly with the existing "voice messages are not configured" style user response or a refined equivalent.

## Acceptance Criteria

- `transcription.enabled` is accepted in config at both scopes.
- Resolved config defaults it to `false`.
- Project override semantics work correctly.
- Voice transcription does not start when the flag is `false`.
- Docs and example config show how to enable transcription explicitly.

## Notes

- Keep this task intentionally small. It is a prerequisite/foundation for follow-up voice work, not a broad redesign.
