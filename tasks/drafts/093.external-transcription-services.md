# 093 — External transcription services for voice messages

## Status

draft

## Summary

Investigate and support external transcription providers for HAL voice messages, in addition to the current local Whisper flow. The first concrete target is a remote service such as WISPR remote, but the design should avoid baking the whole voice pipeline into a single vendor-specific path.

---

## Goals

- Support at least one external transcription service for voice messages.
- Keep local transcription as a supported option.
- Introduce a clear config model for selecting the transcription backend per globals/project.
- Preserve the existing voice UX modes (`confirm`, `inline`, `silent`) independently from the transcription backend.
- Document the privacy and operational tradeoffs between local and remote transcription.

## Problem

HAL currently assumes a local Whisper-based transcription flow. That works well for privacy and offline/local-first usage, but it is a narrow integration point:

- users may prefer a managed remote service for better speed, lower local setup friction, or different quality characteristics;
- local Whisper requires local dependencies (`ffmpeg`, Whisper assets/build), which raises setup cost;
- the current transcription path is not structured as a provider abstraction, so adding a remote service later will be more invasive than it needs to be.

If HAL is going to support voice as a first-class input path, it should be able to route transcription through a configurable backend instead of hard-coding one implementation.

## Desired Behavior

- HAL supports a transcription backend abstraction.
- Config can choose between local transcription and a remote provider.
- The first remote provider explored should be WISPR remote or an equivalent external service.
- Voice handler behavior above the transcription layer stays consistent:
  - transcription enablement still gates voice processing;
  - transcript UX mode remains `confirm`, `inline`, or `silent`;
  - downstream engine dispatch should not care which transcription backend produced the text.
- Remote-provider failures should surface clean, user-friendly errors.

## Scope

- Define the config shape for transcription backend selection.
- Refactor the transcription module so backend-specific logic is isolated behind a stable interface.
- Add one external provider implementation path (target: WISPR remote, pending API/SDK validation).
- Wire provider selection through globals and per-project config with normal inheritance.
- Update docs for setup, config, and privacy implications.

## Out of Scope

- Supporting many remote providers in the first pass.
- Building provider failover / fallback chains unless clearly needed.
- Designing a generic media-processing framework beyond what transcription needs.
- Solving billing/account provisioning UX beyond the config and docs needed to use the provider.

## Requirements

- The transcription backend must be configurable, not hard-coded.
- Local transcription must remain supported.
- The provider selection config must work at both globals and project scope.
- Existing transcript UX modes must continue to work regardless of backend.
- Provider-specific credentials/secrets must stay out of committed config examples when appropriate.
- Errors from remote transcription must be mapped to user-meaningful messages and useful logs.

## Acceptance Criteria

- HAL has a clear abstraction for transcription backends.
- Config can select local vs remote transcription backend.
- At least one remote backend is supported end-to-end for voice messages.
- Existing voice flows still work with the local backend.
- Docs explain how to configure the remote backend and what tradeoffs it introduces.
- The implementation does not couple voice UX mode selection to one specific backend.

## Notes

- This task should align with task 094 so backend selection only matters when transcription is explicitly enabled.
- Provider naming/config details should be validated against the chosen service's real API before implementation; do not guess the final wire format too early.
