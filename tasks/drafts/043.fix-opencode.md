# 043 — Fix OpenCode engine (default free services)

## Status

draft

## Summary

Debug and fix the OpenCode engine integration so it works correctly with the
default free services (e.g. OpenCode Zen, `opencode/gpt-5-nano`). Users
currently see bad or failing responses when using OpenCode with the default
free tier.

## Context

- OpenCode was added in task 010. The adapter runs `opencode run [-m model]
  [-c] "<prompt>"` with `OPENCODE_DISABLE_CLAUDE_CODE_PROMPT=true`, and HAL
  defaults to `opencode/gpt-5-nano` when no model is set (see
  `src/default-models.ts`).
- The adapter is non-streaming: it buffers stdout and returns it on process
  exit; stderr is logged and surfaced as error text on non-zero exit.
- Free services may have different auth requirements, rate limits, or CLI
  behavior than paid providers; the exact failure mode (empty output, wrong
  content, exit code, auth errors) is to be captured during debugging.

## Goals

- Reproduce the bad/failing behavior with the default free setup (no or minimal
  config: `engine.name: "opencode"`, default model or explicit
  `opencode/gpt-5-nano`).
- Identify root cause(s): e.g. CLI invocation, model name, env vars, auth for
  free tier, output parsing, or upstream OpenCode/Zen changes.
- Fix the adapter or config so that OpenCode works reliably with the default
  free services and document any requirements (auth, env, or config) for free
  tier.
- Optionally improve error reporting so that auth/API/rate-limit failures from
  the CLI are clearly visible to the user (e.g. in Telegram).

## Requirements (to be refined after debugging)

- [ ] Reproduce and document the current failure (steps, config, CLI version,
  observed output/exit code).
- [ ] Implement fix(es) in `src/engine/adapters/opencode.ts` and/or config
  (default model, env) so default free usage works.
- [ ] If free tier requires specific auth or setup, document it in
  `docs/engines/opencode/README.md` and in error messages or logs where
  relevant.
- [ ] Ensure HAL default model for OpenCode (`opencode/gpt-5-nano` or
  successor free Zen model) remains valid and is passed correctly to the CLI.
- [ ] (Optional) Surface clearer errors to the user when OpenCode fails (e.g.
  stderr or exit reason in the Telegram response).

## Out of scope

- Adding streaming support for OpenCode (separate task if desired).
- Supporting every OpenCode provider or paid tier; focus is default free
  services.
