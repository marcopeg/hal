# 054 — Changelog generation from conventional commits

## Status

draft

## Summary

Investigate how to automatically generate a changelog based on the conventional commits used in this project, and determine the best tooling and workflow for producing and maintaining a `CHANGELOG.md`.

## Problem

The project follows the Conventional Commits specification, but there is no automated process to produce a changelog from that commit history. Releases and notable changes must be documented manually, which is error-prone and easy to skip.

## Context and constraints

- **Conventional commits**: Used in practice but not enforced. Enforcing them (e.g. commitlint or similar) is desired so bad commit messages cannot be committed accidentally; treat this as a prerequisite or explicit follow-up task in the findings.
- **Release workflow**: Releases are currently manual and local. The `deploy` script (`npm run deploy`) should evolve to:
  1. Ensure git working tree is clean (else fail early before publish).
  2. Run `npm login` (or equivalent auth).
  3. Generate changelog and commit it (so the release commit includes the updated `CHANGELOG.md`).
  4. Publish (e.g. `npm publish` or existing flow).
  Changelog generation must integrate into this `deploy` flow, not CI or a standalone one-off script.
- **Versioning**: Patch-only for now. A future task may introduce proper semver and/or move deploy to CI (e.g. GitHub Actions). Research should assume current patch-based, local-release workflow; semantic-release and tag-based workflows are out of scope for this task.

## Goals

- Identify suitable tools for auto-generating a changelog from conventional commits (e.g. `conventional-changelog`, `release-please`, `git-cliff`, `changie`).
- Evaluate how each tool fits integration into the `deploy` script (pnpm, Node, local run).
- Decide on format and placement for the generated `CHANGELOG.md`.
- Recommend how changelog generation fits the deploy steps (generate then commit before publish).

## Scope

- Research phase only: no implementation required until a tool and approach are chosen.
- Survey at least two or three candidate tools and note pros/cons for this project.
- Assume versioning stays patch-based for this task; do not assume semantic-release or tag-based releases.

## Deliverables

- A short findings document or notes section within this file summarising evaluated tools and the recommended approach.
- A decision on tooling and workflow that fits the target `deploy` flow (clean git → login → changelog + commit → publish), ready to be turned into a Ready task with a concrete implementation plan.
- Explicit note on enforcing conventional commits (e.g. commitlint) as a prerequisite or follow-up task.

## Acceptance criteria

- At least two changelog tools are evaluated with notes on fit for this project and for the `deploy`-script integration.
- A clear recommendation is recorded, including how changelog is generated and committed within the deploy flow.
- Enforcing conventional commit messages is identified and recorded as a prerequisite or follow-up task.
