# 054 — Changelog generation from conventional commits

## Status

draft

## Summary

Investigate how to automatically generate a changelog based on the conventional commits used in this project, and determine the best tooling and workflow for producing and maintaining a `CHANGELOG.md`.

## Problem

The project follows the Conventional Commits specification, but there is no automated process to produce a changelog from that commit history. Releases and notable changes must be documented manually, which is error-prone and easy to skip.

## Goals

- Identify suitable tools for auto-generating a changelog from conventional commits (e.g. `conventional-changelog`, `release-please`, `git-cliff`, `changie`).
- Evaluate how each tool fits the project's workflow (pnpm, GitHub, CI/CD if any).
- Decide on a format and placement for the generated `CHANGELOG.md`.
- Determine whether changelog generation should be a manual script, a git hook, or part of a release workflow.

## Scope

- Research phase only: no implementation required until a tool and approach are chosen.
- Survey at least two or three candidate tools and note pros/cons for this project.
- Consider integration with versioning (`npm version`, semantic-release, or manual tagging).

## Deliverables

- A short findings document or notes section within this file summarising evaluated tools and the recommended approach.
- A decision on tooling and workflow, ready to be turned into a Ready task with a concrete implementation plan.

## Acceptance Criteria

- At least two changelog tools are evaluated with notes on fit for this project.
- A clear recommendation is recorded, including how and when the changelog would be generated.
- Any prerequisites (e.g. enforcing commit message linting) are identified and noted as potential follow-up tasks.
