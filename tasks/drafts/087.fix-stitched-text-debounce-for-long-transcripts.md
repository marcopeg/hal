# 087 - Fix stitched text debounce for long transcripts

**Status**: draft

**Plan**: [087.fix-stitched-text-debounce-for-long-transcripts.plan.md](./087.fix-stitched-text-debounce-for-long-transcripts.plan.md)

## Problem

HAL already has text-message debounce logic, but very long transcript-style inputs can still be processed multiple times instead of being forwarded as one logical prompt. In practice, when a user sends a long transcript that arrives as multiple rapid text messages, HAL appears to buffer parts, yet it still ends up running downstream processing more than once instead of producing one stitched request to the engine/alarm flow.

The current implementation in `src/bot/handlers/text.ts` activates buffering only when the first observed text message has `messageText.length >= 4096`, or when a buffer is already active. That heuristic is too narrow for the real-world behavior now being observed.

## Investigation summary

### Current implementation

- `src/bot/handlers/text.ts`
  - `createTextHandler()` owns the text debounce flow.
  - `buffers: Map<number, BufferEntry>` stores in-flight stitched messages per user.
  - `flush(userId)` sorts parts by `messageId`, concatenates them with `join("")`, and calls `dispatchMessage(...)`.
  - The debounce trigger is currently:
    - `if (messageText.length >= 4096 || existing) { ... }`
- `src/bot.ts`
  - Creates and shares `debounceActiveUsers` between the text handler and rate-limit middleware.
- `src/bot/middleware/rateLimit.ts`
  - Skips rate-limit counting for users already inside an active debounce buffer.
- `src/config.ts`
  - Exposes `debounce.windowMs` with a default of `300`.

### Relevant prior task

- `tasks/completed/056.debounce-messages.md`
  - The original task explicitly assumed that Telegram split long pasted text into consecutive updates and that the first part would be `>= 4096`.
  - The current implementation matches that specification closely.

### Online research summary

The online research is useful, but it is also noisy and partially contradictory. The task should explicitly separate **well-supported facts** from **assumptions that must be validated empirically**.

#### Well-supported findings

- Telegram Bot API and major ecosystem libraries consistently document a **4096-character limit for bot-sent text messages**.
  - Official Telegram Bot API reference: `sendMessage` text is limited to 1-4096 characters.
  - Library/ecosystem references mirror this limit:
    - `python-telegram-bot` constants/documentation
    - `gramjs` documentation
    - community tooling around Telegram message splitting
- Telegram does **not** automatically reconstruct multiple messages into one logical input for bot developers. If a bot wants one logical prompt, it must do that reconstruction in application code.
- Telegram/bot ecosystem guidance for long content focuses on:
  - manually splitting **outgoing** long bot replies,
  - preserving ordering and formatting when split,
  - using files/documents as an alternative when text is too large for normal chat-message handling.

#### Ambiguous or conflicting findings

- Web results conflict on what happens when a **user** sends text longer than `4096` to a bot:
  - some sources claim the client blocks or truncates it,
  - some claim the full content can still reach the bot,
  - some imply client-side or workflow-level splitting may happen.
- I did **not** find strong official Telegram documentation that clearly guarantees any specific incoming behavior for over-limit user text in a bot chat.

#### Practical conclusion for HAL

Because incoming-user behavior is not documented clearly enough to rely on exact-threshold assumptions, HAL should solve this at the **process level**:

- treat rapid consecutive text updates as candidate parts of one logical input,
- stitch them deterministically in arrival/order terms,
- send exactly **one** reconstructed prompt to the engine,
- and avoid coupling correctness to whether Telegram, a client, or a user happened to split the text in a particular way.

### Why this is likely failing

- The current logic only opens the debounce buffer when the first chunk reaches the exact threshold that task 056 assumed.
- If the first delivered message is below that threshold, HAL dispatches it immediately and no buffer is created.
- Any following chunks are then treated as separate prompts unless they happen to arrive while an already-open buffer exists.
- This makes the feature brittle for real transcript-like multi-message inputs, including cases where:
  - the client/user sends several consecutive messages quickly rather than one exact-`4096` split,
  - the first received chunk is below `4096`,
  - the observed flow is a mix of one immediate dispatch plus later buffered parts, causing multiple downstream executions.

### Important discovery to validate during the fix

External Telegram references consistently document a **4096-character message limit** for bots, but they do **not** clearly support the assumption that Telegram automatically splits an overlong user text into multiple bot updates. That means task 056 may have encoded the wrong trigger model. The bug fix should therefore target the observable behavior — rapid multi-message transcript input that should be treated as one logical prompt — rather than relying only on the old "first chunk is 4096" assumption.

## Goal

Make text debounce reliably reconstruct long transcript-style user input into **one single process-level input** and exactly one downstream engine/alarm execution, even when the first received message does not satisfy the current `>= 4096` trigger.

## Scope

### In scope

- Investigate and reproduce the current failure mode for rapid multi-message text input.
- Include targeted online research notes about Telegram/Bot API long-message behavior and use those findings to shape the implementation.
- Redesign the debounce activation heuristic in `src/bot/handlers/text.ts`.
- Preserve per-user buffering and ordering by `message_id`.
- Ensure the downstream engine is executed exactly once per logical stitched input.
- Reconstruct the full logical prompt before it enters the engine execution path.
- Add automated coverage for the text debounce path.

### Out of scope

- Voice-message debounce or transcript confirmation flow.
- Changes to photo/document handlers.
- Outgoing message chunking (`sendChunkedResponse`) except where needed for tests.

## Proposed direction

- Replace the brittle "`first chunk >= 4096`" activation rule with behavior based on the actual arrival pattern of messages.
- Prefer a solution that is grounded in observable Telegram updates, for example:
  - buffer the first candidate text message for a short quiet window before dispatch,
  - or introduce a more reliable heuristic for detecting "continuation" messages than exact length alone.
- Treat the engine boundary as the correctness boundary:
  - multiple Telegram updates may be acceptable at transport level,
  - but HAL must collapse them into one logical prompt before calling `ctx.engine.execute(...)`.
- Preserve the existing UX as much as possible:
  - slash commands still bypass debounce,
  - short normal messages should not feel noticeably slower,
  - rate limiting should still count one logical stitched message, not every buffered fragment.

## Online research deliverable

The fix task should include a short research note in the final implementation or handoff summary covering:

- what Telegram officially documents about long messages,
- what remains ambiguous for incoming user-to-bot text,
- why HAL intentionally reconstructs long transcript-like input at the application layer instead of trusting client/API splitting behavior,
- and whether a document/file upload fallback should be recommended for extremely large transcripts.

## Acceptance criteria

- [ ] A rapid multi-message transcript input is forwarded downstream exactly once as one stitched prompt.
- [ ] The bug is fixed even when the first received chunk is **below** `4096` characters.
- [ ] Messages are stitched in `message_id` ascending order.
- [ ] The reconstructed text is what reaches the engine execution boundary; the engine is not called once per part.
- [ ] The debounce implementation does not regress normal slash-command handling.
- [ ] Rate-limit behavior remains correct for buffered text input.
- [ ] A targeted automated test covers the reproduced bug scenario and fails before the fix.
- [ ] The task documents the final trigger model clearly so the code and backlog description stay aligned.
- [ ] The implementation notes include a concise evidence-based summary of Telegram long-message behavior and any remaining uncertainty.
