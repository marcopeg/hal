# 042 — Users map

## Status

draft

## Summary

Introduce a top-level **users** map (or `globals.users`) keyed by user ID:
`users.{id}.*`. Each entry describes one allowed user in the system. Possible
uses: centralise who is allowed; move ACL to user level by granting access to
specific projects per user; and allow per-user settings (default engine,
preferences, which commands are visible) so that config becomes fine-grained
(user × project).

## Context

- Access is currently **project-centric**: each project has
  `access.allowedUserIds` (and optional `dangerouslyAllowUnrestrictedAccess`),
  with global `globals.access` as fallback when a project omits `access`.
- There is no first-class “user” object: user IDs only appear inside project
  access lists. Per-user defaults (engine, model, commands) are not in config.
- A **users** map would make “user” a first-class entity and could drive both
  who can access what and what each user sees (engine, settings, commands).

## Possible shape (to be decided in refinement)

- **Placement**: top-level `users` vs `globals.users`. Keys: Telegram user ID
  (string or number; consistent with existing `allowedUserIds`).
- **Per-user fields (candidates)**:
  - **Access**: list of project keys (or “*”) this user can use → ACL moves to
    user level; projects could then be “closed by default” and opened per user.
  - **Default engine / model**: override globals for this user.
  - **Commands**: which slash commands this user gets (whitelist/blacklist or
    full list).
  - **Other**: name/label (for logs or UI), optional settings.
- **Interaction with existing access**: if we keep both project-level
  `access.allowedUserIds` and user-level “projects I can use”, refinement must
  define precedence and whether we deprecate one side.

## Refinement: pros and cons to debate

During refinement, the following should be explicitly decided.

### 1. ACL direction: project-centric vs user-centric

| Approach | Pros | Cons |
|----------|------|------|
| **Keep project-centric** (current): project lists allowed users | Simple, one place per project; no need to touch user entries when adding a project. | Adding one user to many projects = repeat their ID in each project. |
| **User-centric**: user lists allowed projects | One place per user; “grant Alice access to A, B, C” in one block. | Adding one project for many users = touch many user entries (or keep a hybrid). |
| **Hybrid**: both; e.g. project has allowlist and user has “projects” override | Flexible: project can say “these users” and user can further restrict “I only use P1, P2”. | More precedence rules; config can be redundant or confusing. |

### 2. Placement: `users` vs `globals.users`

| Option | Pros | Cons |
|--------|------|------|
| **Top-level `users`** | Short, clear; users are a top-level concept. | One more root key; config root grows. |
| **`globals.users`** | Keeps “global” things under `globals`; consistent with `globals.access`. | Slightly longer; “globals” might imply “defaults” rather than “user registry”. |

### 3. Scope of per-user config

| Broad scope (engine, model, commands, access) | Narrow scope (only access / project list) |
|----------------------------------------------|------------------------------------------|
| Single place for all user-specific behaviour; fine-grained user/project experience. | Simpler schema and less to maintain; engine/model/commands can stay global or project-level only. |
| Risk of duplication with project-level and global defaults; precedence gets complex. | Less power; per-user preferences might still need another mechanism later. |

### 4. Backward compatibility

- If we introduce user-level “allowed projects”, do we keep project-level
  `allowedUserIds`? If yes: how do we merge (e.g. “allowed if in project list
  OR in user’s project list”)?
- Or do we deprecate project-level allowlist in favour of “only users in
  `users` with that project in their list” (breaking change)?

### 5. Validation and UX

- Should every ID in any project’s `allowedUserIds` be required to have an
  entry in `users` (so `users` is the source of truth for “who exists”)?
- Or is `users` optional: only needed when you want per-user settings or
  user-centric ACL; existing project allowlists keep working as today?

## Requirements (to be firmed in refinement)

- [ ] Decide: top-level `users` vs `globals.users`.
- [ ] Define schema for `users.<id>` (at least: which projects the user can
  access; optionally: default engine, model, commands, other settings).
- [ ] Define precedence with existing `access.allowedUserIds` and
  `dangerouslyAllowUnrestrictedAccess` (merge vs replace; deprecations).
- [ ] Implement resolution so that at runtime we have a clear “effective
  access” and “effective user settings” per (user, project).
- [ ] Update auth middleware (and any command/settings resolution) to use the
  new model.
- [ ] Docs and examples; consider migration path for existing configs.

## Out of scope (unless refinement brings them in)

- Storing user preferences in persistent storage (e.g. DB or `.hal/users/`)
  instead of config; this task is config-only.
- Per-user rate limits or quotas.
- Multiple identities or roles per user (e.g. “admin” vs “member”).
