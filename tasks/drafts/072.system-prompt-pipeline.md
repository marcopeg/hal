# 072 — System prompt pipeline

## Status

draft

## Summary

Audit all current prompt injection points in HAL, then design and implement a unified system-prompt pipeline: HAL-provided built-in instructions, engine-specific default prompts, user-configurable overrides (global and per-project), and a programmatic hook for dynamic injection.

---

## Research: current injection points

Before any design work, the following injection sites must be fully understood. This is a pre-requisite research phase.

### 1. `formatContextPrompt` — `src/context/resolver.ts:384`

The primary structured injection. All resolved context variables (key-value pairs) are rendered as a `# Context` block, followed by `# User Message`. Every prompt that has a Grammy context goes through this. Result shape:

```
# Context
- project.cwd: /path/to/project
- project.name: my-project
- engine.name: copilot
- bot.userId: 12345
- sys.datetime: 2026-03-14 07:45:00 UTC+1
...

# User Message
<user text>
```

This is purely descriptive. It injects facts, not instructions.

### 2. `buildContextualPrompt` — `src/engine/prompt.ts:41`

The only place where a system-level instruction is currently injected. Appends one directive after the context+message block when a `downloadsPath` is present:

```ts
return `${contextualPrompt}\n\n[System: To send files to the user, write them to: ${downloadsPath}]`;
```

This instruction is appended at the **end** of the prompt — after the user message — which is the weakest position for model attention.

### 3. Document handler — `src/bot/handlers/document.ts:108`

Constructs the prompt directly, embedding the uploads path as a plain-text instruction:

```ts
const prompt = `Please read the file "./uploads/${safeName}" and ${caption}`;
```

The uploads relative path is hardcoded into the prompt string here. It is not routed through any shared system-prompt layer and is not configurable.

### 4. Photo handler — `src/bot/handlers/photo.ts:59`

Same pattern as document:

```ts
const prompt = `Please look at the image file "./uploads/${imageName}" and ${caption}`;
```

Again, hardcoded uploads path, constructed outside any shared pipeline.

### 5. Voice handler — `src/bot/handlers/voice.ts:172`

Uses `downloadsPath` and passes it through to `engine.execute()` like the text handler, so the `[System: ...]` instruction does get appended for voice too via `buildContextualPrompt`.

### Summary of injection sites

| Location | What is injected | Position | Configurable? | Shared pipeline? |
|---|---|---|---|---|
| `formatContextPrompt` | Key-value context facts | Top (before user message) | Via `config.context` | Yes |
| `buildContextualPrompt` | Downloads path `[System: ...]` | Bottom (after user message) | No | Yes |
| `document.ts` handler | Uploads path instruction | Embedded in prompt string | No | No |
| `photo.ts` handler | Uploads path instruction | Embedded in prompt string | No | No |

### Key gaps

- No engine-specific default system prompts (e.g. a Claude-specific instruction that is different from a Codex-specific one)
- No user-configurable system prompt layer in the config schema
- No hook to programmatically modify system instructions at request time
- The `[System: ...]` instruction is appended at the bottom — after the user message — the weakest position; instructions injected before the key-value context block would carry more weight
- The uploads path instructions in `document.ts` and `photo.ts` are ad-hoc strings, not part of the shared pipeline
- No concept of a "preamble" (instructions that precede the context block) vs. a "postamble" (instructions that follow the user message)

## Problem

HAL currently injects two categories of information into every prompt:

1. **Context facts** (descriptive): `project.cwd`, `project.name`, `engine.name`, timestamps, user ID, custom user-defined context values.
2. **System instructions** (imperative): currently only the downloads path `[System: ...]`.

These are treated as one flat block with no distinction, no position control, and no extension point. There is no way for:
- HAL itself to inject engine-specific default instructions (e.g. a cwd anchor instruction, or a memory pattern tailored to Claude vs. Codex)
- A user to add their own system instructions via config without writing custom `AGENTS.md`/`CLAUDE.md` files
- A user to override HAL's built-in instructions for their project
- A hook to dynamically compute system instructions at request time

The result is that each new "imperative" addition (see task 071: enforce project cwd) gets bolted on as another one-off `if (...)` in `buildContextualPrompt`, making the pipeline harder to reason about and impossible for users to customise.

---

## Proposal: a layered system-prompt pipeline

The prompt delivered to every engine should be assembled from four ordered layers:

```
[Layer 1: System preamble — instructions before context]
[Layer 2: Context facts — key-value block as today]
[Layer 3: User message — as today]
[Layer 4: System postamble — trailing instructions]
```

### Layer 1 — System preamble

Imperative instructions that should be read before anything else. This is the highest-attention position. Assembled from, in order:

1. **HAL built-ins** (always included unless overridden): cwd anchor instruction (from task 071), any other cross-engine invariants HAL needs to enforce.
2. **Engine-specific defaults** (HAL-provided, per engine): instructions that make sense for a specific engine's behaviour model. For example, Claude Code might benefit from a memory pattern suggestion; Codex from an explicit workspace reminder. These live in HAL's codebase (not in user space) and are loaded by engine name.
3. **User config overrides** (global, then per-project): `context.systemPrompt` (or `engine.systemPrompt`) in the config file. If provided, **replaces** the HAL built-ins for that scope (not appended — the user takes full ownership). Per-project overrides take precedence over global.

### Layer 4 — System postamble

Instructions that follow the user message. Lower attention than the preamble but appropriate for operational hints (downloads path, etc.). The existing `[System: To send files to the user, write them to: ...]` moves here. Same layering logic as the preamble.

### Hooks

A new hook file: `.hal/hooks/system-prompt.mjs` (global) and `{projectCwd}/.hal/hooks/system-prompt.mjs` (per-project).

Signature (same pattern as the context hook):

```js
export default async function(systemPrompt, context) {
  // systemPrompt: { preamble: string, postamble: string }
  // context: the resolved context vars map (same as today)
  // return: modified { preamble, postamble }
  return systemPrompt;
}
```

The hook receives the fully assembled preamble and postamble (after config overrides have been applied) and can modify either part programmatically. It runs after config resolution, before the prompt is sent to the engine.

## Engine-specific default prompts (HAL-provided)

HAL would ship a default system prompt per engine, stored as static strings or small template files in the codebase (e.g. `src/engine/system-prompts/claude.ts`, `codex.ts`, etc.). These would encode:

- The cwd anchor (from task 071) — unless task 071 is implemented independently
- Any engine-specific quirks worth encoding as a standing instruction (e.g. for Claude: reference to the downloads path convention; for Codex: workspace directory reminder)

These are defaults and are replaced when the user provides `context.systemPrompt` in their config.

## Config schema

New optional fields (exact location TBD based on where it fits cleanest — likely under `context` or as a new top-level `systemPrompt` key):

```yaml
globals:
  context:
    systemPrompt: |
      You are a helpful assistant for the {{project.name}} project.
      Always respond in concise Telegram-friendly markdown.

projects:
  my-project:
    context:
      systemPrompt: |
        You are managing a time tracker. All file operations must stay within the project directory.
```

Variable substitution (using existing `${}` and `@{}` patterns from context resolution) should work inside `systemPrompt` values.

## Relationship to task 071

Task 071 (enforce project cwd) proposes injecting a cwd anchor `[System: ...]` instruction. This task is the generalisation of that idea. The two can be implemented independently:

- Task 071 can be shipped as a one-off `[System: ...]` injection in `buildContextualPrompt` without the full pipeline.
- This task (072) then supersedes that one-off by making it part of the HAL built-ins layer.

If 072 is implemented first, 071 becomes a configuration of the preamble layer rather than a standalone patch.

## Scope

**Research phase (required first):**
- Audit all prompt construction sites thoroughly (handlers, `buildContextualPrompt`, any cron prompt construction)
- Identify any other ad-hoc string injections not captured above
- Confirm the exact prompt shape received by each engine adapter

**Design:**
- Define the exact interface for `{ preamble, postamble }` objects
- Define the hook contract and signature
- Determine where engine-specific defaults live in the codebase
- Decide whether `systemPrompt` lives under `context`, under `engine`, or as a sibling key

**Implementation:**
- Refactor `buildContextualPrompt` to assemble prompts from the four layers
- Add `systemPrompt` to the config schema (Zod + `ResolvedProjectConfig`)
- Add engine-specific default prompt strings to the engine layer
- Implement hook loading (reuse the pattern from `context/resolver.ts` → `loadAndRunHook`)
- Move the uploads path instruction in `document.ts` and `photo.ts` into the shared pipeline
- Update `formatContextPrompt` to accept an optional preamble argument

**Docs:**
- `docs/config/README.md` — new `systemPrompt` field
- `docs/config/reference.yaml` — example entry
- Engine docs — note that HAL injects engine-specific default instructions and how to override them

## Acceptance criteria

- Every prompt is assembled from the four-layer pipeline; no imperative instructions are constructed ad-hoc outside it
- Users can set `context.systemPrompt` globally or per-project to replace HAL's built-in preamble
- Engine-specific defaults are present in the codebase (at least one meaningful default per engine, even if minimal)
- A `.hal/hooks/system-prompt.mjs` hook (global or per-project) can programmatically modify `preamble` and `postamble` at request time
- The cwd anchor instruction (task 071) is either implemented as part of the HAL built-ins layer here, or clearly deferred to task 071
- All existing prompt injection behaviour is preserved (downloads path, uploads path, context facts) — no regression
