# 051 — Telegram skill frontmatter properties

## Status

draft

## Summary

Extend skill frontmatter with a richer nested `telegram` object. Replace the current top-level `telegram: true/false` boolean with structured Telegram-specific properties that control visibility and pre-LLM interaction behavior.

## Context

Task 038 introduced the `telegram: true` flag as the sole criterion for exposing a skill to Telegram (both `/help` and the slash command menu). This single flag is too coarse: there is no way to distinguish between a skill that should appear in the menu but not `/help`, one that needs user input before running, or one that should only be listed in help for discoverability. A richer frontmatter vocabulary is needed.

## Refined schema (agreed)

Use a nested `telegram` object in SKILL.md frontmatter.

| Property | Type | Description |
|---|---|---|
| `telegram.enabled` | boolean | Master visibility switch for Telegram surfaces. Optional; defaults to `true` when `telegram` object is present. |
| `telegram.help` | boolean | Controls inclusion in `/help` (`HAL_COMMANDS` Project Skills). Optional; defaults to `true` when omitted. |
| `telegram.menu` | boolean | Controls inclusion in Telegram slash command menu (`setMyCommands`). Optional; defaults to `true` when omitted. |
| `telegram.input` | boolean | When `true`, skill requires follow-up free-text input before LLM dispatch when initial invocation has no args. |
| `telegram.inputPrompt` | string | Custom prompt message shown when awaiting free-text input. Optional; defaults to `"Please provide input for /{skillName}:"` when omitted. Only relevant when `telegram.input: true`. |
| `telegram.buttons` | `{ label: string, prompt: string }[]` | Optional pre-LLM button choices. Each button has UI label + prompt payload to inject into the final LLM input. Takes precedence over `telegram.input` when both are present. **Validation:** at load time, every entry must have both `label` and `prompt`; otherwise hard fail with actionable error. |
| `telegram.timeout` | number | Optional timeout for waiting on button callback or follow-up input, in seconds. Defaults to `60` when omitted. |

Canonical behavior:

- If `telegram` object is absent → skill is hidden from Telegram surfaces by default.
- If `telegram` object is present and no visibility subflags are provided → visible in both help and menu by default.
- `telegram.help` and `telegram.menu` are primarily used to opt out of one surface.
- If `telegram.enabled: false`, skill is hidden from all Telegram surfaces regardless of `help/menu`.
- Legacy top-level `telegram: true/false` is invalid and must trigger a hard failure (startup/load error), not aliasing.

### Skill invocation flow with pre-LLM user interaction

When HAL resolves a skill to invoke, it must parse the frontmatter **before** dispatching to the LLM to determine whether user interaction is required first:

1. If `telegram.buttons` is present → render an inline keyboard with the listed options **plus an explicit "Cancel" button row**, await callback query, then inject selected button `prompt` into skill context and forward to LLM. If the user taps "Cancel", dismiss the keyboard and abort.
2. Else if `telegram.input: true` and invocation has no args → send follow-up prompt (using `telegram.inputPrompt` or the generic fallback `"Please provide input for /{skillName}:"`) **with an inline "Cancel" button**, await free-text reply, then forward it with skill context. If the user taps "Cancel", dismiss the prompt and abort.
3. Otherwise → pass the skill directly to the LLM without any intermediate interaction.

MVP interaction constraints (agreed):

- Single-step interaction only (no multi-step conversation flows in this task).
- Both `telegram.buttons` and `telegram.input` interactions render an explicit **"Cancel"** button (inline keyboard). Tapping "Cancel" dismisses the keyboard/prompt and aborts the skill invocation. Label is "Cancel" (not "Abort").
- Apply timeout for pending input/button waits; default `60s`, overridable via `telegram.timeout`. On timeout, the keyboard/prompt is dismissed automatically.
- If user sends another slash command while waiting, pending interaction is cancelled and normal command handling continues.
- **One pending skill interaction per user** (or per user+chat): starting a new button/input flow cancels any existing pending skill interaction for that user. No need to support multiple concurrent pending flows in this task.
- **Hot reload:** when the skill/command watcher reloads (e.g. SKILL.md or command list changes), cancel all pending skill interactions (button/input waits) immediately.
- **Reply semantics for `telegram.input`:** only treat a message as the skill input if it is a **direct reply** to the bot's prompt message. Use Telegram message metadata: `message.reply_to_message` must refer to the message we sent (same `message_id`). Other messages in the chat are ignored for this purpose. No AI — a simple handler check.

This means the bot layer needs awareness of the skill's frontmatter at dispatch time, not just at registration time.

> **Refinement scope note:** This task intentionally targets MVP primitives only (inline keyboard callback + single follow-up reply capture). Multi-step conversations and dynamic runtime button generation are out of scope.

### Frontmatter parser upgrade (agreed)

Replace the current regex-based frontmatter parser in `src/bot/commands/loader.ts` (`parseSkillMd`) with the project's existing `yaml` package (v2.8.2, already a dependency). The nested `telegram` object — especially `telegram.buttons` as an array of objects — cannot be reliably parsed with regex.

Scope of change:
- Parse the full frontmatter block (between `---` delimiters) with `yaml.parse()`.
- Extract `name`, `description`, `telegram`, and any future top-level keys from the parsed object.
- Validate the parsed `telegram` value: if it's a plain boolean, reject with a hard failure (legacy format); if it's an object, apply the schema above with defaults. If `telegram.buttons` is present, validate each entry has both `label` and `prompt` and hard fail otherwise.
- This replaces the individual regex matchers (`nameMatch`, `descMatch`, `telegramMatch`) with a single structured parse — cleaner and future-proof for any new frontmatter properties.
