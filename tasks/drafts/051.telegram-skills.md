# 051 — Telegram skill frontmatter properties

## Status

draft

## Summary

Extend skill frontmatter with richer Telegram-specific properties. Replace the current single `telegram` boolean with a structured set of `telegram_*` properties that give fine-grained control over skill visibility and interaction behaviour in the Telegram bot.

## Context

Task 038 introduced the `telegram: true` flag as the sole criterion for exposing a skill to Telegram (both `/help` and the slash command menu). This single flag is too coarse: there is no way to distinguish between a skill that should appear in the menu but not `/help`, one that needs user input before running, or one that should only be listed in help for discoverability. A richer frontmatter vocabulary is needed.

## Proposed properties (open for refinement)

| Property | Type | Description |
|---|---|---|
| `telegram_enabled` | boolean | Master switch — replaces the current `telegram` flag. When `true`, the skill is visible to Telegram (subject to the more specific flags). When absent or `false`, the skill is hidden from all Telegram surfaces. |
| `telegram_input` | boolean | When `true`, the skill requires free-text user input. If the user invokes the skill without providing a message, the bot sends a follow-up prompt asking for input, then waits for the reply before forwarding the combined context to the LLM. |
| `telegram_buttons` | string[] | A list of button labels to present to the user before the skill is forwarded to the LLM. When present, the bot renders an inline keyboard with these choices and waits for the user to tap one before proceeding. Mutually exclusive with `telegram_input` (buttons take precedence if both are set). |
| `telegram_help` | boolean | When `true`, the skill is listed in the `/help` output. Defaults to the value of `telegram_enabled` when omitted (i.e. if enabled, it appears in help by default). |
| `telegram_menu` | boolean | When `true`, the skill is registered in the Telegram slash command menu (`setMyCommands`). Defaults to the value of `telegram_enabled` when omitted. |

> **Refinement note:** The exact property names, defaults, and interaction rules between these flags must be debated during refinement. For example: should `telegram_enabled` be required when using any `telegram_*` property, or should `telegram_help: true` alone imply enablement? Should the existing `telegram` key be a deprecated alias or removed outright?

### Skill invocation flow with pre-LLM user interaction

When HAL resolves a skill to invoke, it must parse the frontmatter **before** dispatching to the LLM to determine whether user interaction is required first:

1. If `telegram_buttons` is present → render an inline keyboard with the listed options, await the callback, then pass the chosen value as part of the skill context to the LLM.
2. Else if `telegram_input: true` → send a follow-up prompt, await a free-text reply, then forward it together with the skill context.
3. Otherwise → pass the skill directly to the LLM without any intermediate interaction.

This means the bot layer needs awareness of the skill's frontmatter at dispatch time, not just at registration time.

> **Refinement research note:** During refinement, **deeply investigate the full Telegram Bot API** (inline keyboards, reply keyboards, `ForceReply`, callback queries, `answerCallbackQuery`, conversation state management in grammY) to ensure the design exploits all available interaction primitives and can be extended to richer patterns (e.g. multi-step button flows, dynamic button lists generated at runtime, button payloads carrying structured data).

## Migration

- Replace `telegram: true` / `telegram: false` in all existing skill frontmatter with `telegram_enabled: true` / omit the key.
- Affected files: all SKILL.md files in `.agents/skills/`, `examples/`, and any other locations that currently use `telegram`.
- Update all docs that describe the `telegram` frontmatter key.

## Requirements

- Define the final set of `telegram_*` properties and their semantics (refinement output).
- Update the skill loader and `CommandEntry` type to parse the new properties; remove (or alias) the old `telegram` key.
- Implement `telegram_buttons` flow: at dispatch time, parse frontmatter for button list, render inline keyboard, await callback query, inject selected value into skill context, then proceed.
- Implement `telegram_input` flow: detect invocation with no user text, send a follow-up prompt via Telegram API, await the reply, then proceed with skill execution.
- Update `/help` filtering to respect `telegram_help`.
- Update `setMyCommands` filtering to respect `telegram_menu`.
- Migrate all existing skill frontmatter from `telegram` to `telegram_enabled`.
- Update all relevant docs under `docs/` and example configs.

## Open questions (for refinement)

1. What is the best naming convention — `telegram_enabled` / `telegram_input` / `telegram_help` / `telegram_menu`, or something else (e.g. nested `telegram.enabled`, `telegram.input`, etc.)?
2. How should the flags interact? If `telegram_help: true` is set but `telegram_enabled` is absent, is the skill implicitly enabled?
3. For `telegram_buttons`: are the listed labels purely presentational, or do they also carry structured payloads? Should the list be static (frontmatter) only, or also dynamically generatable at runtime?
4. For `telegram_input` / `telegram_buttons`: what is the exact UX — inline keyboard cancel button, timeout handling, what if the follow-up is another command? Should multi-step interactions be supported?
5. What Telegram API primitives best fit these interaction patterns? See refinement research note above — inline keyboards, reply keyboards, `ForceReply`, and grammY conversation middleware should all be evaluated.
6. Should the old `telegram` key be a deprecated alias (warn + forward) or a hard removal?

## Acceptance Criteria

- Skill frontmatter supports the agreed `telegram_*` properties; the old `telegram` key is removed (or aliased with a deprecation warning, per refinement decision).
- All existing skills migrated; no remaining `telegram: true/false` in the repo.
- `/help` respects `telegram_help`; menu respects `telegram_menu`.
- `telegram_buttons` skills present an inline keyboard before LLM invocation; the selected value is injected into the skill context.
- `telegram_input: true` skills prompt the user for free-text input and await the reply before LLM invocation.
- Docs and examples updated.
