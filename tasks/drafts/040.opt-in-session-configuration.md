# 040 — Opt-in session configuration

## Status

draft

## Summary

Introduce explicit session configuration object for engines, with session enabled by default and a stateless mode when disabled.

## Requirements

- Replace boolean-only session config:
  - from `globals.engine.session = boolean`
  - to `globals.engine.session.enabled = boolean`
- Default behavior must be `enabled: true` when not specified.
- If `enabled: false`:
  - send messages without session/continue identifiers,
  - do not persist session state,
  - adapters must not write to user `session.json`.
- This mode represents pure stateless messaging.

## Engine-specific session options

- When session is enabled, support engine-specific session flags.
- Example for Claude:
  - `globals.engine.session.shared = boolean`
  - default `shared: true` (current behavior),
  - `shared: false` enables per-user session segregation.
- Example for Codex/Cursor:
  - `globals.engine.session.experimentallyEnableUserSession = boolean`
  - when enabled, adapter may attempt experimental/undocumented session-ID discovery for per-user isolation.

## Acceptance Criteria

- Config schema supports `session.enabled` plus engine-specific session options.
- Omitted `session.enabled` behaves as `true`.
- With `enabled: false`, no session IDs/continue fields are sent and no user session files are written.
- With `enabled: true`, current behavior remains unchanged unless engine-specific flags are set.
- Documentation/examples include the new nested session configuration shape.
