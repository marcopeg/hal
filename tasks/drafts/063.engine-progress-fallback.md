# 063 — Shared engine progress fallback

## Status

draft

## Summary

Implement a shared, reusable progress fallback utility for engine adapters. When an engine emits no streaming signals, show the user a live elapsed-time message (e.g. `"Codex is working on it... 32s"`) updated every 3 seconds, optionally interleaved with random usage tips from a curated list.

## Problem

Several engine adapters (Claude, Codex, Copilot, etc.) need a consistent fallback experience when no structured progress events are available. Today, adapters either show nothing or a static message. The desired UX is:

1. At t=0: `"{Engine} is working..."` (immediate)
2. Every 3s after: `"{Engine} is working on it... {N}s"` — replacing the previous message
3. Optionally, after a few seconds, begin interleaving tips: `"Tip: you can send photos to analyze code. | 45s"`, cycling through a curated list while the elapsed time keeps advancing.

Each adapter currently implements (or omits) this independently. A shared utility prevents duplication and ensures consistent behaviour across all engines.

## Goals

- Implement a reusable `createProgressFallback(engineName, onProgress)` utility (or equivalent) in `src/engine/` that:
  - Fires `onProgress("{Engine} is working...")` immediately at t=0.
  - Starts an interval timer (3s) that fires `onProgress("{Engine} is working on it... {N}s")`.
  - Optionally, after a configurable delay (e.g. 10s), begins cycling through usage tips, appending `| {N}s` to keep elapsed time visible.
  - Returns a `stop()` function that clears the interval (called when the engine process closes or errors).
- Integrate this utility into the Codex adapter (task 046) and any other adapter that needs a fallback.
- Define a curated list of short usage tips (e.g. "send photos to analyze code", "use /reset to start fresh", "try voice messages", etc.) in a shared location.

## Context

- Related: task 046 (Codex log streaming) — the Codex adapter needs this fallback when `--json` mode emits no progress events.
- Existing adapters that may benefit: `src/engine/adapters/claude.ts`, `src/engine/adapters/codex.ts`, and any future adapters.
- The `onProgress` callback contract is defined in `src/engine/types.ts` (`EngineExecuteOptions.onProgress?: (message: string) => void`).

## Constraints

- The utility must be stateless and side-effect-free when `onProgress` is not provided (i.e., no timer should start).
- Elapsed time is computed from when `start()` / the factory is called, not from when `onProgress` is registered.
- Tip cycling is optional — the utility should work without tips (pure elapsed-time mode).
- Tips must be short (fit in one Telegram message line alongside the elapsed time).
- The interval fires at most every 3 seconds; adapters may throttle further but the utility should not fire faster.
- `stop()` must be idempotent and safe to call multiple times.

## Acceptance Criteria

- A shared utility (e.g. `src/engine/progress-fallback.ts`) exists and exports a `createProgressFallback` factory or equivalent.
- The utility fires `onProgress` immediately at t=0, then every 3s with updated elapsed time.
- Tips (when provided) are cycled randomly through the curated list, appended with `| {N}s`.
- The Codex adapter uses this utility as its fallback.
- `stop()` cleanly cancels the timer with no further `onProgress` calls.
- No regressions in existing adapter behaviour.
