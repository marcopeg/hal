# 032b — Project-level crons + logging refinement

## Status

draft

> **Part of [032 — Cron jobs and scheduled actions](./032.cron-jobs-and-scheduled-actions.md).**
> Read the parent task first for shared specifications (cron file formats, source tiers, logging format, lifecycle rules). This file only covers what is specific to 032b.

## Summary

Extend the cron scheduler to support project-level cron definitions (`{projectCwd}/.hal/crons/*.{md,mjs}`), add `runAs` session support, and refine execution logging based on experience from 032a.

## Dependencies

- [032a — Core scheduler + system-level crons](./032a.core-scheduler-and-system-crons.md) must be completed first.

## Scope

### In scope

- Project-level cron source:
  - **File-based only:** `{projectCwd}/.hal/crons/*.{md,mjs}`, loaded at boot for each active project.
  - No inline `projects[n].crons` config key — project crons are file-based only.
- File watcher on `{projectCwd}/.hal/crons/` per active project for hot-reload.
- Per-project scheduler instances: each project runs its own scheduler scope, loading both system-level and project-level crons relevant to it.
- `.md` crons at project tier: `projectId` is **implicit** (the file belongs to that project), so `targets` entries only need `userId?` and `flowResult?`. The Zod schema for project-tier `.md` frontmatter should reflect this.
- `runAs: <userId>` session mode for project-level `.md` crons: execute the anonymous prompt with a specific user's context data injected.
- Cron name uniqueness across system + project tiers within a project scheduler (decision: see Notes).
- Logging to `{projectCwd}/.hal/logs/crons/{job-name}/{timestamp}.{job-name}.txt`.
- Refine execution logging format and contents based on learnings from 032a.
- Clean teardown per project: stop project scheduler and file watcher when project is deactivated or removed from config.

### Out of scope (deferred to 032c)

- User-level crons (`{dataDir}/{userId}/crons/`).
- Slash commands (`/cron_add`, `/cron_list`, etc.).
- `user` session mode (only relevant for user-defined crons).

## Requirements

- On project boot, instantiate a per-project scheduler scope and load:
  1. System crons from `{configDir}/.hal/crons/` (already loaded by 032a; reuse or reference).
  2. Project crons from `{projectCwd}/.hal/crons/`.
- Add file watcher on `{projectCwd}/.hal/crons/` for each active project (add/edit/delete → reschedule).
- Implement a project-tier `.md` frontmatter Zod schema where `targets` entries omit `projectId` (implicit) and only accept `userId?` and `flowResult?`.
- Implement `runAs` session resolution for project-tier `.md` crons: given a `userId`, inject the user's context data into the anonymous `agent.call()`.
- Handle cron name collisions between system and project tiers (see Notes — decide during planning).
- Implement execution logging to `{projectCwd}/.hal/logs/crons/...`.
- Refine log format based on 032a feedback (structure, readability, metadata).
- Ensure clean teardown per project: stop scheduler scope and file watcher when project is deactivated.

## Acceptance Criteria

- A `.md` cron in `{projectCwd}/.hal/crons/` fires on schedule and submits the prompt to that project's engine.
- A `.mjs` cron in `{projectCwd}/.hal/crons/` fires on schedule with a correctly scoped `CronContext`.
- System crons and project crons both run within the same project scheduler without conflict.
- Project cron files are hot-reloaded: add/edit/delete detected and scheduler updated without restart.
- `runAs` correctly injects the specified user's context data into the anonymous prompt call.
- Project-tier `.md` frontmatter without `projectId` is accepted as valid (projectId is implicit).
- Deactivating a project stops its scheduler scope and file watcher.
- Execution logs are written to `{projectCwd}/.hal/logs/crons/...` with the correct format.

## Notes

- **Name collision strategy** (decide during planning): when a project cron and a system cron share the same name, options are: (a) project cron silently overrides system cron within that project's scope, (b) both run (namespaced internally as `system/{name}` vs `project/{name}`), (c) hard error at boot. Leaning toward (b) with internal namespacing, but confirm during planning.
- Logging refinements are intentionally open here — they depend on what we learn from running 032a in practice. The plan phase should revisit after 032a is in use.
- The `CronContext` type extended for project tier may eventually add `ctx.project` (the current project's config/bot) as a convenience shortcut — note for planning.
