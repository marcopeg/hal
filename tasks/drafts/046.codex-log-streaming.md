# 046 — Codex log streaming

## Status

draft

## Summary

Investigate and implement streaming log updates for the Codex engine, sending incremental progress messages to Telegram as the CLI runs rather than waiting for full completion.

## Problem

The Codex adapter currently buffers all stdout until the process exits before returning a result. Users see no feedback during long-running sessions, making the experience feel unresponsive. Claude Code already streams progress via `onProgress` callbacks using its `stream-json` output format — Codex needs an equivalent approach.

## Goals

- Investigate what output Codex emits to stdout/stderr while running (structured JSON, plain text, progress events, etc.).
- Determine whether Codex provides a streaming or event-driven output mode comparable to `--output-format stream-json`.
- Identify meaningful log lines or events that can be surfaced as Telegram progress update messages.
- Implement `onProgress` callback integration in the Codex adapter similar to the Claude adapter.

## Scope

- Research Codex CLI output format and any available streaming flags.
- Prototype log-line parsing to extract actionable progress signals.
- Implement incremental Telegram message updates via the existing `onProgress` mechanism.
- Avoid breaking existing buffered behavior as a fallback when streaming is unavailable.

## Deliverables

- Findings on Codex CLI output structure during execution.
- Updated `src/engine/adapters/codex.ts` with streaming/progress support.
- Test notes covering typical Codex runs with progress updates visible in Telegram.

## Acceptance Criteria

- Telegram users see incremental status messages while a Codex session is running.
- The final response is still delivered correctly after streaming completes.
- Fallback to buffered mode works when no streaming signals are detected.
