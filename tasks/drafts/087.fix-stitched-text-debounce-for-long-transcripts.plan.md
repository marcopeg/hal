# Plan - 087 Fix stitched text debounce for long transcripts

## Objective

Fix the text debounce flow so rapid transcript-like multi-message input is stitched into one logical prompt and executed exactly once downstream.

## Phase 1 - Research Telegram behavior

1. Gather official Telegram Bot API references for long-message limits.
2. Cross-check ecosystem/library references for how long text is treated in practice.
3. Separate well-supported facts from ambiguous incoming-user behavior.
4. Use that research to define the correct HAL requirement: reconstruct one logical process-level input before engine execution.

## Phase 2 - Reproduce and pin down the bug

1. Reproduce the current behavior with automated tests around `createTextHandler()`.
2. Capture the failing scenario where the first chunk is below `4096` but follow-up chunks arrive quickly enough that the user expects a single stitched prompt.
3. Verify how many times the engine execution path runs for the reproduced case.

## Phase 3 - Redesign the debounce trigger

1. Replace the current exact-threshold activation rule with a trigger based on the actual observed arrival pattern.
2. Keep buffering scoped per user and preserve ordering by `message_id`.
3. Ensure slash commands still bypass debounce completely.
4. Make the engine call boundary explicit: only the reconstructed prompt may cross into `ctx.engine.execute(...)`.

## Phase 4 - Protect adjacent behavior

1. Re-check the interaction with `debounceActiveUsers` and `createRateLimitMiddleware()`.
2. Confirm that short ordinary messages do not suffer an unacceptable latency regression.
3. Keep the existing processing-status UX coherent (`Buffering...` / `Processing...`) or intentionally revise it with clear rationale.

## Phase 5 - Validate and document

1. Add or extend Vitest coverage for the bug and the expected fixed behavior.
2. Run the relevant validation commands (`npm run test`, plus any targeted lint/typecheck if touched files require it).
3. Update the task/backlog wording if the final implementation model differs from the old "Telegram 4096 split" assumption.
4. Record a concise research summary in the implementation handoff so the next task does not reintroduce unsupported assumptions.
