## Expected Behaviour

### Message serialization

The bot processes messages one at a time per user. When multiple messages arrive while one is being processed, they are held in an in-memory queue (via `sequentialize`) and executed in order. This is the desired model; the implementation must preserve it.

### `/btw` — interrupt and drain

1. The user sends `/btw` while an agent is running.
2. The running agent is killed **immediately** (SIGINT → SIGTERM → SIGKILL escalation, no `await`). The kill is **fire-and-forget** — the confirmation message is sent before the process fully exits.
3. All messages that were already queued behind the killed agent (earlier `update_id` than `/btw`) are **silently dropped** — no "Processing…" spinner, no LLM call, no reply.
4. The user receives a single reply: the configurable `commands.btw.message.stopped` text (default: `"The process stopped."`).
5. If no agent is running when `/btw` arrives, the user receives `commands.btw.message.nothingToDo` (default: `"No active process to stop, nothing to do."`).
6. If the user writes `/btw do something else`, the `"do something else"` part is immediately dispatched as a new agent call after the kill. It uses the normal UX (progress streaming, replies). Attachments embedded in the same Telegram message are supported because the existing media handlers (`photo`, `document`, `voice`) share the same dispatch pipeline.

### `/clean` extension

`/clean` is extended to also:

1. Stop the running agent (same as `/btw` does — kill + fire-and-forget confirmation).
2. Drain all queued messages (same as `/btw` does — drain watermark + queue file wipe).
3. **Then** clear the session as it already does today.

The net effect: `/clean` resets everything — the active call, the pending queue, and the session state.

## Research and Discoveries

This section consolidates everything learned in tasks 069 and 078. It is the authoritative reference for this task.

### R1 — Grammy's sequential polling loop (root cause of all prior failures)

Grammy's default `bot.start()` calls `getUpdates`, then **awaits each handler to completion** before calling `getUpdates` again. While the LLM handler runs (30–120 s), Grammy never fetches the next update. `/btw` sits unread on Telegram's servers until the LLM finishes. By then, `llmRunRegistry.stop()` is a no-op.

This is not a bug; it is the documented behaviour of Grammy's simple runner.

### R2 — Telegram `getUpdates` constraints

- Passing `offset = N` acknowledges all updates with `update_id < N`. There is no "peek without consuming".
- Two concurrent `getUpdates` callers on the same token trigger a **409 Conflict** error. A second polling loop alongside Grammy's main loop is not viable.
- Update JSON objects **never contain binary data** — photos, voice, and documents are `file_id` references downloaded at processing time from Telegram's CDN. File IDs remain valid for 24+ hours. This means WAL files are tiny (1–5 KB each), and crash-recovered updates can still download their attachments.

### R3 — Grammy Runner (`@grammyjs/runner`)

Grammy's official concurrent runner replaces `bot.start()`:

```ts
import { run } from "@grammyjs/runner";
const runner = run(bot);
await runner.task(); // resolves on clean stop
```

- Decouples the fetch loop from the process loop. Updates are fetched eagerly and dispatched concurrently.
- The runner stops via `runner.stop()`; `runner.task()` resolves (not rejects) on a clean stop.
- If the runner crashes (e.g. 409), `runner.task()` **rejects**.
- `sequentialize` is exported from `@grammyjs/runner` (not from `grammy` core — this tripped up task 069 initially).

### R4 — Grammy `sequentialize` and middleware ordering

`sequentialize` serialises handlers with the same constraint key without blocking handlers with different keys (or handlers registered before it):

```ts
import { sequentialize } from "@grammyjs/runner";

// Registered BEFORE sequentialize — runs concurrently with LLM handlers
bot.command("btw", createBtwHandler(projectCtx));

// sequentialize wraps only LLM-bound handlers
bot.use(sequentialize((ctx) => ctx.from?.id?.toString() ?? undefined));

// These run INSIDE sequentialize — serialised per user
bot.on("message:text", createTextHandler(...));
bot.on("message:photo", createPhotoHandler(...));
bot.on("message:document", createDocumentHandler(...));
bot.on("message:voice", createVoiceHandler(...));
```

Key point: handlers registered **before** `sequentialize` in Grammy's middleware chain are exempt from its queue.

The `reset` callback handler (`callback_query:data`) must remain **after** `sequentialize` because it does destructive user-data operations that must not race with in-flight LLM calls.

### R5 — Grammy 409 resilience

When HAL is restarted while an old instance is still polling (e.g. after a crash with stuck Copilot processes), the new instance gets `409 Conflict`. Grammy Runner does not retry on 409 — it propagates the error and kills the process. The fix is to wrap the runner in a retry loop:

```ts
const MAX_409_RETRIES = 3;
let currentRunner = run(bot);
let stopped = false;

(async () => {
  let retries = 0;
  while (!stopped) {
    try {
      await currentRunner.task();
      break; // clean stop
    } catch (err) {
      if (!stopped && err instanceof Error && err.message.includes("409") && retries < MAX_409_RETRIES) {
        retries++;
        logger.warn({ attempt: retries }, "Grammy 409 — retrying in 2s");
        await new Promise((r) => setTimeout(r, 2000));
        currentRunner = run(bot);
        continue;
      }
      logger.error({ err }, "Grammy Runner crashed — exiting");
      process.exit(1);
    }
  }
})();
```

`BotHandle.stop()` sets `stopped = true` and calls `currentRunner.stop()` (not `bot.stop()`). Clean stop via `runner.stop()` causes `runner.task()` to resolve (not reject), so `stopped = true` prevents re-entering the retry loop.

### R6 — Process-group kill

Node's `child_process.spawn()` does not assign the child a new process group by default. When the child spawns further processes (e.g. Copilot CLI → model worker → tool subprocesses), `proc.kill(signal)` only delivers the signal to the direct child PID. The grandchildren survive, HAL appears frozen.

Fix:

1. Pass `{ detached: true }` to the spawn call. This places the child in its own process group (PGID = `proc.pid`).
2. In the kill escalation, use `process.kill(-proc.pid, signal)` (negative PID = send to entire process group) instead of `proc.kill(signal)`.
3. On Windows, fall back to `proc.kill(signal)` because `process.kill(-pid)` is not supported.

The spawn call is in `src/engine/spawn.ts` (`spawnEngineProcess`), which all adapters use. The kill logic lives in the new `src/bot/llm-run-registry.ts`.

Note: `unref()` must NOT be called on the child — HAL needs the process to stay referenced until it exits or is killed.

### R7 — LLM Run Registry

A per-project, per-user registry that tracks the currently active child process for each user:

```ts
interface LlmRunRegistry {
  // Called when the child process starts (from engine adapter hooks)
  register(userId: number, proc: ChildProcess): void;
  // Called when child process exits normally
  clear(userId: number): void;
  // Check if a process is active
  has(userId: number): boolean;
  // Stop the active process. Returns a Promise<void> that resolves when the process exits.
  // Stores the drain watermark. Pass the /btw update_id as watermark.
  stop(userId: number, watermarkUpdateId: number): Promise<void> | false;
  // Returns true (and clears the watermark) if updateId < stored watermark for userId.
  // Used by queued handlers to skip stale messages.
  shouldDrain(userId: number, updateId: number): boolean;
  // Returns true (and clears the flag) if the process for userId was killed by /btw.
  // Used by the LLM dispatch pipeline to suppress its own reply.
  wasStoppedAndClear(userId: number): boolean;
}
```

Kill escalation in `stop()`:
- t=0: SIGINT (polite shutdown, Claude saves session)
- t=500ms: SIGTERM (if still alive)
- t=1000ms: SIGKILL (final backstop)
- All timers cancelled when process exits

`stop()` returns a `Promise<void>` that resolves when the process emits `close`. It also sets a `stoppedByBtw` flag for `userId` so `wasStoppedAndClear()` returns true.

`shouldDrain(userId, updateId)`: returns `true` and clears the watermark if `updateId < watermarkUpdateId`. Returns `false` if no watermark or `updateId >= watermark`. This drains all messages that were queued before `/btw` arrived. Telegram `update_id` values are monotonically increasing — reliable for this comparison.

### R8 — Engine adapter process hooks

All engine adapters need two optional callbacks added to `EngineExecuteOptions`:

```ts
interface EngineExecuteOptions {
  // ... existing fields ...
  onProcessStart?: (proc: ChildProcess) => void;
  onProcessExit?: (proc: ChildProcess) => void;
}
```

In each adapter's `execute()`:
- Call `options.onProcessStart?.(proc)` immediately after `proc` is created (before any data flows).
- Call `options.onProcessExit?.(proc)` in both the `close` and `error` event handlers.

The handlers (`text`, `photo`, `document`, `voice`) wire these to `llmRunRegistry.register()` and `llmRunRegistry.clear()`.

### R9 — Fire-and-forget stop

`btw.ts` must NOT await `stopResult` before replying. The kill escalation timers are already running in background. The user gets the confirmation immediately:

```ts
if (stopResult !== false) {
  void stopResult; // fire-and-forget
  await clearUserQueue(userDir);
  await gramCtx.reply(btwCfg.message.stopped ?? "The process stopped.");
  if (followUpText) {
    await dispatch(gramCtx, followUpText);
  }
  return;
}
```

Session file race note: session-based engines (Claude) write their session on SIGINT. The new process starts with `--continue`, reading the same session directory. In practice the new process spends longer building the contextual prompt than the old one spends flushing state. SIGKILL fires at t=1s as a hard backstop. No special handling needed.

### R10 — FS Write-Ahead Log (WAL) for crash recovery

Grammy Runner buffers in-flight updates in memory. A hard crash between fetch and handler completion loses those updates. The WAL pattern writes the raw Telegram update JSON to disk before processing and deletes the file only when the handler finishes (in a `finally` block).

**Queue layout:**

```
{config.dataDir}/
  {userId}/
    queue/
      {update_id}.json    ← raw Telegram update JSON (1–5 KB, no binary)
```

**Lifecycle:**

1. Update arrives → `writeFile("{update_id}.json", JSON.stringify(update), { flag: "wx" })` (create-new-only, skip duplicates)
2. Handler runs.
3. Handler completes → `unlink("{update_id}.json")` in `finally`.
4. On crash → file survives.
5. On restart → scan `{userId}/queue/`, sort numerically by `update_id`, replay each via `bot.handleUpdate(update)`.

**WAL + drain interaction:** When `/btw` fires, `clearUserQueue()` deletes all `.json` files in the user's queue directory. This happens **before** the user sees "The process stopped." On WAL replay, if the file is gone, it was intentionally wiped — nothing to replay.

**Debounce interaction:** WAL writes happen per-part (each incoming update). WAL files for buffered parts are deleted when the flush's `finally` block runs after the LLM call completes.

### R11 — Drain watermark detail

After `/btw`, `sequentialize` will dequeue the next waiting message and dispatch it. Without the drain watermark, that message would start a new LLM call the user didn't want. The watermark prevents this:

- `/btw` stores `btwUpdateId` as the drain watermark for `userId`.
- The text handler's `dispatchMessage` and media handlers check `shouldDrain(userId, updateId)` at entry. If `true`, they return immediately (silent). WAL `finally` still cleans up the file.
- The watermark is cleared on the first message that **passes through** (i.e. the next new message after `/btw`, whose `update_id >= watermarkUpdateId`).

### R12 — Drain checks in all handlers

Each handler needs `shouldDrain` checks to handle `/btw` arriving during pre-LLM work:

**`text.ts` (`dispatchMessage`)**: check at entry, before `ensureUserSetup` and before the status message.

**`photo.ts`**:
1. After `ensureUserSetup`, before file download — bail out if already drained.
2. After download, before `engine.execute` — bail out if `/btw` arrived during download.

**`document.ts`**: Same two-point pattern as photo.

**`voice.ts`** (most expensive — has ffmpeg + Whisper before LLM):
1. After `ensureUserSetup`, before download — bail immediately.
2. After transcription completes, before `engine.execute` — bail if `/btw` arrived during ffmpeg/Whisper window. These blocking steps cannot be interrupted; we simply discard the result.

**After `engine.execute` in all handlers**: check `wasStoppedAndClear(userId)`. If `true`, delete the status message and return silently — no reply, no session save. The `/btw` handler has already replied.

### R13 — Skill / .mjs command interaction

Skills (`agent.call()`) and `.mjs` commands run inside the text handler. They also need the `wasStoppedAndClear` check in their catch block — if the kill causes a rejection, the text handler must not reply with "Skill failed:".

### Phase 5 — WAL middleware + replay

**5.1 — Create `src/bot/wal.ts`**

```ts
import { mkdir, unlink, writeFile } from "node:fs/promises";
import { join } from "node:path";
import type { Context, NextFunction } from "grammy";

export function createWalMiddleware(dataDir: string) {
  return async (ctx: Context, next: NextFunction): Promise<void> => {
    const userId = ctx.from?.id;
    if (!userId) return next();
    const updateId = ctx.update.update_id;
    const queueDir = join(dataDir, String(userId), "queue");
    const filePath = join(queueDir, `${updateId}.json`);
    await mkdir(queueDir, { recursive: true });
    await writeFile(filePath, JSON.stringify(ctx.update), { flag: "wx" }).catch(() => {});
    try {
      await next();
    } finally {
      await unlink(filePath).catch(() => {});
    }
  };
}
```

**5.2 — Create `src/bot/wal-replay.ts`**

```ts
import { readFile, readdir, rm } from "node:fs/promises";
import { join } from "node:path";
import type { Bot } from "grammy";

export async function replayWal(bot: Bot, dataDir: string): Promise<void> {
  const userDirs = await readdir(dataDir).catch(() => []);
  for (const userId of userDirs) {
    const queueDir = join(dataDir, userId, "queue");
    const files = (await readdir(queueDir).catch(() => [])).filter((f) => f.endsWith(".json"));
    const sorted = files.sort((a, b) => parseInt(a) - parseInt(b));
    for (const file of sorted) {
      const raw = await readFile(join(queueDir, file), "utf8").catch(() => null);
      if (!raw) continue;
      try {
        await bot.handleUpdate(JSON.parse(raw));
      } catch {
        await rm(join(queueDir, file)).catch(() => {});
      }
    }
  }
}
```

Call `replayWal(bot, config.dataDir)` in `startBot()` before starting the runner.

### Phase 6 — `/btw` command handler

**Create `src/bot/commands/btw.ts`**

```ts
export function createBtwHandler(projectCtx: ProjectContext) {
  return async (gramCtx: Context): Promise<void> => {
    const btwCfg = projectCtx.config.commands.btw;
    const userId = gramCtx.from?.id;
    if (!userId) return;

    const userDir = resolve(join(projectCtx.config.dataDir, String(userId)));
    const btwUpdateId = gramCtx.update.update_id;

    // Text after "/btw " (if any)
    const followUpText = (gramCtx.message?.text ?? "").replace(/^\/btw(@\w+)?(\s*)/, "").trim() || null;

    const stopResult = projectCtx.llmRunRegistry.stop(userId, btwUpdateId);

    if (stopResult !== false) {
      // Case 1: Process was running — fire-and-forget kill, reply immediately
      void stopResult;
      await clearUserQueue(userDir);
      await gramCtx.reply(btwCfg.message.stopped ?? "The process stopped.");
      if (followUpText) {
        await dispatchMessage(gramCtx, followUpText, projectCtx);
      }
      return;
    }

    // Case 2: No process running
    if (followUpText) {
      // Treat as a normal new message — dispatch directly
      await dispatchMessage(gramCtx, followUpText, projectCtx);
    } else {
      await gramCtx.reply(btwCfg.message.nothingToDo ?? "No active process to stop, nothing to do.");
    }
  };
}
```

`dispatchMessage` is the shared LLM dispatch function (extracted or referenced from `text.ts`). See Phase 7.

## Known Failure Modes and Mitigations

| Scenario | Mitigation |
|---|---|
| `sequentialize` dequeues next message after `/btw` kills active call | Drain watermark in registry (`shouldDrain`) drops messages with `update_id < btwUpdateId` |
| Copilot (and others) spawn grandchild processes not killed by `proc.kill()` | `detached: true` + `process.kill(-proc.pid, signal)` kills entire process group |
| New HAL instance starts while old instance still polls → 409 crash | 409-retry loop with 3 retries and 2s delay before each |
| Hard crash mid-LLM call | WAL file survives → replayed on restart |
| Replayed WAL file is for a message before `/btw` wiped the queue | WAL wipe deletes the file; replay finds nothing |
| WAL replay triggers duplicate reply (LLM completed, then crash before WAL delete) | Sub-millisecond race window; acceptable. Future: reply deduplication via last-sent hash in session.json |
| `/btw` fires during voice ffmpeg/Whisper — can't interrupt blocking steps | `shouldDrain` checks bail out after each blocking step completes |
| Session file race: new process reads session while old process is writing it | In practice the new process spends longer building its prompt than the old one takes to flush. SIGKILL at t=1s is the hard backstop |

## Acceptance Criteria

- [ ] `/btw` is available as a built-in command, discoverable in `/help` and Telegram's command menu.
- [ ] `commands.btw.enabled` defaults to `true`; `false` replies "This command is disabled."
- [ ] `commands.btw.message.stopped` and `commands.btw.message.nothingToDo` are configurable at globals and project level with project override.
- [ ] Sending `/btw` while an LLM call is running kills the process and replies the stopped message within 200ms (fire-and-forget).
- [ ] Sending `/btw` while an LLM call is running drains all queued messages (they are silently dropped, not sent to the LLM).
- [ ] Sending `/btw do something else` while an LLM call is running kills the call and dispatches "do something else" as a new call.
- [ ] Sending `/btw` with no active call replies the nothingToDo message.
- [ ] Sending `/btw do something else` with no active call dispatches "do something else" as a new call.
- [ ] `/clean` kills the active agent, drains the queue, and clears the session.
- [ ] A Grammy 409 error does not crash HAL; it retries up to 3 times with 2s delay.
- [ ] After `/btw`, the entire process tree (including grandchildren from Copilot, etc.) is dead within 1s.
- [ ] `npm run lint`, `npm run build`, `npm run test` all pass.
- [ ] No regressions: text, photo, document, voice messages all work normally when no `/btw` is involved.
- [ ] Session continuity works across messages.
