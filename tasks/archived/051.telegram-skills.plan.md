# Plan: 051 — Telegram skill frontmatter properties

## Phase 1: Frontmatter parser and Telegram metadata type

**Goal:** Replace regex-based SKILL.md parsing with YAML and introduce a structured `telegram` type; reject legacy boolean `telegram` with hard failure.

1. **Switch `parseSkillMd` to YAML**  
   In `src/bot/commands/loader.ts`: keep the existing `---` block extraction (regex or split), then parse the frontmatter string with `yaml.parse()` from the existing `yaml` dependency. Read `name`, `description`, and `telegram` from the parsed object. Preserve the prompt body as the content after the closing `---`. Handle YAML parse errors (log and return `null` or throw with file path for startup failure).

2. **Define `TelegramSkillMetadata` type and normalization**  
   Add an interface (e.g. in `loader.ts` or a small types file) for the resolved telegram block: `enabled`, `help`, `menu`, `input`, `inputPrompt`, `buttons`, `timeout` with the types from the task. Add a function that takes the raw parsed `telegram` value and returns normalized metadata or throws:
   - If `telegram` is a boolean → throw with message like: `Legacy telegram: true/false is no longer supported. Use nested telegram: { enabled: true } instead. File: {filePath}`.
   - If `telegram` is an object: apply defaults (`enabled: true`, `help: true`, `menu: true`, `inputPrompt` fallback when `input: true`, `timeout: 60`). If `telegram.buttons` is present, validate every element has both `label` and `prompt` (and are non-empty strings); otherwise throw with actionable error including file path.
   - If `telegram` is absent/undefined → skill is not exposed to Telegram; represent as `undefined` or a sentinel so callers know "no telegram block".

3. **Wire parsed `telegram` into `parseSkillMd` return**  
   `parseSkillMd` should return `{ name, description, prompt, telegram }` where `telegram` is the normalized `TelegramSkillMetadata` or `undefined` when the frontmatter has no `telegram` key. All call sites (e.g. `scanSkillsDir`, `resolveSkillEntry`) already use the return value; ensure they pass `telegram` through.

4. **Update `CommandEntry`**  
   Change `telegram?: boolean` to `telegram?: TelegramSkillMetadata` (or the chosen type name). Ensure `filePath` is available where errors are thrown so messages can include the skill file path.

## Phase 2: Visibility filtering (help and menu)

**Goal:** Use the new `telegram` object for "show in /help" and "show in menu" instead of a single boolean.

1. **`commandsForTelegramMenu`**  
   In `src/bot/commands/loader.ts`: include a skill only when `entry.telegram` is present, `entry.telegram.enabled !== false`, and `entry.telegram.menu !== false`. Non-skill commands unchanged.

2. **`buildHalCommands` (Project Skills section)**  
   In `src/bot/commands/message.ts`: push a skill into the "Project Skills" list only when `entry.telegram` is present, `entry.telegram.enabled !== false`, and `entry.telegram.help !== false`. This keeps `/help` and `${HAL_COMMANDS}` in sync with the new schema.

## Phase 3: Pending skill interaction state

**Goal:** One pending skill interaction per user; support timeout, cancel, and hot-reload cancel.

1. **New module `src/bot/commands/skillPending.ts`**  
   Similar to `resetPrompt.ts`: maintain a single pending entry per user (e.g. `Map<userId, PendingSkillEntry>`). Entry type: at least `chatId`, `messageId`, `skillName`, `skillPrompt`, `telegram` (for timeout seconds), `type: 'buttons' | 'input'`, `timer` (Node timeout). Store enough to either continue with button payload or with reply text and to dismiss the message (edit reply_markup to undefined or delete).

2. **API**  
   - `trackSkillPending(userId, chatId, messageId, skillName, skillPrompt, telegram, type, api)` — set entry, start timer (telegram.timeout * 1000 ms). On timeout: remove from map, call `api.editMessageReplyMarkup(chatId, messageId, { reply_markup: undefined })` (catch and ignore errors).
   - `invalidateSkillPending(userId, api)` — cancel timer, remove entry, clear keyboard for that message (same edit as timeout). Used when user starts another slash command or a new skill flow.
   - `resolveSkillPending(userId)` — cancel timer and remove entry (after we've received the callback or reply and are about to run the skill).
   - `cancelAllSkillPending(api)` — cancel all timers and clear all keyboards; used on hot reload. Requires passing `Api` or having a way to get it (e.g. from project context when watcher runs).

3. **One-per-user rule**  
   When starting a new button or input flow, call `invalidateSkillPending(userId, api)` before sending the new message so any previous pending skill interaction is cancelled.

## Phase 4: Text handler — pre-LLM branching and "another slash cancels"

**Goal:** Before calling the LLM for a skill, check `telegram.buttons` and `telegram.input`; if interaction is required, send the prompt/keyboard and wait instead of calling the agent.

1. **Cancel existing pending at slash-command entry**  
   At the start of the slash-command block in `src/bot/handlers/text.ts`, after resolving `userId`, if there is a pending skill for this user, call `invalidateSkillPending(userId, gramCtx.api)` so any previous button/input wait is cancelled and the keyboard is cleared.

2. **Branch after `resolveSkillEntry`**  
   When `skillEntry` is found and has `skillPrompt`:
   - If `skillEntry.telegram?.buttons` is present and non-empty: do not call the agent. Send a message with an inline keyboard: one row per option (from `telegram.buttons`, use `label` for button text), then a row with a single "Cancel" button. Callback data format: e.g. `skill:{commandName}:{index}` for option at index, `skill:{commandName}:cancel` for Cancel. Call `trackSkillPending(..., 'buttons', ...)` with timeout from `skillEntry.telegram.timeout`. Return.
   - Else if `skillEntry.telegram?.input === true` and `args.length === 0`: do not call the agent. Send a message with text = `skillEntry.telegram.inputPrompt ?? \`Please provide input for /${commandName}:\`` and an inline keyboard with a single "Cancel" button (callback `skill:{commandName}:cancel`). Call `trackSkillPending(..., 'input', ...)`. Return.
   - Else: keep current behavior — build prompt from `skillPrompt` and optional args, call agent, send response (existing code path).

3. **Extract "execute skill with prompt" helper**  
   Factor a shared function used by both the text handler (direct invocation with args) and the callback/reply handlers (after button or reply): e.g. `executeSkillWithPrompt(projectCtx, gramCtx, skillEntry, userInputText)` that sends "Processing…", calls `agent.call(prompt, { onProgress })`, then sends the result (and handles errors). This avoids duplicating the agent call and chunked response logic.

## Phase 5: Skill callback handler (buttons) and reply handling (input)

**Goal:** Handle button callbacks and reply-to-message for skills; then run the skill with the chosen payload or reply text.

1. **Register skill callback handler in `bot.ts`**  
   Add a `callback_query:data` handler that runs before the MJS dispatcher (so it takes precedence for skill prefixes). If `data` starts with `skill:` (e.g. `skill:chuck:0` or `skill:chuck:cancel`), handle it and do not call `next()`. Parse `skill:{commandName}:{rest}`; if `rest === 'cancel'`, call `resolveSkillPending(userId)`, answerCallbackQuery, edit message to remove reply_markup, return. Otherwise treat `rest` as button index (number); get pending entry for this user, verify it's for the same skill and type `buttons`, then `resolveSkillPending(userId)`, re-resolve skill with `resolveSkillEntry(commandName, ...)` to get `buttons[index].prompt`, build full prompt (skillPrompt + user input from that prompt), call `executeSkillWithPrompt(..., buttons[index].prompt)`, answerCallbackQuery, remove keyboard.

2. **Reply-to-message handling for `telegram.input`**  
   In the text handler, before the "slash command" block (or in a path that runs for every text message): if the message has `reply_to_message` and the bot has a pending skill interaction of type `input` for this user, and `reply_to_message.message_id === pending.messageId` (and same chat), then treat `messageText` as the user input. Call `resolveSkillPending(userId)`, resolve skill again by name, build prompt with skillPrompt + user input, call `executeSkillWithPrompt(..., messageText)`. Return so the message is not processed as a normal user message or slash command. If there is no such pending or reply doesn't match, fall through to normal handling (including slash and LLM).

## Phase 6: Hot reload cancels pending skill interactions

**Goal:** When the command/skill watcher triggers a republish, cancel all pending skill interactions so no stale keyboards or waits remain.

1. **Call `cancelAllSkillPending` from the watcher**  
   In `src/bot/commands/watcher.ts`, at the start of `republish()` (before `loadCommands`), call `cancelAllSkillPending(api)`. The watcher does not have `api` today; it only has `bot`. Use `bot.api` and pass it into `cancelAllSkillPending`. Ensure `skillPending` is imported in the watcher.

   Update `.agents/skills/draft_task/SKILL.md` and `examples/obsidian/.agents/skills/chuck/SKILL.md`: replace `telegram: true` with the nested form, e.g. `telegram:` with no subkeys (defaults) or `telegram: { enabled: true }`. Ensure no other SKILL.md files in the repo use a top-level `telegram` boolean (grep and fix any found).

2. **Update `docs/config/commands/README.md`**  
   Replace the "Project Skills" bullet and "Making skills visible" section with the new schema: `telegram` as an object, `enabled`, `help`, `menu`, optional `input`, `inputPrompt`, `buttons`, `timeout`. Document that absence of `telegram` means the skill is hidden from Telegram; presence with defaults means visible in both help and menu. Document legacy boolean as invalid (hard failure). Add a short example of the nested object and, if desired, a minimal example for `telegram.buttons` or `telegram.input`.

3. **Update any other docs that mention skill `telegram`**  
   Search `docs/` and READMEs for "telegram: true" or "telegram frontmatter" for skills; update to the new schema and link to the commands config doc if appropriate. Ensure `docs/skills/README.md` (if it mentions exposure) is consistent.

## Phase 8: Testing and acceptance

**Goal:** Manually (or with tests) verify all acceptance criteria.

1. **Parser and visibility**  
   - Start bot with migrated skills; confirm `/help` and menu show the expected skills.  
   - Add a skill with legacy `telegram: true` in frontmatter; confirm startup fails with a clear error and file path.  
   - Add a skill with invalid `telegram.buttons` (e.g. missing `prompt`); confirm load fails with actionable error.

2. **Buttons flow**  
   - Use a skill with `telegram.buttons`; confirm keyboard appears with options + Cancel, that selecting an option runs the skill with the injected prompt, and that Cancel dismisses the keyboard and does not run the skill.  
   - Confirm timeout removes the keyboard and that sending another slash command while waiting cancels the flow.

3. **Input flow**  
   - Use a skill with `telegram.input: true` and no args; confirm prompt message with Cancel appears, that replying to that message with text runs the skill with that text, and that messages that are not a reply to the prompt are ignored for this purpose.  
   - Confirm Cancel and timeout behave as for buttons; confirm one pending per user (new flow cancels previous).

4. **Hot reload**  
   - Start a skill button or input flow, then change a SKILL.md or command file to trigger the watcher; confirm the pending interaction is cancelled (keyboard cleared or prompt no longer active).

5. **Docs**  
   - Skim updated docs for consistency and correct examples.
