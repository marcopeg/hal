# 088 — Slash command handlers can yield to the agent

## Status

completed

## Summary

Extend `.mjs` slash command handling so a command implementation can decide at runtime whether HAL should handle the message programmatically or yield and let the original message continue through the normal LLM/skill pipeline.

This enables hybrid commands such as `/todo`:

- `/todo` with no arguments is handled directly by the `.mjs` command, which can inspect local state, render a formatted response, and potentially expose pagination or action buttons.
- `/todo buy milk` is recognized by the same `.mjs` command but deliberately yielded so the raw message can continue to the agent, where an appropriate skill can update the to-do list.

The refinement for this task locks in a typed command return contract so `.mjs` commands can act either as direct responders or as filters that selectively forward a message to the engine.

## Problem

Current `.mjs` command dispatch is effectively all-or-nothing: if a matching slash command exists, HAL treats that as the handler path. That makes it hard to build commands that are partly programmatic and partly AI-assisted.

For some command families, the best UX depends on the arguments:

- no arguments: inspect data locally and return a deterministic response;
- arguments present: let the agent/skill interpret the request and perform a higher-level action.

Without an explicit yield capability, a command author has to either:

- implement both behaviors entirely in JavaScript, duplicating skill logic; or
- avoid adding the command at all, losing the programmatic UX for the no-argument case.

The current slash-command pipeline also has an extra ambiguity around Telegram-enabled skills. Today, when there is no `.mjs` command, HAL can detect a same-name Telegram-enabled skill and execute it directly. For this task, yielded `.mjs` commands must not re-enter that skill shortcut path. They should go straight to the engine with the selected message.

## Goal

Allow a `.mjs` slash command to make an explicit routing decision at runtime:

- `handle` the message and return a programmatic response;
- `yield` the message to the engine so HAL continues through the normal agent path with the original or transformed text.

The yield path must preserve the agent-facing behavior where the message reaches the engine unchanged unless the command explicitly supplies a replacement message.

## Context & Constraints

- This builds on the existing `.mjs` slash command system introduced in [004 — Project-Specific Slash Commands](../completed/004.commands.md).
- This must align with the routing semantics being clarified in [082 — Built-in command visibility and enabled semantics](../082.command-visibility-and-enabled-semantics.md), especially the idea that unmatched or non-intercepted slash commands can fall through to lower-precedence handling and then to the agent.
- The decision to yield must be made by the loaded `.mjs` handler itself, not only by static config.
- Yielding should pass the full original incoming slash-command message to the engine by default, including the leading slash command and arguments.
- A command may optionally provide a replacement message when yielding, so the command can act as a filter or prompt transformer before the engine sees the input.
- A yielded command should not send an intermediate “handled” response from HAL before the message reaches the agent.
- When a matching `.mjs` command yields, HAL should not execute the Telegram skill shortcut lookup for that slash command. Skill exposure remains useful for command discovery/menu/help surfaces, but yielded `.mjs` commands should forward directly to the engine path.
- The task should introduce the typed return object as the preferred API and keep legacy return styles only for backward compatibility.

## Desired Behavior

### Command contract

Define a typed `.mjs` handler contract for routing decisions. The refined target API is an explicit result object rather than implicit string/null semantics.

Preferred typed results:

- `{ type: "agent", message?: string }`
  - forward to the engine
  - if `message` is omitted, HAL forwards the full original user message verbatim
  - if `message` is present, HAL forwards that replacement message to the engine instead
- `{ type: "assistant", message: string }`
  - HAL sends `message` back to the user as the command response
- `{ type: "void" }`
  - HAL does nothing further for that message
  - no assistant reply is sent
  - nothing is forwarded to the engine
  - intended for commands that manage their own multi-step interaction flow, callbacks, or button-driven UI state

Backward compatibility requirements:

- returning a raw `string` must continue to work as a direct assistant reply, but HAL should log a warning telling command authors to migrate to the typed return object
- returning `undefined` / no return must be treated as equivalent to `{ type: "agent" }`, forwarding the original full message to the engine, and HAL should log a warning telling command authors to migrate to the typed return object
- returning `null` or any other falsy non-typed legacy result must also be treated as equivalent to `{ type: "agent" }`, forwarding the original full message to the engine, and HAL should log a warning telling command authors to migrate to the typed return object

The important constraint is that the preferred long-term contract is the typed object and all new documentation/examples should use it.

### Dispatch flow

When a slash command matches a project or global `.mjs` command:

1. HAL loads and executes the `.mjs` handler.
2. The handler inspects args/context and returns either:
   - `{ type: "assistant", message }`, which HAL sends back to the user; or
   - `{ type: "agent", message? }`, which HAL forwards to the engine.
   - `{ type: "void" }`, which HAL treats as fully handled with no reply and no engine forwarding.
3. If `{ type: "agent" }` omits `message`, HAL forwards the original full slash-command text unchanged.
4. If `{ type: "agent", message }` provides `message`, HAL forwards that message to the engine instead of the original text.
5. If `{ type: "void" }` is returned, HAL stops routing for that message and performs no further action.
6. Yielding from a matched `.mjs` command does not invoke the direct Telegram skill execution shortcut; it goes straight into the engine path.

### Example: `/todo`

- `/todo`
  - handled by `.mjs`
  - reads the note or backing store
  - parses items in JavaScript
  - returns the current list, possibly with pagination buttons
- `/todo buy milk`
  - `.mjs` recognizes there are arguments
  - returns `{ type: "agent" }`
  - HAL forwards the original `/todo buy milk` message to the agent
  - the to-do skill interprets the command and updates the list

### Filter-style example

- `/summarize last 10 commits`
  - `.mjs` inspects local state and expands shorthand into a richer prompt
  - returns `{ type: "agent", message: "Summarize the last 10 commits in this repository and group them by feature area." }`
  - HAL forwards that transformed message to the engine

## Scope

### In scope

- Add an explicit typed return mechanism to the `.mjs` slash command runtime contract.
- Update runtime dispatch so yielded commands continue through the normal engine path.
- Preserve existing direct-response behavior for commands that do handle the message.
- Support command-authored message transformation before forwarding to the engine.
- Support a no-op `void` result for commands that fully manage their own interaction flow.
- Document the contract for command authors, including the typed return object and legacy compatibility behavior.
- Update the user-facing/project-facing command documentation to describe the new return contract and migration path.
- Add at least one test covering argument-sensitive yield behavior.
- Add backward-compatibility warnings for legacy return styles.

### Out of scope

- Designing a full button/callback protocol for paginated command UIs.
- Replacing skill behavior with JavaScript implementations for existing commands.
- Changing Telegram command registration/menu behavior.
- Reworking the separate skill-discovery/menu exposure mechanism beyond what is needed to clarify routing.

## Explicit Expectations

- `.mjs` command handlers should move toward a single typed result contract.
- Commands may act as a filter in front of the engine by returning `{ type: "agent", message }`.
- If a matched `.mjs` command yields to the engine, HAL must not try to execute a same-name Telegram-enabled skill directly in the slash-command branch.
- The default yield behavior must preserve the full original message exactly as received, including the leading slash command.
- Commands that fully manage their own UI or callback flow should be able to stop routing with `{ type: "void" }`.
- Legacy return styles should keep working during migration, but the runtime should emit clear warnings so command authors know the typed API is now preferred.
- The task must update the relevant documentation so command authors know how to migrate and how each return type behaves.

## Requirements

- A matching `.mjs` command must be able to explicitly forward to the engine after inspecting `args` and available context.
- The preferred `.mjs` return type must support at least:
  - `{ type: "assistant", message: string }`
  - `{ type: "agent", message?: string }`
  - `{ type: "void" }`
- `{ type: "agent" }` with no `message` must forward the original full user message, including the slash command.
- `{ type: "agent", message }` must forward the provided replacement message to the engine.
- `{ type: "void" }` must stop routing without replying and without forwarding anything to the engine.
- Yield/agent forwarding must continue processing through the engine path and must not invoke the direct Telegram skill execution shortcut for that command.
- The implementation must distinguish between:
  - assistant reply;
  - engine forwarding with original message;
  - engine forwarding with transformed message.
-  - fully handled no-op / self-managed command flow.
- Errors thrown by a command before yielding remain command execution errors and should continue to be logged/reported consistently.
- Yield semantics must work for both project-scoped and global fallback `.mjs` commands.
- Legacy raw-string returns must remain supported as assistant replies with a deprecation warning.
- Legacy `undefined` / no return must remain supported as engine forwarding of the original full message with a deprecation warning.
- Legacy `null` or other falsy non-typed returns must remain supported as engine forwarding of the original full message with a deprecation warning.
- The command author API must be simple enough to document with short examples.
- The implementation must update the relevant docs/examples that describe `.mjs` slash command authoring.

## Acceptance Criteria

- A `.mjs` command can return `{ type: "agent" }` to forward the original full slash-command message to the engine.
- A `.mjs` command can return `{ type: "agent", message: "..." }` to forward a transformed message to the engine.
- A `.mjs` command can return `{ type: "assistant", message: "..." }` and HAL sends that message back to the user.
- A `.mjs` command can return `{ type: "void" }` and HAL stops routing without replying or forwarding.
- When a matched `.mjs` command returns an agent result, HAL does not send a command response and does not execute the direct Telegram skill shortcut before forwarding to the engine.
- A sample command pattern like `/todo` can handle the no-argument case programmatically and forward the argument-bearing case to the agent.
- Returning a raw string still works, but logs a migration warning.
- Returning `undefined` / no return still works as forwarding the original full message to the engine, but logs a migration warning.
- Returning `null` or another falsy legacy non-typed value still works as forwarding the original full message to the engine, but logs a migration warning.
- Tests cover at least:
  - typed assistant response
  - typed agent forward with original message
  - typed agent forward with replacement message
  - typed void/no-op handling
  - legacy string response warning
  - legacy undefined/yield warning
  - legacy null/falsy yield warning
- Command author documentation explains the typed return contract, the legacy compatibility behavior, and when to use `assistant`, `agent`, or `void`.
- Relevant README/docs/examples for `.mjs` commands are updated to use the typed return object rather than raw string/empty return examples.

## Notes

- This feature is primarily about routing control, not UI. Buttons/pagination are an important motivator, but they can remain follow-up work.
- This task now assumes standardizing command result objects is the right direction. Backward compatibility remains important, but new examples and docs should use the typed return object only.
