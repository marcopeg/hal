# Plan: ASCII art HAL banner on startup

## Phase 1 — Design a portable banner output ✅

### Step 1.1: Define banner text payload ✅

- Implemented a compact multiline ASCII banner with HAL-eye motif and `HAL` label.
- Kept width small to avoid dominating standard terminal startup output.
- Ensured full readability without ANSI color (plain ASCII remains clean).
- Iteration update: replaced the eye motif with a larger `HAL` wordmark per user feedback and added `====` framing above/below to separate banner block from logs.

### Step 1.2: Define color fallback behavior ✅

- Decision: color support is opt-in by terminal capability, not by config/env flag for this task.
- Implemented red accent only for the eye character when ANSI is supported.
- Implemented plain fallback when `stdout` is non-TTY, `NO_COLOR` is set, or `TERM=dumb`.

## Phase 2 — Implement startup-only rendering in CLI flow ✅

### Step 2.1: Add a dedicated banner helper ✅

- Added helper functions in `src/cli.ts`:
	- `supportsAnsiColor(...)`
	- `renderStartupBanner(...)`
	- `printStartupBanner(...)`
- Decision: write directly to `process.stdout` to preserve ASCII layout and avoid logger formatting side effects.
- Iteration update: banner renderer now emits a framed block with a project-themed subtitle (`MULTI-ENGINE TELEGRAM COMMAND DECK FOR AI CODING AGENTS`).

### Step 2.2: Invoke banner at the correct boot point ✅

- Added `printStartupBanner()` at the top of `runStart()` before the first startup log (`Loading configuration`).
- `init` path is untouched, so no banner is printed there.

### Step 2.3: Preserve hot-reload behavior ✅

- Banner emission is only in initial `runStart()` entry path.
- Reload callback path does not call banner helper; no repeated banner on hot-reload.
