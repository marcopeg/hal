# 082 - Built-in command visibility and enabled semantics

**Status**: completed

## Problem

The current command config mixes together three different concerns:

- whether HAL itself should intercept a command;
- whether a command should be published in the Telegram slash-command menu;
- whether a command should appear in HAL-generated help output.

This makes it hard to keep Telegram-specific commands like `/start` available without also exposing them in user-facing menus, and it prevents hidden built-ins from falling through to the agent when HAL is not supposed to handle them.

## Goal

Separate command routing from command visibility.

Add explicit per-command visibility flags so HAL can:

- intercept a built-in command when enabled;
- hide it from the Telegram menu when desired;
- hide it from HAL-generated help when desired;
- stop intercepting disabled built-ins so lower-precedence custom commands or the agent can receive the raw slash-command text.

## Context & Constraints

- Config inheritance is assumed: every new command option must be supported at both `globals.commands.*` and `projects.<key>.commands.*`, with project-level values overriding globals when set.
- `/start` remains a Telegram-native command and should continue to be supported by HAL even if it is hidden from menu/help surfaces.
- `commands.<cmd>.enabled` should mean only whether HAL intercepts that command. When disabled, HAL should stop treating that built-in as routable and fall through to lower-precedence custom commands or the agent.
- When a built-in command is disabled, routing precedence should be: project custom commands first, then system/global custom commands, then the agent. Disabling a built-in must not block a same-name custom command from handling that slash command.
- The existing help/template pipeline should use the new visibility flags when building `${HAL_COMMANDS}`.
- Telegram menu publication via `setMyCommands` should use the new visibility flags independently of help visibility.
- Existing internal availability rules still apply. For example, `/model` and `/engine` remain conditionally unavailable when there is only one model/engine choice, even if their visibility defaults are `true`.
- `/new` is out of scope and should not be added or reintroduced by this task.

## What to do

### Add visibility flags

Add two new config keys for built-in command visibility:

- `commands.<cmd>.showInMenu`
- `commands.<cmd>.showInHelp`

Both should default to `true` unless a command defines a command-specific default override.

### Default built-in behavior

Define the following built-in defaults:

- `/start`
	- `enabled: true`
	- `showInMenu: false`
	- `showInHelp: false`
- `/help`
	- `enabled: true`
	- visible by default
- `/clear`
	- `enabled: true`
	- visible by default
- `/reset`
	- `enabled: false`
	- when enabled, visible by default unless explicitly overridden
- `/info`
	- `enabled: true`
	- visible by default
- `/model`
	- `enabled: true`
	- visible by default, subject to existing runtime availability rules
- `/engine`
	- `enabled: true`
	- visible by default, subject to existing runtime availability rules

### NPM command-family behavior

- The npm command family should remain controlled by `commands.npm` plus its whitelist/blacklist configuration.
- `commands.npm.enabled` should default to `false`.
- `commands.npm` is a single config entry that enables a derived npm command surface based on whitelist/blacklist resolution; it is not a requirement to keep or introduce a single `/npm` launcher command.
- When npm command handling is enabled and a command is exposed by the whitelist/blacklist rules, each allowed npm-derived command should be listed in both Telegram menu and HAL help by default.

### Routing semantics

- Enabled built-ins are intercepted and handled by HAL.
- Disabled built-ins are not HAL commands for routing purposes. HAL should first allow project custom commands, then system/global custom commands, and only pass the raw slash-command text to the agent when no higher-precedence custom command matches.
- This enabled/disabled meaning should be treated as a routing concern only, not as an implicit visibility toggle.

## Acceptance criteria

- [ ] `commands.<cmd>.showInMenu` and `commands.<cmd>.showInHelp` are supported in both global and project command config.
- [ ] `showInMenu` and `showInHelp` default to `true` unless explicitly overridden by command-specific defaults.
- [ ] `/start` is enabled by default but hidden from both Telegram menu and HAL help by default.
- [ ] `/help`, `/clear`, `/info`, `/model`, and `/engine` are enabled and visible by default.
- [ ] `/reset` is disabled by default.
- [ ] `/model` and `/engine` continue to respect their existing runtime availability constraints even when enabled and visible by default.
- [ ] `commands.npm.enabled` defaults to `false`.
- [ ] When npm handling is enabled, each npm-derived command allowed by whitelist/blacklist rules is shown in both Telegram menu and HAL help by default.
- [ ] When a built-in command is disabled, HAL routes matching same-name project commands before system/global commands, and forwards the raw slash-command text to the agent only if no custom command matches.
- [ ] Telegram menu publication and HAL help rendering use their respective visibility flags independently.
- [ ] No `/new` command or alias is introduced as part of this task.
