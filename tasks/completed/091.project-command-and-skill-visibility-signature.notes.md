# Notes - 091 Project command and skill visibility signature

## Decisions

- Normalized visibility into `enabled`, `showInMenu`, and `showInHelp` on `CommandEntry` rather than keeping the old skill-only `telegram?: boolean` flag. This keeps routing, menu publication, and help rendering on one shared model.
- Parsed skill frontmatter as YAML in [`/Users/marcopeg/dv/hal/src/bot/commands/loader.ts`](/Users/marcopeg/dv/hal/src/bot/commands/loader.ts) instead of extending the previous regex parsing. The object-form `telegram` contract requires structured validation, unknown-key rejection, and boolean field checks that are brittle with regex.
- Kept validation strict only for Telegram visibility metadata. Existing skip-on-warning behavior for unrelated issues like missing `description` or invalid command names was preserved to avoid broadening startup failure semantics beyond the task.
- Made `resolveCommandPath` async and visibility-aware so slash-command routing and callback routing both respect disabled `.mjs` commands. This was preferable to duplicating metadata parsing in each caller.
- Allowed `loadCommands` to omit disabled custom commands and Telegram-disabled skills from the active command surface entirely. This makes same-name fallback to lower-precedence skills happen naturally in menu/help composition.
- Left watcher reload failure behavior as "log error and do not republish" rather than clearing Telegram commands. This matches the existing degraded-failure model more closely and avoids surprising destructive menu updates on transient invalid edits.

## Problems encountered

- The first loader implementation still inserted disabled project/global `.mjs` commands into the merged command map. That would have continued to shadow lower-precedence skills in menu/help. Resolved by filtering disabled custom commands before they are inserted into the precedence map.
- The first pass also gave built-in command entries local `showInMenu`/`showInHelp` defaults that would have bypassed config-driven visibility overrides. Resolved by keeping config visibility precedence ahead of per-entry flags for built-ins/git entries.
- Changing `resolveCommandPath` to return a promise introduced TypeScript failures in [`/Users/marcopeg/dv/hal/src/bot/handlers/text.test.ts`](/Users/marcopeg/dv/hal/src/bot/handlers/text.test.ts), because older mocks were still using `mockReturnValue(...)`. Resolved by converting those mocks to `mockResolvedValue(...)`.
- Visibility-aware command resolution now imports modules before dispatch, which moved some potential failures earlier in the request path. Resolved by adding explicit error handling in both [`/Users/marcopeg/dv/hal/src/bot/handlers/text.ts`](/Users/marcopeg/dv/hal/src/bot/handlers/text.ts) and [`/Users/marcopeg/dv/hal/src/bot/handlers/mjs-callback.ts`](/Users/marcopeg/dv/hal/src/bot/handlers/mjs-callback.ts).

## Divergence from plan

- The plan called out reviewing startup and hot-reload publication flows in [`/Users/marcopeg/dv/hal/src/bot.ts`](/Users/marcopeg/dv/hal/src/bot.ts) and [`/Users/marcopeg/dv/hal/src/bot/commands/watcher.ts`](/Users/marcopeg/dv/hal/src/bot/commands/watcher.ts). No direct code changes were needed there because both already consume `loadCommands(...)` and `commandsForTelegramMenu(...)`; updating the shared loader path was sufficient.
- The plan mentioned distinguishing "file exists but is disabled" vs "file does not exist" in runtime resolution. The shipped solution does that indirectly by making `resolveCommandPath(...)` return only enabled command files. Callers do not receive a richer enum/state object because they only need the active routing surface, not diagnostic state branching.

## Known limitations

- Hot-reload validation failures still leave the previously published Telegram menu in place until the invalid metadata is fixed and republished successfully. The failure is logged and does not silently publish bad metadata, but it is not an active rollback/clear strategy.
- Skill frontmatter parsing remains limited to the `SKILL.md` frontmatter block and only validates Telegram visibility semantics added by this task. It is not a generalized schema validator for all possible skill metadata fields.
- The callback dispatcher now respects disabled commands, which means inline buttons from previously rendered messages for a command that has since been disabled will stop resolving. That is consistent with the new enabled semantics, but older Telegram messages can become inert after a disable.
