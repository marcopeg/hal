# 054 — Changelog generation from conventional commits

## Status

completed

## Summary

Investigate how to automatically generate a changelog based on the conventional commits used in this project, and determine the best tooling and workflow for producing and maintaining a `CHANGELOG.md`.

## Problem

The project follows the Conventional Commits specification, but there is no automated process to produce a changelog from that commit history. Releases and notable changes must be documented manually, which is error-prone and easy to skip.

## Context and constraints

- **Conventional commits**: Used in practice but not enforced. Enforcing them (e.g. commitlint or similar) is desired so bad commit messages cannot be committed accidentally; treat this as a prerequisite or explicit follow-up task in the findings.
- **Release workflow**: Releases are currently manual and local. The `deploy` script (`npm run deploy`) should evolve to:
  1. Ensure git working tree is clean (else fail early before publish).
  2. Run `npm login` (or equivalent auth).
  3. Generate changelog and commit it (so the release commit includes the updated `CHANGELOG.md`).
  4. Publish (e.g. `npm publish` or existing flow).
  Changelog generation must integrate into this `deploy` flow, not CI or a standalone one-off script.
- **Versioning**: Patch-only for now. A future task may introduce proper semver and/or move deploy to CI (e.g. GitHub Actions). Research should assume current patch-based, local-release workflow; semantic-release and tag-based workflows are out of scope for this task.

## Goals

- Identify suitable tools for auto-generating a changelog from conventional commits (e.g. `conventional-changelog`, `release-please`, `git-cliff`, `changie`).
- Evaluate how each tool fits integration into the `deploy` script (pnpm, Node, local run).
- Decide on format and placement for the generated `CHANGELOG.md`.
- Recommend how changelog generation fits the deploy steps (generate then commit before publish).

## Scope

- Research phase only: no implementation required until a tool and approach are chosen.
- Survey at least two or three candidate tools and note pros/cons for this project.
- Assume versioning stays patch-based for this task; do not assume semantic-release or tag-based releases.

## Deliverables

- A short findings document or notes section within this file summarising evaluated tools and the recommended approach.
- A decision on tooling and workflow that fits the target `deploy` flow (clean git → login → changelog + commit → publish), ready to be turned into a Ready task with a concrete implementation plan.
- Explicit note on enforcing conventional commits (e.g. commitlint) as a prerequisite or follow-up task.

## Acceptance criteria

- At least two changelog tools are evaluated with notes on fit for this project and for the `deploy`-script integration.
- A clear recommendation is recorded, including how changelog is generated and committed within the deploy flow.
- Enforcing conventional commit messages is identified and recorded as a prerequisite or follow-up task.

## Findings

### 1. conventional-changelog-cli (Node)

- **Ecosystem**: Node/npm; install as dev dependency and run via `pnpm exec conventional-changelog` or add to `package.json` scripts.
- **Invocation**: CLI only. Example: `conventional-changelog -p angular -i CHANGELOG.md -s` (preset, in-place update, prepend so existing content is kept).
- **Reads**: Git history; by default uses commits since the last semver tag. `-r 0` regenerates full history; `-u` / `--output-unreleased` includes unreleased commits.
- **Writes**: To a file (e.g. `CHANGELOG.md` at repo root); `-s` prepends new release block so the file is not overwritten.
- **Version source**: Release headings come from git tags (e.g. the tag created by `npm version patch`). The standard workflow is to run the generator inside the npm `version` script, so it runs after `package.json` is bumped but before the version commit is made; the new tag then matches the new version.
- **Docs**: [npm: conventional-changelog-cli](https://www.npmjs.com/package/conventional-changelog-cli), [GitHub: conventional-changelog](https://github.com/conventional-changelog/conventional-changelog).

**Pros for this project**: Pure Node/pnpm, no extra binaries; integrates naturally with `npm version` (add a `version` script that runs the CLI and `git add CHANGELOG.md`, so the release commit includes the updated changelog); works with patch-only workflow because `npm version patch` creates the tag the tool uses for the new section. Widely used and well documented.

**Cons**: Preset-based (e.g. angular or conventionalcommits); customizing output requires preset config or a custom preset. Depends on tags for release boundaries (which we already get from `npm version`).

---

### 2. git-cliff (Rust, npm wrapper available)

- **Ecosystem**: Rust binary; can be installed via npm/pnpm as a dev dependency (`pnpm add -D git-cliff`) and run with `pnpm exec git-cliff` (npm package provides the binary).
- **Invocation**: CLI. Example: `git cliff -o CHANGELOG.md` or `git cliff --prepend CHANGELOG.md` to prepend; config via `cliff.toml` (e.g. repo root).
- **Reads**: Git history; supports conventional commits via config (`conventional_commits = true`). Can scope by tag (`--latest`), unreleased (`--unreleased`), or range.
- **Writes**: To a path (e.g. `CHANGELOG.md`); `--prepend` merges new content into existing file.
- **Version source**: Configurable; can use `--tag` to set the version label for unreleased changes (e.g. from `package.json` via `$(node -p "require('./package.json').version")`).
- **Docs**: [git-cliff.org](https://git-cliff.org/), [npm: git-cliff](https://www.npmjs.com/package/git-cliff).

**Pros for this project**: Very flexible (templates, commit parsers, filtering); can run via pnpm; supports conventional commits and unreleased section; single binary, fast.

**Cons**: Config is TOML and more involved; version script integration requires passing the new version (e.g. from `package.json`) into `--tag` so the unreleased block is labeled correctly; Rust binary may add install size/time vs pure Node.

---

### 3. release-please

- **Ecosystem**: Node; designed around GitHub Actions and Release PRs.
- **Invocation**: Typically runs in CI; creates/updates a “Release PR” that updates `CHANGELOG.md` and version files; on merge, tags and optionally publishes.
- **Reads**: Conventional commits from the default branch.
- **Writes**: Updates `CHANGELOG.md` and version (e.g. `package.json`) inside the Release PR.
- **Version source**: Determines next version from commit types (feat → minor, fix → patch, etc.) and creates the tag on merge.

**Pros for this project**: Full automation in CI; good if we later move to tag-based, semver, and GitHub releases.

**Cons**: Geared to CI and Release PRs, not to a local `deploy` script. Overkill and poor fit for “run changelog generation and commit inside `pnpm run deploy`” as specified for this task. **Not recommended for the current local, patch-only deploy flow.**

---

### 4. changie (Go binary, file-based)

- **Ecosystem**: Go binary; not Node. Workflow is file-based: developers add change files (e.g. under `.changie/`), then run `changie batch` to build the changelog and optionally bump version.
- **Invocation**: CLI; `changie batch`, `changie merge`, etc.
- **Reads**: Not from git log; reads from change files written by developers.
- **Writes**: Merges change files into `CHANGELOG.md` and can update version files.

**Pros**: Flexible, version-agnostic, no dependency on commit message format.

**Cons**: Different workflow (maintain change files instead of deriving from conventional commits); requires installing a separate binary and changing team workflow. **Does not match the requirement to generate changelog from conventional commits in git.** Not recommended for this project’s stated goal.

## Recommendation

- **Tool**: **conventional-changelog-cli** with the **conventionalcommits** preset (or **angular** if preferred). Use it inside the npm **version** script so the release commit includes the updated `CHANGELOG.md` without a separate changelog commit step.
- **Placement**: `CHANGELOG.md` at repo root. No mandatory config file; optional `.changelogrc` or preset options if we want to tweak later.
- **Deploy flow** (target):  
  1. Ensure git working tree is clean (e.g. `preversion` script or manual check), then fail fast if not.  
  2. Run `npm login` (or equivalent) as needed.  
  3. Run **`npm version patch`**. This will:  
     - Bump `package.json` version.  
     - Run the **version** script: `conventional-changelog -p conventionalcommits -i CHANGELOG.md -s && git add CHANGELOG.md`.  
     - Create the release commit (including `CHANGELOG.md` and version bump) and tag (e.g. `v1.0.27`).  
  4. Run **`npm publish --access public`** (or existing publish command).

So “changelog + commit” is done inside `npm version`, not as a separate step. The `deploy` script can remain conceptually “version then publish”, with changelog generation and commit handled by the `version` script.

- **Preset**: `-p conventionalcommits` aligns with Conventional Commits; `-p angular` is also common and a bit stricter. Either is acceptable; document the choice in the implementation task.

---

## Prerequisites / follow-up

- **Enforce conventional commit messages** so the generated changelog has consistent, parseable input. Treat this as a **prerequisite or a dedicated follow-up task**, e.g.:
  - **commitlint** (+ **husky** pre-commit or commit-msg hook) to reject non-conventional commits, or  
  - A pre-push / CI check that fails if recent commits don’t match the convention.  
- Do not block the changelog implementation on this, but record it clearly so a follow-up task can add enforcement (and optionally extend the deploy script with a clean-tree check and login step as in the target flow above).

## Implementation outline (for future Ready task)

- Add **conventional-changelog-cli** (and optionally **conventional-changelog-conventionalcommits** preset) as a dev dependency.
- Add a **version** script in `package.json`, e.g.  
  `"version": "conventional-changelog -p conventionalcommits -i CHANGELOG.md -s && git add CHANGELOG.md"`  
  so that `npm version patch` (or minor/major) generates and commits the changelog.
- Keep **deploy** as “version then publish” (e.g. `npm version patch && npm publish --access public`), and optionally add a **preversion** script that checks for a clean git tree and exits with an error if not.
- Document in README or contributing docs: use conventional commits; changelog is generated at release time from git history; optionally mention that commitlint/husky for enforcement is a follow-up.
- Create a **follow-up task** (or prerequisite) to introduce commitlint (and optionally husky) to enforce conventional commits.
