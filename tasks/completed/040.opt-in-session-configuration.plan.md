# Plan: 040 — Opt-in session configuration

**Execution notes (2026-03-05):** All phases implemented. Schema uses Zod refinement to reject boolean `engine.session` with a clear message; `resolveProjectConfig` also guards and throws `ConfigLoadError` for defence in depth. Handlers use `usePerUserSession` (enabled && !(claude && shared)) for get/save sessionId. Session command includes Claude in active-reset when `engineSession.shared` is true and guards on `engineSession.enabled`. Claude adapter: shared → `--continue` and omit sessionId from result; per-user → `--resume <id>`; enabled=false → no session args. Codex: new `src/engine/codex-sessions.ts` for post-run UUID discovery; experimental flag uses `resume <UUID>` when stored, else `resume --last` and discover UUID after success. Cursor: experimental parses `session_id` from stdout JSON lines and uses `--resume <id>` when stored. OpenCode, Copilot, Antigravity: all use `config.engineSession.enabled` only. Examples: reference.yaml updated; examples/hal.config.yaml had no session key. Docs: config README session section, reference.yaml, all engine READMEs updated. Verified: config with `engine.session: true` is rejected with the expected error message; build and lint:fix pass.

## Overview

Replace the boolean `engine.session` with a structured `engine.session` object (`enabled`, `shared`, `experimentallyEnableUserSession`), update config resolution and all consumers (handlers, session command, adapters), implement Claude shared mode, Codex/Cursor experimental per-user session ID handling, and update examples and docs. No backward compatibility for the old boolean.

---

## Phase 1: Config schema and session object

**Goal:** Zod schema accepts only the new session object; old `session: true/false` is rejected with a clear error. Resolved defaults: `enabled: true`, `shared: false`, `experimentallyEnableUserSession: false`.

### Steps

1. **Define session config schema**
   - In `src/config.ts`, add `SessionConfigSchema`: `z.object({ enabled: z.boolean().optional(), shared: z.boolean().optional(), experimentallyEnableUserSession: z.boolean().optional() }).optional()`.
   - Do **not** allow `z.boolean()` for `session`. Use a refinement on `EngineConfigSchema` (or a discriminated parse): if `engine.session` is present and is a boolean, throw a `ConfigLoadError` with a message like: `engine.session must be an object (e.g. { enabled: true }), not a boolean. See docs for the new session configuration.`
   - Ensure the schema is used in both `GlobalsFileSchema` and `ProjectFileSchema` via `EngineConfigSchema` (replace `session: z.boolean()` with `session: SessionConfigSchema`).

2. **Reject boolean at load time**
   - During config load (after parse, before or inside `resolveProjectConfig`), if any `project.engine?.session` or `globals.engine?.session` is a boolean, throw `ConfigLoadError` with the message above. Alternatively, use a Zod transform/refine on the merged engine config so that a boolean `session` fails validation with a clear path in the error (e.g. `globals.engine.session` or `projects.<key>.engine.session`).

3. **Resolve session in `resolveProjectConfig`**
   - Compute merged session from `project.engine?.session` and `globals.engine?.session` (project over globals). If omitted entirely, treat as `{}`.
   - Resolved values: `enabled: merged.enabled ?? true`, `shared: merged.shared ?? false`, `experimentallyEnableUserSession: merged.experimentallyEnableUserSession ?? false`.
   - Add to the returned config object a new field `engineSession` of type `{ enabled: boolean; shared: boolean; experimentallyEnableUserSession: boolean }`.
   - Remove the old flat `engineSession: boolean` from `ResolvedProjectConfig` and from the return object of `resolveProjectConfig`.

## Phase 2: ResolvedProjectConfig type and handler/command guards

**Goal:** All code that reads `config.engineSession` uses the new object shape and guards on `config.engineSession.enabled` (and, where needed, `shared`).

### Steps

1. **Update `ResolvedProjectConfig` type**
   - In `src/config.ts`, change `engineSession: boolean` to:
     - `engineSession: { enabled: boolean; shared: boolean; experimentallyEnableUserSession: boolean }`.

2. **Update handlers (session read/write and pass-through)**
   - **text.ts:** Replace `config.engineSession` with `config.engineSession.enabled` for: (a) deciding whether to call `getSessionId(userDir)` (use `null` when disabled); (b) deciding whether to call `saveSessionId` after result (and only when enabled). When enabled and Claude with `config.engineSession.shared === true`, do not read/write session (handled in adapter; see Phase 3).
   - **voice.ts, photo.ts, document.ts:** Same pattern: guard all `getSessionId` / `saveSessionId` and `sessionId` pass-through on `config.engineSession.enabled`. For Claude shared mode, do not pass `sessionId` and do not save (adapter will use `--continue` and not return a sessionId to persist).

3. **Update session command (`session.ts`)**
   - Use `config.engineSession.enabled` where the current code checks session (e.g. for whether to run the "active reset" engine call). 
   - **Claude shared mode:** When `config.engine === "claude"` and `config.engineSession.shared === true`, include Claude in the "active reset" path: after `clearSessionData`, send `config.engineSessionMsg` via `agent.call(..., { continueSession: false })` so the shared session is started fresh (no `--continue`). Do not save any returned sessionId. When `config.engine === "claude"` and `config.engineSession.shared === false`, keep current "passive" behaviour (clear session, reply "New session started.").
   - Ensure `clearSessionData` is still called for all engines (no-op for shared Claude in terms of engine state, but clears per-user file if it existed).

## Phase 3: Claude adapter — shared vs per-user

**Goal:** When `engineSession.shared === true`, use `--continue` and do not use per-user session ID. When `engineSession.shared === false`, use `--resume <id>` and return session_id from stream-json for persistence.

### Steps

1. **Shared mode (`config.engineSession.shared === true`)**
   - In `execute()`, if `config.engineSession.shared` is true: do not add `--resume <sessionId>`; add `-c` / `--continue` instead (so the most recent session in `cwd` is resumed). Do not require `sessionId` from options.
   - In the result, when shared is true, do not set `sessionId` on the resolved result (so handlers do not persist a per-user session).
   - Ensure `-c` is used in the same place as `-p` (e.g. `claude -c -p "query"`).

2. **Per-user mode (`config.engineSession.shared === false`)**
   - Keep current behaviour: pass `--resume <sessionId>` when `sessionId` is provided; capture `session_id` from the `result` (or system/init) event in stream-json and return it in `EngineResult.sessionId`.
   - Existing code already captures `event.session_id` on result and `currentSessionId` from system/init; ensure `parse()` exposes it when needed (already returns `sessionId: parsed.session_id`).

3. **Stateless mode**
   - When `config.engineSession.enabled === false`, do not pass `--resume` or `--continue`; do not return a sessionId.

## Phase 4: Codex adapter — experimental per-user session

**Goal:** When `engineSession.experimentallyEnableUserSession === true`, use a real UUID for resume (from options) and after a successful run scan `~/.codex/sessions/` to extract the latest session UUID for the project `cwd` and return it so the handler can persist it.

### Steps

1. **Resume by UUID when experimental flag is true**
   - In `execute()`, when `config.engineSession.enabled` is true and `config.engineSession.experimentallyEnableUserSession` is true and `sessionId` is a non-placeholder string (e.g. not `"active"`), use `args.push("resume", sessionId)` instead of `resume`, `"--last"`.
   - When experimental is true but no valid stored sessionId, use `resume --last` for the first run; after the run, try to resolve a UUID from the filesystem (see step 2) and return it.

2. **Post-run session ID discovery**
   - Add a helper (e.g. in the adapter or a small `codex-sessions.ts` under `engine/`) that: (1) reads `~/.codex/sessions` (use `os.homedir()` for portability); (2) recursively lists files matching `**/rollout-*-*.jsonl` (or the pattern from the task); (3) for each file, optionally read the first line (or a small chunk) to get `cwd` from the JSONL payload; (4) filter to files where `cwd === config.cwd`; (5) sort by mtime or filename (newest first); (6) return the UUID from the first matching filename (e.g. extract from `rollout-<timestamp>-<UUID>.jsonl`).
   - Call this helper after a successful `codex exec` run when `experimentallyEnableUserSession` is true; set `result.sessionId` to the discovered UUID so the handler can call `saveSessionId`.
   - If no matching file is found, leave `sessionId` undefined or fall back to returning nothing (next run will use `--last` again).

3. **Guards**
   - When `config.engineSession.enabled` is false, do not add resume or session-related args (use fresh `exec` with `-C` cwd only).
   - When enabled and experimental is false, keep current behaviour: `resume --last` when `continueSessionRequested`.

## Phase 5: Cursor adapter — experimental per-user session

**Goal:** When `engineSession.experimentallyEnableUserSession === true`, use `--resume <session_id>` when a stored sessionId exists, and parse Cursor's stream-json (or json) output to capture `session_id` and return it for persistence.

### Steps

1. **Use `--resume` when experimental and sessionId present**
   - In `execute()`, when `config.engineSession.enabled` and `config.engineSession.experimentallyEnableUserSession` are true and `sessionId` is a non-placeholder string, pass `--resume`, `sessionId` instead of `--continue`.
   - When experimental is false, keep current behaviour: `--continue` when sessionId and continueSession, and return `"active"` (no persistence of real ID).

2. **Capture session_id from output**
   - When experimental is true, Cursor must be invoked with an output format that includes `session_id` (e.g. `--output-format json` or `stream-json` per Cursor docs). Parse stdout line-by-line (NDJSON) or single JSON object; from the terminal `result` event or `system/init` event, read `session_id` and set it on `EngineResult.sessionId` so the handler can call `saveSessionId`.
   - If the CLI does not support stream-json in print mode, use the non-stream json output and parse the final object's `session_id` after process exit.
   - Ensure `parse()` does not strip sessionId when experimental is true (currently `parse()` does not return sessionId; the handler uses `result.sessionId` from `EngineResult`, so no change to `parse()` is strictly required if we set `result.sessionId` in execute).

3. **Stateless and disabled**
   - When `config.engineSession.enabled` is false, do not pass `--continue` or `--resume`.

## Phase 6: OpenCode, Copilot, Antigravity — enabled-only

**Goal:** These adapters only need to guard on `config.engineSession.enabled` (no new flags). Antigravity already returns a real sessionId; OpenCode and Copilot keep current continue behaviour.

### Steps

1. **opencode.ts**
   - Replace every `config.engineSession` with `config.engineSession.enabled`. Logic unchanged: when enabled and hasActiveSession and continueSession, use `-c`; when enabled, return `sessionId: "active"` (or undefined when disabled).

2. **copilot.ts**
   - Same: replace `config.engineSession` with `config.engineSession.enabled` for the continue check and for setting `result.sessionId`.

3. **antigravity.ts**
   - Replace `config.engineSession` with `config.engineSession.enabled` for the `--resume` and session logic. No change to how sessionId is captured or returned.
