# 029 — Top-level providers config and /model validation

## Status

completed

## Summary

Move `globals.providers` to a top-level `providers` config key. Validate `/model {name}` against the configured list when available. Enhance `/model` (no args) to show current model, default, and available models from config.

## Requirements

- Promote `providers` from inside `globals` to a sibling of `globals` and `projects` in the config schema.
- Update config resolution: `project.providers[engine]` > `providers[engine]` > `[]`.
- Update merge logic so `hal.config.local` can override top-level `providers`.
- Update validation to reference `providers.<engine>` (not `globals.providers.<engine>`).
- `/model {name}`: when a `providers` list exists for the active engine, reject names not in the list. When no list exists, allow any name.
- `/model` (no args, list available): show engine, current model, default model (if known), and the full model list with indicators for current and default.
- `/model` (no args, no list): show engine and current model with a tip to use `/model <name>`.
- Model callback (inline button selection): validate against `providerModels` when non-empty.
- Update docs, example configs, and the `refresh_models` skill to reflect the new top-level key.

## Acceptance Criteria

- Config parser accepts `providers` at the root level (not inside `globals`).
- Per-project `providers` overrides still work.
- `providerDefaultModel` is resolved from the entry with `default: true` in the active engine's list.
- `/model {name}` rejects unknown models when a providers list is configured.
- `/model` (no args) shows default indicator when info is available.
- Model callback validates against the list.
- Docs, examples, and skill reference top-level `providers`.
