# Development plan — 055 NPM commands wrapper

## Phase 1 — Config schema + resolution ✅

### Step 1.1 — Zod schema for `commands.npm` ✅

Added `NpmConfigSchema` in `src/config.ts` with: `enabled`, `whitelist`, `blacklist`, `timeoutMs`, `maxOutputChars`, `sendAsFileWhenLarge`. Wired into `CommandsConfigSchema`.

### Step 1.2 — Extend `ResolvedProjectConfig.commands` ✅

Added `npm` block to the `commands` type. Resolved in `resolveProjectConfig()` with project > globals > defaults (enabled defaults to `false`, timeoutMs to 60000, maxOutputChars to 4000, sendAsFileWhenLarge to true).

### Step 1.3 — Extend `CommandEnabledFlags` ✅

Added `npm: boolean` to `CommandEnabledFlags` and `BUILTIN_ENABLED_MAP` in `src/bot/commands/loader.ts`.

### Step 1.4 — Register built-in command entry ✅

Added `npm` entry to `BUILTIN_COMMANDS` with `source: "builtin"`.

## Phase 2 — Bot command wiring ✅

### Step 2.1 — Create handler file ✅

Created `src/bot/commands/npm/index.ts` exporting `createNpmHandler` and `createNpmCallbackHandler`.

### Step 2.2 — Wire in `src/bot.ts` ✅

Imported handlers, wired behind `cmd.npm.enabled`, added `npm` to `enabledFlags`. Also updated `src/bot/commands/message.ts` (used by `/help`) to include the `npm` flag.

## Phase 3 — Script discovery + allowlist resolution ✅

### Step 3.1 — `package.json` reader ✅

Created `src/bot/commands/npm/scripts.ts` with `readPackageScripts()`. Throws `NpmScriptError` when `package.json` is missing, unreadable, invalid JSON, or has no scripts.

### Step 3.2 — Allowlist resolver ✅

Implemented `resolveAllowedScripts()` in the same file. Follows the decided algorithm: whitelist intersection first, then blacklist subtraction.

### Step 3.3 — Validation in handler ✅

Handler validates script against allowed list. Bare `/npm` (no script) replies with an inline keyboard of allowed scripts using `npm:<script>` callback data.

### Step 3.4 — Callback handler for script buttons ✅

`createNpmCallbackHandler` listens for `npm:` prefix, extracts script name, answers the callback query, removes the keyboard, and delegates to the shared `runScript()` function.

## Phase 4 — Subprocess execution + timeout ✅

### Step 4.1 — npm runner ✅

Created `src/bot/commands/npm/exec.ts` with `npmExec()`. Uses `child_process.spawn` with stdio piped. Timeout escalation: SIGINT → 3s → SIGTERM → 2s → SIGKILL. Returns `NpmResult` with stdout, stderr, exitCode, timedOut, durationMs. Disables color output via `FORCE_COLOR=0`.

### Step 4.2 — Concurrency lock ✅

Module-level `Map<string, boolean>` keyed by project slug. Lock set before spawn, cleared in `finally`. Concurrent requests rejected with a clear message.

## Phase 5 — Output formatting + delivery ✅

### Step 5.1 — Format the result message ✅

Created `src/bot/commands/npm/format.ts` with `formatNpmResult()`. Header shows script name, status icon (✅/❌/⏱), exit code, and duration. Logs wrapped in code block, truncated from tail.

### Step 5.2 — Large output as file ✅

When output exceeds `maxOutputChars` and `sendAsFileWhenLarge` is true, full log is attached as a `.txt` document via `gramCtx.replyWithDocument()` using Grammy's `InputFile`.

### Step 5.3 — Immediate feedback ✅

Before spawning, a `⏳ Running npm run <script>…` message is sent. After completion, the running message is edited with the result (falls back to a new reply if edit fails).

## Phase 6 — Docs + config reference ✅

### Step 6.1 — Update `docs/config/reference.yaml` ✅

Added `npm` key under `commands` with all options and their defaults.

### Step 6.2 — Update example config ✅

Added commented-out snippet in `examples/hal.config.yaml` showing how to enable `/npm` with a whitelist.

## Phase 7 — Testing ⏭️ (skipped)

No test framework exists in the project (no vitest/jest/mocha in package.json, no test files). Adding tests would require installing infrastructure. The allowlist resolver and package.json reader are pure functions ready to test once a framework is added.
