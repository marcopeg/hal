# Plan — 054 Changelog generation from conventional commits

## Overview

This task is **research only**. The outcome is a documented tool choice and workflow recommendation so a future Ready task can implement changelog generation inside the `deploy` script. No code or config changes are required in this task.

## Phase 1 — Survey candidate tools ✅

- List 2–3 (or more) tools that generate changelogs from conventional commits.
- For each tool, capture: name, ecosystem (Node vs other), how it's invoked (CLI / API), and where it reads/writes (tags, `package.json` version, file path).
- Candidates to consider: `conventional-changelog` (and preset-cli), `release-please`, `git-cliff`, `changie`, or others that match conventional commits + Node/pnpm-friendly usage.
- Document in the task file (or a "Findings" section): one short paragraph per tool with source links and basic usage.

## Phase 2 — Evaluate fit for deploy-script integration ✅

- For each surveyed tool, assess:
  - **Runnable from `pnpm run deploy`**: Can be invoked from a Node/pnpm script (e.g. `pnpm exec …` or `npx …`) without extra infra.
  - **Patch-only**: Works with current workflow (no requirement for git tags or semantic-release); or note if it assumes tags and how to adapt.
  - **Output**: Where does it write (e.g. `CHANGELOG.md` at repo root), and is the format acceptable (e.g. prepend new section, append, or overwrite).
  - **Version source**: Does it use `package.json` version, or tags, or something else; ensure it aligns with `npm version patch` (or current version bump step).
- Add a short "Pros / cons for this project" subsection per tool in the findings.
- Cross-check with target deploy flow: clean git → npm login → **changelog + commit** → publish (changelog step must run before publish and commit the updated file).

## Phase 3 — Document recommendation and prerequisites ✅

- Choose one recommended tool and workflow (how and when changelog is generated and committed in the deploy flow).
- In the task file:
  - Write a **Recommendation** section: chosen tool, exact step in deploy (e.g. "after login, run X; then `git add CHANGELOG.md && git commit -m 'chore: update changelog'`; then bump version and publish" or equivalent).
  - Note **CHANGELOG.md** placement (e.g. repo root) and any config file (e.g. `.changelogrc`) if needed.
  - Add a **Prerequisites / follow-up** section: explicitly state that enforcing conventional commits (e.g. commitlint + husky) is a prerequisite or a separate follow-up task, so the changelog has consistent input.
- Ensure the write-up is enough for a future Ready task to implement without re-doing the research.

## Phase 4 — Optional: outline future implementation task ✅

- Optionally add a short "Implementation outline" or "Ready task bullets" at the end of the task file: high-level steps for the follow-up implementation (e.g. add dependency, add `changelog` script, wire into `deploy`, document commitlint follow-up). This is not mandatory but helps the next task start quickly.

## Exit criteria (match task acceptance criteria)

- At least two changelog tools are evaluated with notes on fit for this project and for deploy-script integration.
- A clear recommendation is recorded (tool + how and when changelog is generated and committed in the deploy flow).
- Enforcing conventional commit messages is identified and recorded as a prerequisite or follow-up task.
