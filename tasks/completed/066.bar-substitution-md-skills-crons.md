# 066 — Prompt body substitution for cron `.md` files

## Status

completed

## Summary

Apply `${}` (context + env) and `@{}` (shell) substitution to cron `.md` prompt bodies at execution time, using the same resolved context map already passed to `formatContextPrompt()`.

## Context

HAL already supports three substitution patterns in config context values:

| Pattern | Evaluated | Description |
|---------|-----------|-------------|
| `${expr}` | Per message | Looks up `expr` in the full context map, then `process.env` |
| `#{cmd}` | Once at boot | Runs shell command, caches result |
| `@{cmd}` | Per message | Runs shell command fresh each time |

These patterns work inside the `context:` block of the HAL config (i.e. for context header values). However, cron `.md` prompt bodies are currently passed to the engine as-is — no substitution is applied.

The `substituteMessage(template, vars, logger)` helper in `src/context/resolver.ts` already implements `${}` + `@{}` substitution on arbitrary strings. It is currently used only for custom message templates (`message.ts`).

### What already works

- `${VAR}` in cron **frontmatter** — already resolved via `substituteVars()` before YAML parsing (resolves from env files and `process.env`)
- `${}` / `@{}` in config **context values** — fully supported

### The gap

- `${}` / `@{}` in cron `.md` **prompt bodies** — not applied; prompt is sent raw

## Scope

### In scope

- Apply `substituteMessage(def.prompt, contextVars, logger)` to cron `.md` prompt bodies in both `executeMdCron` and `executeMdProjectCron`, after `contextVars` is fully built (including `cron.runs` / `cron.lastRun`), and before `formatContextPrompt()` wraps the prompt with the context header.
- Cover both system-tier (`executeMdCron`) and project-tier (`executeMdProjectCron`) executors.
- Add tests and fixtures for cron prompt body substitution, including unresolved keys and shell expansion.
- Update documentation.

### Out of scope

- Any substitution in skill bodies — HAL exposes skills via the Telegram API as-is; skill prompt bodies must not be modified.
- Any changes to `.mjs` cron handlers.
- Changes to `${VAR}` frontmatter substitution (already works via `substituteVars()`).
- Changes to config-level `#{}` / `${}` / `@{}` semantics.
- New syntax — no bar-style `|var|` or any other new pattern.

## Requirements

1. In `executeMdCron` and `executeMdProjectCron`, after the `contextVars` map is fully built, call `substituteMessage(def.prompt, contextVars, logger)` to resolve `${}` and `@{}` in the prompt body.
2. Pass the resolved prompt (not the raw one) to `formatContextPrompt()`.
3. Unresolved `${expr}` keys (absent from both contextVars and `process.env`) become empty string — this is existing `resolveAppVars` behaviour; document it clearly.
4. Add test coverage for:
   - `${context.key}` resolved from contextVars
   - `${ENV_VAR}` resolved from `process.env`
   - `@{cmd}` shell expansion in prompt body
   - Unresolved key → empty string (existing behaviour, verify and document)
5. Update `docs/crons/` to describe prompt body substitution.

## Acceptance Criteria

- `${bot.userId}`, `${sys.date}`, `${MY_ENV_VAR}` and similar expressions inside a cron `.md` prompt body are expanded at execution time using the fully-resolved context map.
- `@{date}` and similar shell expressions inside a cron prompt body are evaluated fresh at execution time.
- Unresolved keys produce an empty string (no crash, no silent data corruption).
- Existing `${VAR}` frontmatter substitution continues to work unchanged.
- Skill prompt bodies are unaffected.
- Test coverage exists for all cases above.

## Notes

- Execution order: 1. build context (`buildCronContextVars` + cron state keys), 2. `substituteMessage` on prompt body (`${}` then `@{}`), 3. `formatContextPrompt` (prepend context header), 4. `agent.call`.
- `substituteMessage` must NOT be applied to the context header block — only to the raw prompt body, before `formatContextPrompt` is called.
