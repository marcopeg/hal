## Description

Replace the current non-interactive `init` command with a full interactive setup wizard. The wizard reads the current configuration (if any), identifies missing mandatory information, and interactively guides the user through completing it. It is the primary entry point for new users and the fallback for incomplete configurations.

### Invocation

- **Explicit:** `npx @marcopeg/hal wiz` — runs the wizard directly.
- **Implicit (auto-trigger):** When `npx @marcopeg/hal` (i.e. `start`) is invoked, the startup sequence first loads and analyzes the config. If no config file exists, the wizard runs from scratch. If a config file exists but any wizard-coverable field is missing or unresolved (i.e., any field that corresponds to a wizard step — project `cwd`, bot token, user ID, engine, session — is absent or still a placeholder), the wizard kicks in to fill only the gaps. Fields that are outside the wizard's scope (e.g., logging, rate limiting, advanced engine-specific options) do not trigger the wizard — those are manual-edit concerns. Once all wizard-coverable fields are satisfied, `start` proceeds normally without prompting.
- **Legacy:** The `init` command remains functional for now but is considered deprecated. It will be removed in a future release once the wizard fully supersedes it.

### Idempotency

- Skip questions whose answers are already present in the config.
- `--reset` flag re-asks all questions, overriding current values.

## Requirements

### CLI Pre-fill Flags

The wizard accepts flags that pre-fill specific configuration values. When a flag is provided, the corresponding wizard step is skipped entirely (the value is applied silently). This enables concise, shareable one-liners for documentation and help text:

```
npx @marcopeg/hal wiz --engine cursor
npx @marcopeg/hal wiz --engine cursor --model claude-sonnet-4-5
npx @marcopeg/hal wiz --bot-key 123456:ABC-DEF
```

Supported flags:

| Flag | Wizard step skipped |
|------|---------------------|
| `--engine <name>` | Engine / model picker (engine selection) |
| `--model <name>` | Engine / model picker (model selection) |
| `--bot-key <value>` | Telegram bot token |
| `--user-id <value>` | Telegram user ID |
| `--session <mode>` | Session configuration |

- Flags are validated the same way wizard inputs are; if a value is invalid the wizard falls back to prompting for that step instead of failing hard.
- Pre-filled values are shown in the confirmation summary at the end so the user can review them before writing.

### Steps (minimum)

1. **Project name and `cwd`** — Ask for the project's display name (optional). Derive the project key as a slug from it and set `projects[slug].name` to the full name. If the user skips or provides an empty name, use key `prj1` and do not set the `name` property. Then list subdirectories of the config directory as selectable options; also offer "Use current directory (.)". Store the chosen path as that project's `cwd`.
2. **Telegram bot token** — prompt for the API key; offer help ("do you need help?"), and if yes, open the GitHub docs page with setup instructions in the browser.
3. **Telegram user ID** — same as above, with optional help link.
4. **Additional allowed users** — ask if more users should be added; repeat until the user selects "no, I'm done".
5. **Engine / model** — let the user pick the engine; show the model picker only if the engine exposes an explicit list or supports model selection.
6. **Session configuration** — configure session using the **current** `engine.session` schema as implemented after task 040 (see Context): boolean or `"shared"` / `"user"` where the engine supports it. Present engine-appropriate options (e.g. on/off for all; shared vs per-user for Claude; per-user for Codex/Cursor where supported). Default to sensible values and skip options the selected engine doesn't support.

### UX

- **Interactive CLI library:** Use [`@clack/prompts`](https://github.com/bombshell-dev/clack) — modern, TypeScript-native, 80% smaller than alternatives, beautiful default styling, with `select`, `multiselect`, `text`, `confirm`, `spinner`, and `group` primitives. Actively maintained (7.4k+ stars, 6M+ weekly downloads as of early 2026).
- Display progress at every step ("Step 3 of 5" or a progress bar).
- Where multiple choices exist, offer an interactive picker (keyboard arrow keys to navigate, Enter to select).
- At the end, display the resulting config and ask the user to confirm before writing (same format as the existing config when present, otherwise YAML).
- After writing, ask the user whether they want to start the bot or exit the wizard.

### Architecture: Modular & Extensible

The wizard must be designed as a **modular step pipeline** so that new steps can be added, removed, or reordered without touching the core wizard logic:

- Each wizard step is a self-contained module (e.g. `src/wizard/steps/engine.ts`, `src/wizard/steps/bot-token.ts`) that exports a standard interface: check whether its value is already configured, prompt the user if needed, and return the result.
- The wizard runner iterates through a registry of steps, handling progress display, skip logic (idempotency + pre-fill flags), and cancellation uniformly.
- Engine-specific steps (like session sub-options) conditionally activate based on the engine selected in a prior step.
- This architecture allows future expansion — new configuration dimensions (e.g., engine-specific permissions like Codex's `networkAccess`/`fullDiskAccess` or Antigravity's `approvalMode`/`sandbox`) can be added as new step modules without modifying the wizard core.

## Context & Constraints

- **Session schema (task 040):** In 040 we evolved to the signature that is implemented now in `src/config.ts`: `engine.session` is optional and accepts `boolean` or `"shared"` | `"user"` (see `SessionSchema` and `SessionMode`). The wizard's session step must use this schema — no separate structured object.
- The current `init` command (`src/cli.ts` `runInit()`) generates YAML from `src/init-template.yaml` with placeholder substitution (`{{ENGINE_NAME}}`, `{{ENGINE_MODEL}}`, `{{PROJECT_CWD}}`). Task 052 removed doc-link placeholders; the template now has hardcoded doc URLs. The wizard replaces the init flow with interactive steps but can reuse template rendering and file-writing (e.g. engine instructions file creation).
- **Config format and existing config:** The wizard **respects and extends the current configuration**, honoring the existing format when a config file already exists. When gap-filling or completing an existing config, write back to the **same file** in the **same format** (JSON, JSONC, or YAML). When creating from scratch (no config), create a new file in the default format (YAML, e.g. `hal.config.yaml`). Do not prompt to "replace or keep" the old file — always update in place when a config exists. `resolveConfigFile(configDir, "hal.config")` in `src/config.ts` is already exported for the analyzer.
- **Projects shape:** `projects` is a map keyed by project slug. When creating new, the wizard asks for an optional project display name, derives the key as a slug, and sets `projects[slug].name`; if the user skips or leaves the name empty, use key `prj1` and omit `name`. When filling gaps, the wizard operates on the single project (e.g. the first or only key in the map).
- The wizard must handle the case where a config file exists but is incomplete — read existing values, skip already-configured steps, and prompt only for what's missing.
- **Secrets:** Prefer writing the bot token to the config directory's `.env` or `.env.local` (task 041: custom env file); wizard can create/append there and write `botToken: "${TELEGRAM_BOT_TOKEN}"` in the YAML.

## Notes

- The pre-fill flags are the primary mechanism for making the wizard shareable in docs — e.g. `npx @marcopeg/hal wiz --engine cursor` is a clean, copy-pasteable command that guides the user through only the remaining steps.
- Engine-specific permission configuration (Codex: `networkAccess`, `fullDiskAccess`, `dangerouslyEnableYolo`; Antigravity: `approvalMode`, `sandbox`) is **not** part of this task. These are engine-dependent and complex; they should be added as separate wizard step modules in a follow-up task once the modular architecture is in place.
