# 058 — Fix broken session continuity across adapters

**Status**: completed

## Description
Recent changes appear to have broken session behavior for adapters that rely on shared/cwd-scoped continuation markers. This was first observed on Copilot and Codex, and the same handler gating pattern affects other adapters using non-user session continuation semantics.

## Goals
- Reproduce and characterize the regression for all affected adapters (Copilot, Codex, Cursor, OpenCode) and document what changed and where.
- Define adapter-level session capability metadata so the system can determine whether an engine supports true user-isolated sessions.
- Restore consistent continuation behavior for shared/cwd-scoped engines while preserving per-user isolation where engines truly support it.
- Fail fast during config load when config requests `engine.session: "user"` for an adapter that cannot support user-isolated sessions.
- Keep this validation compatible with config hot reload: once config is fixed, project runtime recovers without process restart.

## Acceptance Criteria
- Copilot, Codex, Cursor, and OpenCode maintain continuation across multiple messages when configured with `engine.session: true` or `"shared"`.
- For adapters that support true user-isolated sessions, `engine.session: "user"` maintains per-user continuity without cross-user leakage.
- For adapters that do not support true user-isolated sessions, startup/config resolution fails with a clear error when `engine.session: "user"` is requested.
- When config is invalid due to unsupported `engine.session: "user"`, hot reload keeps reporting load failure until fixed; after fixing config, hot reload successfully applies it and the project resumes normally.
- Adapter interface exposes session capability information (for example: supports user-isolated sessions) and configuration validation/runtime logic uses it instead of hardcoded engine-name checks.
- `session.json` is treated as the continuation marker for shared/cwd-scoped adapters now, and remains available to store real session IDs in future adapters.
- A regression test and/or manual test checklist is documented so future changes don’t re-break session handling.

## Findings (what broke, when, and why)

### Symptom
Even with sessions enabled, Copilot and Codex behave as if every message is a fresh run: no continuity, no reuse of prior context.

### Root cause
This regression was introduced when session handling was refactored for task **040 (opt-in session configuration)**.

In the message handlers (e.g. `src/bot/handlers/text.ts`, `voice.ts`, etc.) the code now only **loads `session.json`** (via `getSessionId(userDir)`) for engines that are treated as *per-user resumable*:

- Claude
- Antigravity
- or when `engine.session` is explicitly set to `"user"`

That means **shared/cwd-scoped adapters in default session modes (`true` / `"shared"`) never receive a `sessionId` marker**.

However, adapters currently only enable their continuation mechanisms if a `sessionId` exists:

- **Copilot** (`src/engine/adapters/copilot.ts`): adds `--continue` only when `sessionId != null`.
- **Codex** (`src/engine/adapters/codex.ts`): uses `codex exec resume --last` only when `sessionId != null`.
- **Cursor** (`src/engine/adapters/cursor.ts`): continuation path is similarly gated by session state.
- **OpenCode** (`src/engine/adapters/opencode.ts`): continuation path is similarly gated by session state.

So the end-to-end chain is broken:

1) adapters may return a placeholder `sessionId` (`"active"`) and handlers may even **save** it to `session.json`, but
2) handlers don’t **read** it back for Copilot/Codex, so
3) adapters never see `sessionId`, therefore never pass `--continue` / `resume --last`.

### “What decision got broken?”
Task **008 (Engine session continuity)** explicitly decided that:

- Copilot session continuity should be achieved via `--continue` (no real session ID available), and
- Codex continuity should be achieved via `resume --last` (unless/until per-user session ID discovery is implemented).

The refactor in 040 unintentionally turned “has an engine session already started” into “did we load a per-user session ID?”, which is not true for shared/cwd-scoped default modes.

### Likely regression commit
`579d46c` ("feat: 040 support confi options") added the `usePerUserSession` gating in handlers; after this, Copilot/Codex stop continuing because they never get `sessionId`.

## Fix direction (implementation options)

### Chosen direction — Treat `session.json` as a *continuation marker* for shared engines
For engines that only support cwd-scoped continuation (Copilot/OpenCode/Codex default/Cursor default), still **load `session.json`** and pass the placeholder `"active"` back into adapters so they can enable `--continue` / `-c` / `resume --last`.

- Pros: minimal change, preserves “first message starts fresh, subsequent continue” behavior.
- Cons: does **not** create true per-user isolation for these engines (not possible with their current CLIs).

### Capability contract requirement
The adapter contract must expose session capability metadata so configuration/session policy can be validated against actual adapter support.

- Required behavior:
	- if config requests `engine.session: "user"` and adapter capability says unsupported, HAL must throw a clear error and refuse to start that project.
	- this rejection happens at config load/resolve time (including watcher-triggered reload), not deferred to first user message.
	- if config is `true` or `"shared"` for shared/cwd-scoped engines, use `session.json` as continuation marker (`"active"`) so continuation flags are sent from second message onward.
	- this task does not require introducing synthetic per-user isolation for CLIs that do not support it.

## Repro / validation checklist
- **Copilot**: `engine=copilot`, `engine.session=true` → send two messages; verify the *second* invocation includes `--continue` (currently it doesn’t).
- **Codex**: `engine=codex`, `engine.session=true` → send two messages; verify the second invocation uses `codex exec resume --last` (currently it uses `codex exec -C <cwd>`).
- **Cursor**: `engine=cursor`, `engine.session=true` → send two messages; verify second invocation uses continuation mode instead of fresh-start path.
- **OpenCode**: `engine=opencode`, `engine.session=true` → send two messages; verify second invocation uses continuation mode instead of fresh-start path.
- **Unsupported user mode guard**: for each adapter that cannot do user-isolated sessions, set `engine.session="user"` and verify startup/config error is explicit and adapter-capability-driven.
- **Hot reload recovery**: while bot is running with config watcher, change config to unsupported `engine.session="user"` and verify reload fails clearly; then fix config (`true` or `"shared"`) and verify next reload succeeds and message handling works without restart.

> Note: for adapters that cannot provide true user-isolated sessions, the expected behavior is explicit rejection of `engine.session="user"`, not silent fallback.
