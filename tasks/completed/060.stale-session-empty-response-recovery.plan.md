# Plan: 060 — Stale session empty-response recovery

## Overview

Bring Claude's continuation behavior in line with Copilot and Codex by detecting stale-session continuation failures, retrying the same prompt once in a fresh session, and surfacing a warning to the Telegram user instead of returning a blank or opaque failed reply.

## Execution status

- Status: completed
- Date: 2026-03-16
- Validation: `npm run lint && npm run build` passed

## Current code context

- `src/engine/adapters/claude.ts` currently has a single execution path and does not distinguish between fresh runs and continuation retries once the CLI arguments are built.
- Claude currently uses `--resume <sessionId>` for user mode and unconditional `--continue` for shared mode, but it does not honor `continueSession: false` from `EngineExecuteOptions`.
- Warning delivery already exists in message handlers through `EngineResult.warning`.
- Existing Copilot behavior in `src/engine/adapters/copilot.ts` is the reference pattern for: attempt resume, detect stale continuation failure, retry fresh once, and attach a warning.
- For Claude shared mode there is no persisted continuation marker to clear locally; recovery relies on forcing one fresh run so the stale continuation attempt is not reused for the current message.

## Phase 1 — Define Claude stale-session recovery contract ✅

- Done: treated both Claude continuation modes as retryable candidates:
  - user-mode `--resume <sessionId>`
  - shared-mode `--continue`
- Done: implemented adapter-local stale detection for:
  - success with blank assistant output after a continuation attempt
  - continuation-specific failures based on resume/session/conversation wording plus invalid/missing-state signals from result/stderr
- Done: added Claude warning copy aligned with the existing Copilot/Codex recovery pattern.
- Decision: retry remains capped at one forced fresh rerun per incoming user message.

## Phase 2 — Refactor Claude adapter execution flow ✅

- Done: refactored `src/engine/adapters/claude.ts` around a reusable `runClaudeProcess()` helper so a continued attempt and a forced fresh attempt can share the same parsing path.
- Done: updated Claude argument building to honor `continueSession === false`, which suppresses both `--resume` and `--continue`.
- Done: preserved existing progress logging, tool progress updates, and session ID extraction while moving execution into the reusable helper.
- Decision: kept the refactor adapter-local instead of introducing a cross-adapter retry abstraction.

## Phase 3 — Add stale-session retry and warning behavior ✅

- Done: detect-and-retry logic now runs only when the first Claude attempt actually used continuation.
- Done: stale-session recovery logs a `warn` event with project slug, user ID, continuation mode, error/output signal, and stderr.
- Done: successful fresh retries return the retried Claude response with `EngineResult.warning` set.
- Done: failed fresh retries stop after one retry and return the fresh failure normally.
- Done: malformed stored Claude session IDs are now treated as stale local state before invoking the CLI, so HAL no longer leaks invalid `--resume` UUIDs to the user.
- Decision: user-mode session cleanup after failed retries still relies on the existing handler behavior that clears `session.json` when no replacement `sessionId` is returned.
- Decision: shared mode still uses Claude's native latest-session behavior after a forced fresh rerun; no new shared marker persistence was introduced.

## Phase 4 — Verify command/session interactions stay correct ✅

- Done: the Claude adapter now respects `continueSession: false`, which aligns shared `/clear` renewal with the intended fresh-session behavior.
- Done: fresh-session failures remain non-retrying because stale-session recovery is gated on an actual continuation attempt.
- Done: handler-side session cleanup now uses the effective session mode, so engines whose `engine.session: true` resolves to user mode still clear stale `session.json` state correctly.
- Done: no other adapters were modified.

## Risks and sequencing notes

1. Claude stale-session detection must stay narrow enough that normal model refusals or unrelated engine failures are not silently retried.
2. Shared Claude continuation has no local session marker, so recovery should rely on a forced fresh rerun rather than introducing new persistence unless implementation proves that is necessary.
3. Honoring `continueSession: false` in Claude is likely required for both this task and existing shared reset semantics; treat that change as part of the same adapter refactor so behavior stays coherent.
4. Keep the implementation adapter-local unless a second code path proves a reusable helper is justified; this task does not need a cross-adapter retry framework.

## Execution notes

- The detection heuristic intentionally stays conservative: automatic retry only happens when Claude was already attempting continuation and either produced blank output or surfaced continuation-related invalid-session signals.
- Follow-up: malformed UUID-like values persisted in `session.json` are now intercepted before Claude CLI execution and force the same warning + fresh-session recovery path.
- The warning path reuses existing handler behavior via `EngineResult.warning`, so no Telegram callback or command-layer changes were needed.
- Manual validation is still required to confirm the exact Claude CLI error strings seen with stale non-native backends and to verify the retry path against a real stale session.
