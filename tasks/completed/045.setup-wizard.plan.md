## Overview

**Goal:** Install `@clack/prompts`, create the `src/wizard/` directory structure with the step interface and type definitions.

### Steps

1. **Install `@clack/prompts`**
   - `pnpm add @clack/prompts`
   - Verify it's added to `dependencies` in `package.json` (it's a runtime dependency, not dev-only â€” the wizard runs from the published CLI).

2. **Create wizard types and step interface**
   - Create `src/wizard/types.ts` with:
     ```ts
     import type { ConfigFormat } from "../config.js";

     export interface WizardContext {
       cwd: string;                    // working directory (configDir)
       existingConfig: PartialConfig | null; // parsed existing config (null = fresh)
       existingConfigPath: string | null;    // path to the existing config file (null = fresh)
       existingConfigFormat: ConfigFormat | null; // format of the existing config (null = fresh â†’ default YAML)
       prefill: PrefillFlags;          // CLI pre-fill flags
       reset: boolean;                 // --reset flag
       results: Record<string, unknown>; // accumulates step outputs
     }

     export interface PrefillFlags {
       engine?: string;
       model?: string;
       botKey?: string;
       userId?: string;
       session?: string;
     }

     export interface WizardStep {
       id: string;
       label: string;
       isConfigured(ctx: WizardContext): boolean;
       shouldSkip?(ctx: WizardContext): boolean;
       run(ctx: WizardContext): Promise<void>;
     }
     ```
   - `PartialConfig` is the loosely-typed raw config object (pre-Zod), read directly from the config file (YAML, JSON, or JSONC). Define it as a minimal type covering the fields the wizard cares about (engine name, bot token, user IDs, project cwd, session, project name).

   - Stub implementation: iterates steps, calls `isConfigured()` to skip, shows progress ("Step N of M"), calls `run()`.
   - Handles `@clack/prompts` `isCancel()` for graceful Ctrl+C exit at any point.
   - When `ctx.reset` is true, skip the `isConfigured` check (always prompt).

4. **Create step registry**
   - Create `src/wizard/steps/index.ts` â€” exports an ordered array of all wizard steps. Initially empty; steps are added in subsequent phases.

5. **Create wizard entry point**
   - Create `src/wizard/index.ts` â€” exports `startWizard(cwd: string, prefill: PrefillFlags, reset: boolean): Promise<void>`.
   - Uses `resolveConfigFile(cwd, "hal.config")` to detect existing config; reads it into `PartialConfig` (raw parse, no env substitution), captures its `path` and `format` into `WizardContext.existingConfigPath` / `existingConfigFormat`.
   - Builds `WizardContext`, imports step registry, calls `runWizard()`.

## Phase 2: Config analyzer (completeness check) âś…

**Goal:** Detect whether an existing config is "complete" from the wizard's perspective, and determine which fields are missing.

### Steps

1. **Create config analyzer**
   - Create `src/wizard/analyzer.ts`.
   - Export `analyzeConfig(cwd: string): { configExists: boolean; configPath: string | null; configFormat: ConfigFormat | null; missingFields: string[] }`.
   - Logic:
     1. Use `resolveConfigFile(cwd, "hal.config")` from `src/config.ts` (already exported).
     2. If no file found â†’ `configExists: false`, `missingFields: ["all"]`.
     3. If file found â†’ parse it raw (no env substitution â€” just structural check), inspect wizard-coverable fields on the first project key in the map:
        - project `cwd` present and not a placeholder
        - project `telegram.botToken` present and not a literal `${...}` placeholder without a resolving env var
        - `globals.access.allowedUserIds` or the project's `access.allowedUserIds` has at least one real ID (not the template placeholder)
        - `globals.engine.name` or project's `engine.name` is set
        - `engine.session` is present (boolean or `"shared"` / `"user"`)
     4. Return list of missing field IDs (matching step IDs: `"project-name"`, `"cwd"`, `"bot-token"`, `"user-id"`, `"engine"`, `"session"`).

2. **Export `needsWizard` helper**
   - `needsWizard(cwd: string): boolean` â€” returns `true` when `missingFields.length > 0` or `configExists === false`.
   - This is the function `runStart()` in `cli.ts` will call to decide whether to auto-trigger the wizard.

## Phase 3: Wizard steps (project name, cwd, bot token, user ID, engine/model) âś…

**Goal:** Implement the core wizard steps (1â€“5 from task requirements), each as a self-contained module.

### Steps

1. **Step: Project name** (`src/wizard/steps/project-name.ts`)
   - `isConfigured`: check `ctx.existingConfig?.projects` has at least one entry (gap-fill mode â€” we already have a project key).
   - `run`: use `@clack/prompts` `text` to ask for the project's display name (optional, user can press Enter to skip). If a name is provided, derive a slug from it (lowercase, replace spaces/special chars with hyphens, trim) and store `ctx.results.projectKey = slug`, `ctx.results.projectName = displayName`. If empty/skipped, store `ctx.results.projectKey = "prj1"`, `ctx.results.projectName = undefined` (omit `name` property in config).

2. **Step: Project `cwd`** (`src/wizard/steps/cwd.ts`)
   - `isConfigured`: check `ctx.existingConfig?.projects` has at least one entry with a non-placeholder `cwd`.
   - `shouldSkip`: skip if prefill flag not applicable (cwd uses `--cwd` from the global CLI args, not a wizard-specific flag).
   - `run`: use `@clack/prompts` `select` to list subdirectories of `ctx.cwd` (via `readdirSync` + `statSync` to filter dirs only), plus a "Use current directory (.)" option. Store chosen path in `ctx.results.cwd`.

3. **Step: Telegram bot token** (`src/wizard/steps/bot-token.ts`)
   - `isConfigured`: check existing config has a bot token that is not a `${...}` placeholder, OR the `TELEGRAM_BOT_TOKEN` env var is set.
   - `shouldSkip`: skip if `ctx.prefill.botKey` is set and valid.
   - `run`: use `@clack/prompts` `text` with validation (non-empty, matches Telegram bot token format `digits:alphanumeric`). Offer a `confirm` follow-up: "Need help creating a bot?" â†’ if yes, open `HAL_DOCS_TELEGRAM_CREATE_BOT` in the default browser (reuse `open` / `xdg-open` logic from existing `openFileInEditor`). Store in `ctx.results.botToken`.
   - Suggest storing the token in `.env` rather than inline in the config file, with a note explaining why (secrets shouldn't be committed).

4. **Step: Telegram user ID** (`src/wizard/steps/user-id.ts`)
   - `isConfigured`: existing config has at least one numeric entry in `access.allowedUserIds`.
   - `shouldSkip`: skip if `ctx.prefill.userId` is set and valid.
   - `run`: use `text` prompt with numeric validation (1â€“0xFFFFFFFFF range, reuse `parseTelegramUserId` logic). Offer help link similar to bot token step. Store in `ctx.results.userId`.

5. **Step: Additional allowed users** (`src/wizard/steps/additional-users.ts`)
   - `isConfigured`: always returns `false` (this step is additive â€” you can always add more users). But skip if not in `--reset` mode and existing config already has users configured.
   - `run`: loop â€” `confirm` "Add another user?", if yes â†’ `text` prompt for ID, validate, accumulate. When user says no, store array in `ctx.results.additionalUserIds`.

6. **Step: Engine / model** (`src/wizard/steps/engine.ts`)
   - `isConfigured`: existing config has `engine.name` set.
   - `shouldSkip`: skip if `ctx.prefill.engine` is set and valid.
   - `run`:
     1. `select` prompt with all valid engines (`claude`, `copilot`, `codex`, `opencode`, `cursor`, `antigravity`).
     2. If `ctx.prefill.model` is set, apply it silently. Otherwise, show a `text` prompt for model name with the engine's default pre-filled (from `DEFAULT_PROVIDER_MODEL`). User can accept the default or type a custom model.
     3. Store `ctx.results.engine` and `ctx.results.model`.

7. **Register steps in the step registry**
   - Update `src/wizard/steps/index.ts` to import and export all six steps in order.

## Phase 4: Config generation, confirmation, and file writing âś…

**Goal:** After all steps run, build the config (respecting existing format), show a summary, ask for confirmation, write to disk, and offer to start the bot.

### Steps

1. **Config builder** (`src/wizard/config-builder.ts`)
   - Export `buildConfigFromResults(ctx: WizardContext): { content: string; targetPath: string; envEntries?: Record<string, string> }`.
   - Construct a config object from `ctx.results` + any existing config values (for idempotent runs where some fields were preserved). Use `ctx.results.projectKey` as the project map key and set `projects[key].name` to `ctx.results.projectName` when defined.
   - **Format-aware serialization:** When an existing config file exists (`ctx.existingConfigFormat` is set), serialize to the **same format** (YAML, JSON, or JSONC). When creating from scratch, default to YAML.
     - YAML: use the `yaml` package (already a dependency) to serialize with comments (doc-link header, section comments similar to `init-template.yaml`).
     - JSON: use `JSON.stringify(obj, null, 2)`.
     - JSONC: use `JSON.stringify(obj, null, 2)` (comments from the original file will be lost â€” acceptable; JSONC writing with preserved comments is out of scope).
   - Target path: when updating, write to `ctx.existingConfigPath`; when creating, write `hal.config.yaml` to `ctx.cwd`.
   - For bot token: if the user provided it inline, write `botToken: "${TELEGRAM_BOT_TOKEN}"` in the config and return `envEntries` with `TELEGRAM_BOT_TOKEN: <actual value>` for writing to `.env`. If user explicitly wants inline, allow that too.

2. **Confirmation + write step** (`src/wizard/confirm-and-write.ts`)
   - Show the generated config using `@clack/prompts` `note` (or plain `console.log` with syntax-highlighted box).
   - Show a summary of all values (including pre-filled ones) before the config.
   - `confirm` prompt: "Write this configuration?"
   - If yes: write the config to the target path. If `envEntries` were returned, append to `.env` / `.env.local` in `ctx.cwd`.
   - If no: abort with a friendly message.
   - Create engine instructions file if missing (reuse logic from `runInit`).

3. **Post-write action**
   - `select` prompt: "Start the bot now?" / "Exit"
   - If start: return a signal to the caller so `cli.ts` can proceed to `runStart()`.
   - If exit: `process.exit(0)`.

## Phase 6: CLI integration (wiz command + auto-trigger) âś…

**Goal:** Wire the wizard into `src/cli.ts` as both an explicit `wiz` command and an auto-trigger on incomplete config.

### Steps

1. **Extend `ParsedArgs` and arg parser**
   - Add `"wiz"` to the command union: `command: "start" | "init" | "wiz"`.
   - Parse `wiz` as a command in the args loop.
   - Parse wizard-specific flags: `--bot-key`, `--user-id`, `--session`, `--reset`. (Existing `--engine`, `--model`, `--cwd` are already parsed.)
   - Extend `ParsedArgs` with `botKey?: string`, `userId?: string`, `session?: string`, `reset?: boolean`.

2. **Add `wiz` command handler in `main()`**
   - After `parseArgs()`, if `command === "wiz"`, import and call `startWizard()` from `src/wizard/index.ts` with the parsed prefill flags and reset.
   - If the wizard signals "start the bot", call `runStart(cwd)`.

3. **Add auto-trigger in `runStart()`**
   - Before `tryLoadMultiConfig()`, call `needsWizard(configDir)` from the analyzer.
   - If it returns `true` and `process.stdin.isTTY` (interactive terminal), import and run the wizard in gap-filling mode (only missing fields).
   - If not a TTY (e.g., running in CI/Docker), skip the wizard and let the existing `ConfigLoadError` flow handle it (so non-interactive environments don't hang).
   - After the wizard completes, retry `tryLoadMultiConfig()`.

4. **Update help text**
   - Add `wiz` to the commands list in `showHelp()`.
   - Add wizard-specific flags to the options list.
   - Update the "no config found" error message in `loadMultiConfigInternal` to suggest `npx @marcopeg/hal wiz` instead of (or in addition to) `init`.

5. **Deprecation notice for `init`**
   - When `init` is invoked, print a deprecation notice: "Note: `init` is deprecated. Use `wiz` for an interactive setup experience."
   - Keep `init` functional â€” no behavioral changes.

6. **Build script update**
   - Verify `pnpm run build` succeeds with the new `src/wizard/` directory.
   - No special handling needed â€” TypeScript will compile the new files. The `init-template.yaml` copy in the build script is unaffected.
