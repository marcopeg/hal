# 086 — Copilot per-user session isolation

## Status

completed

## Summary

Mirror the per-user session isolation model introduced for Codex in task 034 onto the Copilot engine. Copilot CLI exposes a structured JSON output mode that includes the session UUID in the final `result` event. HAL can parse that event after each successful run, persist the UUID per Telegram user, and resume with `--resume <UUID>` on later turns. This task changes Copilot's default session mode to `"user"`, implements real UUID recovery, removes the stale `"active"` placeholder, surfaces recovery failures as Telegram warnings, and updates all affected docs.

## Problem

Copilot currently uses a shared-continuation marker model: on every successful run HAL stores `"active"` in `session.json` and issues `--continue` on the next message. This means all Telegram users who share the same project are resumed into the same Copilot session, with no per-user isolation. The broader concern matches what was already addressed for Codex but has not yet been applied to Copilot.

Additionally, the `"active"` marker model prevents HAL from ever storing a real session identifier for Copilot, which makes the stored session.json meaningless as a restart or continuity anchor. The Copilot CLI already supports `--resume <UUID>` for per-session continuation — HAL simply does not use it today.

## Current Context

- Copilot adapter: `src/engine/adapters/copilot.ts`.
- `sessionCapabilities.supportsUserIsolation: false`, `defaultMode: "shared"`, `sharedContinuationRequiresMarker: true`.
- On success, returns `sessionId: "active"` (placeholder) unconditionally when sessions are enabled.
- Continuation uses `--continue` (most-recent shared session), not `--resume <UUID>`.
- Codex precedent: `src/engine/codex-sessions.ts` implements `snapshotCodexSessionPathsForCwd` / `findLatestCodexSessionForCwd` snapshot diffing; same pattern is applicable here.
- Copilot prompt mode supports `--output-format json`, which emits JSONL events and includes the session UUID in the final `result` event.
- Resume flag: `copilot --resume <UUID> -p <prompt>`.
- Per-user mode equivalent for Copilot: run with structured JSON output → extract `sessionId` from the final `result` event → persist per-user.

## Product Decisions

- Copilot should default to per-user session mode. Omitting `engine.session` for Copilot should resolve to `"user"` (via `sessionCaps.defaultMode`), consistent with Codex behavior.
- Shared Copilot sessions remain available as an explicit opt-in via `engine.session: "shared"`. Shared mode continues to use `--continue`.
- In Copilot `"user"` mode, `/clear` should only delete the user's local `session.json` and must not perform an active reset call to Copilot.
- In Copilot `"user"` mode, if HAL cannot discover a real Copilot session UUID after a fresh run, it must not silently fall back to shared continuation (`--continue`). Instead: warn the Telegram user and log for the operator.
- In Copilot `"user"` mode, if HAL has a previously stored UUID but `--resume <UUID>` can no longer recover that session, HAL should clear the stale local session, warn the Telegram user, log for the operator, and degrade to fresh runs until a new UUID is discovered.
- Session files may rotate, be compacted, or be deleted by Copilot — treat them as best-effort continuation, with graceful fresh-start degradation on any failure.
- This task covers implementation and documentation only. The developer should add a manual test checklist, but the actual two-device Copilot validation run will be performed afterward by the user.

## Expected Behavior

- In Copilot `"user"` mode, HAL runs prompts with structured JSON output enabled.
- After a successful run, HAL extracts `sessionId` from the final `result` event and persists it in that user's `session.json`.
- On subsequent messages, if a stored UUID exists, HAL runs `copilot --resume <UUID> -p <prompt>`.
- If a stored UUID is stale or no longer resumable, HAL clears the stale local session record, warns the Telegram user, logs the recovery failure, and falls back to fresh-run behavior for later messages rather than failing closed or switching to shared continuation.
- If no UUID is recoverable after a fresh run, HAL sets `warning` on `EngineResult`, clears any stale `session.json` entry, and informs the Telegram user that session continuity is temporarily unavailable.
- In Copilot `"shared"` mode, behavior is unchanged: `--continue` is used and `/clear` performs an active reset.
- In Copilot `"user"` mode, `/clear` deletes the user's `session.json` only — no CLI call is made.

## Acceptance Criteria

- Copilot advertises `supportsUserIsolation: true` and resolves omitted or `true` `engine.session` to `"user"` by default.
- In Copilot `"user"` mode, HAL persists a real Copilot UUID in the user's `session.json` after a successful fresh run when discovery succeeds.
- In Copilot `"user"` mode, HAL resumes with `copilot --resume <UUID>` and never uses the stale `"active"` placeholder as a per-user continuation marker.
- In Copilot `"shared"` mode, HAL continues to use `--continue`, and `/clear` still performs an active reset.
- In Copilot `"user"` mode, `/clear` only deletes the user's local `session.json` and does not make a Copilot reset call.
- If Copilot UUID discovery fails after a fresh run, HAL clears stale local session state, warns the Telegram user, logs the failure for the operator, and does not silently fall back to shared continuation.
- If a previously stored Copilot UUID becomes stale or invalid, HAL clears stale local session state, warns the Telegram user, logs the failure for the operator, and degrades to fresh runs until a new UUID is discovered.
- All affected user-facing docs in the repo are updated so Copilot session defaults and per-user support are described consistently.
- A manual test checklist for two-device Copilot isolation is added under `manual-tests/`; executing that checklist is explicitly deferred to post-implementation user validation.
