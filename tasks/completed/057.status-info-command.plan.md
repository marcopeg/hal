# 057 Plan - Status/info command with feature flags

## Objective
Implement a built-in `/info` command with configurable section visibility, Telegram-optimized formatting, and docs updates.

## Scope
- Add built-in command `/info`.
- Add config node `commands.info` with flags:
  - `enabled` (default `true`)
  - `cwd` (default `true`)
  - `engineModel` (default `true`)
  - `session` (default `true`)
  - `context` (default `true`)
- Output format:
  - Message 1: compact summary (`Project`, `CWD`, `Engine`, `Model`, `Session`).
  - Message 2: context only, fenced code block, sorted keys.
  - If context is large, split into multiple fenced code-block messages.
- Disabled behavior:
  - When `commands.info.enabled: false`, `/info` replies `This command is disabled.`
- Docs updates for config and command behavior.

## Out of scope
- Env/source-aware secret redaction in context output.

## Phase 1 - Config support
1. Extend command config schema in `src/config.ts` to accept `commands.info` fields in globals and project scopes.
2. Extend `ResolvedProjectConfig` type in `src/config.ts` to include `commands.info` resolved booleans.
3. Resolve defaults and precedence (project > globals > defaults) for all `commands.info` fields.
4. Ensure config loading remains backward compatible when `commands.info` is absent.

## Phase 2 - Command handler and wiring
1. Create `src/bot/commands/info.ts` with `createInfoHandler(projectCtx)`.
2. In `src/bot.ts`, wire `/info` when enabled.
3. In `src/bot.ts`, add disabled fallback for `/info` and `/info@botname` using same behavior as `/model` and `/engine`.

## Phase 3 - Command discovery/menu/help integration
1. Update `src/bot/commands/loader.ts`:
   - Add `info` to `CommandEnabledFlags`.
   - Add `/info` to `BUILTIN_COMMANDS`.
   - Add `info` to `BUILTIN_ENABLED_MAP`.
2. Update `enabledFlags` in `src/bot.ts` to include `info`.
3. Update `${HAL_COMMANDS}` path in `src/bot/commands/message.ts` so `/info` appears/disappears based on enabled state.

## Phase 4 - Output formatting behavior
1. Build summary message with stable field order and Markdown-safe formatting.
2. Include fields conditionally using `commands.info` section flags.
3. Build context message from resolved context key-value pairs, sorted by key.
4. Render context in fenced code blocks.
5. When context exceeds Telegram limits, split into multiple fenced code-block messages while preserving valid Markdown in each chunk.

## Phase 5 - Documentation updates
1. Update `docs/config/commands/README.md`:
   - Add `/info` section and examples.
   - Document `commands.info` flags and defaults.
   - Document disabled behavior and two-message format.
   - Document current limitation: no env/source-aware redaction.
2. Update `docs/config/reference.yaml` with `commands.info` structure and defaults.
3. Update `docs/config/README.md` where built-in command lists/notes need `/info` coverage.

## Phase 6 - Validation
1. Run `pnpm run lint`.
2. Run `pnpm run build`.
3. Manual checks:
   - Default `/info` shows summary + context code-block message.
   - Per-flag toggles show/hide only intended sections.
   - `commands.info.enabled: false` returns disabled message for `/info` and `/info@botname`.
   - Long context splits into multiple valid code-block messages.
   - `/info` appears/disappears in Telegram menu and `${HAL_COMMANDS}` according to enabled state.

## Risks and notes
- Context provenance is not currently available after resolution; env-origin redaction cannot be implemented reliably in this task.
- Markdown-safe splitting must avoid broken code fences and malformed messages.
