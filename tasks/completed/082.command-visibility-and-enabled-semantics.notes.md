# Notes — 082 Built-in command visibility and enabled semantics

## Decisions

### `/npm` kept as a direct launcher despite not being in BUILTIN_COMMANDS
The task said _"it is not a requirement to keep or introduce a single `/npm` launcher command"_. Rather than removing the `/npm` Grammy handler entirely, I kept `bot.command("npm", createNpmHandler)` registered in bot.ts. This means users who already know `/npm` still get the script-picker keyboard. The change is that `/npm` no longer appears in the Telegram menu or in `${HAL_COMMANDS}` — individual script entries replace it. This was the least disruptive path and matched the spirit of the task (replace the surface, not break the shortcut).

### npm script entries use sanitized raw script names (no prefix)
I considered prefixing npm script names (e.g. `npm_build`) to avoid any collision with built-in command names. I decided against it because:
- The map deduplication already handles conflicts: if `build` is a project custom command, the npm entry is never added.
- Enabled built-ins are added to the map _before_ npm entries; when a built-in is enabled, the npm script with the same name would overwrite it in the map — so I reversed the order: built-ins overwrite npm entries. Actually, this was handled naturally by the existing precedence: npm entries are added _after_ the disabled-built-in filter, so they only fill slots not already occupied.
- Using the raw name gives cleaner UX in the Telegram menu.

### `bot.hears(...)` removal for disabled `/model`, `/engine`, `/info`
The original code intercepted disabled built-ins with `bot.hears(/^\/model.../i, ctx => ctx.reply("This command is disabled."))`. This blocked the text handler from seeing those slash commands, preventing fallthrough to custom commands or the agent. I removed all three `bot.hears` interceptors. The Grammy command handler for enabled built-ins still registers first (correct precedence). When disabled, no handler is registered and the message reaches the text handler which routes through custom commands → skills → agent.

The stale inline-keyboard callback handler (`en:` / `md:` prefixes) was preserved — it answers old inline buttons with "This command is disabled." This is correct because callback queries have no fallthrough mechanism equivalent to slash commands.

### `commandsForHelp` is separate from `commandsForTelegramMenu`
The plan called for these to be independent. I added `commandsForHelp(commands, visibility?)` alongside the updated `commandsForTelegramMenu(commands, visibility?)`. Both accept the same `CommandVisibility` map but filter on different keys (`showInHelp` vs `showInMenu`). The `message.ts` `buildHalCommands` function was updated to use `commandsForHelp`. This satisfies the requirement that menu and help visibility are fully independent.

### Visibility map is built at call site, not passed through `loadCommands`
I considered embedding visibility into `CommandEntry` itself so `loadCommands` could attach it. I went with a separate `CommandVisibility` map instead, built in `bot.ts` and `message.ts` from the resolved config. This keeps `CommandEntry` lean and `loadCommands` focused on routing (enabled/disabled filtering), with the visibility concern handled by the filter functions that wrap its output.

### npm NpmScriptError silently omitted from menu/help; logged only for unexpected errors
When `readPackageScripts` throws `NpmScriptError` (missing or empty `package.json`), npm entries are silently omitted from the menu. When an unexpected error (non-NpmScriptError) is thrown, a `logger.warn` is emitted. This matches the plan guidance: _"prefer graceful omission plus runtime error messaging over startup failures"_.

### `/reset` default changed from `true` to `false`
The task spec was explicit: `/reset` should be disabled by default. This is a breaking change for users who relied on the previous default. I changed the default in `resolveProjectConfig` and updated docs and CHANGELOG accordingly. Existing configs that explicitly set `reset.enabled: true` are unaffected.

### `CommandVisibility` uses `Partial<Record<string, {...}>>` not a typed struct
I could have defined a fully typed `CommandVisibility` with named keys for each command. I used `Partial<Record<string, { showInMenu?: boolean; showInHelp?: boolean }>>` instead. This allows npm-derived script names (which are dynamic) to also be carried in the same map without requiring a separate lookup path.

## Problems encountered

### TypeScript error when spreading `never` in tests
The test helper `createProjectContext()` returns `as never`. Spreading it (`{ ...createProjectContext(), config: { ... } }`) caused TS2698. The fix was to duplicate the full context object into a `createNpmProjectContext()` helper rather than trying to spread or mutate the `never`-typed return value.

### `npm` entry in `BUILTIN_ENABLED_MAP` caused double-filtering
When I removed `npm` from `BUILTIN_COMMANDS` but left `npm: "npm"` in `BUILTIN_ENABLED_MAP`, the filter loop tried to `map.delete("npm")` on every load even though npm was never added as a builtin. Harmless but confusing — I removed the `npm` entry from `BUILTIN_ENABLED_MAP` at the same time.

### `executeNpmScript` was not exported
The text handler needed to call the npm runner for individual script commands. `runScript` in `npm/index.ts` was private. I added `export async function executeNpmScript(ctx, gramCtx, script)` that delegates to `runScript`. This avoids duplicating the concurrency lock, error formatting, and file-sending logic.

### `git_*` commands needed to be mapped individually in the visibility map
The `git` config entry is a single `{ enabled, showInMenu, showInHelp }` object, but there are four separate `CommandEntry` records (`git_init`, `git_status`, `git_commit`, `git_clean`). The visibility map is keyed by command name, so I expanded the single `cmd.git.showInMenu` value into four entries in the map. Slightly verbose but correct.

## Divergence from original plan

### Step 3.2 — npm surface is individual script entries, `/npm` kept for direct invocation
The plan discussed whether to keep or remove the single `/npm` launcher. I kept it as a registered Grammy handler (not in the menu) and exposed individual scripts in the menu/help. This is consistent with the plan's note: _"If the implementation still keeps `/npm` as an execution entry point for some flows, document exactly how that coexists with the exposed derived commands"_.

### Step 4.1 — No separate "disabled" callback handling was needed for `/info`
The plan mentioned updating `bot.ts` so disabled `/info` doesn't respond with "This command is disabled." I removed the `bot.hears` for `/info` (same as model/engine). `/info` never had a callback handler, so no additional change was needed.

### No `showInMenu`/`showInHelp` on `CommandEntry` itself
The plan mentioned potentially embedding visibility on `CommandEntry`. I kept it out of the struct and handled it only in filter functions. Simpler and avoids the need to pass visibility through the full `loadCommands` call.

### npm script routing in `text.ts` uses a sanitized name reverse-match
When a user clicks `/tunnel_start` (sanitized from `tunnel:start`), the text handler receives `tunnel_start` as the command name. The npm check loops through allowed scripts and applies the same sanitization to each, then compares. This correctly maps `tunnel_start` → `tunnel:start`. The approach is slightly more expensive than a precomputed lookup map but the script list is small and the operation is rare.

### Post-execution: `examples/copilot/` missing `package.json`

After task completion, the user reported that the npm commands defined in `examples/hal.config.yaml` (copilot project, `npm.enabled: true`, whitelist `[tunnel:stop, tunnel:start, tunnel:logs, hello]`) did not appear in either the Telegram menu or HAL help.

Root cause: both `loadCommands` and `text.ts` called `readPackageScripts(cwd)` unconditionally to resolve which scripts to expose. When `package.json` doesn't exist at the configured `cwd`, `NpmScriptError` is thrown and silently caught, resulting in zero npm entries — even when an explicit `whitelist` was configured. The visibility logic itself was correct (npm entries default to visible), but they never made it into the command list.

An intermediate incorrect fix (creating `examples/copilot/package.json` with fake scripts) was reverted. Adding a fake `package.json` to the examples directory was wrong.

Correct fix: when `whitelist` is non-empty, treat it as the authoritative list of scripts — no `package.json` required. Only read `package.json` when no whitelist is provided (auto-discovery mode). This is the correct semantic: the user who configures a whitelist knows exactly which scripts to expose; they shouldn't need a `package.json` present for HAL to honor that config. Both `loadCommands` and the `text.ts` routing were updated consistently.
