# Plan for 034 — Validate Codex session ID isolation

## Goal

Make Codex default to per-user session isolation in HAL, keep explicit shared mode available, remove unsafe fallback from per-user mode, warn users when UUID recovery fails, document the new semantics, and verify the behavior with a real two-client manual test.

## Phase 1 — Confirm current behavior and lock the target semantics

### Step 1.1 — Trace the existing Codex session flow

- Review how `engine.session` resolves today in `src/config.ts`.
- Confirm how handlers load/save `session.json` in `src/bot/handlers/*.ts` and `src/user/setup.ts`.
- Confirm how Codex currently chooses between fresh execution, `codex exec resume --last`, and `codex exec resume <UUID>` in `src/engine/adapters/codex.ts`.
- Confirm how `/clear` currently behaves for Codex in `src/bot/commands/session.ts`.

### Step 1.2 — Align implementation targets with the refined task

- Treat Codex omitted `engine.session` as default per-user behavior.
- Keep `engine.session: "shared"` as the only way to request project-level shared continuation.
- In Codex `"user"` mode, treat missing or unrecoverable UUIDs as no-continuity state, not as permission to use shared continuation.
- In Codex `"user"` mode, define `/clear` as local session-file deletion only.
- Preserve existing shared-mode reset behavior for Codex `"shared"`.

## Phase 2 — Change resolved session defaults and mode-specific reset behavior

### Step 2.1 — Make Codex default to per-user mode

- Update config resolution in `src/config.ts` so Codex resolves omitted `engine.session` to `"user"` while other engines keep their existing defaults.
- Ensure explicit `true`, `false`, `"shared"`, and `"user"` still behave predictably and remain validated against adapter capabilities.
- Review whether any helper functions or adapter capability defaults need to change so runtime behavior matches resolved config semantics.

### Step 2.2 — Split `/clear` behavior by Codex session mode

- Update `src/bot/commands/session.ts` so Codex in `"user"` mode only deletes `session.json` and does not send the active reset message.
- Keep Codex `"shared"` mode in the active-reset path so shared session reset still creates a fresh shared Codex conversation.
- Verify that `/start` reuse of the same reset helper inherits the intended behavior.

## Phase 3 — Harden Codex per-user session recovery

### Step 3.1 — Make `session.json` authoritative for per-user Codex continuation

- In Codex `"user"` mode, only resume with `codex exec resume <UUID>` when `session.json` contains a real session ID.
- Ensure placeholder marker flows such as `"active"` are not reused in per-user Codex mode.
- Keep shared mode using the existing marker-based continuation path.

### Step 3.2 — Tighten post-run session discovery

- Review and harden `src/engine/codex-sessions.ts` so HAL reliably finds the latest matching Codex session transcript for the configured `cwd`.
- Verify the helper assumptions against current Codex session file layout and contents.
- If needed, improve matching, ordering, and extraction logic to reduce the risk of attaching the wrong transcript UUID.

### Step 3.3 — Remove unsafe fallback in Codex `"user"` mode

- Ensure a user without a recoverable UUID never falls through into `resume --last` shared behavior.
- Treat the no-UUID case as fresh-session execution only.
- After a successful fresh run, attempt transcript discovery and persist the recovered UUID when available.

## Phase 4 — Surface UUID recovery failures safely

### Step 4.1 — Add operator-visible logging

- Log when Codex `"user"` mode fails to recover a session UUID after a successful fresh run.
- Keep logging in normal HAL logs/terminal output with enough context to diagnose the failure without dumping excessive transcript data.

### Step 4.2 — Add user-visible warning behavior

- Define how the engine/handler returns a user-facing warning when Codex `"user"` mode cannot recover a UUID.
- Ensure Telegram users are told that user-specific continuity is unavailable and later messages will run in fresh anonymous sessions until recovery succeeds.
- Keep the warning scoped to the failure condition so normal successful runs are unaffected.

### Step 4.3 — Preserve successful-response behavior

- Decide whether the warning is prepended/appended to the normal assistant response or sent separately.
- Make sure warning delivery works across text/photo/document/voice flows if they all share the same execution/result path.

## Phase 5 — Update documentation and examples

### Step 5.1 — Update session configuration docs

- Update `docs/config/session/README.md` to document Codex defaulting to per-user mode, explicit shared mode, and failure behavior when UUID discovery fails.
- Update any config reference docs/examples that describe Codex default session behavior.

### Step 5.2 — Update Codex engine docs

- Update `docs/engines/codex/README.md` to reflect:
  - default per-user behavior,
  - explicit `"shared"` mode,
  - `/clear` semantics in each mode,
  - the dependency on `~/.codex/sessions/` transcript files,
  - the warning/degraded behavior when UUID discovery fails.
- Keep the experimental caveat explicit.

### Step 5.3 — Update example docs/config where needed

- Update example documentation under `examples/codex/README.md` and any example config files if they imply shared-by-default Codex behavior.
- Ensure docs do not claim that `session.json` contains real IDs in modes where it does not.

## Phase 6 — Verify with targeted testing and a real manual run

### Step 6.1 — Add or update automated coverage where practical

- Add focused tests for config resolution and Codex default session mode if existing test structure makes that practical.
- Add tests for `/clear` mode split if there is already command/session test coverage.
- Add focused unit coverage for Codex session discovery helper behavior if feasible.

### Step 6.2 — Run local validation

- Run lint/build/test commands needed to validate touched code.
- Fix only issues introduced by this task.

### Step 6.3 — Execute the manual Codex isolation scenario

- Run HAL against a real Codex installation with `engine.session` omitted or explicitly `"user"`.
- Use two Telegram clients/users in the same project and confirm:
  - each user gets their own `session.json`,
  - stored session IDs are real Codex UUIDs, not `"active"`,
  - each user resumes their own prior Codex transcript,
  - `/clear` for one user only deletes that user's local session continuity,
  - the other user's continuity remains intact,
  - forced/observed UUID recovery failure produces the Telegram warning and normal log output.

### Step 6.4 — Capture implementation notes back into the task/docs

- Record the observed manual-test outcome and any residual caveats.
- If the implementation still shows unsafe ambiguity, document the limitation explicitly before considering the task complete.

## Risks to watch

- Accidentally changing session semantics for engines other than Codex.
- Reintroducing shared continuation via `resume --last` in a path labeled as per-user.
- Matching the wrong Codex transcript file when multiple sessions exist for the same `cwd`.
- Warning delivery being lost because current engine result shapes do not carry structured notices.
- Docs drifting from the actual CLI behavior or from the final failure semantics.
