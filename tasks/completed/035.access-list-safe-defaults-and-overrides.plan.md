# 035 — Access list safe defaults and overrides — Development Plan

## Phase 1: Schema and type changes ✅

### Step 1.1 — Add `dangerouslyAllowUnrestrictedAccess` to the access schema ✅

**File:** `src/config.ts`

Extracted a shared `AccessSchema` used by both `GlobalsFileSchema` and `ProjectFileSchema`, adding the `dangerouslyAllowUnrestrictedAccess` boolean field alongside `allowedUserIds`. Both fields remain `.partial().optional()` at schema level.

### Step 1.2 — Update `ResolvedProjectConfig` type ✅

**File:** `src/config.ts`

Added `dangerouslyAllowUnrestrictedAccess: boolean` to the `access` field on the resolved config type.

### Step 1.3 — Implement full replacement semantics for `access` in `resolveProjectConfig` ✅

**File:** `src/config.ts`

Changed the resolution logic to check `project.access !== undefined` — if the key is present at all (even as `{}`), only project values are used. If the key is omitted, globals are used. Defaults: `[]` for IDs, `false` for the flag.

## Phase 2: Boot-time access validation ✅

### Step 2.1 — Create `validateAccessPolicies` function ✅

**File:** `src/config.ts`

Added `validateAccessPolicies` that iterates all resolved projects, collects errors for any project missing both `allowedUserIds` and `dangerouslyAllowUnrestrictedAccess`, and throws a single `ConfigLoadError` listing all misconfigured projects.

### Step 2.2 — Wire validation into boot and hot-reload paths ✅

**File:** `src/cli.ts`

Added `validateAccessPolicies(resolvedProjects)` call right after `validateProjects(resolvedProjects)` in `runBotsForConfig`. Both initial boot and hot-reload paths are covered since they share this function.

## Phase 3: Auth middleware update ✅

### Step 3.1 — Update `createAuthMiddleware` to use the new access model ✅

**File:** `src/bot/middleware/auth.ts`

Rewrote middleware with three branches:
1. `allowedUserIds` non-empty → only listed users pass (dangerous flag ignored).
2. `allowedUserIds` empty + `dangerouslyAllowUnrestrictedAccess: true` → everyone passes.
3. Neither → deny (defensive; boot validation prevents this at runtime).

Extracted the denial message to a `DENIED_MSG` constant to avoid duplication.

## Phase 4: Templates and examples ✅

### Step 4.1 — Update `buildConfigTemplate` in CLI ✅

**File:** `src/cli.ts`

Changed global `access.allowedUserIds` from `[]` to `[0]` (placeholder) so the generated config passes boot validation. Removed the redundant per-project `access` block — projects now inherit from globals. Updated "next steps" output to tell the user to replace `0` with their Telegram user ID.

### Step 4.2 — Update example config ✅

**File:** `examples/hal.config.json`

Changed global access from `"allowedUserIds": []` to `"dangerouslyAllowUnrestrictedAccess": true` to demonstrate the flag and keep the example valid.

### Step 4.3 — Update CLI help and "next steps" output ✅

**File:** `src/cli.ts`

The help text already showed `"access": { "allowedUserIds": [123456789] }` which is valid under the new rules. Updated "next steps" step 3 to explicitly mention replacing the placeholder `0` with a real Telegram user ID.
