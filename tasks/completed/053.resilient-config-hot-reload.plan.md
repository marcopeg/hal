# Plan: 053 — Resilient config hot-reload

## Overview

Make the hot-reload path resilient: on config load/validation or bot startup failure during a reload, do not exit the process; keep the config watcher running and enter a degraded state (no bots). Next config file change triggers another reload attempt. Startup behaviour remains unchanged (exit on error).

## Phase 1 — Shared mutable state for current bot handles ✅

**Goal:** Reload callback and shutdown must always act on the *current* set of bot handles, which may be empty when degraded.

- **Step 1.1:** Introduce a single source of truth for "current bot handles" that both the reload callback and `shutdown()` use. Options: (a) a mutable object like `const state = { botHandles: BotHandle[] }` updated on every reload (success or failure), or (b) a variable that is reassigned. Option (a) keeps a stable reference for `shutdown()` and the watcher callback. — **Done:** `const state = { botHandles: [] }` added in `runStart()`.
- **Step 1.2:** After successful initial run, set `state.botHandles = runResult.botHandles` (or construct initial state so it holds the first run's handles). — **Done:** `state.botHandles = runResult.botHandles` after first successful `runBotsForConfig`.
- **Step 1.3:** In `shutdown()`, stop bots from `state.botHandles` (not from a closed-over `runResult`). This allows shutdown to work when degraded (`state.botHandles.length === 0`): no bots to stop, watcher still closes, process exits. — **Done:** `shutdown()` now uses `state.botHandles`; "All bots stopped" only logged when `state.botHandles.length > 0`.

**Outcome:** Shutdown and reload both use the same mutable handle list; when degraded that list is empty.

## Phase 2 — Reload callback: catch errors and avoid exit ✅

**Goal:** On reload failure (config load or bot startup), do not call `process.exit(1)`; set current handles to empty and log.

- **Step 2.1:** In the hot-reload callback, after stopping current bots (from `state.botHandles`), wrap the entire reload sequence in a single try/catch: load config (sync) and run bots (async). Catch any error (config load/validation or `runBotsForConfig` rejection). — **Done:** Inner try/catch wraps `tryLoadMultiConfig` + `runBotsForConfig`; catch sets `state.botHandles = []` and logs, no exit.
- **Step 2.2:** On success: assign the new handles to `state.botHandles`, log that bots are running (existing or new message). Optionally log "Exiting degraded state" when `state.botHandles` was previously empty. — **Done:** On success `state.botHandles = nextResult.botHandles`; we log "Exiting degraded state; all bots running" only when `wasDegraded`; otherwise `runBotsForConfig` already logs "All bots running".
- **Step 2.3:** On failure: set `state.botHandles = []`, log the error with a clear message that the process is in degraded state and waiting for a corrected config (e.g. "Reload failed; in degraded state, waiting for corrected config"). Do not call `process.exit(1)`. — **Done:** Catch block sets `state.botHandles = []`, logs at error with message "Reload failed; in degraded state, waiting for corrected config", no `process.exit`.
- **Step 2.4:** Keep the `reloading` guard so concurrent reloads do not run; ensure `finally` still resets `reloading` so the next file change can trigger a new attempt. — **Done:** Guard and `finally { reloading = false }` unchanged.

**Outcome:** Failed reload leaves the process running with no bots and watcher active; next change triggers another reload.

## Phase 3 — Degraded path: no bots to stop ✅

**Goal:** When already degraded, a config file change should still run the reload logic (no bots to stop, then load and start).

- **Step 3.1:** Ensure the callback always runs "stop current bots" then "load + run bots". When `state.botHandles` is empty, `Promise.all([])` is a no-op — no code change required if Phase 1/2 use `state.botHandles` for the stop step. — **Done:** Callback uses `state.botHandles` for stop; empty array is a no-op.
- **Step 3.2:** (Optional) If we want to avoid "Config change detected" + "All bots stopped" when there are no bots, add a conditional log or skip the stop step when `state.botHandles.length === 0`; otherwise leave behaviour as-is for simplicity. — **Done:** "All bots stopped" only logged when `hadBots` (i.e. `state.botHandles.length > 0` before stop).

**Outcome:** Degraded → fix config → file change → reload runs → success restores bots.

## Phase 4 — Logging and observability ✅

**Goal:** Clear log messages for degraded vs healthy transitions.

- **Step 4.1:** When entering degraded state (reload failed), log at error level with: (1) the underlying error (message or stack), (2) a fixed message such as "Reload failed; in degraded state, waiting for corrected config". — **Done:** `startupLogger.error({ error: message, err }, "Reload failed; in degraded state, waiting for corrected config")`.
- **Step 4.2:** When leaving degraded state (reload succeeds after a previous failure), log at info level that all bots are running (existing "All bots running" or a short "Exiting degraded state; all bots running"). — **Done:** When `wasDegraded` we log "Exiting degraded state; all bots running".
- **Step 4.3:** Ensure "Config change detected" and "All bots stopped" remain so operators see the sequence of events. — **Done:** "Config change detected" always; "All bots stopped" when we had bots (conditional).

**Outcome:** Logs clearly show when the process is degraded and when it recovers.

## Phase 5 — Verification and edge cases ✅

**Goal:** Confirm startup unchanged, shutdown correct in both states, no regression.

- **Step 5.1:** Leave `runStart()` startup path unchanged: first load still uses `tryLoadMultiConfig` + `printConfigError` / `process.exit(1)` on config failure, and existing try/catch around `runBotsForConfig` still exits on bot startup failure. — **Done:** No changes to startup path; state is populated only after successful first run.
- **Step 5.2:** Manually verify: (1) invalid config at startup → process exits. (2) Valid start, then break config and save → reload fails, process stays up, degraded log. (3) Fix config and save → next reload starts bots. (4) SIGINT/SIGTERM in healthy and in degraded state → clean exit (watcher closed, no lingering work). — **Left to operator:** Run with invalid startup config, then valid → break config → fix config; test SIGINT/SIGTERM in both states.
- **Step 5.3:** Optionally add a short note in code or in the task's "Notes" section documenting: startup = hard failure; reload failure = degraded, watcher stays up; fix and save = auto-recover. — **Done:** Comment block above `state` in `src/cli.ts` documents startup vs reload behaviour.

**Outcome:** Acceptance criteria met; no change to startup behaviour; shutdown and normal hot-reload unchanged.
