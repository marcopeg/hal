# Plan for 086 — Copilot per-user session isolation

## Goal

Make Copilot default to per-user session isolation in HAL, persist real Copilot UUIDs instead of the shared `"active"` marker, keep explicit shared mode available via `engine.session: "shared"`, degrade safely when UUID recovery fails, and update the user-facing docs to match the new behavior.

- Completed: traced the existing Copilot/session flow in config resolution, handlers, adapter runtime, and `/clear` handling.
- Completed: replaced filesystem probing with `src/engine/copilot-output.ts`, which parses Copilot JSONL output and extracts `sessionId` from the final `result` event.
- Completed: updated `src/engine/adapters/copilot.ts` so Copilot now advertises per-user support, defaults to `"user"`, resumes with `--resume <UUID>`, keeps explicit shared mode on `"shared"`, and retries stale UUIDs as fresh runs with warnings instead of switching to shared continuation.
- Completed: updated `src/bot/commands/session.ts` so Copilot active reset only applies in shared mode.
- Completed: added focused coverage in `src/engine/copilot-output.test.ts` and `src/config.test.ts` for Copilot structured-output parsing and session-mode resolution.
- Completed: updated maintained user-facing docs and examples to reflect Copilot's new default per-user behavior and explicit shared-mode opt-in.
- Completed: added the manual verification checklist in `manual-tests/061.copilot-user-session-isolation.md` for post-implementation two-device validation.

## Phase 1 — Lock runtime semantics and affected surfaces

### Step 1.1 — Trace the current Copilot session flow

- Confirm how `engine.session` resolves today in `src/config.ts` and how adapter capabilities drive `true`/omitted defaults.
- Confirm how handlers load and persist `session.json` in `src/bot/handlers/*.ts` and `src/user/setup.ts`.
- Confirm how Copilot currently decides whether to append `--continue` and where it returns the placeholder `"active"` marker in `src/engine/adapters/copilot.ts`.
- Confirm how `/clear` currently routes through `src/bot/commands/session.ts` for Copilot.

### Step 1.2 — Align implementation targets with the refined task

- Treat omitted or `true` `engine.session` for Copilot as `"user"` by default.
- Keep `engine.session: "shared"` as the explicit shared-continuation mode using `--continue`.
- In Copilot `"user"` mode, only use `--resume <UUID>` when HAL has a real stored UUID.
- Define both UUID discovery failure and stale-resume failure as warning-and-fresh-degradation paths, never as permission to switch into shared continuation.
- Keep `/clear` local-only in Copilot `"user"` mode and active-reset in Copilot `"shared"` mode.

## Phase 2 — Add Copilot session discovery helpers

### Step 2.1 — Introduce structured-output parsing for Copilot JSONL

- Add `src/engine/copilot-output.ts` to parse the JSONL emitted by `copilot --output-format json`.
- Extract the final assistant reply from `assistant.message` events and the resumable UUID from the final `result` event.
- Make the parser tolerant of non-JSON lines or missing fields so failures collapse to `undefined` instead of throwing.

### Step 2.2 — Define the helper contract around best-effort recovery

- Keep the helper API small and adapter-oriented: parse stdout after the run, return response text plus `sessionId` when present.
- Avoid coupling recovery to Copilot's on-disk session layout.
- Preserve a clean seam so Copilot-specific recovery logic stays out of shared handler code.

## Phase 3 — Switch Copilot runtime behavior to per-user UUID continuation

### Step 3.1 — Update adapter capabilities and session-mode branching

- Change Copilot adapter capabilities to `supportsUserIsolation: true` and `defaultMode: "user"`.
- Preserve `sharedContinuationRequiresMarker: true` so explicit shared mode still uses the existing marker-based continuation flow.
- Re-check config resolution paths so `engine.session: "user"` no longer fails at boot for Copilot while OpenCode remains unchanged.

### Step 3.2 — Implement fresh-run snapshot/discovery flow

- In Copilot `"user"` mode, run with `--output-format json --stream off` so the final stdout includes a `result` event with `sessionId`.
- After a successful run, parse stdout and persist the discovered UUID in `EngineResult.sessionId`.
- Stop returning `"active"` in Copilot `"user"` mode.
- Keep shared mode returning the marker only where handlers still need it for `--continue`.

### Step 3.3 — Implement UUID resume flow and stale-session degradation

- When a real Copilot UUID exists in user state, resume with `copilot --resume <UUID>` instead of `--continue`.
- Detect the stale-resume path when Copilot rejects or cannot use the stored UUID.
- On stale-resume failure, clear the stale local session marker, log the problem, and surface a warning so the next message runs fresh and can attempt new UUID discovery.
- Ensure Copilot `"user"` mode never falls through to shared `--continue`, even after recovery failures.

### Step 3.4 — Preserve current shared-mode behavior

- Keep Copilot `"shared"` mode using `--continue` and the existing marker-based continuation semantics.
- Ensure `continueSession: false` still suppresses continuation flags for one-off reset/new-session flows.
- Verify parse/output behavior remains unchanged aside from session ID and warning propagation.

## Phase 4 — Update reset and handler persistence behavior

### Step 4.1 — Split `/clear` behavior for Copilot by effective session mode

- Update `src/bot/commands/session.ts` so Copilot is only in the active-reset path when `effectiveMode === "shared"`.
- In Copilot `"user"` mode, `/clear` should only remove the user’s local `session.json` and send the normal confirmation reply.
- Confirm `/start` reuse of the same reset helper inherits the same semantics where applicable.

### Step 4.2 — Ensure warning/session persistence behaves correctly in handlers

- Verify existing text/photo/document/voice handlers already clear stale local session data when `warning` is set and no replacement `sessionId` is returned.
- If Copilot-specific gaps appear, patch the shared persistence flow rather than adding engine-specific cleanup in multiple handlers.
- Ensure successful Copilot user-mode runs persist the real UUID and shared-mode runs continue to persist the shared marker only when intended.

## Risks to watch

- Accidentally changing shared-session semantics for engines other than Copilot.
- Returning the shared `"active"` marker in Copilot user mode and silently reintroducing cross-user continuation.
- Failing to parse the final `result` event from Copilot's JSONL output and losing the resumable session UUID.
- Treating stale-resume failures as hard request failures instead of warning-and-fresh degradation.
- Updating source docs but forgetting other maintained user-facing references that still claim Copilot has no per-user session support.
- Editing generated example/blog output unnecessarily and creating avoidable churn.
