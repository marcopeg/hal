# Plan: 058 — Fix broken session continuity across adapters

## Overview

Restore session continuity for shared/cwd-scoped adapters (Copilot, Codex, Cursor, OpenCode) by treating `session.json` as the continuation marker, while introducing explicit adapter session capabilities for validating `engine.session` modes. Unsupported `"user"` isolation must fail during config load (including hot reload), and recover automatically after config correction.

## Phase 1 — Define adapter session capabilities ✅

- Done: added `EngineSessionCapabilities` + `sessionCapabilities` to `EngineAdapter` in `src/engine/types.ts`.
- Done: added capabilities to all adapters with semantics:
  - user isolation supported: `claude`, `antigravity`, `codex`, `cursor`
  - user isolation unsupported: `copilot`, `opencode`

- Done: exposed capability lookup via `getEngineSessionCapabilities()` in `src/engine/registry.ts` and exported from `src/engine/index.ts`.
- Decision: capabilities are static metadata on adapters so config validation stays deterministic and side-effect free.

## Risks and sequencing notes

1. Type-level adapter contract changes will require all adapters to compile before moving on; complete Phase 1 first to avoid piecemeal breakage.
2. Config validation should remain deterministic and side-effect-free; avoid calling external CLIs during validation.
3. Shared-marker behavior can unintentionally affect first-message semantics; verify that continuation is only attempted after marker exists.
4. Keep backward compatibility for existing valid configs (`false`, `true`, `"shared"`, and supported `"user"`).

## Execution notes

- Rationale: regression fixed at the policy boundary (handlers decide whether to load session state) rather than forcing risky retries/fallback logic inside adapters.
- Rationale: capability metadata makes policy explicit and reusable for both config validation and runtime session decisions.
