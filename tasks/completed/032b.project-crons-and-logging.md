## Summary

Extend the cron scheduler to support project-level cron definitions loaded from `{projectCwd}/.hal/crons/*.{md,mjs}`. Project crons run with a narrowed `ProjectCronContext` (single project), use implicit `projectId` with no `targets` array, and log to the centralised log directory under `crons/projects/{slug}/{name}.{type}/`.

## Dependencies

- [032a — Core scheduler + system-level crons](./032a.core-scheduler-and-system-crons.md) must be completed first.

## Cron source

Project crons are **file-based only**: `{projectCwd}/.hal/crons/*.{md,mjs}`.

There is no inline `projects[n].crons` config key.

---

## `.md` frontmatter — project tier

At project tier there is **no `targets` array**. The project is implicit (the file belongs to that project's directory). Routing is controlled by two top-level fields:

| Field | Type | Required | Description |
|---|---|---|---|
| `enabled` | boolean | no | Defaults to `false`. Must be `true` to schedule. |
| `schedule` | string | one of | Cron expression (5- or 6-field). |
| `runAt` | string | one of | ISO 8601 absolute datetime (one-off). |
| `runAs` | number | no | User ID whose context (`bot.userId`) is injected into the prompt AND who receives the result via Telegram DM. |
| `notify` | number[] | no | Additional user IDs that receive the result via Telegram DM. No context injection for these users. |

`runAs` and `notify` are both optional. If neither is set, the cron runs silently (log-only).

`notify` can be set without `runAs` (broadcast result without any user context injection).

`runAs` can be set without `notify` (inject user context and send result to that user only).

**Validation rules:**
- Exactly one of `schedule` or `runAt` must be present.
- A past `runAt` is silently skipped at load time (debug log written, not scheduled).
- `runAs` and each entry in `notify` must be valid integer user IDs.

```markdown
---
enabled: true
schedule: "0 8 * * *"
runAs: 123456
notify:
  - 789012
  - 345678
---

Summarise yesterday's commits and send me a morning briefing.
```

In this example: the prompt is executed with `bot.userId = 123456` in context, and the result is sent via DM to `123456`, `789012`, and `345678`.

## `ProjectCronContext` — shape for `.mjs` project crons

Project-tier `.mjs` handlers receive a `ProjectCronContext` instead of the system-tier `CronContext`. The shape is flat (single project, not a map):

```ts
export interface ProjectCronContext {
  /** Full resolved config for this project. */
  project: ResolvedProjectConfig;
  /** Grammy Bot instance for this project only. */
  bot: Bot;
  /**
   * Resolved context vars — the same key/value map injected into .md cron prompts.
   * Built fresh on every execution (includes time-sensitive @{} values).
   */
  context: Record<string, string>;
  /** Call this project's AI engine with a prompt and return the response. */
  call(prompt: string): Promise<string>;
}
```

- `ctx.project` — full resolved project config (slug, name, cwd, engine, model, session, context vars, etc.)
- `ctx.bot` — Grammy Bot already scoped to this project; no need to index into a map
- `ctx.context` — resolved context vars built fresh on every execution (includes time-sensitive `@{}` values such as current date/time, git status, etc.)
- `ctx.call(prompt)` — calls this project's AI engine anonymously

## Context injection for `.md` crons

`buildCronContextVars()` is called **fresh on every execution** (not at scheduler startup). This ensures time-sensitive `@{}` dynamic lookups (current date/time, shell output, file reads) are always up-to-date.

The resulting vars (`sys.*`, `project.*`, `engine.*`, `bot.userId` when `runAs` is set) are prepended to the prompt via `formatContextPrompt()`, identical to 032a.

`ctx.context` in `ProjectCronContext` is also built fresh per execution by the same call, then passed into the handler.

---

## `runAs`

`runAs` means: inject the given user ID into the context block as `bot.userId` (exactly as 032a does for system-tier `.md` crons with `userId` on a target). The result is also sent as a Telegram DM to that user.

No user data directory or user session loading — that is deferred to 032c.

---

## `notify`

`notify` is an optional list of additional user IDs that receive the cron result as a Telegram DM. No context injection for these users — `runAs` is the only source of context injection. All recipients (the `runAs` user plus all `notify` entries) are sent the same output string via `bot.api.sendMessage()`.

---

## Execution logging

Logs are written to the **same centralised log base** as system crons (`configDir`), under a `projects/` subfolder:

```
{configDir}/.hal/logs/crons/
  system/
    {name}.md/
    {name}.mjs/
  projects/                        ← plural, mirrors "projects" in config
    {slug}/
      {name}.md/
      {name}.mjs/
```

- The extension in the folder name (`.md` / `.mjs`) is kept — prevents collisions when a `.md` and `.mjs` cron share the same name.
- Log file format and contents are identical to 032a (header, `--- prompt ---`, `--- context ---`, `--- project config ---`, `--- output ---`, `--- error ---`).
- The `scope` field in `CronLogEntry` is the project slug; `writeCronLog` derives the path as `projects/{scope}/{name}.{type}/` when `scope !== "system"`.

**Note**: this changes the path prefix from `project/` (singular, from 032a placeholder) to `projects/` (plural). 032b canonicalises it.

## Per-project scheduler lifecycle

- On project boot, instantiate a `CronScheduler` for the project with `scope = project.slug`.
- Load project crons from `{projectCwd}/.hal/crons/` at startup.
- Start a file watcher on `{projectCwd}/.hal/crons/` for hot-reload.
- **Teardown**: full shutdown/reboot cycle (same as 032a). A config change stops all bots + schedulers and restarts everything. Granular per-project hot reload is out of scope — tracked separately.
- System crons (from `startSystemCrons()`) continue to run independently in their own scheduler.

## Scope

### In scope

- `ProjectCronTargetSchema` / `ProjectMdFrontmatterSchema` in `src/crons/schema.ts` (no `targets`, adds `runAs?` and `notify?`).
- `.md` loader extension: accept a `tier` option to pick the right schema.
- `ProjectCronContext` interface in `src/crons/types.ts`.
- `buildProjectCronContext()` in `src/crons/context.ts`.
- `executor-md` extension: project-tier definitions with implicit `projectId`, `runAs` context injection, result dispatch to `runAs` user + `notify` list.
- `executor-mjs` extension: call handler with `ProjectCronContext`.
- Per-project `CronScheduler` instance, file watcher, and lifecycle wiring in `cli.ts`.
- Execution logging to `{configDir}/.hal/logs/crons/projects/{slug}/{name}.{type}/`.
- Update `writeCronLog` scope path from `project/` → `projects/`.
- `ctx.context` built fresh per execution inside the executors.
- `startProjectCrons(options)` function in `src/crons/index.ts`.

### Out of scope (deferred)

- User-level crons (`{dataDir}/{userId}/crons/`) — 032c.
- Slash commands (`/cron_add`, `/cron_list`, etc.) — 032c.
- Full user-scoped session injection (`userDir`, user config) — 032c.
- Granular per-project hot reload (stop/restart only affected projects) — separate task.

## Requirements

1. Add `ProjectMdFrontmatterSchema` to `src/crons/schema.ts`: no `targets`, adds `runAs?: number` and `notify?: number[]`.
2. Extend `.md` loader to accept `tier: "system" | "project"` and select the appropriate schema.
3. Define `ProjectCronContext` interface in `src/crons/types.ts`.
4. Add `buildProjectCronContext()` in `src/crons/context.ts`: builds `ProjectCronContext` for a single project. Note: `ctx.context` is not pre-built here — it is built fresh inside each executor call.
5. Extend `executor-md.ts` for project-tier: resolve implicit `projectId` from scheduler scope, call `buildCronContextVars()` fresh per execution, dispatch result to `runAs` user and all `notify` users.
6. Extend `executor-mjs.ts` for project-tier: build `ProjectCronContext` fresh per execution (including `ctx.context`) and call `handler(ctx)`.
7. Add `startProjectCrons(options)` in `src/crons/index.ts`, analogous to `startSystemCrons`.
8. Wire `startProjectCrons()` into `cli.ts` for each active project; wire teardown into the existing full-reload shutdown sequence.
9. Update `writeCronLog` path: `projects/{scope}/` instead of `project/{scope}/`.
10. File watcher: one per project pointing to `{projectCwd}/.hal/crons/`.

## Acceptance Criteria

- A `.md` cron in `{projectCwd}/.hal/crons/` fires on schedule and submits the prompt (with fresh project-scoped context) to that project's engine.
- A `.mjs` cron in `{projectCwd}/.hal/crons/` fires on schedule and the exported `handler` is called with a correctly shaped `ProjectCronContext`.
- `ctx.project` contains the full resolved project config.
- `ctx.bot` is the Grammy Bot instance for that project.
- `ctx.context` contains fresh resolved context vars (time-sensitive `@{}` values reflect the time of execution, not boot time).
- `ctx.call(prompt)` invokes the project's AI engine and returns the response string.
- Project-tier `.md` frontmatter without `targets` is accepted as valid.
- A cron with `runAs: 123456` injects `bot.userId = 123456` into context AND sends the result as a DM to user `123456`.
- A cron with `notify: [789012]` sends the result to user `789012` (in addition to `runAs` user if set).
- A cron with neither `runAs` nor `notify` runs silently (log-only).
- Project cron files are hot-reloaded: add/edit/delete detected and scheduler updated without restart.
- Deactivating all projects (full reload) stops all project schedulers and file watchers; system crons are unaffected.
- Execution logs are written to `{configDir}/.hal/logs/crons/projects/{slug}/{name}.{type}/`.
- Log path uses `projects/` (plural).
- `ctx.context` and `.md` context vars are built fresh on every execution tick.

## Notes

- **Name collision between system and project crons**: each project has its own `CronScheduler` instance (scope = slug), and the system scheduler is separate. No collision is possible. No additional namespacing needed.
- **Strict vs. non-strict at boot for project crons**: leaning toward non-strict (log and skip invalid files, do not fail the whole project boot). Decide during planning.
- **`runAs` field naming**: canonical field name is `runAs` (not `userId`) at project tier, to distinguish from the system-tier `targets[].userId` pattern and to make the intent clearer.
- **`ProjectMjsCronDefinition` type**: `MjsCronDefinition.handler` is typed as `(ctx: CronContext) => Promise<void>`. Project-tier `.mjs` handlers receive `ProjectCronContext` instead. A separate `ProjectMjsCronDefinition` interface should be introduced in `src/crons/types.ts` with `handler: (ctx: ProjectCronContext) => Promise<void>`. The loader receives a `tier` option (same as the `.md` loader) and returns the appropriate definition type. `CronDefinition` and `ProjectCronDefinition` become distinct union types.
