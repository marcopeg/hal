# 055 — NPM commands wrapper

## Status

completed

## Summary

Add a built-in Telegram slash command `/npm` that runs an npm command (primarily npm **scripts**) in the active project, captures stdout/stderr, and sends the result back to the user. The command must be **opt-in per project** via config and **disabled by default**.

Plan: [`055.npm-commands-wrapper.plan.md`](./055.npm-commands-wrapper.plan.md)

## Context

- HAL already has a notion of “built-in” commands that are toggled via config (e.g. `commands.git.enabled`).
- This task adds a new built-in command family, but with **subcommands/args** (e.g. `/npm start`).
- Telegram messages have strict size limits, so logs must be truncated and/or uploaded as a document.

## Proposed config shape (to confirm in refinement)

- `globals.commands.npm.enabled: boolean` (default `false`)
- `projects.<key>.commands.npm.enabled: boolean` (override)
- `commands.npm` additional options (project overrides globals):
  - `whitelist?: string[]` (if present: only these scripts are allowed)
  - `blacklist?: string[]` (if present: these scripts are removed)
  - `timeoutMs?: number` (default `60000`)
  - `maxOutputChars?: number` (default: safe Telegram limit; upload file when exceeded)
  - `sendAsFileWhenLarge?: boolean` (default `true`)

## Requirements

- `/npm` must be listed in the system commands list only when enabled for that project.
- Running `/npm <script>` must:
  - run in the target project’s `cwd`
  - execute via `npm run <script>` (scripts-only; no extra args for v1)
  - capture both stdout and stderr
  - channel the command output back to the user (with exit code + duration)
- If `<script>` is missing, reply with an **inline keyboard** listing all allowed scripts as buttons so the user can run a script by touch without typing anything.
- The bot must:
  - locate `package.json` in the project `cwd`
  - fail hard if `package.json` is missing/unreadable or has no `scripts`
  - validate `<script>` against `package.json` scripts and the configured allowlist rules
  - reject unknown/disallowed scripts with a clear error message
- Concurrency:
  - only one `/npm` run may be active at a time (per project)
  - if another run is already active, reject the new request with a clear message
- Output handling:
  - always send a short summary message (success/failure, exit code, duration)
  - include logs inline up to a limit
  - if logs exceed limits, attach full logs as a file (and include a short tail inline)

## Script discovery + allowlist resolution (decided)

This command is scripts-only and must consult `package.json` scripts. Resolution for the “effective allowed scripts” must follow:

- Start from `available = Object.keys(packageJson.scripts)`.
- If `whitelist` is present:
  - allowed = available ∩ whitelist
- Else:
  - allowed = available
- If `blacklist` is present:
  - allowed = allowed \ blacklist
- If `allowed` is empty:
  - fail hard with a message that includes the discovered `available` scripts and the effective filters.

Notes:
- If both whitelist and blacklist are present, both apply (whitelist first, then blacklist removal).
- If only blacklist is present, allow all scripts except the blacklisted ones.

## Long-running command policy (decided)

Hard timeout only for v1:

- Default `timeoutMs = 60000`.
- If the process does not exit within `timeoutMs`, send partial logs + “timed out” status, then stop the process (SIGINT → SIGTERM → SIGKILL).
- No detach/background jobs in v1.

## Open questions to resolve in refinement

- **Output formatting**: Plain text vs code blocks vs file attachment-first.
- **Environment**: Should the subprocess inherit the bot env as-is, or support extra env vars in config?

## Acceptance Criteria

- [ ] `commands.npm.enabled` exists in config (globals + project override) and defaults to `false`.
- [ ] When enabled, `/npm` appears in `/help` and in the Telegram slash menu.
- [ ] `/npm <script>` executes in project `cwd` using `npm` from PATH and returns exit status + logs.
- [ ] Missing/unreadable `package.json` or missing `scripts` fails hard with a clear error message.
- [ ] `/npm` with no argument replies with an inline keyboard of allowed scripts (one button per script).
- [ ] Tapping a script button triggers the same execution flow as `/npm <script>`.
- [ ] Disallowed/missing scripts produce clear, actionable error messages (including the discovered available scripts).
- [ ] Long-running behavior is implemented: hard timeout at 60s with termination sequence and partial logs returned.
- [ ] Only one `/npm` process can run at a time per project; concurrent requests are rejected cleanly.
- [ ] Output size limits are handled without Telegram API errors (truncate + optional file upload).

## Out of scope (unless brought in during refinement)

- Supporting pnpm/yarn/bun (separate tasks).
- Background job manager for detached long-running commands.
- Streaming live logs line-by-line (can be revisited once log streaming tasks land).
