# 061 — Copilot JSONL output + engine minimum version checks

**Status:** draft

## Goal

Update the copilot adapter to use `--output-format json` (available since v0.0.422, released 2026-03-05), which emits JSONL — one JSON record per interaction. Parse the stream to extract only assistant message lines, discarding tool activity records. This replaces the fragile `extractFinalAnswer()` regex heuristics with structured data.

Introduce a minimum version registry for engine CLI tools. On each execution (or adapter init), check the installed CLI version against the declared minimum and throw a clear, user-facing error if it's too old.

No backward compatibility with older copilot versions.

## Context

The copilot CLI previously wrote all output — tool traces (`● Glob`, `● Read`, `$ ...`, `└ ...`) and the final assistant response — mixed together in stdout. The existing `extractFinalAnswer()` function in `src/engine/adapters/copilot.ts` attempted regex-based extraction but fell back to raw output when no lead-in pattern matched, sending the full trace to Telegram.

`--output-format json` (v0.0.422+) separates interactions as distinct JSONL records. Assistant messages and tool events are distinct types, making clean extraction trivial.

## Scope

- Update `copilot` adapter spawn flags to include `--output-format json`
- Parse JSONL stdout; collect only records where the type indicates an assistant message
- Remove `extractFinalAnswer()` — no longer needed
- Create `src/engine/versions.ts` (or similar) as the minimum version registry
- Add a version check utility: run `copilot --version`, parse semver, compare against registry
- On version mismatch at boot: send a Telegram message prompting the user to update (interactive yes/no), run the upgrade command if confirmed, then exit. See Notes for full flow.
- Apply the same version-check pattern to other engines as a foundation (even if their minimum is currently `0.0.0`)

## Out of scope

- Backward compatibility with copilot < v0.0.422
- Streaming / progressive JSONL delivery to Telegram (separate task — see 046/050)
- `--acp` or `@github/copilot-sdk` integration

## Notes

### JSONL schema
`--output-format json` emits one JSON object per line. The schema is reverse-engineered from community tooling (no official spec). All known field names:

**Base structure (every record):**
```json
{ "type": "event.type", "data": { ... }, "id": "uuid", "timestamp": "ISO-8601", "parentId": "uuid | null" }
```

**Relevant record types:**
| `type` | Notes |
|---|---|
| `assistant.message` | Model response — extract `data.content` for the text |
| `assistant.turn_start` / `assistant.turn_end` | Turn demarcation — no text content |
| `tool.execution_start` | `data.toolName`, `data.toolArgs` |
| `tool.execution_complete` | Tool result |
| `user.message` | User input — ignore |
| `session.start` / `session.resume` / `session.shutdown` | Lifecycle — `session.shutdown` may include token stats but is unconfirmed in stdout stream |

**Extraction rule:** collect `data.content` from every `assistant.message` record in order. Concatenate if multiple are emitted. There is no confirmed `result`-type terminal record (unlike Claude Code); use `assistant.turn_end` or process close as the end signal.

For non-JSON lines or unknown record types: log to terminal, do not forward to the user.

**Fallback:** if the stream is fully parsed but zero `assistant.message` records are found (e.g., schema changed upstream), fall back to raw stdout concatenation and log a warning: `"No assistant.message records found in JSONL stream — schema may have changed"`. Do not surface an error to the user.

### Multiple assistant message records
Collect all `assistant.message` records in order and concatenate their `data.content`. See `src/engine/adapters/claude.ts` (`lastAssistantText`) for the reference pattern.

### Version check — boot flow
Run `check()` at boot before the bot is fully initialized. If the version is below the minimum:
1. Log the error to the terminal (including the upgrade URL).
2. Exit the process.

Terminal log is the authoritative output — this is a crash for the operator, not an end-user event. The bot does not start while the version is unsupported.

### Version registry
`src/engine/versions.ts` (or similar) is the minimum version registry. It doubles as living documentation of engine CLI requirements. Apply the same version-check hook to all other engines (Claude Code, Codex, etc.) even if their current minimum is `0.0.0`.

### Upgrade docs URL
`https://docs.github.com/en/copilot/how-tos/set-up/install-copilot-cli`
