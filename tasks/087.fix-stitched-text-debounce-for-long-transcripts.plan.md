## Execution notes

- Adopted a fixed quiet-window debounce for all non-command text messages.
- Hardened the debounce so flush happens only after a full idle window since the latest buffered part, even if an older timeout callback fires near the boundary.
- Registered the per-user buffer before any asynchronous status work so rapid early follow-up parts cannot create a second buffer.
- Set the preferred config surface to `telegram.message.debounceMs` with a `1000ms` default baseline after real-client validation.
- Kept `debounce.windowMs` as a compatibility fallback rather than breaking existing configs immediately.
- Preserved the slash-command bypass by dispatching slash-prefixed text immediately when no command/skill handles it.
- Implemented a hybrid stitcher that prefers transcript fidelity:
	- uncertain boundaries fall back to `join("")`,
	- strong near-limit word boundaries insert a space,
	- stronger burst boundaries insert a newline.
- Delayed visible processing status until the final flush so the bot does not announce work before the chain closes.
- Added focused tests for handler behavior, middleware rate-limit interaction, and config resolution/inheritance.
- Validation completed successfully with full Vitest, TypeScript typecheck, and Biome lint.

## Objective

Fix the text debounce flow so rapid transcript-like multi-message input is stitched into one logical prompt and executed exactly once downstream, using a validated `1000ms` baseline quiet window and a documented distinction between fragment-style concatenation and burst-style joining.
