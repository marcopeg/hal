# Plan: 032b — Project-level crons

## Context

- **All 032a files are in place** — `src/crons/{types,schema,loader-md,loader-mjs,loader-dir,log,executor-md,executor-mjs,context,scheduler,watcher,index}.ts`
- **`CronScheduler`** currently takes `(cronCtx, internalProjectCtxs, logBaseDir, logger, scope)`. Needs a light refactor to accept an executor callback so the project-tier can reuse the same timer logic without carrying system-tier state.
- **`writeCronLog`** path uses `project/` (singular) — must be changed to `projects/` (plural). Only one consumer (`executor-md.ts` and `executor-mjs.ts`), so the rename is safe.
- **`loader-md.ts`** uses the system-tier schema (`MdFrontmatterSchema`) with a `targets` array. Project tier needs a separate schema and loader path.
- **`loader-mjs.ts`** hard-codes the handler type as `(ctx: CronContext) => Promise<void>`. Project tier needs `(ctx: ProjectCronContext) => Promise<void>`.
- **`cli.ts`** `RunResult` currently holds `{ botHandles, cronHandle }`. Must be extended to hold per-project cron handles.

### Step 1.1 — `src/crons/types.ts`

Add four new exports alongside the existing system-tier types. Do not rename or remove existing exports.

```ts
/** Parsed project-tier .md cron definition (no targets, implicit projectId). */
export interface ProjectMdCronDefinition {
  type: "md";
  tier: "project";
  name: string;
  sourceFile: string;
  schedule?: string;
  runAt?: Date;
  enabled: boolean;
  runAs?: number;       // userId for context injection AND primary DM recipient
  notify?: number[];   // additional DM recipients (no context injection)
  prompt: string;
}

/** Parsed project-tier .mjs cron definition. */
export interface ProjectMjsCronDefinition {
  type: "mjs";
  tier: "project";
  name: string;
  sourceFile: string;
  schedule?: string;
  runAt?: Date;
  enabled: boolean;
  handler: (ctx: ProjectCronContext) => Promise<void>;
}

export type ProjectCronDefinition = ProjectMdCronDefinition | ProjectMjsCronDefinition;

/**
 * Context passed to project-tier .mjs handlers.
 * Flat (single project), built fresh on every execution.
 */
export interface ProjectCronContext {
  /** Full resolved config for this project. */
  project: ResolvedProjectConfig;
  /** Grammy Bot instance for this project only. */
  bot: Bot;
  /**
   * Resolved context vars — same key/value map injected into .md cron prompts.
   * Built fresh on every execution; includes time-sensitive @{} values.
   */
  context: Record<string, string>;
  /** Call this project's AI engine with a prompt and return the response. */
  call(prompt: string): Promise<string>;
}
```

## Phase 2 — Loaders ✅

### Step 2.1 — `src/crons/loader-md.ts`

Add a new `loadProjectMdCron(filePath, options)` function. Keep the existing `loadMdCron` unchanged (other than adding `tier: "system"` to the returned object).

```ts
export function loadProjectMdCron(
  filePath: string,
  options: { strict: boolean },
): ProjectMdCronDefinition {
  const raw = readFileSync(filePath, "utf-8");
  const match = raw.match(/^---\r?\n([\s\S]*?)\r?\n---\r?\n?([\s\S]*)$/);
  if (!match) throw new Error(`No YAML frontmatter found in ${filePath}`);

  const [, frontmatterRaw, body] = match;
  const parsed = parseYaml(frontmatterRaw);
  const result = ProjectMdFrontmatterSchema.safeParse(parsed);
  if (!result.success) {
    throw new Error(`Invalid frontmatter in ${filePath}: ${result.error.message}`);
  }

  return {
    type: "md",
    tier: "project",
    name: basename(filePath, ".md"),
    sourceFile: filePath,
    schedule: result.data.schedule,
    runAt: result.data.runAt ? new Date(result.data.runAt) : undefined,
    enabled: result.data.enabled,
    runAs: result.data.runAs,
    notify: result.data.notify,
    prompt: body.trim(),
  };
}
```

No `flowResult`/`userId` co-presence validation needed at project tier (no `flowResult` field).

### Step 2.2 — `src/crons/loader-mjs.ts`

```ts
export async function loadProjectMjsCron(
  filePath: string,
): Promise<ProjectMjsCronDefinition> {
  const url = `${filePath}?t=${Date.now()}`;
  const mod = await import(url);

  if (typeof mod.handler !== "function") {
    throw new Error(`Missing exported handler() function in ${filePath}`);
  }
  if (!mod.schedule && !mod.runAt) {
    throw new Error(`Missing schedule or runAt export in ${filePath}`);
  }
  if (mod.schedule && mod.runAt) {
    throw new Error(`Only one of schedule or runAt may be exported in ${filePath}`);
  }

  return {
    type: "mjs",
    tier: "project",
    name: basename(filePath, ".mjs"),
    sourceFile: filePath,
    schedule: typeof mod.schedule === "string" ? mod.schedule : undefined,
    runAt: mod.runAt ? new Date(mod.runAt as string) : undefined,
    enabled: mod.enabled === true,
    handler: mod.handler as (ctx: ProjectCronContext) => Promise<void>,
  };
}
```

## Phase 4 — Executors ✅

### Step 4.1 — `src/crons/executor-md.ts`

Add `executeMdProjectCron(def, projectCtx, bot, logBaseDir, logger, scope)`.

Key differences from the system-tier executor:
- No `targets` loop — single implicit project.
- `buildCronContextVars()` called fresh with `userId: def.runAs` (may be `undefined`).
- Result dispatched to `def.runAs` user (if set) and all `def.notify` users.

```ts
export async function executeMdProjectCron(
  def: ProjectMdCronDefinition,
  projectCtx: ProjectContext,
  bot: Bot,
  logBaseDir: string,
  logger: pino.Logger,
  scope: string,
): Promise<void> {
  const startedAt = new Date();
  let output = "";
  let error: string | undefined;
  let contextVars: Record<string, string> | undefined;

  try {
    const { config, logger: pLogger, bootContext } = projectCtx;
    const defaultModel = config.engineModel
      ? undefined
      : (getDefaultEngineModel(config.engine) ?? "engine-defaults");

    contextVars = await buildCronContextVars({
      configContext: config.context,
      bootContext,
      configDir: config.configDir,
      projectCwd: config.cwd,
      projectName: config.name,
      projectSlug: config.slug,
      logger: pLogger,
      engineName: config.engine,
      engineCommand: projectCtx.engine.command,
      engineModel: config.engineModel,
      engineDefaultModel: defaultModel,
      userId: def.runAs,
    });

    const contextualPrompt = formatContextPrompt(contextVars, def.prompt);
    const agent = createAgent(projectCtx);
    output = await agent.call(contextualPrompt);

    // Send result to runAs user + all notify users
    const recipients = [
      ...(def.runAs != null ? [def.runAs] : []),
      ...(def.notify ?? []),
    ];
    for (const userId of recipients) {
      try {
        await bot.api.sendMessage(userId, output);
      } catch (sendErr) {
        logger.warn(
          { jobName: def.name, userId, error: sendErr instanceof Error ? sendErr.message : String(sendErr) },
          "Cron result DM failed",
        );
      }
    }
  } catch (err) {
    error = err instanceof Error ? err.message : String(err);
    logger.error({ jobName: def.name, error }, "Cron .md project execution failed");
  }

  const cfg = projectCtx.config;
  writeCronLog(logBaseDir, {
    jobName: def.name,
    sourceFile: def.sourceFile,
    scope,
    type: def.type,
    startedAt,
    finishedAt: new Date(),
    output,
    error,
    prompt: def.prompt,
    context: contextVars,
    projectId: cfg.slug,
    projectConfig: {
      slug: cfg.slug,
      name: cfg.name,
      cwd: cfg.cwd,
      engine: cfg.engine,
      engineModel: cfg.engineModel,
      engineSession: cfg.engineSession,
      context: cfg.context,
    },
  });
}
```

### Step 4.2 — `src/crons/executor-mjs.ts`

```ts
export async function executeMjsProjectCron(
  def: ProjectMjsCronDefinition,
  projectCtx: ProjectContext,
  bot: Bot,
  logBaseDir: string,
  logger: pino.Logger,
  scope: string,
): Promise<void> {
  const startedAt = new Date();
  let output = "";
  let error: string | undefined;

  try {
    // Build ProjectCronContext fresh — context vars include time-sensitive @{} values
    const { config, logger: pLogger, bootContext } = projectCtx;
    const defaultModel = config.engineModel
      ? undefined
      : (getDefaultEngineModel(config.engine) ?? "engine-defaults");

    const contextVars = await buildCronContextVars({
      configContext: config.context,
      bootContext,
      configDir: config.configDir,
      projectCwd: config.cwd,
      projectName: config.name,
      projectSlug: config.slug,
      logger: pLogger,
      engineName: config.engine,
      engineCommand: projectCtx.engine.command,
      engineModel: config.engineModel,
      engineDefaultModel: defaultModel,
      userId: undefined,
    });

    const cronCtx: ProjectCronContext = {
      project: config,
      bot,
      context: contextVars,
      call: (prompt: string) => createAgent(projectCtx).call(prompt),
    };

    await def.handler(cronCtx);
    output = "(programmatic handler completed)";
  } catch (err) {
    error = err instanceof Error ? err.message : String(err);
    logger.error({ jobName: def.name, error }, "Cron .mjs project handler threw an error");
  }

  writeCronLog(logBaseDir, {
    jobName: def.name,
    sourceFile: def.sourceFile,
    scope,
    type: def.type,
    startedAt,
    finishedAt: new Date(),
    output,
    error,
  });
}
```

## Phase 5 — Scheduler refactor ✅

### Step 5.1 — `src/crons/scheduler.ts`

Refactor `CronScheduler` to accept an `executeJob` callback instead of carrying the runtime context itself. This makes the scheduler a pure timer manager and lets both system-tier and project-tier share the same class.

```ts
export class CronScheduler {
  private readonly jobs = new Map<string, JobEntry>();

  constructor(
    /** Called by the scheduler when a job fires. Closure captures tier-specific state. */
    private readonly executeJob: (def: AnyDefinition) => Promise<void>,
    private readonly logger: pino.Logger,
    private readonly scope: string,
  ) {}

  load(definitions: AnyDefinition[]): void { ... }
  add(def: AnyDefinition): void { ... }
  remove(name: string): void { ... }
  replace(def: AnyDefinition): void { ... }
  stop(): void { ... }

  private async execute(def: AnyDefinition, jobId: string): Promise<void> {
    try {
      await this.executeJob(def);
    } catch (err) {
      this.logger.error({ jobId, error: ... }, "Unhandled error in cron execution");
    }
  }
}
```

**Update `startSystemCrons` in `index.ts`** to construct `CronScheduler` with an `executeJob` closure:

```ts
const scheduler = new CronScheduler(
  async (def) => {
    if (def.type === "md") {
      await executeMdCron(def as MdCronDefinition, internalProjectCtxs, cronCtx, configDir, logger, "system");
    } else {
      await executeMjsCron(def as MjsCronDefinition, cronCtx, configDir, logger, "system");
    }
  },
  logger,
  "system",
);
```

The `as` casts are safe because system-tier scheduler is only ever loaded with `CronDefinition` (tier `"system"`).

## Phase 7 — Public API ✅

### Step 7.1 — `src/crons/index.ts`

Add `startProjectCrons(options)`:

```ts
export interface StartProjectCronsOptions {
  projectCtx: ProjectContext;
  bot: Bot;
  configDir: string;   // for log base dir
  logger: pino.Logger;
}

export async function startProjectCrons(
  options: StartProjectCronsOptions,
): Promise<CronHandle> {
  const { projectCtx, bot, configDir, logger } = options;
  const { config } = projectCtx;
  const cronDir = join(config.cwd, ".hal", "crons");
  const scope = config.slug;

  const { definitions } = await loadProjectCronsFromDir(cronDir, logger, {
    strict: false,  // non-strict: log and skip bad files, don't fail project boot
  });
  logger.info({ count: definitions.length, project: scope }, "Project crons loaded");

  const scheduler = new CronScheduler(
    async (def) => {
      if (def.type === "md") {
        await executeMdProjectCron(
          def as ProjectMdCronDefinition,
          projectCtx,
          bot,
          configDir,
          logger,
          scope,
        );
      } else {
        await executeMjsProjectCron(
          def as ProjectMjsCronDefinition,
          projectCtx,
          bot,
          configDir,
          logger,
          scope,
        );
      }
    },
    logger,
    scope,
  );
  scheduler.load(definitions);

  const watcher = startCronWatcher(cronDir, scheduler, logger, { tier: "project" });

  return {
    stop: async () => {
      await watcher.stop();
      scheduler.stop();
      logger.info({ project: scope }, "Project cron scheduler stopped");
    },
  };
}
```

Export the new types and functions:

## Open issues / notes for implementation

1. **`tier` discriminator migration**: Adding `tier: "system"` to `MdCronDefinition`/`MjsCronDefinition` means updating every construction site in `loader-md.ts` and `loader-mjs.ts`. Both return objects that need `tier: "system"` added. Straightforward but don't miss it.
2. **`AnyDefinition` cast safety**: The `as MdCronDefinition` casts in `startSystemCrons` and `as ProjectMdCronDefinition` in `startProjectCrons` are safe because each scheduler is only ever loaded with its own tier's definitions. If the `tier` discriminator is present, consider using it to narrow instead of casting.
3. **`buildCronContextVars` import in `executor-mjs.ts`**: Currently `executor-mjs.ts` does not import from `context/resolver.ts`. The project-tier executor needs it. Add the import.
4. **DM send failure isolation**: If `bot.api.sendMessage()` throws for one recipient in `executeMdProjectCron`, the others should still be attempted. The plan shows a per-recipient try/catch with a warning log — confirm this is the desired behaviour.
5. **`getDefaultEngineModel` import in `executor-mjs.ts`**: Needed for `buildCronContextVars`. Already imported in `executor-md.ts`; add to `executor-mjs.ts` as well.
