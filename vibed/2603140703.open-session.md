### 2026-03-14 07:10 — Fix Copilot path escape via --allow-all-paths removal

Copilot was escaping the project cwd because `--allow-all` (= `--allow-all-tools --allow-all-paths --allow-all-urls`) disables all path verification. Copilot is git-aware and auto-discovers the git root; with `--allow-all-paths` it could write anywhere. Fix: replaced `--allow-all` with `--allow-all-tools --allow-all-urls` in the adapter (cwd-restricted by default). Added `engine.copilot.allowAllPaths: true` as an opt-in flag for users who need cross-directory access. Updated `src/config.ts`, `src/engine/adapters/copilot.ts`, `docs/engines/copilot/README.md`, `docs/engines/README.md`, and `docs/config/reference.yaml`.

### 2026-03-14 07:35 — Deep analysis: cwd escape risk across all adapters

Full cross-engine audit of path restriction. Findings:
- **Codex**: safe by default — `--full-auto` sandbox restricts to `-C <cwd>`. Escape is explicit opt-in via `fullDiskAccess` or `dangerouslyEnableYolo`. No change needed.
- **Antigravity**: same risk class as Copilot — `sandbox: false` default + `approvalMode: yolo` = unrestricted access. `--sandbox` flag already exists and is exposed in HAL as `engine.antigravity.sandbox`; just needs a strong doc warning. Default left as `false` (sandbox requires Docker/macOS tools, would break setups without it).
- **Claude Code**: same risk class but no HAL CLI lever. Path restrictions come from `.claude/settings.json` `allowedPaths`, which is outside HAL's control. Documented.
- **Cursor**: `--workspace <cwd>` is a context directive, not a sandbox. Agent can reach outside. No CLI flag available. Documented.
- **OpenCode**: no path restriction mechanism at all in the CLI. Documented.
Added "cwd sandboxed by default" row to feature table in `docs/engines/README.md`, added a "cwd boundary" section to each engine's README, expanded the cross-cutting cwd boundary paragraph in the index.
