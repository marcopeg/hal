### 2026-03-04 10:00 — HAL logo: 500ms delay before logs

The HAL ASCII banner is printed at the very start of `runStart()` in `src/cli.ts`. ### 2026-03-04 10:05 — MJS callback dispatcher: what it is and why

**What changed:** A new handler `createMjsCallbackDispatcher` was added in `src/bot/handlers/mjs-callback.ts`. It is registered in `bot.ts` on `callback_query:data` *after* the engine callback handler, and exported from `handlers/index.ts`. The file is currently untracked (new).

**What it does:** It’s a generic dispatcher for *inline keyboard callbacks* that belong to custom .mjs commands. When a user taps an inline button, Telegram sends a `callback_query` with a `data` string. Built-in handlers already use prefixes: `en:` (engine switch), `md:` (model), `r:` (reset), `gc:` (git). The MJS dispatcher looks for data in the form `commandName:rest` (e.g. `tasks:drafts` or `tasks:open:42`). It takes the part before the first colon as the command name, resolves the corresponding `.hal/commands/{commandName}.mjs` file via `resolveCommandPath(projectCwd, configDir)`, and dynamically imports that file. If the module exports a `callbackHandler` function, it’s called with `{ data, gram, projectCtx }`. So any .mjs command can show its own inline keyboard and handle the button tap without new code in bot.ts.

**Why:** Custom commands (e.g. `/tasks`) can now send messages with inline buttons (e.g. "Drafts", "Ready") and handle the tap in the same .mjs file that defines the command. Previously, only built-in features (engine, model, reset, git) had callback handlers; adding a new one required editing bot.ts and the command loader. This change keeps bot.ts stable and lets each .mjs command own its callback protocol (e.g. `tasks:drafts` vs `tasks:open:123`). Failure handling: if the file doesn’t exist, doesn’t export `callbackHandler`, or the handler throws, the dispatcher answers the callback query with a short user-visible message and logs the error, so the Telegram button doesn’t stay in a pending state.

### 2026-03-04 — /tasks inline button hang fix

User reported: after tapping a section button (e.g. "Ready") on `/tasks`, the chat hung and the button looked like it was waiting — nothing happened. Cause: Telegram keeps the inline button in a loading state until the bot calls `answerCallbackQuery`. If the MJS dispatcher didn’t find the command file (`resolveCommandPath` returned null) it called `next()` and no other handler answered, so the callback was never answered. Fixes applied: (1) **mjs-callback.ts** — when we match `commandName:` but no `.mjs` file exists, we now call `gramCtx.answerCallbackQuery()` before `next()` so the button always stops spinning; added a debug log (commandName, cwd, configDir) when that path is taken. (2) **tasks.mjs** — replaced `gram.api.answerCallbackQuery(queryId)` with `gram.answerCallbackQuery()` so the context-aware grammY method is used and the spinner is dismissed reliably when the handler runs. The separate "sendDocument failed / invalid HTTP URL" error was noted as likely a different code path (e.g. engine or another command); no change made for that in this session.

### 2026-03-04 — OpenCode and Cursor model self-discovery; /model always exposed for those engines

**Problem:** When `providers.opencode` (or `providers.cursor`) was omitted or empty, the /model command was disabled and no longer appeared in the Telegram menu, because `modelEnabled` required `providerModels.length > 1`.

**Changes:** (1) **Config** (`src/config.ts`): `/model` is now enabled when we have a config list *or* when the engine supports model self-discovery and the config list is empty. Engines with self-discovery: `opencode` (runs `opencode models`), `cursor` (runs `agent models`). So for opencode and cursor, /model is always exposed and the list is resolved at runtime from the CLI when `providers.*` is not set. (2) **CLI model discovery** (`src/engine/opencode-models.ts`): Added `getCursorModelsFromCli(cwd, command?)` — runs `agent models` (default command `agent`), parses stdout into `ProviderModel[]`, returns `[]` on failure. `getEffectiveProviderModels(config)` now handles both opencode and cursor: when `providerModels.length === 0` and engine is opencode → `getOpencodeModelsFromCli`; when cursor → `getCursorModelsFromCli`. Model command and callback already use `getEffectiveProviderModels(config)`, so they show the CLI-derived list. Result: if `providers.opencode` or `providers.cursor` exists, that list is used; otherwise the adapter shows only models returned by the CLI.

### 2026-03-04 — CLI availability check for model self-discovery

Added `src/engine/cli-available.ts` with `isCliAvailable(command)`: fast check that runs `command --version` with 3s timeout, returns boolean. Used in `src/config.ts` so that `/model` is enabled for opencode/cursor only if the CLI is actually installed — if the binary isn't on PATH, `/model` stays hidden rather than showing an empty list. Previously only checked whether engine was opencode or cursor; now also verifies the CLI exists.

### 2026-03-04 — Friendly config error on startup

When config is missing or invalid, the process now prints a nicely formatted error instead of a raw stack trace or dangling process. `printConfigError(err)` in `cli.ts` shows: (1) bold red "Configuration error" heading, (2) the actual error message, (3) a "Need help?" block with the GitHub docs URL and `npx @marcopeg/hal init` quick start. The function calls `process.exit(1)` — the process always exits cleanly.

### 2026-03-04 — Init command rewritten: YAML template, --model flag, confirmation prompt

**Complete rewrite of `runInit`** in `src/cli.ts`. Overlaps with draft [045 — Interactive Setup Wizard](../tasks/drafts/045.setup-wizard.md).

**Template file:** Extracted the init config to `src/init-template.yaml` — a YAML file with `{{PLACEHOLDER}}` syntax (`{{ENGINE_NAME}}`, `{{ENGINE_MODEL}}`, `{{PROJECT_CWD}}`, `{{HAL_DOCS_*}}`). `buildYamlInitConfig` reads it and does string replacement. The template is copied to `dist/` at build time (`cp src/init-template.yaml dist/` in the build script). Easy to edit without touching TS code.

**Config shape:** Minimal — only `globals` (engine name + access with `${YOUR_TELEGRAM_USER_ID}`) and one project (`prj1`, active, engine + model, telegram with `${TELEGRAM_BOT_TOKEN}`, access with user ID). No providers, no logging, no rateLimit. Comments with links to relevant GitHub doc pages (config, telegram create bot, telegram configure token).

**Flags:** `--engine` (default: codex), `--model` (default: engine default from `DEFAULT_PROVIDER_MODEL`), `--cwd`. Default engine changed from claude to codex.

**Flow:** (1) Check no config file exists already. (2) Print proposed YAML between `---` markers. (3) Prompt `Y/n, Enter = yes`. (4) Write `hal.config.yaml`. (5) Create engine instruction file (e.g. `AGENTS.md`). (6) Print next steps with doc links. Process always exits.

**Prompt fix:** Replaced unreliable `readline.createInterface` + `rl.question` (which looped/hung under `tsx watch`) with direct `process.stdin` read: writes prompt to stdout, reads exactly one line from stdin, removes listener and pauses stdin immediately, resolves once. Enter/empty = yes, "n"/"no" = no.

### 2026-03-04 — bip.mjs: "find my Mac" command

`.hal/commands/bip.mjs` — a custom slash command (`/bip`) that beeps the Mac at full volume so you can locate it in a room. Saves the current output volume via `osascript`, cranks it to 100%, runs `beep N` (default 6, configurable via `/bip <count>`, capped at 20), then restores the original volume in a `finally` block. Returns a confirmation message with the beep count and restored volume. Handy when the Mac is somewhere nearby but you can't see it — send `/bip` from Telegram and follow the sound.
