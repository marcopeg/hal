### 2026-03-03 10:20 — Fix Cursor adapter session tracking

The Cursor adapter was blindly passing `--continue` whenever `engineSession` was true in config, even on the first message when no session existed. This caused "No previous chats found." errors. Root cause: unlike Claude (which returns a real `sessionId`), Cursor never wrote `session.json` and never checked for an active session before deciding on `--continue`.

**Fix (2 files):**
- `src/engine/adapters/cursor.ts`: Changed `--continue` decision from `config.engineSession && continueSession !== false` to also require `sessionId != null` (proving a prior session exists). On success, returns synthetic `sessionId: "active"` so the text handler saves `session.json`.
- `src/user/setup.ts`: Enriched `SessionData` with `active` and `lastMessageAt` fields; `saveSessionId` now records both.

**Flow after fix:** First message → no session.json → no `--continue` → fresh start → saves session.json. Subsequent messages → session.json exists → `--continue`. `/clean` → deletes session.json → next message starts fresh.

### 2026-03-03 10:25 — Apply same session fix to Copilot, Codex, OpenCode

Extended the session tracking fix to all `--continue`-style adapters. Same pattern in each: gate the continue flag on `sessionId != null` (evidence of prior session from session.json), and return `sessionId: "active"` on successful execution so the text handler persists the marker.

- `src/engine/adapters/copilot.ts`: `--continue` gated on `hasActiveSession`, returns `sessionId: "active"`
- `src/engine/adapters/codex.ts`: `resume --last` gated on `hasActiveSession`, returns `sessionId: "active"`
- `src/engine/adapters/opencode.ts`: `-c` gated on `hasActiveSession`, returns `sessionId: "active"`

Claude and Antigravity are unaffected — they use real per-user session IDs (`--resume <id>`) already returned by their CLIs.

### 2026-03-03 ~15:05 — Fix crash on config hot-reload (bot stop race)

Switching model via `/model` triggers a config file write → config watcher fires → reload handler stops all bots then restarts. Grammy's `bot.stop()` does a final `getUpdates` to acknowledge the last offset, which can race with an in-flight long-poll and throw a 409 Conflict error. The shutdown handler already swallowed this with `.catch(() => {})` but the reload handler on `src/cli.ts:320` did not, causing an unhandled crash. Fixed by adding the same `.catch(() => {})` to the reload path.
